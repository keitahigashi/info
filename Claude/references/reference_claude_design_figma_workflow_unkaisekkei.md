## 概要

公開日：2026-04-23。「Claude Designは初稿構造生成に、Figmaは最終調整と資産化に」という鉄則のもと、3ステップワークフローを実務ベースで解説。従来比1/3のコスト削減実績。

## 役割分担表

| 工程 | Claude優位 | Figma優位 |
|------|-----------|---------|
| 情報設計・ワイヤー | ◎初稿生成 | △手作業 |
| ビジュアルデザイン | ○大枠配色 | ◎ピクセル精度 |
| コンポーネント化 | △概念レベル | ◎Variants/Properties |
| デザインシステム | △プロンプト制御 | ◎Variables/Libraries |
| 開発連携 | ○コード生成 | ◎Dev Mode |

役割混在は「トークン崩れ・命名規則不統一・Auto Layout不整合」を招く。

## 実務ワークフロー（3ステップ）

**Step 1：Claude Designで初稿生成**
- デザイントークンを最初にプロンプトに提供
- 出力形式：HTML + TailwindCSS + JSONコンポーネント構造
- 既存システムとの乖離を防止

**Step 2：Figmaへの取り込み（3経路）**

| 経路 | 特徴 | 推奨度 |
|------|------|--------|
| html.to.design プラグイン | 手軽・Auto Layout再構成は手動 | 簡易 |
| Figma Dev Mode MCP | 2026年主流・既存コンポーネント読取可 | 推奨 |
| JSONスキーマ＋自作プラグイン | 大規模プロジェクト向け厳密管理 | 高精度 |

**Step 3：デザインシステム反映**
- HEX直指定を意味トークン（`color/primary/default`）に置換
- 8ptグリッド統一
- 既存Libraryインスタンスへ差替
- Text Style適用

実例：中堅SaaS企業でこの工程を省略→6ヶ月後にカラー定義57種に膨張し2週間の再整備が必要になった事例あり。

## 失敗パターンと対策

| 失敗 | 原因 | 対策 |
|------|------|------|
| トークン逸脱 | プロンプトに基準未提示 | 先行提供必須 |
| 既存ファイル上書き | MCP全権限付与 | Sandbox専用ファイル作成 |
| 再利用不可能 | Variable置換後回し | 完了条件に組込 |
| アクセシビリティ破綻 | AI出力をそのまま流用 | AA レベル人間最終検証 |

## 作例：B2B SaaS ログイン画面

- 従来フロー：90分 → AI活用フロー：25〜30分
- 主な手動調整：ボタン高さ調整・アイコン素材差替・余白tokenへの置換・エラー状態Variants追加

## 2026年ベストプラクティス チェックリスト

1. Claude にデザイントークンを最初に提供
2. 取込経路は Figma MCP 優先
3. Sandbox ファイルで受け→本番手動マージ
4. Variable 置換を完了条件に組み込む
5. アクセシビリティは人間が最終確認（AA以上）
6. レビューはFigma、開発はDev Modeで完結
