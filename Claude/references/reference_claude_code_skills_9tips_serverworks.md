## 出典

サーバーワークスエンジニアブログ: https://blog.serverworks.co.jp/claude-code-skills-tips

## 概要

Skillsは「マークダウンファイルだけでなく指示・スクリプト・リソースをまとめたフォルダ」。9つのコツをAnthropicの社内実践に基づいて解説。

## 何を書くか（コツ1〜2）

**コツ1：当たり前を書かない**
- Claudeが通常実行することを書いてもコンテキストを消費するだけ
- 「Claudeのデフォルト判断を変える情報だけ」を記載

**コツ2：ハマりどころを最優先で書く**
- 仕様書から気づけず、知らないと失敗する環境固有のクセ
- 例：「subscriptions テーブルは追記専用」「Stripe Webhook はステージングで200を常に返す」

## どう構造化するか（コツ3〜5）

**コツ3：ファイルシステムと段階的開示**
- 詳細（API仕様・テンプレート・トラブル対応）を別ファイルに分離
- 普段は軽い情報のみ読み込み、必要時のみ詳細参照

**コツ4：レールを敷きすぎない**
- 手順を1〜10まで固定化すると想定外で破綻
- 「マニュアル」より「先輩メモ」アプローチが応用性を確保

**コツ5：Descriptionはモデル向けに書く**
- ユーザーの実際の語彙（「調査して」「原因を調べて」）を意識的に含める
- 人間向けではなくClaudeがSkillを呼ぶかを判断するトリガー文

## どう動かすか（コツ6〜9）

**コツ6：セットアップ設計**
- ユーザー固有設定を `config.json` に保存
- `AskUserQuestion` ツールで選択肢形式の設定を促す

**コツ7：Claudeに状態を覚えさせる**
- ログファイルやJSONで状態を追記記録
- 例：朝会投稿Skillで `standups.log` から過去内容を参照

**コツ8：スクリプトを同梱してコード生成させない**
- 定型コードをライブラリとして含める→Claudeは組み合わせと高度分析に集中

**コツ9：オンデマンドフックでガードレール**

```bash
# /careful で破壊的コマンドをブロック
rm -rf、DROP TABLE 等

# /freeze で特定ディレクトリ外のファイル操作を制限
```

## 配布・運用

| 規模 | 方法 |
|------|------|
| 小規模 | リポジトリの `./.claude/skills` にチェックイン |
| 大規模 | プラグインマーケットプレイス方式 |

- `PreToolUse` フックで全社Skill利用をログ記録
- Anthropic社内：中央承認なし・ボトムアップで有用なSkillが自然に拡散
- 最も効果的だったSkill：コード検証系（書いたコードが動くか確認）
