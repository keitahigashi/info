## Claude Codeを"優秀な新卒部下"として活用するワークフロー

### 基本思想

- Claude Code を「優秀な新卒エンジニア」として扱い、実装業務を委譲
- プロンプト技術より「**構造化された前提条件**」が速度向上の本質
- 「AIが迷わない環境設計」が成功のカギ

### Step 1: CLAUDE.md（行動規範の明文化）

プロジェクトルートに配置し、以下を記載：

- ワークフロー（計画フェーズ→実装→検証）
- 自己改善ループ（ミスを `lessons.md` に記録）
- バグ修正の自律対応ルール

### Step 2: ai/rules ディレクトリ（ルールの分割管理）

「ルール長が増えるほど注意散漫になる」ため役割ごとに分割：

```
ai/rules/
├── PROJECT_ARCHITECTURE.md
├── GIT_WORKFLOW.md
└── REACT_NATIVE_GUIDELINES.md
```

### Step 3: GitHub MCP の活用

Issue作成 → ブランチ作成 → 実装 → PR提出までを自動化。  
単なるAPI呼び出しではなく、開発フロー全体を Claude に委譲。

### 生産性の変化

| タスク | Before | After |
|--------|--------|-------|
| 新機能1つ | 2〜3時間 | 30〜60分 |
| バグ修正 | 30分〜1時間 | 5〜15分 |

### Claude に任せない領域（重要）

- 本番デプロイ判断
- セキュリティ関連の実装
- 破壊的な Git 操作

### 哲学

環境整備に数時間投資することで、その後の開発で何十倍もの時間を回収できる。  
「どう使うか」ではなく「迷わせない環境をどう整えるか」が問い。
