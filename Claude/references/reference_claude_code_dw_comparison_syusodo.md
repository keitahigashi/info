## Dynamic Workflowの本質

Claudeがタスクに応じてJavaScriptのオーケストレーションスクリプトを自動生成し、そのスクリプトをランタイムがバックグラウンドで実行する仕組み。中間結果をコンテキストウィンドウではなくスクリプト変数に保持するため、ノイズ削減・コスト最適化・再現性向上を実現。

## 4手段の比較表

| 手段 | 並列処理 | コンテキスト共有 | 再利用性 | ユースケース |
|------|----------|------------------|----------|------------|
| 通常のClaude Code | なし | 共有 | 低 | 単純タスク |
| サブエージェント | 限定的 | 独立 | 低 | 中規模タスク |
| Agent Teams | あり | 独立 | 中 | チーム型タスク |
| Dynamic Workflow | 最大1,000並列 | スクリプト変数 | 高 | 大規模並列 |

## スクリプトの構成要素

- `parallel`: 並列実行するタスクの定義
- `pipeline`: 順次実行パイプライン
- `phase`: フェーズ単位の実行制御
- `merge`: 並列結果のマージロジック

## 4つの起動方法

1. プロンプトに「ultracode」と記述
2. `/effort ultracode` スラッシュコマンド
3. `/deep-research` コマンド（リサーチ特化）
4. 自然言語で「ワークフローを作って」と依頼

## ユースケース

- コードベース全体監査（大量ファイルの並列チェック）
- 大規模マイグレーション（ファイル単位並列変換）
- クリティカルな検証（複数エージェントによる相互検証）
- SEO記事の内部リンク一括チェック・提案

## サブエージェントとの使い分け判断軸

- **サブエージェント推奨**: タスク数が少ない・中間結果を参照しながら進行する
- **Dynamic Workflow推奨**: 独立したタスクが多数・バッチ処理・再現性が必要

## 制約（research previewゆえ）

- 本番リポジトリへの直接実行は推奨しない
- コスト試算なしの常用は避ける（通常より大幅にトークン消費）
- 許可リスト設計を事前に実施する必要あり
- Pro/Max/Team/Enterpriseで利用可能（無料プラン不可）
