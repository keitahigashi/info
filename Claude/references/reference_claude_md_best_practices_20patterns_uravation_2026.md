## CLAUDE.md ベストプラクティス 設定の書き方20選

## コア概念
CLAUDE.mdはClaudeがセッション開始時に読む設定ファイル。プロジェクト固有のルール、コーディング標準、ワークフローを定義

## 4層スコープ構造

| 層 | パス | 用途 |
|---|---|---|
| **組織ポリシー** | `/Library/Application Support/ClaudeCode/CLAUDE.md` | システム全体 |
| **ユーザー層** | `~/.claude/CLAUDE.md` | 全個人プロジェクト |
| **プロジェクト層** | `./CLAUDE.md` | チーム共有（git管理） |
| **ローカル上書き** | `./CLAUDE.local.md` | 個人設定（gitignore） |

**ルール**: 下層が上層を上書き; プロジェクトルールがユーザー設定を優先

## クリティカル設計原則
"1ファイル200行以内が最大値 — より長いファイルはコンプライアンス率が著しく低下"

コンテンツ: Claudeがコードを読んで推測できない情報のみ

## 8つのコンテンツカテゴリ

| カテゴリ | 目的 | 例 |
|---|---|---|
| **ビルド・テストコマンド** | プロジェクト固有の実行方法 | `pnpm vitest` vs `npm test` |
| **コーディング標準** | 非自明なパターンのみ | "ESM必須、CommonJS禁止" |
| **アーキテクチャ** | ディレクトリ構造ルール | `src/api/handlers/` 配置 |
| **ワークフロー** | Git・コミット規約 | Conventional Commits形式 |
| **禁止事項** | 禁じられた操作 | "コミット時console.log禁止" |
| **セキュリティ** | データハンドリング規則 | "APIキーをコードに含めない" |
| **ドキュメント** | `@filename`による参照 | スキーマ、デプロイガイドへのリンク |
| **パス固有ルール** | `.claude/rules/`サブディレクトリ | `src/components/**`のReactルール |

## 4つの一般的な失敗パターン

1. **ファイル肥大化**: 200行超過で重要ルール希釈 → `.claude/rules/`に移動
2. **冗長情報**: 可視コードの情報を再述（言語、フレームワークバージョン）
3. **強制不可能なルール**: "console.log禁止"にはESLint必須（ドキュメント不可）
4. **Git露出**: `CLAUDE.local.md`のgitignore忘れで個人設定漏洩

## 5つの有効な実装パターン

1. **パターン1**: `/init`コマンドでベースライン自動生成
2. **パターン2**: Claudeがエラー繰り返す時に毎セッション1ルール追加
3. **パターン3**: `/compact`セッション向け重要ルール保持（プロジェクトルート）
4. **パターン4**: チームCLAUDE.md版控管理で説明コメント保持
5. **パターン5**: `.claude/rules/` + `paths`フィールドでディレクトリ固有ルール活用

## モノレポ構造

```
├── CLAUDE.md (50行以内 — 共有必須事項)
├── .claude/rules/ (git-workflow.md, security.md)
└── packages/
    ├── frontend/CLAUDE.md (React固有)
    └── backend/CLAUDE.md (API固有)
```

親ルールは子が継承; 兄弟パッケージは独立

## CLAUDE.md vs オートメモリ

- **CLAUDE.md**: 静的チームルール（git管理、永続的）
- **オートメモリ**: Claude自動学習（マシンローカル、一時的）

## コピペテンプレート（最小実行可能）

- 正確なビルド・テストコマンド
- 3-5個の重要ルール
- 禁止操作リスト
- ディレクトリ配置ルール
- `@filename`構文による参照リンク

## 実装ロードマップ

**週1**: `/init`実行 → ≤200行確認 → 非自明コンテンツのみ検証

**週2**: ビルドコマンド、禁止事項、アーキテクチャルール追加 → `CLAUDE.local.md`gitignore

**週3以降**: 月次レビュー → インシデント1件あたり1ルール追加 → git PR経由で管理

## キーテイクアウト
"CLAUDE.mdは一度に完成させるべき完全仕様ではなく、プロジェクト経験とともに段階的に成長する生きたドキュメント"
