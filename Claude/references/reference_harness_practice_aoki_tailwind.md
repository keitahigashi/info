## ハーネスエンジニアリングの定義（著者解釈）

> 「ハーネスは馬を制御する道具から転じて、生成AIをうまく動かすための周辺環境や仕組みを整える考え方。AI本体だけに頼らず、周りの環境で品質を上げる発想。」

## AIのみに依存する危険性

Claude Codeで開発を続けると以下の問題が発生しやすい：
- 同じような関数が複数箇所に作られる（重複コード増大）
- エージェント切り替え時に前のセッションの記憶が消える
- コンテキスト不足によって後半の指示品質が低下する

「途中で検証できる環境を整えることが大事」と製造業の工程管理になぞらえて説明。

## 実践した3つのアプローチ

### 1. チェックリストによる多軸レビュー

Claude Codeに以下の観点でコードレビューを実行させるチェックリストを作成：
- リファクタリング（重複・命名・責務の分離）
- セキュリティ（認証・バリデーション・SQLi）
- パフォーマンス（N+1・不要なループ）
- 保守性（テスト容易性・ドキュメント）

結果をCLAUDE.mdやルールへ反映するフィードバックサイクルを構築。

### 2. 重複コードを防ぐ共通関数チェックリスト

```
配置場所: utils/ helpers/ library/
共通化の対象:
  - 日付フォーマット関数
  - バリデーション処理
  - ID生成ロジック
  - 非同期エラーハンドリング
検知ツール: jscpd（3回以上出現で共通化を検討）
```

SOLID原則（特に単一責任・開放閉鎖原則）との連携を意識した設計。

### 3. Claude Codeの苦手なことと対応策

| 苦手なこと | 対応策 |
|-----------|--------|
| コンテキスト限界 | セッション前に要約ドキュメントを読み込ませる |
| セッション記憶喪失 | フィードバックループで成功・失敗情報を引き継ぐ |
| 長大なコードベース理解 | ファイル分割と責務の明確化 |

## 実装したファイル構造

```
src/
├── types/      # 型定義
├── utils/      # 共通ユーティリティ
├── errors/     # エラー定義
├── config/     # 設定
└── hooks/      # カスタムフック
```

自動フォーマット・型チェック・テスト実行・危険コマンド停止をフックで組み込み。

## 著者の評価

「体感として、かなり品質が上がる。ただし、もう一度レポートを読み込ませて再チェックするひと手間が効果を最大化する」と結論。小規模チームでも導入しやすい軽量なハーネス設計の例として参考になる。
