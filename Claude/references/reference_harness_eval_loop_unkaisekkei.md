## ハーネスエンジニアリングの定義

> 「LLMを業務に耐える形で繋ぎ込み、評価・改善し続ける足回り全体を設計する技術」

プロンプトエンジニアリングが「文の中身」を磨く技術であるのに対し、ハーネスは**配管・計測器・安全弁を組む技術**と位置づけ。

## Claude Codeが選ばれる理由

CursorやDevinが「コードを書く」に最適化されているのに対し、Claude Codeは**「コードを書きながら評価ループを回す」**ことに最適化された唯一のツールと記述。

## 3点セット構成

| 機構 | 役割 | 配置場所 |
|------|------|---------|
| サブエージェント | 批評・採点・別視点検証 | `.claude/agents/` |
| Hook | ツール呼び出し前後の検証・遮断 | PreToolUse / PostToolUse / Stop |
| カスタムコマンド | 評価ループ一括起動 | `.claude/commands/` |

## 評価ループの5段階

```
実行 → 自動採点 → 失敗ケース蓄積 → プロンプト改修 → 再評価
```

**最小データセット要件:** 敵対的15件 + 境界ケース15件 + ハッピーパス20件（合計50件推奨、最低30件で稼働）

## サブエージェントの採点フォーマット

```json
{
  "score": 0.0-1.0,
  "reason": "失敗理由の説明",
  "failure_category": "hallucination|missing_step|wrong_format|..."
}
```

「迷ったら低い点をつける」を採点エージェントのシステムプロンプトに明記することを推奨。

## Hookの配置パターン

- **PreToolUse:** 危険操作遮断（`rm -rf /`、`DROP TABLE`などのパターンマッチ）
- **PostToolUse:** 変更後の自動検証（lint / typecheck / test実行）
- **Stop:** セッション終了時に変更ファイル一覧をログ記録

## 本番投入前チェックリスト（8項目）

1. 評価データセットの準備完了
2. 採点サブエージェントの動作確認
3. PreToolUse hookの遮断テスト
4. PostToolUse hookの自動検証確認
5. カスタムコマンドの一括起動テスト
6. 失敗ケースのログ保存確認
7. MCP接続の安定性確認
8. 監査ログの保存先設定

## まとめ

モデル更新が月単位で行われる2026年において、手動採点では追いつかない。自動評価ループを持たないエージェントは「運に依存した開発」になるという立場を強調。
