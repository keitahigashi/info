## Dynamic Workflows とは

Claude Code が「数十〜数百のサブエージェントを並列実行させ、大規模タスクを自動処理する」機能。  
従来のサブエージェント・スキルと異なり、**計画（オーケストレーション）をコードに移す**ことでコンテキスト膨張を抑えられる。

- 対応プラン: Pro / Max / Team / Enterprise（全有料プラン）
- 必要バージョン: Claude Code v2.1.154 以降
- 最大エージェント数: 1,000 エージェント/実行、同時実行上限 16

## ワークフロー化の3つの方法

### 1. `ultracode` キーワード（v2.1.160 以降）

```
ultracode: このリポジトリ全体のセキュリティ監査をして
```

プロンプトに `ultracode` を含めると自動でワークフロー計画が立案される。

### 2. `/effort ultracode` コマンド

```
/effort ultracode
```

セッション開始後にこのコマンドを入力すると、以降のタスクに対してワークフロー計画を自動生成。

### 3. 保存済みワークフロー再実行

```
/<ワークフロー名>
```

以前に定義したワークフローを名前で呼び出して再実行。

## トークン消費の実態

**通常セッションより明確に多く消費される。**

| 要因 | 内容 |
|------|------|
| 複数エージェントの独立実行 | 各エージェントが独自のコンテキストを持つ |
| 相互検証フェーズ | エージェント間の結果検証でトークンが積み上がる |
| オーケストレーション層 | 計画・調整に追加トークンが必要 |

## コスト制御の方法

### 1. 実行前承認プロンプト

ワークフロー実行前に「このタスクで何エージェントを使用するか」の確認が表示される。承認しない限り実行されない。

### 2. `/workflows` コマンドでの可視化

```
/workflows
```

各フェーズのトークン使用量をリアルタイムで確認できる。

### 3. モデル選択によるコスト管理

セッションで選択しているモデル（Opus 4.8 / Sonnet 等）がそのまま各エージェントに使われるため、**モデル選択がコスト管理の鍵**。

## 実用的な推奨事項

1. **スコープを絞った小規模タスク**から始めて使用感を掴む
2. 初回は必ず `/workflows` でトークン消費を確認
3. 本番運用前にテスト環境でコスト試算を実施
4. 大規模タスクは Sonnet モデルでプロトタイプ → 問題なければ Opus に切り替え

## Subagents・Skills との違い

| 機能 | オーケストレーション場所 | コンテキスト管理 | 並列数 |
|------|------------------------|----------------|-------|
| Subagents | Claude のコンテキスト内 | 膨張しやすい | 数個 |
| Skills | 定義済みフロー | 中程度 | 限定的 |
| Dynamic Workflows | コード（JS）に移管 | 抑制しやすい | 最大 1,000 |
