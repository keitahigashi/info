## 定義

> エージェントの出力品質を決めるのは、モデルそのものではなく、モデルを取り巻くハーネスの設計

## 10の主要実装トピック

### 1. プロジェクトルールファイル

**書くべき**: エージェントが推測不可のコマンド、デフォルトと異なるスタイル、ブランチ命名規則、固有アーキテクチャ、非自明なハマりどころ
**避ける**: コード読了で判明する情報、言語標準慣習、詳細APIドキュメント（リンク化）

CLAUDE.md構成例: ビルド・テストコマンド → コードスタイル → アーキテクチャ → Git規約
強調表現（`IMPORTANT`, `YOU MUST`）がエージェント遵守率を向上

### 2. コンテキストエンジニアリング（4戦略）

| 戦略 | 内容 |
|---|---|
| Write | claude-progress.txt、AGENTS.md、Git履歴に外部保存 |
| Select | スクラッチパッド読込、メモリ活性化、セマンティック検索 |
| Compress | 自動コンパクション（95%発動）、手動 `/compact` |
| Isolate | サブエージェント分離、要約のみ返却 |

Spotify Honkの方針: 「静的で包括的なプロンプトを優先」し動的検索の不確実性を排除

### 3. ツール設計

Spotify Honkは**わずか3ツール**構成:
1. 検証ツール（フォーマッター/リンター/テスト）
2. Gitツール（安全サブコマンドのみ）
3. Bashツール（許可リスト制限）

Vercelは「エージェントツール80%削除」で効率改善。**less is more**

### 4. プロンプト設計

**Explore → Plan → Implement パターン**:
- Explore: Plan Modeでファイル読了のみ（変更なし）
- Plan: 詳細実装計画作成
- Implement: 計画遵守でコード記述・テスト

**インタビューパターン**: 大規模機能向け。技術実装・UI/UX・エッジケース・トレードオフを質問→仕様書作成→新セッションで実装

### 5. テスト駆動フィードバックループ

テスト = 仕様。エージェントの基本動作: Edit → Test → Fix ループ

Spotify 3チャンネル検証:
- 決定論的検証器（正規表現エラー抽出）
- ストップフック（PR送信前検証）
- LLMジャッジ（別モデルが品質評価、約25%却下）

### 6. ガードレール（多層防御）

- 権限システム: 最小権限原則
- サンドボックス: gVisor/Firecracker/WASI
- LLMジャッジ: 独立コンテキストの別モデルが評価
- 原則: リンターの役割をLLMに委譲しない、PR自動マージしない

### 7. ワークフロー

**Anthropic 2エージェント構成**:
- 初期化エージェント（1回）: 機能リスト・進捗ログ・init.sh作成
- コーディングエージェント（各セッション）: pwd確認→進捗確認→未完了機能選択→init.sh実行→**新機能実装前にベースラインE2E検証**

**ダブルループ**: Vibing（高速プロトタイプ）→ Refinement（エンジニアリングモード）

### 8. 評価

LangChain事例: モデル変更なし、ハーネス最適化だけでTerminal Bench 30位→5位改善
**結論: モデル改善より先にハーネス最適化が効果的**

### 9. コスト最適化

- プロンプトキャッシュ: 入力コスト最大90%・レイテンシ最大80%削減
- モデルルーティング: 単純クエリ（80%）を安価モデルへ→40-60%コスト削減
- 積極的コンテキスト管理: `/clear`・`/compact`・サブエージェント分離

### 10. エージェントフレンドリーなコードベース

- テストスイート高速化（キャッシュ付き）
- 明快な検証手段（ワンコマンド）
- 明示的依存関係（暗黙DI回避）
- 直接実装（巧妙な抽象化回避）
- ORM より素SQL
- ヘッドレスCLIツール

## 実績

- OpenAI: 約100万行コード手書きなし構築
- Spotify Honk: 1,500件以上PR本番マージ
- LangChain: ハーネス最適化だけで30位→5位
