## 概要
「完璧なハーネスを最初から構築するのではなく、枠組みだけを先に作成し、チームで段階的に育てる」アプローチの実践ガイド。CADDiでのClaude Code導入における具体的な設計パターンとスキル実装。

## 詳細

### コアコンセプト
ルール内容の充実度より「誰でも更新しやすい状態」が優先。枠組みが明確だと更新・拡張のハードルが低下し、チームのフィードバックが自動的に統合される仕組みが生まれる。

### Rules の2分類
1. **汎用ルール**: レイヤー横断的ポリシー（coding-style.md、testing.md等）
2. **レイヤー別ルール**: 実装・テストを分離（`usecase/usecase.md`、`usecase/usecase_testing.md`等）

### インデックスファイル戦略
`.claude/rules/`下のファイルは軽量なインデックスとして機能し、実体は別の場所に配置。YAML frontmatterで`paths`を指定して条件付きルールを実現:
```yaml
paths: ["src/usecase/**/*.spec.ts"]
```
コンテキストウィンドウへの負荷を最小化しながら必要なルールを動的に読み込む。

### `/update-coding-rule` スキル
PRレビューコメントや一般的なフィードバックをルール化するカスタムスキル:
1. フィードバック収集（PR URLまたはテキスト）
2. 汎用化（具体例を抽象化、背景原則を抽出）
3. 配置先決定（既存ルールとの重複確認）
4. ユーザー確認（AskUserQuestion）
5. ファイル更新

### 導入ステップ
1. **枠組みの作成**: 見出し階層だけのファイルで開始
2. **段階的充実**: 開発過程での気づきを反映
3. **更新の民主化**: `/update-coding-rule`で誰もが追加可能に
4. **継続的進化**: ルール更新のハードル低下により自然な育成

### 設計原則
- Rules以外のSkills（API stub生成等）やHooksにも同じ「枠組み先行」原則を適用可能
- 枠組みを構築するプロンプト要件: プロジェクトアーキテクチャ分析、ルール種別の明確化、各ドキュメントに概要セクション、既存コードからの明らかなルール1〜2個の記載、インデックスファイルによる参照構造
