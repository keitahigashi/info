## 実装した4つのルーチン

| ルーチン | トリガー | 内容 |
|---------|---------|------|
| 朝のトレンド収集 | Schedule（毎朝） | 技術トレンドを自動収集・要約 |
| 夜間の経理入金確認 | Schedule（夜間） | 入金状況の自動チェック |
| GitHub Issueの自動トリアージ | GitHub Event（Issue opened） | 優先度分類・担当者提案 |
| 週次活動レポート | Schedule（週次） | 活動サマリーを自動生成 |

## 3件の「事故」

1. **停止忘れによる課金増加**: 不要なルーチンを止め忘れて予期しない課金が発生
2. **通知の連続発火**: GitHub Eventトリガーが短時間に多数発火して通知が大量に届いた
3. **意図せぬセッション重複**: 同じルーチンが重複起動してコンフリクトが発生

## 教訓・ベストプラクティス

- **監視・停止手順を事前に整備**する（停止コマンドを記録しておく）
- **Hooksによるレート制限**を設定する
- **週次棚卸し**でルーチンの生死確認を習慣化
- 「安定運用のコストと事故のコストがほぼ同オーダー」という実感
- **初利用者は1本のルーチンから始める**ことを強く推奨

## 料金感

プラン別日次上限（Pro=5回・Max=15回・Team/Enterprise=25回）内での運用が基本。上限外での意図しない実行に注意。
