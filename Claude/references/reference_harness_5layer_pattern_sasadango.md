## ハーネスエンジニアリングの定義

LLMはステートレスであるため、「AIの振る舞いをコードのように設計・管理する仕組み」が必要。エージェントがミスをするたびに環境側に恒久的修正を施すことで一貫性を確保する。

## 5層実装パターン

| 層 | 役割 |
|----|------|
| CLAUDE.md | 憲法。全体の行動制約・目的を定義 |
| Rules | ドメインルール。プロジェクト固有の規約 |
| Skills | ワークフロー定義。再利用可能な手順書 |
| Agents | 専門エージェント。役職別の実行体 |
| Settings | 権限制御。allowedTools・deny設定 |

## 責務分離による効率化

- **決定的処理**（データパース・API呼び出し）→ Python スクリプトに委譲
- **判断と生成のみ** → LLMに委譲
- 結果: トークン消費を1/6に削減した実例

## 「やらないこと」の明示

意思決定の自動化・自動ファイル整理などをスコープ外と明示することで、LLMの予期しない行動を劇的に削減。ルールは「してはならないこと」の列挙が効果的。

## 設計原則

- 最小限から始めて実運用で改善（段階的アプローチ）
- パーミッション管理は deny で十分なうちは Hooks を導入しない
- 各層の責任境界を明確に分離し、変更影響を局所化する
