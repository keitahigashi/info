## 概要

Claude Code のグローバル設定ファイル（`~/.claude/CLAUDE.md`）のベストプラクティスを、Anthropic公式ドキュメント、SFEIR Institute のコンサルティングデータ、HumanLayer の分析を引用しながら体系的に解説。「毎セッション読み込まれるコストに見合う指示かどうか」を判断基準として設定している。

## 数値エビデンス付きの推奨フォーマット原則

| 原則 | 効果（引用元: SFEIR Institute） |
|---|---|
| 箇条書き形式（散文比） | 遵守率が60%向上 |
| 命令形（「〜する」） | 精度が25%向上 |
| 150行以下の制限 | 遵守率が25%向上 |

Anthropic社内レポートより「IMPORTANT」「YOU MUST」等の強調表現が重要指示に有効であることを確認。

## 推奨グローバルCLAUDE.mdテンプレート（実例）

```markdown
# コミュニケーション
- 日本語で応答する（コード・変数名は英語）
- 簡潔に回答し、自明な説明は省略する
- 複雑なタスクでは実装前に計画を提示し、承認後に着手する

# コードスタイル
- 関数型アプローチを優先し、副作用を最小化する
- 厳密な型付け（anyは使わずunknownを使う）
- エラーは握りつぶさず、意味のあるメッセージ付きで処理する

# Git規約
- Conventional Commits形式、本文は日本語
- 確認なしに自動コミット・自動pushしない

# 禁止事項
- README・ドキュメントを勝手に生成・変更しない
- テストコードを確認なしに削除・コメントアウトしない
- 既存の動作するコードを理由なくリファクタリングしない
```

## 指示の希釈効果（重要な注意点）

HumanLayerの分析として「システムプロンプトに含まれる約50個の指示に加えて新たな指示を追加するたびに、すべての指示の遵守率が低下する」という「指示の希釈効果」を紹介。グローバルCLAUDE.mdへの記述は毎回トークンを消費するため、費用対効果の高い指示に絞り込む判断が重要。

## 意図的に除外した項目と理由（負の知識）

| 除外項目 | 除外理由 |
|---|---|
| console.log 制限 | ESLint `no-console` で検出可能 |
| 関数行数制限 | `max-lines-per-function` ルールで管理 |
| thinkingモード指定 | セッション毎の判断で十分 |
| 1コミット1論理変更 | Claudeのデフォルト動作 |

HumanLayerの原則「リンターやフォーマッターで強制できるルールはCLAUDE.mdに書くな」を採用。

## Progressive Disclosure パターン

`~/.claude/rules/` ディレクトリに YAML フロントマターで `globs` パターンを指定し、該当ファイル操作時のみ自動ロードする段階的開示手法を紹介。グローバル設定の肥大化を防ぐ高度な最適化手法。
