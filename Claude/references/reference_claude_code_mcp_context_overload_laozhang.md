## 問題の概要

Claude Code に MCP サーバーを多数接続すると **コンテキストウィンドウが圧迫**される。サーバー数だけでなく「4つのコスト」が複合的に影響する。

## コンテキストを圧迫する4つのコスト

| コスト種別 | 説明 |
|----------|------|
| **ツール定義** | 接続サーバー数 × ツール定義のスキーマ |
| **ツール結果** | MCP ツール実行の出力（大量になりがち） |
| **会話履歴** | 長期セッションで蓄積する過去ターン |
| **選択面** | ツール数が増えると Claude の選択精度が低下 |

**Tool Search** はツール定義の遅延ロードでコスト削減（85〜95%削減）するが、ツール結果・古い会話・重複サーバー・広すぎるカスタムツールは別途対処が必要。

## MCP サーバーの取捨選択フレームワーク

各サーバーを以下で分類:

| 判断 | 条件 |
|------|------|
| **Keep** | 毎日頻繁に使い、代替手段がない |
| **Defer** | 用途が限定的で Tool Search で遅延可能 |
| **Disable** | 月に数回しか使わない（必要時だけ有効化） |
| **Split** | 重い探索タスクは subagent に分離 |
| **Redesign** | ツール定義が広すぎる → より絞ったツールに再設計 |

## ツール結果の圧縮テクニック

### 4つのアプローチ

1. **Summary**: ツール結果全体でなく要約のみ返す
2. **Filter**: 必要なフィールドのみ抽出（例: ファイル一覧→名前のみ）
3. **Limit**: 返す件数を制限（例: 検索結果の上位10件）
4. **Pagination**: 大量データは複数ターンに分割

## 重い探索を subagents に分離する設計

コンテキストを汚染しやすいタスク（大量ファイル検索・長いログ解析等）は:
- Subagent に分離して別コンテキストで実行
- Subagent が結果サマリーのみを返す
- メインエージェントのコンテキストは「指示と結論」のみ保持

## MCP とスキルの使い分け

| 用途 | 推奨 |
|------|------|
| 外部サービス連携（DB・API・SaaS） | MCP サーバー |
| 繰り返す Claude Code 内作業フロー | スキル（SKILL.md） |
| 頻繁でない外部ツール呼び出し | 必要時のみ有効化 MCP |

## 実用的な上限目安

- Cursor: アクティブツール数の上限は**約40**
- Claude Code: 50ツール以上で選択精度が低下し始める
- 複数サーバー接続時は1メッセージあたり15,000〜20,000トークンのオーバーヘッドが発生することがある

## まとめ

Tool Search だけでは解決しない部分を意識し:
1. Keep/Defer/Disable でサーバー数を管理
2. ツール結果を Summary/Filter/Limit/Pagination で圧縮
3. 重い探索は subagents に分離
4. MCP とスキルを適切に使い分ける
