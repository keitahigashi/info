## 核となる考え方

大規模コードベースでの精度はモデル単体よりも「コードベースの見通しの良さ・補助情報の置き方・運用体制」に大きく左右される。必要な文脈へ素早くたどり着ける構造が鍵。

- ハーネス（CLAUDE.md・フック・スキル・プラグイン・MCPサーバー・LSP連携）が成功を左右する
- 派手な拡張より前に基礎となるCLAUDE.mdの階層化やコマンド整理を優先する

## 3つの成功パターン

### 1. コードベース探索の最適化

- ルートの `CLAUDE.md` は全体像と重要事項のみ（200行以内が目安）
- サブディレクトリごとにローカルな慣習を記載
- 生成ファイル・外部コードは `.ignore` で除外
- LSP（Language Server Protocol）導入でシンボル単位検索を実現
- テスト・lint・ビルドをサブディレクトリ単位で分離

### 2. 設定の継続的な見直し

- 3〜6ヶ月ごと、またはモデル更新後に設定を棚卸し
- 「不要になった補助輪を外していく」運用思想
- 新モデルでは以前の制約が妨げになることもある
- コードベースを分かりやすくすることと、Claudeが見やすくすることは異なることを認識する

### 3. 運用責任の明確化（DRI制）

- DRI（Directly Responsible Individual）を配置
- 設定・権限ポリシー・プラグイン配布を一元管理
- 初期段階から統制体制を整備
- 組織内のノウハウ蓄積・横展開を担当者が主導

## 導入ステップ（推奨優先順）

1. **CLAUDE.md構造** — 階層的な文脈整理（ルート + サブディレクトリ）
2. **コマンド分離** — 領域ごとのテスト/ビルド/lint環境
3. **ノイズ除外** — 生成ファイル・無関係ファイルのフィルタリング
4. **LSP統合** — 言語ごとの解析器接続でシンボル単位ナビゲーション
5. **拡張機能** — スキル・プラグイン・MCPサーバーは後付けで追加

## 元記事の背景

Anthropicが2026年5月14日に公開した英語記事「How Claude Code works in large codebases: Best practices and where to start」を日本語で解説したもの。数百万行のモノレポや数十年ものレガシーコードでClaude Codeを運用するチームから抽出されたベストプラクティスに基づく。
