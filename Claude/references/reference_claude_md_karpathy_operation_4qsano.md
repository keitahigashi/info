## 概要

Claude Codeを既存コードベースで安全に長時間動かすための運用術。GitHub「multica-ai/andrej-karpathy-skills」（16万スター超）を参考に、CLAUDE.mdによる行動制御の4原則と実践的な運用ガイドを解説。

## 問題提起

Claude Codeが既存コードベースで起こしがちな問題行動:
- 頼んでいないリファクタリング
- 無関係なファイルの変更
- 過剰なdiff生成
- 不必要な抽象化
- 曖昧な要件の拡大解釈

## 4つの行動原則

### 1. Think Before Coding（考えてから書く）

- 目的・対象ファイル・前提条件を実装前に明確化
- 「既存の設計意図に沿った実装」が技術的に正しいコードより優先

### 2. Simplicity First（シンプル第一）

不必要な汎用化の防止:
- 明示的に要求されていない機能を追加しない
- 単一用途のコンポーネントを抽象化しない
- 既存コードのパターンに合わせる

### 3. Surgical Changes（外科的変更）

「ついで修正」の防止（著者が最重要視するルール）:
- 無関係なファイルに触れない
- 不要なリファクタリングをしない
- 既存コードの再フォーマット禁止
- 明示的指示なしのデッドコード削除禁止

### 4. Goal-Driven Execution（ゴール駆動実行）

曖昧な指示を検証可能な成功基準に置き換える。
- NG: 「バグを直して」
- OK: 「このバグを再現するテストを追加し、そのテストが通るように修正して」

## グローバル vs プロジェクト固有の配置

### `~/.claude/CLAUDE.md`（グローバル）に置くもの

- 推測で実装しない
- 最小限のdiff
- 無関係なファイルを変更しない

### グローバルに置かないもの

- プロジェクト固有のビルドコマンド
- API仕様
- ディレクトリ構造

## plugin/skills と CLAUDE.md の違い

- **plugin / skills** = 能力追加
- **CLAUDE.md** = 行動制御

能力拡張だけで行動制約がなければ、暴走範囲が広がるだけ。

## 追加推奨ルール

- 「無関係なファイルを変更しない」（特に強力）
- 依存関係追加前に明示的な許可を求める
- タスクの正当な理由なしに公開APIを変更しない

## 重要な注意点

CLAUDE.mdは「書けば必ず守る魔法のファイルではない」。コンテキストとして機能するが、法的拘束力はない。重要タスクでは:
- プロンプト自体に制約を明示的に再記述する
- 実装前に計画を要求する
