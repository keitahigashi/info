公開日：2026年6月5日

Claude Maxの「5時間ごとの利用トークン上限」という制約に対する実践的な最適化戦略を3軸で整理。

## 課題：Claude Maxのトークン上限

- Claude Maxプランには5時間サイクルのトークン利用上限が存在
- 上限到達後は5時間待機するか、別の手段で作業継続する必要がある
- 高額プラン導入が難しい組織でも「組織的な最適化により戦える」

## 最適化戦略 3軸

### 軸1：Claude以外のモデル活用

| ツール | 用途 |
|--------|------|
| Codex | サブエージェント的に利用。シンプルなタスクをCodexに分散 |

- 単純なリファクタリング・定型コード生成はCodexへオフロード
- Claude Codeのトークンを複雑・創造的なタスクに集中投下

### 軸2：言語・出力の圧縮

| ツール | 削減効果 |
|--------|---------|
| `genshijin` | 日本語レスポンスを単純化。約75%のトークン削減 |
| `Serena` | LSP活用によりコードベースを効率的に把握、冗長な説明を抑制 |

- `genshijin`：Claudeの出力を「原始人的な簡潔さ」に制限するカスタムプロンプト/ツール
- `Serena`：Language Server Protocol経由でコードコンテキストを圧縮して渡す

### 軸3：Claude Codeネイティブ機能の活用

| 機能 | 設定・コマンド | 効果 |
|------|-------------|------|
| effortパラメータ | `max` → `auto` | 不要な深い推論を抑制 |
| opusplan | `/opusplan` コマンド | 計画フェーズだけOpus切替、実装はSonnetで節約 |
| /compact | `/compact` コマンド | セッション履歴を圧縮、コンテキスト再利用 |

## トークン最適化の優先順位

```
1. /compact でセッション圧縮（無料・即効）
2. effort=auto 設定（デフォルト変更）
3. genshijin/Serena 導入（初期設定コストあり）
4. Codex へのタスク分散（複数ツール管理コストあり）
```

## 対象読者

Claude Maxプランを利用しているが、トークン上限で作業が中断されることに悩んでいる開発者・チーム。
