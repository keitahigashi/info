## Dynamic Workflowsの本質

> 「処理構造をコードで宣言する」こと

- Claudeがその場で判断・実行するのではなく、**オーケストレーションロジックを事前にコード化**
- 「段取りはコードで確定、中身はAIで柔軟」な確信度向上が主な価値
- 速度向上は副産物であり、品質・信頼性向上が本来の目的

## Subagents vs Dynamic Workflows 比較表

| 側面 | Subagents | Dynamic Workflows |
|------|-----------|-------------------|
| **処理計画** | Claudeがその場で判断 | 人間が事前にコード化 |
| **制御方式** | 人間が逐次指示 | 自動オーケストレーション |
| **適用タスク** | 単発・定型処理 | ループ・分岐・大規模枝分かれ |
| **主な価値** | 柔軟性 | 確信度の向上 |
| **トークン消費** | 少ない | 多い（複数エージェント並列稼働） |

## スキル・ワークフロー・サブエージェントの使い分け基準

1. **スキル**: 繰り返し使う定型処理をSKILL.mdで定義→再利用性重視
2. **サブエージェント**: 単発・探索的なタスク→柔軟性重視
3. **Dynamic Workflows**: 大規模並列・品質保証が必要なタスク→信頼性重視

## トークンコスト削減のための選択基準

- タスクが**明確に分解可能**かつ**独立性が高い**場合にのみDynamic Workflowsを選択
- 探索的・対話的なタスクはSubagentsで十分
- コスト感の把握のために小規模スコープから開始することを推奨
