## 記事概要

**公開日:** 2026年6月12日

GENDA開発部のエンジニアが実践するClaude Fable 5のコスト最適化戦略。高性能だが高コストなFable 5を効率的に運用するための「タスク別モデル委譲」アプローチを解説。

## モデル別コスト比較（Fable 5 比）

| モデル | 相対コスト | 推奨タスク |
|--------|-----------|-----------|
| Claude Fable 5 | 100% | 設計判断・方針決定 |
| Claude Opus 4.8 | 50% | 複雑な複数ファイル実装・デバッグ |
| Claude Sonnet 4.6 | 30% | 広範なコード調査・仕様確定済み実装 |
| Claude Haiku | 10% | テスト実行・定型処理 |

**注意点:** 新トークナイザの影響で最大35%のトークン増加があるため、実質コストは単価以上に変わる場合がある。

## タスク別モデル割り当て設計

| タスク種別 | 推奨モデル | 理由 |
|-----------|-----------|------|
| 広範なコード調査 | Sonnet 4.6 | 読み取り中心でFableの精度は不要 |
| 仕様確定済みの実装 | Sonnet 4.6 | 判断要素が少ない |
| 複数ファイル実装＋デバッグ | Opus 4.8 | 中程度の複雑さに最適 |
| テスト実行・確認 | Haiku | 定型処理に高機能モデルは過剰 |
| 設計・方針判断・レビュー | Fable 5 | 最も価値が出るコア業務に集中 |

## 実装上のポイント

### 委譲ルールの管理
- `.claude/references/` 配下の共有ファイルに委譲ルールを記述
- `@` 参照でCLAUDE.mdに常時適用し、構造的に確実性を担保

### plan ファイルへの注記
- 各タスクに `owner: sonnet` / `owner: opus` / `owner: haiku` の注記を付ける
- 誰が（どのモデルが）何を担当するかを明示化

### サブエージェントからの報告制限
- 報告内容を「変更ファイル一覧」等に絞り込む
- 冗長な説明トークンの消費を抑制

## 効果

- メインセッションをFableで維持しながら実装コストを大幅削減
- 「最強モデルを全タスクに使う」アンチパターンを回避
- モデル切り替えの意思決定をルール化することで認知負荷も低減
