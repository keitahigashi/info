## Claude Tag 実践レビュー：設定・テスト・監査ログ

**著者**: Shinji Saito（CloudNative代表取締役社長、2026年6月24日）

### 先に結論：4つの警告点

1. **過大な権限付与リスク**: 適切に制限しないとSlack全体へのアクセス権が付与される
2. **従量課金による予算管理の必要性**: Opus 4.8使用、コスト設計が必要
3. **モデル信頼性の課題**: タスクを自律的に進めるため、結果の品質検証が必要
4. **自律動作の統制困難性**: Ambientモードでは人間介在なしでタスクが進行する

### 設定手順

#### 二層設計のアーキテクチャ

```
Layer 1: スコープ手順（Scope Procedure）
  └─ どのチャンネルにどのツールを許可するか
  
Layer 2: バンドル手順（Bundle Procedure / System Prompt）
  └─ Claudeの行動指針・ガードレール定義
```

#### ワークスペース設定

1. Slack管理画面 → Claude Tagアプリをインストール
2. **アクセスバンドル**を作成し、ツール・チャンネルスコープを定義
3. バンドルをチャンネルに割り当て

#### ガードレール設定例

```
ガードレール項目：
- 機微情報の非公開チャンネルへの転記禁止
- 本番環境直接変更の事前確認要件
- 読み取り専用アクセス制限
- ドメイン許可リスト実装（外部APIアクセス制御）
```

### 主要機能

| 機能 | 内容 |
|------|------|
| マルチプレイヤーモデル | チャンネル単位の共有アイデンティティ（全員が同じClaudeと対話） |
| 文脈の永続化 | チャンネル履歴から学習し、毎回説明不要 |
| Ambient（能動監視）モード | 呼ばれなくても関連情報を先回りして提供 |
| スケジュール実行 | タスクを時間をかけて自律的に進行（数時間～数日） |

### セキュリティ上の重要ポイント

- **チャンネル分離**: チャンネルごとに独立したメモリ・権限を持つ（リーガルチームの情報をエンジニアチャンネルに持ち込めない）
- **監査ログ**: 全操作がログ記録され、Enterprise管理ダッシュボードで確認可能
- **最小権限原則の適用**: ツール・チャンネル・情報源を必要最小限に絞る

### 利用条件

- **対象プラン**: Enterprise・Teamプラン（ベータ）
- **使用モデル**: Claude Opus 4.8（従量課金）
- **既存Claude Slackアプリからの移行**: 30日間の移行ウィンドウあり
