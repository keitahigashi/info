## Dynamic Workflows の本質

「Claudeがタスク専用のハーネスをその場で書く仕組み」。単一コンテキストウィンドウアプローチで発生する3つの問題を構造的に解決する：

1. **エージェント的怠慢** — コンテキストが長くなるほど手抜きが増える
2. **自己優先バイアス** — 自分の過去出力に引っ張られる
3. **目標ドリフト** — 長いタスクで当初の目的からずれていく

## 3つのコアAPI

| API | 役割 |
|-----|------|
| `agent()` | サブエージェントを1つ起動し、指定タスクを実行 |
| `parallel()` | 複数サブエージェントを同時起動し、結果を収集 |
| `pipeline()` | エージェントをチェーンしてデータを逐次変換 |

## 6つの実践パターン

### 1. 分類して実行
入力データを種別に分類し、それぞれ専門エージェントへ振り分ける。

### 2. 展開して統合（Map-Reduce）
大量タスクを並列展開（map）→結果を集約（reduce）。コードレビューやドキュメント生成に最適。

### 3. 敵対的検証
複数エージェントに同じ問題を独立解析させ、矛盾点を別エージェントが裁定する。セキュリティ監査や仕様レビューで威力を発揮。

### 4. 生成してフィルタリング
多数の候補を生成エージェントが作成し、評価エージェントが選別する。創造的タスクやテストケース生成に有効。

### 5. トーナメント
候補同士を対戦評価し、勝者を逐次選出。最適解探索やアーキテクチャ設計選定に使用。

### 6. 完了まで繰り返す（Retry-Until）
品質基準を満たすまでエージェントを再実行する。テスト合格・ビルド成功など客観的完了基準がある場合に適用。

- `/goal` コマンドで目標を明示的に固定し、目標ドリフトを防ぐ
- `/loop` スケジューリングで定期実行を自動化
- 明示的なトークン予算設定でコスト暴走を防止
