## 「完成宣言問題」の3層対策（2026年5月3日）

### 問題の本質

Claude Codeは型チェック合格時に「ビルド成功=完成」と判断してしまう。実際には以下が未検証:

- イベントハンドラの`await`忘れ
- フレームワーク固有設定（Next.jsの`'use client'`指定）
- CSSによるポインタ無効化
- 環境変数の設定漏れ

**根本原因**: 「観測できないもの（実際の挙動）は推論の対象にならない」。目に見えるシグナル（ビルド成功）に飛びつく構造的バイアス。

### 3層防御アーキテクチャ

| 層 | 役割 | 強制力 |
|---|---|---|
| L1 | Hooks：ビルドコマンド後に警告・ブロック | 強 |
| L2 | Skill：チェックリスト形式の完成確認 | 中 |
| L3 | 自動E2E：Playwrightで動線確認 | 強 |

### L1実装（Hooks）

settings.jsonのPostToolUse設定:

```json
"PostToolUse": [{
  "command": "ビルド/型チェック/lint成功後に『実画面を踏むまで完成と言うな』という警告をstderrに出力"
}]
```

stderr出力で警告を流す方式により、Claudeに次のアクションを強制。

### L2実装（Skill）

チェックリスト化により遵守率向上:

1. ビルド通過確認
2. 型チェック通過確認
3. dev server起動
4. HTTP 200応答確認
5. **Playwrightで実画面1動線実行**（必須）
6. console無エラー確認

### 運用知見

**効くルール**:
- ツール割り込み可能（PostToolUseなど）
- チェックリスト形式
- 失敗コストが目視化される

**効かないルール**:
- 禁則表現（「〜するな」）
- 抽象的な指針
- 発動タイミングが不明確

### 重要な示唆

- 48個のメモリルール運用経験から、メモリは「思い出させる装置」に過ぎない
- 実行ルート強制にはHooks/Skillが必須
- LLMへの制約は「プロンプト禁止」より「実行フロー設計」で実現する
