## 主要な実績・数値

| 指標 | 改善前 | 改善後 | 変化 |
|------|--------|--------|------|
| 脆弱性検出率 | 60% | 95% | +35% |
| レビュー時間 | 基準値 | 15%に短縮 | -85% |
| 見逃し率 | 基準値 | 20%に低下 | -80% |

著者は「5分で15個の脆弱性を発見」した体験を冒頭で紹介。

## 実践ワークフロー：4フェーズアプローチ

1. **認証・認可層のレビュー** — JWTトークン検証、権限チェックのバイパス可能性
2. **入力値検証層のレビュー** — SQLi、XSS、XXEなどインジェクション系
3. **データアクセス層のレビュー** — N+1クエリ、不適切なORM使用
4. **出力層のレビュー** — レスポンスに機密情報が含まれていないか

## 効果的なプロンプト設計のポイント

- 「このコードを確認して」ではなく、「SQLインジェクション対策・認証認可実装・入力値検証を重点的に確認して」と重点項目を明示
- コンテキストに業務要件と脅威モデルを含める
- 出力フォーマット（CVSSスコア・修正例付き）を指定する

## ツール精度比較

| ツール | 精度 | 偽陽性率 |
|--------|------|---------|
| Claude Code | 95% | 低 |
| 静的解析ツール（SAST） | 80% | 高（68%超） |
| 手動レビュー | 60% | - |

## 重要な失敗事例（3つの教訓）

1. AI指摘を**無検証で全て適用**し、正常機能を破損
2. **機密情報を含むコードをそのまま送信**（社内規定違反）
3. フロントエンド側のセキュリティチェックを見落とし（バックエンドのみ対応）
