# ハーネスエンジニアリング

出典: [ハーネスエンジニアリングとは何か](https://qiita.com/miruky/items/155f3b5a0dcde72fcd10) (@miruky, 2026-03-21)

## 定義

AIエージェントが**複数のコンテキストウィンドウにまたがって一貫した品質で作業を継続する**ための、環境・制御構造・フィードバックループを設計する技術体系。

## パラダイムの進化

| 観点 | プロンプトEng. | コンテキストEng. | ハーネスEng. |
|------|--------------|-----------------|-------------|
| 時期 | 2022年〜 | 2025年前半〜 | 2025年後半〜 |
| 対象 | 単発のLLM呼び出し | コンテキストウィンドウ | 複数セッション |
| 比喩 | 報告書の書き方 | 会議資料の準備 | プロジェクト運営体制 |
| 成果物 | プロンプトテンプレ | RAG、Few-shot | 進捗ファイル、CI設計 |

関係は**包含**（置き換えではない）: Prompt ⊂ Context ⊂ Harness

## AIエージェントの5つの失敗パターン

| パターン | 説明 |
|---------|------|
| One-shot問題 | 全機能を一度に実装しようとしてコンテキスト切れ |
| 早期完了宣言 | いくつか動いた時点で「完了」と判断 |
| コンテキスト喪失 | compact後に前セッションの内容が不明確 |
| テスト不足 | curl等の軽度テストのみでE2E未実施 |
| 技術的負債の増幅 | 不適切なパターンを「正規」として学習・拡散 |

## 5つの柱

### 1. 環境の初期化（Initializer Agent）

プロジェクト開始時に以下を生成:

| 成果物 | 目的 |
|--------|------|
| **feature_list.json** | 機能をJSON形式で構造化。各機能にテスト手順と合否フラグ |
| **init.sh** | 開発サーバー起動手順の文書化。毎セッションの環境復元用 |
| **claude-progress.txt** | エージェントの作業履歴を時系列記録 |
| **初期gitコミット** | 変更の起点を明確化 |

**重要**: Markdownではなく**JSON形式**を使用する。エージェントによる不適切な書き換えが少ない。

```json
{
  "category": "functional",
  "description": "新規チャットボタンで新しい会話を作成できる",
  "steps": [
    "メインインターフェースに移動する",
    "「新規チャット」ボタンをクリックする",
    "新しい会話が作成されることを確認する"
  ],
  "passes": false
}
```

### 2. インクリメンタル進捗

**核心ルール: 1セッション1機能**

各セッションの7ステップサイクル:
1. 進捗ファイルとgitログで現状把握
2. フィーチャーリストから次の1機能を選択
3. init.shで開発サーバー起動・基本動作確認
4. 選択機能を実装
5. E2Eテスト実施（合格のみ `passes: true` に更新）
6. gitコミットと進捗ファイル更新
7. 環境を「クリーン状態」に整備

**クリーン状態** = mainマージ可能レベル（重大バグなし、コード整理済み、新機能実装に即対応可能）

### 3. フィードバックループ（4層の検証）

| 層 | 検出対象 | レイテンシ | ツール例 |
|----|---------|-----------|---------|
| 第1層: コンパイル | 構文・型エラー | 数秒 | TypeScript/Go |
| 第2層: ユニットテスト | ロジック・リグレッション | 数分 | Jest, pytest |
| 第3層: E2Eテスト | 統合・UI不整合 | 数分〜 | Playwright |
| 第4層: CI/CD | クロスプラットフォーム | 数十分 | GitHub Actions |

**知見**: 強い型付け言語（Go + TypeScript + Protocol Buffers）はエージェント品質ゲートになる。コンパイル時にミスを検出。

### 4. コードベースそのものがコンテキスト

外部RAGよりも**コード構造自体の高品質化**が重要:

- **ディレクトリ構造がドキュメント**: 製品概念→コードパスの直接マッピング
- **命名規則の統一**: フロント・バックエンドで同じ概念に同じ名前
- **クリーンアーキテクチャ**: DDD層分けがエージェントに「どこに何を書くか」を伝える

### 5. 技術的負債の制御

**AIエージェントは技術的負債を指数関数的に増幅する**

| 状況 | 人間エンジニア | AIエージェント |
|------|---------------|--------------|
| サービス層バイパス | 一時的回避と認識 | 正当なパターンとして学習 |
| マジックナンバー | 将来リファクタ対象 | 他ファイルにも複製 |
| 不適切な命名 | 文脈から意図推測 | 新規コードでも採用 |

**対策**: 定期的にリファクタリング専用セッションを設ける。
**逆の好循環**: 良いパターンを配置すれば、エージェントはそれを増幅・適用する。

## 実践的な構成

### 2エージェント構成

| エージェント | 役割 | タイミング |
|-----------|------|----------|
| イニシャライザー | 環境構築、フィーチャーリスト作成 | プロジェクト開始時1回 |
| コーディング | インクリメンタル実装、テスト、コミット | 以降の全セッション |

### 3つのプリミティブ

1. **分離（Isolation）**: git worktreeとサンドボックスで並列作業時のコンフリクト排除
2. **分解（Decomposition）**: スコープ・受け入れ基準・完了定義を明確化
3. **協調（Coordination）**: 役職ではなくコミュニケーション・権限で管理

### ツール非依存の構成テンプレート

```
project-root/
  feature_list.json        # 機能リストと合否フラグ
  progress.md              # セッション間の進捗メモ
  init.sh                  # 環境初期化スクリプト
  .github/workflows/ci.yml # CIパイプライン
  tests/unit/              # ユニットテスト
  tests/e2e/               # E2Eテスト
  docs/architecture.md     # アーキテクチャ記録
```

## 実績データ

352日間・356Kライン・600コミットの実プロジェクトから得た知見:

- **並列上限**: 3つのワークツリーが人間の判断力の上限。4つ目で品質低下
- **実験コストの劇的低下**: アーキテクチャの失敗が2日で修正（従来チームなら2週間）
- **フィードバック必須**: 700以上のユニットテスト + E2E + CIパイプラインで検証
- **性能比較**: ハーネス違いで22ポイント性能差、モデル交換で1ポイント差（Can Boluk, 2026）

## Claude Codeでの実装（3フェーズモデル）

### Phase 1: 調査（PLAN MODE）

PLAN MODEでコードベースを読み込み、出力は「読書メモ」。

### Phase 2: 計画

実装計画を書かせる。変更ファイル・順序・テスト方針を明示。

### Phase 2.75: アノテーション

計画の誤りに注釈を入れ再計画させる。1〜6回の往復で共通理解を構築。

### Phase 3: 実装

計画をレビュー後にNormal/Auto-Acceptへ切替。

### 3原則

1. **土台なき自律は暴走する** — まず制約を敷く
2. **思考と実行を混ぜない** — フェーズ分離で追跡可能性を保つ
3. **信頼は段階的に積む** — 実績を見てから自律度上げ

## 6つの構成要素

### 指示と制約（3層）

| 層 | 役割 | ポイント |
|----|------|---------|
| **CLAUDE.md** | プロジェクト索引 | 30-60行目安。構造図・ビルドコマンド・rules参照のみ |
| **rules/** | 関心事ごとの制約 | `paths: **/*.ts`で対象ファイル編集時のみ自動適用 |
| **skills/** | ドメイン知識パッケージ | Progressive Disclosure で必要時のみロード |

### 自動化と委譲（3層）

| 層 | 役割 | ポイント |
|----|------|---------|
| **hooks/** | 自動検証 | 非同期基本、100ms以内、`exit 0`を死守 |
| **agents/** | サブエージェント | モデルルーティング: haiku(探索) < sonnet(実装) < opus(複雑推論) |
| **MCP** | 外部連携 | 10以下に抑える（コンテキスト消費回避） |

### 三重防御

rules（意図伝達）→ hooks（不変条件強制）→ deny（物理的ブロック）

## AI時代のテスト戦略

**TDDはエージェント時代にこそ効く**: テスト先行でエージェントの「テスト改竄」チート防止。

テストの役割変化:
- **仕様書として機能**: プロンプトより曖昧性が低い
- **生きたドキュメント**: 実行可能で嘘がつけない
- **人間とエージェントの契約**: 「完了」定義を機械判定

### フィードバック速度の階層

| 層 | 速度 | 検証内容 |
|----|------|--------|
| PostToolUse hook | ミリ秒 | format/lint |
| Pre-commit | 秒 | 型チェック・テスト |
| CI/CD | 分 | 統合テスト・E2E |
| 人間レビュー | 時間〜 | 設計判断・要件適合 |

---

## 生成速度と理解速度の非対称

- **生成は並列化可能**（複数エージェント同時実行）
- **理解は並列化不可**（個人の認知作業）

「書いた人が直す」前提が崩壊する。エージェント生成コードの障害時、設計意図を知る人間がいない。

**解決策**: ハーネス（テスト・linter・型チェック・ADR）が人間理解を代替する構造を作る。

## PostToolUse Hookの実装

出典: [ハーネスエンジニアリングを自分のプロジェクトで実装してみた](https://qiita.com/nogataka/items/a8306067788798975fa7) (@nogataka)

### 設定例（settings.json）

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "bash -c 'FILE=$(jq -r .tool_input.file_path); npx biome format --write \"$FILE\"; npx oxlint --fix \"$FILE\" 2>&1 | head -20'"
          }
        ]
      }
    ]
  }
}
```

### Linterスタック選定（速度優先）

| ツール | 速度 | 用途 |
|--------|------|------|
| Biome | ESLint比50-100倍高速 | フォーマット・基本リント |
| Oxlint | 10-20ミリ秒 | 追加的なルール検出 |
| ESLint | 基準速度 | PostToolUse向きでない（遅い） |

**原則**: PostToolUse Hookは「秒単位の遅延で効果が激減」するため、速度優先で選定する。リンターエラーがエージェントのコンテキストに即注入され、次アクションで自己修正が成立する。

### カスタムリンターのエラーメッセージ構造化

```
ERROR: [何が間違っているか]
  WHY: [理由 + ADRリンク]
  FIX: [具体的な修正手順]
  EXAMPLE:
    // Bad: const data: any = fetchUser()
    // Good: const data: User = await fetchUser()
```

プロジェクト文脈を含むメッセージほど、エージェントの修正精度が向上する。

## Minimum Viable Harness（MVH）ロードマップ

### 第1週（半日で完成可）

- 50行以内のCLAUDE.md作成（ポインタのみ）
- Lefthook pre-commit設定
- PostToolUse Hook導入（自動フォーマット）
- 最初のADR（Architecture Decision Record）記述

### 第2-4週

- エージェント失敗時にテストまたはルール追加
- Plan→Review→Execute ワークフロー確立
- E2Eテストツール導入
- Stop Hook設定（テスト通過を完了条件化）

### 2-3ヶ月目

- カスタムリンター作成（修正指示付き）
- ADRとリンタールールのマッピング整備
- 記述的ドキュメントをテスト+ADRで置換

## CLAUDE.mdポインタ設計

### 非推奨（記述型）

```markdown
# プロジェクト概要
Next.js 15 + TypeScript 5.7で構築。
PostgreSQL 16とDrizzle ORM使用。

## ディレクトリ構造
src/
├── app/ Next.js App Router
├── components/ UIコンポーネント
...（60行の規約記述続く）
```

### 推奨（ポインタ型）

```markdown
# プロジェクト
Next.js 15 + TypeScript 5.7 SaaSアプリ

## コマンド
- テスト: `pnpm vitest run`
- 型チェック: `pnpm tsc --noEmit`
- リント: `pnpm biome check src/`
- フォーマット: `pnpm biome format --write src/`

## 設計判断
- ADR: `/docs/adr/` 配下を参照
- APIルート追加時: ADR-003確認
- 新コンポーネント: ADR-007確認

## 禁止事項
リンターが強制。`biome.json`を参照。
```

### ポインタ設計の3原則

1. **実行可能なソースへの参照**を記述（`npm test を実行せよ`）
2. **システムの現状を記述しない**（コード+テストが真実の源泉）
3. **存在しないパスは明示的に失敗させる**（エージェントが検出可能）

**Vercelの事例**: 約40KBのプロンプトを8KBに圧縮しても評価テストパス率100%維持。情報量削減が性能向上につながる逆直感的結果。

| パターン | 問題点 | 対策 |
|---------|--------|------|
| TypeScript `any` 乱用 | 型安全性崩壊 | リンターで自動ブロック |

## 大規模実践事例: 5ヶ月・100万行の知見

出典: [「人間はコードを1行も書かない」という縛りで5ヶ月間プロダクトを作り続けた結果](https://qiita.com/nogataka/items/43c01957fa1e54d9a079) (@nogataka)

### 実績データ

- **5ヶ月間**: 100万行のコード、1,500件のPR
- **開発効率**: 手作業比で約1/10の開発時間
- **チーム規模**: 初期3人→7人
- **PR頻度**: 1人あたり1日平均3.5件

### 段階的開示パターン（ドキュメント構成）

```
AGENTS.md（約100行の目次）
ARCHITECTURE.md
docs/
├── design-docs/（索引付き）
├── exec-plans/（アクティブ/完了/技術的負債）
├── generated/（自動生成）
├── product-specs/
├── references/
├── DESIGN.md, FRONTEND.md, PRODUCT_SENSE.md等
```

短いAGENTS.mdから始まり、必要に応じてより深い層のドキュメントをたどる設計。

### アーキテクチャ自動強制

各ドメインは固定レイヤー（Types → Config → Repo → Service → Runtime → UI）に分割。依存方向をカスタムリンターと構造テストで厳密に検証。横断的関心事はProvidersインターフェース経由で一元化。

### AIスロップ対策（ガベージコレクション）

**黄金原則**をリポジトリにエンコード:
1. 手作りヘルパーより共有ユーティリティパッケージを使う
2. バウンダリでのバリデーション、型付きSDK依存

バックグラウンドタスクが定期的に逸脱をスキャンし、リファクタリングPRを自動作成。ほとんどが1分以内にレビュー可能。

### 技術選定: 退屈な技術（Boring Technology）を優先

APIが安定しており、LLMのトレーニングデータに多く含まれている既知技術を優先。新規ライブラリより既知パターンの再実装を選択（例: p-limitの代わりにmap-with-concurrencyを自前実装）。

### 未解決の課題

- 長期的アーキテクチャ一貫性（数年単位）
- 人間判断の最適配置とエンコード方法
- モデル進化への適応

### 「最小構成」ハーネス設計: CLAUDE.md + Skills + Hooks（自動収集 2026-03-28）

- テストフレームワーク・ガバナンスツール不要の3層アーキテクチャ
- **Hook**（settings.json）= いつ発火するか。プロンプト送信時の自動トリガー・前処理
- **CLAUDE.md** = 何をすべきか。「Claudeの行動制約の憲法」として制約と原則を定義
- **Skills**（SKILL.md）= どうやるか。再利用可能なプロシージャとしてモジュール化

> 詳細: メモリ内 reference_claude_code_minimal_harness.md を参照

### CoDD（整合性駆動開発）— ハーネスの次の進化（自動収集 2026-03-30）
Prompt→Context→Harnessの次に来る方法論。設計書間の依存グラフを構築し、変更影響分析と整合性検証を自動化。Wave概念（上流→下流の波状処理）で設計書を依存順に生成。実案件LMSで設計書18本・全コード・全テストを自動生成。`pip install codd-dev`でインストール可能。Claude Code統合: `/codd-*`スキル群 + PostToolUse Hookで自動scan。比較: バイブコーディング(初速◎/変更耐性✕) vs ハーネス(初速○/変更耐性△) vs CoDD(初速△/変更耐性◎)。
> 詳細: メモリ内 `reference_codd_coherence_driven_dev.md` を参照

### ハーネスエンジニアリング入門 — 5要素と段階的導入（自動収集 2026-03-30）
CLAUDE.mdの次に来る制御パラダイムとして、5要素（ルール・スキル・フック・メモリ・フィードバックループ）の統合制御を提唱。everything-claude-code（100K+ Stars）では28エージェント定義・125+スキル・25+フック。段階的導入: Day1にCLAUDE.md→Week1-2にスキル化→Week2-4にフック設定。エスカレーションラダー（3回ルール）で違反3回ごとに強度を上げる（L1:ドキュメント→L2:AI検証→L3:ツール検証→L4:構造テスト）。
> 詳細: メモリ内 `reference_harness_engineering_intro.md` を参照

### 「狩りから稲作へ」— 4層ハーネス体系（自動収集 2026-04-04）
4層ハーネス体系: 第1層hooks（AI行動リアルタイム制御）、第2層Lefthook（git操作ゲート、Clippy pedantic+ast-grep）、第3層skills（/pre-push-reviewで4エージェント並列レビュー）、第4層GitHub Actions+CodeRabbit（最終防衛）。
