## 定義

**エージェントにプロンプトを打つ人（自分自身）をループから外し、「プロンプトを打ってくれる仕組み」を設計すること。**

ハーネスエンジニアリングが「エージェントが安定して動く環境」を整える技術であるのに対し、ループエンジニアリングはその一段上で、**自動発見（何をやるべきかの発見）と検証（できたかどうかの判定）を自動化し、人間の手動トリガーなしにサイクルが回り続ける構造**を設計する技術である。

> 「AIにループを任せるなら、人間の仕事は prompt 作成から**ハーネス設計・ループ設計**へ移る」

---

## パラダイムの進化

各エンジニアリング手法は**置き換えではなく包含関係**にある。ループエンジニアリングは最も外側の層に位置する。

| 手法 | 対象 | 主眼 |
|------|------|------|
| Prompt エンジニアリング | 単一プロンプトの設計 | 1回のLLM呼び出しを最適化 |
| Context エンジニアリング | コンテキストウィンドウの管理 | 何を見せ、何を圧縮するか |
| Harness エンジニアリング | エージェントの動作環境 | 制約・検証・フィードバックの土台 |
| **Loop エンジニアリング** | **自動発見と検証の自動化** | **人間を介さず回り続けるサイクル** |

関係: Prompt ⊂ Context ⊂ Harness ⊂ Loop

## 組織マネジメントとの類似

ループエンジニアリングは、部下（エージェント）に仕事を任せる**組織マネジメント**の発想と重なる。両極端を避けることが要点。

| 悪いパターン | 問題 | ループ設計での対策 |
|-------------|------|-------------------|
| マイクロマネジメント | 逐一プロンプトを打ち続け、人間がボトルネックになる | トリガーと作業サイクルを仕組み化して手を離す |
| 丸投げ | 目標も検証もないまま放置し、暴走・空回りする | 停止条件と検証機構で「完了」を機械判定する |

すなわち、**目標設定・レビュー体制・引き継ぎの仕組み**を整えるという組織運営の考え方を、そのままエージェント運用に持ち込む。

## ループの4構成要素

自律ループは次の4要素で構成される。**このうち「検証」が最重要**である。検証が甘いと、見かけ上の成功や無限ループを招く。

| # | 構成要素 | 役割 |
|---|---------|------|
| 1 | トリガー | ループを開始するきっかけ（Issue起票、スケジュール、ファイル変更など） |
| 2 | 停止条件 | いつループを止めるか（成功判定、最大反復回数、タイムアウト） |
| 3 | 作業サイクル | 実際に繰り返す処理内容（実装→テスト→修正など） |
| 4 | **検証（最重要）** | 進捗と成功を判定する機構。人間判断に依存しない客観的な判定を置く |

---

## ループ実装が有効になる4条件

すべてのタスクがループ化に向くわけではない。次の4条件を満たすタスクほど自動ループの効果が高い。

1. **反復性** — 同じ処理が繰り返されるパターンである
2. **機械的失敗検出** — ツール（テスト・lint・型チェック）が自動で失敗を検出できる
3. **エージェントの自律完結** — 人間の介入なしにエージェントが処理を完了できる
4. **客観的完成定義** — 人間の主観判断に依存しない、明確な「完成」の定義がある

裏を返せば、**失敗を機械的に検出できない／完成が主観的なタスクは、ループ化の前に検証機構そのものの設計が必要**になる。

## 失敗しないための5層ハーネス設計

自律ループが失敗を量産するのは「ハーネスが弱い」ことに起因する。以下の5層でガードを設計する。

| 層 | 役割 |
|----|------|
| Workflow guard | 工程制御、分岐、戻り先、停止条件 |
| Step guard | 各工程内の手順規定 |
| Verdict guard | LLMの判断を構造化出力（合否など）に変換 |
| Quality gate guard | 決定論的チェック（テスト、lint、型チェック） |
| Standards guard | コーディング規約、文書化規約 |

### AIエージェントがループで陥る典型的な失敗

弱いハーネスのまま自律ループを回すと、以下が頻発する。

- テスト実行の「見かけ上の成功」（実質は未検証）
- 修正が新たな問題を生む
- 無限の修正ラリー（終わらない fix ループ）
- コーディング規約からの逸脱
- 完了根拠の欠落（「なぜ完了と言えるのか」が説明できない）

---

## Verdict guard と Quality gate guardの分離

5層設計で最も重要なのが、**「LLMの判断」と「LLMを通さない検証」を明確に分けること**。

| ガード | 判定の主体 | 例 |
|--------|-----------|-----|
| Verdict guard | LLM（構造化出力で受け取る） | 「レビュー観点を満たすか」をJSON等で回答させる |
| Quality gate guard | 決定論的ツール（LLM非依存） | `make check`、pytest、lint、型チェックの exit code |

**なぜ分けるか**: 判定をLLMの自由記述や文字列マッチングに頼ると、判定がぶれて無限ループや誤完了の原因になる。決定論的に判定できる部分（Quality gate）はツールに委ね、LLMの判断が必要な部分（Verdict）は構造化出力で受け取ることで、ループの停止条件を信頼できるものにする。

## 実装例: kaji（workflow harness）

`kaji` は、Claude Code などの agent CLI を**ワークフローYAMLに沿って実行する** workflow harness。

- GitHub Issue を起点に、`design → implement → review → fix → verify → PR` までを**再開可能な closed loop** として管理する
- ワークフロー定義で**最大反復回数**と**終了条件**を明示的に指定できる（無限ループの防止）

### kaji-starter-python

- GitHub リポジトリテンプレート形式のスターターキット
- Python プロジェクト骨格、5本のワークフローYAML、23の汎用skill を同梱
- 作成直後から `make check` が通る**完成状態**でスタートできる

セットアップの流れ（6ステップ）:

```
1. テンプレートからリポジトリ作成
2. 設定編集
3. sync
4. 検証（make check）
5. commit
6. ラベル設定 → Issue作成（ループのトリガー）
```

## 実装から得た知見

`kaji` の実装・運用から得られた教訓（失敗の再発防止に直結）:

- **判定機構が脆弱（文字列マッチング頼り）だと無限ループの危険がある** → 修正PRで、ツールが埋め込む決定論的な目印による判定方式へ改良
- **工程のやり直し回数に上限を設けていないとバグになる** → 各工程に最大反復回数を明示する
- 判定の信頼性は、Verdict（LLM判断）と Quality gate（決定論的検証）の分離にかかっている

---

## 核心メッセージ

> 自律ループを回すこと自体は難しくない。難しいのは「**いつ止めるか**」と「**本当に完了したか**」を、人間の手動確認なしに信頼できる形で判定する検証機構を設計すること。ループエンジニアリングの本質は、コードを書くことではなく「正しく回り、正しく止まる構造」を設計することにある。
