# Claude Code はじめてガイド

Claude Codeを初めて使う人が「最短で日常に組み込む」ためのガイド。詳細は各ファイルを参照。

---

## Claude Code とは

ターミナル上でClaudeと対話しながら、**コードの読み書き・実行・Git操作**を行えるAnthropicの公式CLIツール。チャットAIに「手足」が付いたイメージ。

```
あなた（指示）→ Claude Code → ファイル読み書き・コマンド実行・Web検索
```

---

## インストールと起動

```bash
# インストール
npm install -g @anthropic-ai/claude-code

# 認証
claude auth login

# 起動（プロジェクトのルートで実行）
cd /path/to/your-project
claude
```

起動するとチャット画面が開く。あとは日本語で指示するだけ。

## 最初に覚える5つの操作

### 1. 普通に話しかける

```
> このプロジェクトの構成を教えて
> src/auth.ts の認証ロジックを説明して
> ユーザー登録のバリデーションにメールアドレスチェックを追加して
```

ファイルを読む・書く・コマンドを実行する、すべて自然言語で指示できる。

### 2. ファイルを指定して渡す（`@`）

```
> @src/index.ts このファイルのエラーハンドリングを見直して
> @src/models/ このフォルダの全モデルにバリデーションを追加して
```

`@` に続けてパスを書くと、そのファイル/フォルダをコンテキストとして渡せる。

### 3. 計画してから動く（`/plan`）

```
> /plan 認証機能をJWT方式で実装したい
```

いきなりコードを書かず、まず計画を提示してくれる。**3ステップ以上のタスクは計画から始めるのがコツ**。計画に納得したら「実行して」と伝える。

### 4. セッションを再開する（`-c`）

```bash
# 前回の続きから
claude -c
```

Claude Codeを閉じても、会話の続きから再開できる。

### 5. 困ったら質問攻めにしてもらう

```
> 認証機能を実装して。あいまいな点があれば、クリアになるまで質問して。
```

Claudeが「A案とB案どちらにしますか？」のように聞いてくれる。自分で気づかなかった考慮漏れも洗い出せる。

## 知っておくと便利なコマンド

| 操作 | コマンド/キー | 場面 |
|------|-------------|------|
| コンテキスト圧縮 | `/compact` | 会話が長くなって回答品質が下がったとき |
| 権限モード切替 | `Shift+Tab` | 編集の自動許可を切り替えたいとき |
| 巻き戻し | `Esc` → `Esc` | Claudeが間違った方向に進んだとき |
| Bash直接実行 | `! npm test` | Claudeを介さず素早くコマンドを打ちたいとき |
| Git差分表示 | `/diff` | 変更内容を確認したいとき |
| コスト確認 | `/cost` | トークン使用量を知りたいとき |
| セッション終了 | `Ctrl+D` | 作業を終えるとき |

## 権限モード — 最初は「default」でOK

Claude Codeはファイルを書き換えたりコマンドを実行したりできるため、**権限モード**で安全性を調整する。

| モード | 概要 | おすすめ場面 |
|--------|------|------------|
| **default** | 全操作の前に確認を求められる | **最初はこれ**。動きを理解するまで |
| **acceptEdits** | ファイル編集は自動、コマンド実行のみ確認 | 慣れたら日常使い |
| **plan** | 読み取り専用。計画だけ出力 | 設計・探索 |

`Shift+Tab` で切り替えられる。最初は `default` で全承認しながら、Claudeが何をしているか観察するのが学習効率が高い。

## CLAUDE.md — Claudeへの「申し送り事項」

プロジェクトに `CLAUDE.md` ファイルを置くと、毎回のセッションで自動的に読み込まれる。

```markdown
# プロジェクト概要
ECサイトのバックエンドAPI（ASP.NET Core 8）

# コマンド
- テスト実行: dotnet test
- ビルド: dotnet build

# ルール
- テストなしで完了としない
- async/awaitを使い、.Result/.Wait()は禁止
```

**最初は5〜10行で十分**。使いながら「毎回同じことを言っている」と気づいたら追記する。

→ 詳細: [02-setup.md](02-setup.md)

## 効果的な指示の出し方

### 結論ファーストで伝える

```
○ 「認証機能のテストを追加したい。理由はXXバグの再発防止。既存テストを壊さないこと」
× 「えーと、先日XXというバグがあって、認証周りが怪しいので、テストがあるといいかなと...」
```

**最初の1文で「何をしたいか」**、次に「なぜ」「制約」を添える。

### 1回の指示は1〜2タスクまで

3つ以上やりたいことがあるなら `/plan` で全体像を先に固める。

### コンテキストを明示する

```
> @src/auth/login.ts このファイルの Loginメソッドを見てから修正方針を提案して。
> 制約: データベースのスキーマは変更できない
```

## よくあるつまずきと対処

| 症状 | 原因 | 対処 |
|------|------|------|
| 回答が雑になってきた | コンテキスト枯渇 | `/compact` で圧縮、または新セッション |
| 編集を許可できない | 権限モードが `default` | `Shift+Tab` で `acceptEdits` に変更 |
| 間違ったコードを書いた | — | `Esc+Esc` で巻き戻し |
| 毎回同じ説明をしている | CLAUDE.mdがない | CLAUDE.md を作成して指示を永続化 |
| 大きなファイルで重い | 全行読み込み | `@` で必要なファイルだけ渡す |

→ 詳細: [09-troubleshooting.md](09-troubleshooting.md)

## 心構え: Why を手放すな

Claude Codeは「どう書くか（How）」を高速に処理してくれる。しかし「**なぜ作るのか（Why）**」「**そもそもやるべきか（Whether）**」は人間が保持する。

1. AIに投げる前に「なぜこれを作るのか」を一行書く
2. AIの出力に対して「これでいい理由」を一文で言語化する
3. 全部任せるのではなく、判断と検証は自分でやる

→ 詳細: [08-tips.md](08-tips.md) の「Why/Whether を手放すな」
