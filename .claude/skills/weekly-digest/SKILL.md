# Weekly Digest Skill

直近1週間にリポジトリへ追加された情報を分析し、「今週読むべき情報・活用案・ゲーム企画の種」を1枚の Markdown にまとめるワークフロー。

単なる更新ファイル一覧ではなく、利用者（ゲーム開発会社のマネージャー兼エンジニア）が**日曜に Digest を1本読むだけで、今週の追加情報・読むべき3件・実際の活用方法が分かる**状態を目的とする。

## 目的

- 直近7日間に追加された情報の概要とカテゴリ別件数を提示する
- 利用者との関連度が高い情報を最大5件、活用案付きで紹介する
- 今週優先して読む3件を提示する
- 寓話・格言・技術情報を組み合わせたゲーム企画の種を1件生成する

## トリガー（起動条件）

以下のいずれかでこのワークフローを開始する:

- 「Weekly Digestを作成して」「今週のDigestを作って」「今週追加された情報をまとめて」「今週の情報を整理して」
- 「Weekly Digestを再生成して」（→ 再生成モード。後述）
- `routine/WeeklyDigest.txt` を用いた定期実行（毎週日曜 08:00 JST）

## 利用者プロフィール（関連度評価の基準）

ゲーム開発会社のマネージャー兼エンジニア。主な関心領域:

ゲーム企画 / ゲーム開発 / Unity / プログラミング / AIチャットゲーム / LLM API活用 / AIエージェント / Claude Code / 開発効率化 / プロジェクト管理 / ゲーム会社のマネジメント / 受託ゲーム開発 / 個人ゲーム制作 / カードゲーム制作 / App Store向けゲーム開発

## 制約（初版方針）

- 情報量を増やすより「今週何を読むべきか・何に使えるか」が短時間で分かることを優先する
- すべてを詳しく紹介しない。注目情報は**最大5件程度**
- メール/Slack通知・Web化・ベクトルDB・永続スコア保存・利用履歴追跡は初版では実装しない
- **コンテキスト制限を理由に途中で中断しない**。集計・要約が重い場合はサブエージェント（Explore）に委譲する

## ワークフロー

### Step 1: 対象期間の計算

- 基準時刻 = 実行日（原則、日曜）の 08:00 JST
- **対象期間 = 実行日の7日前 00:00 JST 〜 実行日 07:59 JST**
  - 例: 実行 `2026-07-12 08:00 JST` → 対象 `2026-07-05 00:00` 〜 `2026-07-12 07:59`
- Digest には対象期間（`YYYY-MM-DD 〜 YYYY-MM-DD`）と生成日時（`YYYY-MM-DD HH:mm JST`）を必ず明記する
- 本リポジトリのローカル時刻は JST。git の `--since/--until` はローカル時刻で解釈される

### Step 2: 追加情報の取得（Git履歴ベース）

マージコミットを除外して追加ファイルを取得する（PRマージコミットと元コミットの**二重集計を防ぐため必ず `--no-merges`**）:

```bash
# 追加された「内容ファイル」の一覧（実行日と7日前を埋める）
git log --no-merges --since="<7日前> 00:00" --until="<実行日> 08:00" \
  --diff-filter=A --name-only --pretty=format: | sort -u
```

- `--until` を実行時刻（08:00）に置くことで、Digest 自身のコミットや当日昼以降の変更を巻き込まない
- 件数だけでなく、コミットメッセージ（`git log --no-merges --since=... --until=... --pretty="%s"`）で傾向を把握する
- 履歴が不正確・取得困難な場合は、各カテゴリの `log/YYYY-MM-DD.md`（デイリー収集ログ）を併用して補正する

### Step 3: 対象カテゴリと除外対象

**集計対象（内容ファイルのみ）:**

| カテゴリ | 集計する内容ファイル |
|---|---|
| `Claude/` | `references/reference_*.md` と 章ファイル `NN-*.md` |
| `GameDev/` | `references/reference_*.md` と 章ファイル `NN-*.md` |
| `Fables/` | `fables/{origin}/*.md` |
| `Maxims/` | `maxims/{category}/*.md` |

**集計対象外（除外）:**

- `Digests/*`
- 各カテゴリの `log/`・`index/`・`references/` 直下以外のメタ、`REFERENCES.md`・`INDEX.md`・`README.md`・`CLAUDE.md`・`_template.md`
- ルート `README.md` のみの変更、設定ファイル・自動生成ログのみの変更

「新規追加件数」は上記の**追加された内容ファイル数**で数える。既存章ファイルの更新（M）は件数に含めず、内容が顕著なら「今週の傾向」で言及する。

### Step 4: 情報の要約（サブエージェント委譲を推奨）

- 追加ファイル数が多い場合、各ファイルの内容確認・要約は **Explore サブエージェント**に委譲し、メインは指揮官役に徹する（コンテキスト節約）
- 各内容ファイルは冒頭のフロントマター（`name`/`description`/`title`/`moral` 等）と先頭数行で要約できる。捏造せず、必ず実ファイルの記述に基づく

### Step 5: 関連度評価

追加情報について、利用者との関連度を5段階で評価する:

| 評価 | 基準 |
|---|---|
| 5 | 現在の業務・企画・個人制作へ**直接**利用できる |
| 4 | 半年以内に利用する可能性が高い |
| 3 | ゲーム開発者・マネージャーとして有益 |
| 2 | 将来的に利用する可能性がある |
| 1 | 興味はあるが現状の活動との関連性は低い |

- Digest には**原則として評価4以上**を掲載する
- ただしカテゴリが偏りすぎないよう調整する

### Step 6: 注目情報の選定

- 全体で**最大5件程度**
- カテゴリごとの掲載上限（目安）: Claude 最大3件 / GameDev 最大3件 / Fables 最大1件 / Maxims 最大1件
- 評価が高く、活用案が具体的に書けるものを優先する

### Step 7: Digest生成

出力先: `Digests/YYYY/YYYY-MM-DD.md`（例: `Digests/2026/2026-07-12.md`）。年ディレクトリがなければ作成する。

以下の構成を基本とする（必要に応じて改善可）。

**リンク規約**: 対象ファイルへの参照（注目情報の「関連ファイル」、ゲーム企画の「元情報」、実行提案など）は、クリック可能な相対 Markdown リンクにする。Digest は `Digests/YYYY/` 配下にあるため、リポジトリルート起点のパスには `../../` を前置する。
例: `[Claude/12-loop-engineering.md](../../Claude/12-loop-engineering.md)`

```markdown
# Weekly Digest

対象期間: YYYY-MM-DD 〜 YYYY-MM-DD
生成日時: YYYY-MM-DD HH:mm JST

---

## 今週の概要

| カテゴリ | 新規追加 |
|---|---:|
| Claude | XX件 |
| GameDev | XX件 |
| Fables | XX件 |
| Maxims | XX件 |
| 合計 | XX件 |

## 今週の傾向

（3〜5行で傾向を説明）

---

## 今週の注目情報

### 1. タイトル
- カテゴリ: Claude
- 関連度: 5 / 5
- 概要: （短く）
- 利用者との関連: （関心領域との対応）
- 活用案: （業務・個人制作でどう使うか）
- 関連ファイル: [表示パス](リンク)

（最大5件）

---

## 今週読むならこの3件

1. タイトル — 理由
2. タイトル — 理由
3. タイトル — 理由

---

## ゲーム企画への応用

- 元情報: （利用した寓話・格言・技術情報）
- 企画案: （短いゲーム企画案）
- 想定ジャンル: （パズル / カードゲーム / シミュレーション等）

---

## 今週の実行提案

- 今週試すこと: （具体的な行動を1件）
- 想定作業時間: （30分・1時間など）
- 期待できる効果: （何が確認できるか）
```

### Step 8: 情報がない場合

対象期間に追加情報がなくても Digest を生成する。これにより「正常実行だが更新なし」と「ルーチン失敗」を区別できる:

```markdown
# Weekly Digest

対象期間: YYYY-MM-DD 〜 YYYY-MM-DD
生成日時: YYYY-MM-DD HH:mm JST

今週追加された情報はありませんでした。
```

### Step 9: INDEX更新

`Digests/INDEX.md` を作成/更新する。新しい Digest を各年セクションの**上**に追加する:

```markdown
# Weekly Digest 一覧

## 2026
- [2026-07-12](2026/2026-07-12.md)
- [2026-07-05](2026/2026-07-05.md)
```

### Step 10: 重複生成防止 / 再生成

- 同一日付の `Digests/YYYY/YYYY-MM-DD.md` が**既に存在する場合、通常実行では新規作成せず処理終了**する（INDEXも変更しない）
- ただし「Weekly Digestを再生成して」と手動依頼された場合は**再生成モード**とし、既存ファイルを上書き更新する（INDEXの行は重複追加しない）

### Step 11: コミット

既存ルーチン運用に合わせる。

- コミットメッセージ（Digest生成時）: `docs(digest): add weekly digest YYYY-MM-DD`
- Skill/ルーチンの新規追加時: `feat(digest): add weekly digest routine`
- main 以外のブランチで作業した場合は push 後に PR を作成し auto-merge ラベルを付与する:

```bash
gh pr create --base main --head "$(git branch --show-current)" \
  --title "<コミットメッセージと同じ内容>" --body "<変更概要を1〜3行>" --label "auto-merge"
```

- `--label` でエラーが出たらラベルなしで作成後 `gh pr edit <PR番号> --add-label auto-merge`
- **ログ・Digestに PR情報（URL・番号・ラベル）を記載しない**（マージ後ブランチへの追記は main に反映されず不要差分を生むため）

#### リモート実行のオーバーライド

`routine/WeeklyDigest.txt` からの定期実行では、確認なしで `git add → commit → push → PR(auto-merge)` まで自動完了する。ローカル対話実行では、ファイル更新までを行い commit/push はユーザー確認を待つ。

### エラー時の扱い

- 集計・要約の一部が失敗しても、取得できた分で Digest を生成し、失敗箇所を Digest 末尾に注記する（中途半端な未生成で放置しない）
- Git 履歴が取得できない場合は各カテゴリ `log/` を代替ソースにする
- 対象期間の解釈に迷う場合は「実行日の7日前 00:00 〜 実行日 08:00 直前」を採用する

## 実装後チェックリスト

1. Weekly Digest Skill が認識される
2. 手動実行できる（トリガーワード）
3. 対象期間が正しい
4. マージコミットによる二重集計がない（`--no-merges`）
5. 4カテゴリを正しく分類できる
6. Digest Markdown が生成される
7. `Digests/INDEX.md` が更新される
8. 同一日付で二重生成されない
9. 再生成指示では更新できる
10. 情報がない場合も正常終了する（空 Digest 生成）
