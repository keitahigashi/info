# ゲームプログラミングパターン

エンジン非依存の設計パターン・アーキテクチャ。

---

## Entity Component System (ECS)

### 概要

Entity-Component-System (ECS) は、データとロジックを分離することでゲームオブジェクトを構造化するアーキテクチャパターン。従来のオブジェクト指向における「継承の深いツリー」問題を、コンポジション (Composition) で解決する。

| 要素 | 役割 | 性質 |
|------|------|------|
| **エンティティ (Entity)** | ゲームオブジェクトを追跡する一意の識別子 | IDのみ。振る舞いもデータも持たない |
| **コンポーネント (Component)** | オブジェクトの状態を保持するプレーンなデータ構造 | ロジックを持たない値型・構造体 |
| **システム (System)** | マッチするコンポーネントセットに対して動作するロジック | データを横断的に処理する関数群 |

### ECSが解決する問題

従来の継承ベース設計では、`Enemy → FlyingEnemy → FireFlyingEnemy` のように継承ツリーが爆発する。ECSでは、エンティティに `Position`, `Flying`, `FireBreathing` といったコンポーネントを付与するだけで同等の振る舞いを表現できる。結合度を下げ、テスタビリティを向上させ、キャッシュフレンドリーなメモリアクセスによる予測可能なパフォーマンスを実現する。

### アーキテクチャ判断

#### Push vs. Pull 処理モデル

| アプローチ | 適したケース | トレードオフ |
|-----------|------------|------------|
| **Pull（ポーリング）** | 多数のエンティティに対する頻繁な更新 | 静的データへの無駄なスキャンが発生 |
| **Push（イベント駆動）** | スパース・バースト的な変更 | キュー管理の複雑さが増す |

基本方針: シミュレーションループにはPullをデフォルトとし、入力やネットワークメッセージなど境界イベントでPushを使う。

#### アーキタイプ (Archetype) 設計

コンポーネントの組み合わせ（アーキタイプ）をどう設計するかは、パフォーマンスに直結する判断。

- **広いアーキタイプ**: クエリ変更を最小化するが、過剰フェッチやメモリチャーンのリスク
- **細粒度アーキタイプ**: ホットクエリのローカリティが向上するが、テーブル数が増加しリシェイプオーバーヘッドが発生

戦略として、高チャーンフィールド（頻繁に変更されるデータ）を静的データから分離し、コンポジション変更率をプロファイリングして判断する。

### コンポーネント設計ガイドライン

- シンプルな値型またはプレーン構造体を維持する
- 仮想ディスパッチや重いインダイレクションを避ける
- 親子関係・所有権には明示的なエンティティ参照リンクを使用する
- ライフサイクルタグ（`PendingDestroy` など）とペアにしてダングリング参照を防止する

### システムスケジューリング

- 明示的なフェーズ名を確立し、各システムの読み書き関係を文書化する
- コンポーネントタイプごとにどのシステムが読み書きするかの依存テーブルを構築する
- 同期ポイントはエンジンライフサイクルが要求する箇所のみに挿入する

### よくある落とし穴

| 問題 | 症状 | 解決策 |
|------|------|--------|
| **隠れた順序依存** | 同じコンポーネントに書き込む2システムが未定義順序で動作 → フレーム依存バグ | コンポーネントごとの書き込み所有権を宣言し、名前付きフェーズにシステムを固定。力の蓄積にはバッファを使用 |
| **エンティティライフタイム** | 削除がリレーション内のダングリング参照を残す | `PendingDestroy` タグでマークし、削除前に明示的なティアダウンシステムを実行。1フレームのレイテンシと引き換えにnull参照を防止 |
| **アーキタイプチャーン** | 揮発性データの頻繁な変更でテーブル再構築が多発 | 揮発性データを別の小テーブルに分離、またはビットセットを使用 |
| **キャッシュミス** | 小さなコンポーネントが分散しキャッシュ効率が低下 | 一緒にクエリされる小さなコンポーネントをマージ。プロファイリングで検証 |

### ハイブリッドアプローチ

実際のプロジェクトでは、完全なECSとエンジンネイティブ機能を組み合わせるハイブリッドが現実的。

- ゲームプレイシミュレーションはECSスタイルのシステムで処理
- レンダリング・オーディオはネイティブエンジンオブジェクトに維持
- レジストリレイヤーを通じて参照をマッピング
- コンポーネントごとに明確な所有権ルールを定義し、重複ミューテーションを回避

## ステートマシン (State Machine)

### 概要

ステートマシン（有限状態機械, FSM: Finite State Machine）は、システムが取り得るすべての状態と、それらの間の遷移を体系的にモデル化するパターン。ゲーム開発では**キャラクター状態管理・画面遷移・ゲーム進行管理・ターン構造**など、幅広い場面で適用される。

### FSMが解決する問題

状態遷移のスパゲッティ `if-else` を排除する。各状態を独立したクラスとして定義し、遷移条件を明示的に管理することで、コードの可読性と保守性が飛躍的に向上する。「明確で排他的な振る舞いモード」を持つあらゆる場面で有効。

### 状態遷移設計の手法

#### 状態遷移図 (State Transition Diagram)

UMLステートマシンまたは有限オートマトンを使用し、状態・イベント・遷移条件・アクティビティを視覚的に表現する。全体像の把握に優れる。

#### 状態遷移表 (State Transition Table)

「拡張階層化状態遷移表」形式により、すべての状態とトリガーの組み合わせを網羅的に検証する。仕様漏れの防止に有効。

**両者はセットで運用すべき。** 図で全体像を俯瞰し、表で網羅性を検証する。

### 設計時の重要原則

1. **上流工程を優先**: 要求分析 → ユースケース分析を経て初めて状態遷移設計に取り組む
2. **システム全体視点**: 個別の状態遷移でなく、全体の振る舞いをユーザー観点で定義する
3. **過度な複雑化を避ける**: 状態爆発 (State Explosion) や放射状遷移が生じたら設計を見直す
4. **協働開発**: 一人設計ではなく、チーム・ユーザーを巻き込んで複数観点から検証する
5. **アクティビティ図との区別**: 順序性ある処理はアクティビティ図として分離すべき

### メリットとデメリット

| メリット | デメリット |
|---------|----------|
| 分析段階で仕様漏れや不可能な遷移を早期に検出 | 未経験メンバーへの教育コスト |
| チーム内のコミュニケーション向上 | 最適な設計に「正解がない」場合がある |
| 十数個程度の状態で「不具合を出したことがない」水準の品質維持 | 相互変換やコード自動生成には有料ツールが必要な場合も |
| 仕様変更時の対応が迅速かつ低コスト | — |
| 完成したモデルはそのまま実装可能 | — |

### 実践事例: カードゲームのターン管理

大規模なステートマシンの実例として、カードゲーム対戦機能では **33種類のState** で複雑なターン構造を管理した事例がある。

- バトル開始・終了（5種類）
- ターン制御（4種類）
- 移動・攻撃・召喚フェーズ（各複数）
- 能力割り込み機構

自動遷移と入力待機状態を組み合わせ、能力による中断・連鎖発動にも対応。各状態を独立したクラスとして実装することで、33種類でも可読性を維持している。

## オブザーバーパターン (Observer) とメッセージ駆動

### 概要

オブザーバーパターンは、あるオブジェクトの状態変化を、そのオブジェクトを知らない複数のオブジェクトに通知する仕組み。ゲーム開発では「1つのトリガーに複数システムが反応する」場面で頻繁に使われる。メッセージ駆動アーキテクチャ (Message-Driven Architecture) はこれを拡張し、システム全体の疎結合化を実現する。

### 直接呼び出しの問題

従来の直接メソッド呼び出し（例: `GameInformation.AddScore(10)`）では、呼び出し元がクラス定義やメソッド内容に依存し、結合度 (Coupling) が高まる。「相互に知るべきでないシステム間の密結合」が課題。

### オブザーバーの基本構造

- **サブジェクト（発行者）**: イベントの発生を通知する
- **オブザーバー（購読者）**: イベントに反応して処理を実行する
- 送信側は具体的な受取先を指定しない

例: 「敵全滅」イベントを発行すると、スコア計算・UI更新・サウンド再生がそれぞれ独立に反応する。

### メッセージ駆動の利点

| 利点 | 説明 |
|------|------|
| **低結合度 (Low Coupling)** | 処理をコールする側は、コール先のクラス構成やその内容に依存しない |
| **受信意思表明 (Subscription)** | モジュール側が「こういうメッセージが来たら配信してくれ」と宣言する方式 |
| **ブロードキャスト配信** | 単一メッセージで複数モジュールに配信可能 |
| **スレッド・ネットワーク透過性** | 同一プロセス内、スレッド間、ネットワーク越しの配信を透過的に処理可能 |
| **テスタビリティ** | モジュール間が独立しているためテストが容易 |

### イベントキュー (Event Queue / Message Bus) との組み合わせ

オブザーバーパターンの同期的な通知に対し、イベントキューは非同期的にメッセージをバッファリングして処理する。

- **同期オブザーバー**: イベント発火と同時にリスナーが実行される。シンプルだが、重いリスナーがフレームヒッチを起こしうる
- **非同期イベントキュー**: メッセージをキューイングし、適切なタイミングで順次処理。入力処理、ネットワーク、重い非同期操作に適する

**実践的な組み合わせ:**
- **Observer + Event Queue** = 非同期イベント処理（大規模プロジェクト向け）
- **Command + Queue** = 入力バッファリング（格闘ゲームのコマンド入力など）

### 実践事例: 能力発動のTriggerシステム

カードゲームの事例では、40種類のトリガー (Trigger) をオブザーバーパターンで管理し、能力発動条件の判定とUI通知を疎結合に実現している。発行側は具体的な購読者を知る必要がなく、新たな能力やUIコンポーネントの追加が容易。

> 📎 出典: reference_message-driven-game-architecture.md, reference_10-game-design-patterns.md, reference_card-game-command-architecture.md

## イベントキュー (Event Queue)

### 概要

イベントキューは「メッセージやイベントの送信と処理を時間的にデカップルする」パターン。リクエスターとプロセッサー間のバッファとしてキューを導入し、送信者はエンキューして即座にリターンする。別プロセスが自分のペースでキューからアイテムを取得し非同期処理を行う。Robert Nystrom の『Game Programming Patterns』で体系的に解説されている。

### 同期 API の 3 つの問題

イベントキューが解決する、同期 API が直接リクエストを処理する際の問題:

| 問題 | 説明 |
|------|------|
| **ブロッキング実行** | API 呼び出しが完了まで呼び出し元を停止する |
| **個別リクエスト処理** | 複数リクエストの最適化・集約ができない |
| **スレッド問題** | 同期呼び出しが呼び出し元のスレッドで実行され、複数システム同時呼び出しで並行性問題が発生 |

### リングバッファ実装

配列要素のシフトではなく、head と tail ポインタが modulo 演算でラップアラウンドする。キャッシュ効率を維持しつつ連続的エンキュー/デキューをサポートする。リクエストは処理されるまで持続する離散データ構造（例: `PlayMessage` に SoundId と volume）に**具象化 (Reification)** される。

### 設計判断

#### キュー内容タイプ

| タイプ | 説明 | 特徴 |
|--------|------|------|
| **イベント（通知型）** | 過去の出来事を通知（「敵が死亡」） | 複数リスナーにブロードキャスト。広いスコープ |
| **メッセージ（要求型）** | 望む未来のアクションを要求（「音を再生」） | 単一受信者。より限定的ルーティング |

#### 読み取りアクセスモデル

| モデル | 説明 |
|--------|------|
| **シングルキャスト** | キューは読み取り側のプライベート |
| **ブロードキャスト** | 全リスナーが全アイテムを受信。イベントフィルタリングが必要 |
| **ワークキュー** | 各アイテムを 1 リスナーに配布。スケジューリングロジックが必要 |

#### 書き込みアクセスモデル

| モデル | 説明 |
|--------|------|
| **単一ライター** | 暗黙的な起点。サイクル防止だが柔軟性に制限 |
| **複数ライター** | 任意システムがアクションを要求。メッセージ内での送信者 ID 追跡が必要 |

#### オブジェクト寿命

| 戦略 | 説明 |
|------|------|
| **所有権移転** | 送信者がメッセージを放棄、受信者がクリーンアップ担当 |
| **共有所有権** | 参照カウントで全参照が消えるまでメッセージを維持 |
| **キュー所有権** | 事前割当プールスロットにメッセージが永続。送信者が事前割当オブジェクトを参照 |

### 重大な考慮事項

#### 状態の陳腐化 (State Staleness)

キューに入れたイベントは未知の間隔で遅延通知されるため、その間に世界が変化している可能性がある（エンティティ解放、環境変更など）。エンキュー時に必要なコンテキストをすべてキャプチャすることが重要。

#### フィードバックループ

非同期システムは同期コードならクラッシュするサイクルを隠す可能性がある。一般的な安全策として、イベント処理中のイベント送信を禁止する。ログで問題パターンを特定する。

#### グローバル状態の危険性

中央イベントキューはグローバル変数として機能し、隠れた依存関係・予測不能な状態相互作用・追跡困難なバグのリスクがある。

### いつ使うか

**使うべき場面:**

- 送信者が不規則なタイミングでリクエストをプッシュする
- 受信者が便利な瞬間にプルする必要がある
- 複数リクエストの集約が有益
- スレッドがコア間で関心を分離する

**使わないべき場面:**

- 送信者が即座の応答を必要とする
- 状態がリクエスト時点の世界条件を反映すべき

### 関連パターン

- **Observer**: リスナーを送信者からデカップルする同期的代替。本ナレッジベースのオブザーバーパターンセクションで詳述
- **Command**: キューに入れて遅延実行可能。Command + Queue = 入力バッファリング
- **Update Method**: キュー評価の処理サイクルを提供（ゲームループ参照）
- **Object Pool**: 固定サイズキューの効率的バッキングストア

## コマンドパターン (Command)

### 概要

コマンドパターンは、操作を不変のオブジェクト (Immutable Object) としてカプセル化するパターン。入力処理・Undo/Redo・リプレイ・入力バッファリング・ネットワーク同期など、ゲーム開発の広範な場面で活用される。

### コマンドパターンが解決する問題

- 入力処理のハードコード化（キーバインドの柔軟な変更）
- Undo/Redo機能の実装
- リプレイシステムの構築
- 入力バッファリング（格闘ゲームのコマンド入力先行入力）

### 基本構造

| 要素 | 役割 |
|------|------|
| **Command** | 盤面更新の不変オブジェクト。実行内容を記録可能 |
| **Invoker** | コマンドの実行を制御する（キューイング・遅延実行） |
| **Receiver** | コマンドの対象となるオブジェクト |

### 決定論的リプレイ (Deterministic Replay)

コマンドパターンの最も強力な応用の一つ。すべての操作をコマンドとして記録し、再実行することで完全な再現が可能。

- **コマンド履歴の共有**: 各クライアントが同じコマンド列を実行し、決定論的に同一の結果を得る
- **再接続・復元**: 切断時に過去のコマンド履歴をサーバーから取得 → 高速リプレイで盤面復元
- **チート対策**: コマンド署名の検証により不正な操作を検出
- **JSON直列化**: コマンドをシリアライズ可能にすることで、保存・送信・再生が容易

### 実践事例: カードゲームの4つのコアデータ構造

| 構造 | 役割 | 規模 |
|------|------|------|
| **BattleData** | HP・攻撃力・位置情報・状態異常の統合管理 | — |
| **Operation** | 「実行前の可能性」を表現。選択可能マスの事前バリデーション | 12種類 |
| **Command** | 盤面更新の不変オブジェクト。全進行を記録可能 | 29種類 |
| **Trigger** | イベント。能力発動条件判定とUI通知の二重用途 | 40種類 |

OperationとCommandの分離が特徴的。Operationは「これから何ができるか」をUIに提示し、Commandは「実際に何が起きたか」を記録する。

### パターンの組み合わせ

- **Command + Queue** = 入力バッファリング（格闘ゲームの先行入力）
- **Command + Observer** = コマンド実行時にUI・サウンド・エフェクトへ自動通知
- **Command + State** = ステートに応じた有効コマンドの制御

## ゲームループ (Game Loop)

### 概要

ゲームループは「ユーザー入力をブロックせずに処理し、ゲーム状態を更新し、ゲームを描画する」連続サイクル。異なるハードウェア間で一貫したゲームプレイ速度を維持するために時間を追跡する、ゲームプログラミングの最も基盤的なパターン。

### 基本動作の3フェーズ

1. **入力処理 (Process Input)** — 入力を待たずにユーザーコマンドを処理
2. **更新 (Update)** — AI・物理を含むゲームシミュレーションを進行
3. **描画 (Render)** — 現在のゲーム状態をプレイヤーに表示

### タイムステップの進化5段階

ゲームループのタイムステップ管理は、段階的に洗練されてきた。各段階は実際の問題を解決するために発展した。

#### 第1段階: 固定デルタタイム（ナイーブ）

1/60秒のような定数タイムステップ。更新ループが1フレーム内に完了する場合のみ動作する。ディスプレイリフレッシュレートの違いやマシン性能不足で破綻し、シミュレーションが速すぎるか遅すぎる状態になる。

#### 第2段階: 可変デルタタイム（問題あり）

実際のフレーム時間を計測しデルタタイムとして渡す。「物理シミュレーションの振る舞いは渡されるデルタタイムに依存する」ため、任意のタイムステップで物理が不安定になる。バネが爆発、オブジェクトが壁を貫通、床を突き抜けるなどの問題が発生する。非決定論的で、丸め誤差が蓄積する。

#### 第3段階: セミ固定タイムステップ（制限付き）

大きなフレームを小さなステップに分割して最大デルタタイムを制限する。しかし、フレームごとの複数シミュレーションステップが **死のスパイラル (Spiral of Death)** を引き起こしうる — 処理が追いつかず、シミュレーションステップが増加し、さらに処理が遅延する悪循環。

#### 第4段階: 固定タイムステップ + アキュムレータ（レート分離）

**ブレイクスルー**: レンダラーが時間を生産し、シミュレーションが離散的な固定サイズのステップで消費する。アキュムレータがフレーム時間をバッファリングし、物理エンジンがレンダーフレームレートに関係なく一定間隔でステップする。**決定論的再現性** を実現。

#### 第5段階: 補間（スムーズなビジュアル出力）

未シミュレートの残余時間からのスタッターを解決。前回と現在の物理状態間のブレンディングファクター (0.0〜1.0) で補間し、シミュレーションと表示レートが一致しなくてもスムーズな表示を実現。

### タイムステップアプローチの比較

| アプローチ | 説明 | 問題 | 推奨度 |
|-----------|------|------|--------|
| **固定（同期なし）** | 最もシンプル | ハードウェアで速度が変化 | 非推奨 |
| **固定（同期あり）** | sleep/delayでFPSをキャップ | 処理超過時にゲームプレイ減速 | 限定的 |
| **可変タイムステップ** | フレームごとの経過時間でスケール | 非決定論的物理、不安定性 | 非推奨 |
| **固定更新+可変描画** | 固定ステップでゲームロジック、任意タイミングで描画 | 安定性とビジュアルの滑らかさを両立 | **推奨** |

### 更新/描画分離 (Update-Render Decoupling)

固定タイムステップ更新と可変レート描画の分離が、物理の安定性問題を解決しつつスムーズなビジュアルを維持する鍵。更新と描画間の「残留ラグ」は補間 (Interpolation) で補償する。

### 電力管理とプラットフォーム差異

| プラットフォーム | 戦略 |
|----------------|------|
| デスクトップ | 最大速度で実行。余裕があればVSyncで同期 |
| モバイル | 30-60 FPSにクランプし残りをスリープ。バッテリー消費を抑制 |

### ループの所有者

| 方式 | 利点 | 欠点 |
|------|------|------|
| **プラットフォームのイベントループ** | シンプル | 制御が限定的 |
| **ゲームエンジンのループ** | 実装負担なし | 柔軟性低下 |
| **カスタム実装** | 最大限の制御 | プラットフォーム調整が必要 |

## 入力処理アーキテクチャ

### 概要

入力処理は、ゲームループの最初のフェーズであり、プレイヤーの操作をゲーム内のアクションに変換する橋渡しの役割を担う。コマンドパターンとの組み合わせが定石。

### 入力とコマンドの分離

入力処理の基本原則は、物理的な入力（キーボード・コントローラーのボタン）とゲーム内の操作（ジャンプ・攻撃・メニュー操作）を分離すること。

- **入力マッピング層**: 物理入力をゲームアクションに変換
- **コマンド層**: ゲームアクションを実行可能なオブジェクトとしてカプセル化

この分離により、キーリマッピング、リプレイ、AIによる自動入力、ネットワーク経由の入力注入がすべて同じ仕組みで実現可能になる。

### 入力バッファリング

格闘ゲームに代表される先行入力の仕組み。入力をキューに蓄積し、適切なタイミングで消費する。**Command + Queue** パターンの典型的な応用。

- 入力をフレーム単位でタイムスタンプ付きキューに格納
- ゲームロジック側が適切なフレームでキューから取り出して実行
- 入力ドロップ（取りこぼし）やフレームヒッチを防止

### Facilitator/Facilitatee パターン

プレイモードによる入力ソースの抽象化。対戦ロジック本体は純粋なゲームロジックに専念し、入力ソースの差異はFacilitator層で吸収する。

| Facilitator | 入力ソース |
|-------------|-----------|
| オンライン | ネットワーク経由のプレイヤー入力 |
| ローカル | コントローラー/キーボード |
| AI | CPU判断ロジック |
| リプレイ | 記録済みコマンド履歴 |
| チュートリアル | スクリプト化された操作ガイド |
| デバッグ | 開発者用テスト入力 |

この設計により、ネットワーク/ローカル、プレイヤー/AIの差分をFacilitator層で吸収し、ゲームモデル本体には手を加えずに新しいプレイモードを追加できる。

> 📎 出典: reference_10-game-design-patterns.md, reference_card-game-command-architecture.md

## オブジェクトプール (Object Pool)

### 概要

再利用可能なオブジェクトのコレクションを事前に確保し、ゲームプレイ中のヒープ割り当てをゼロにするパターン。弾丸、パーティクル、エフェクト、敵など「毎秒10以上の同一オブジェクトの生成・破棄」が発生する場面で必須のパターン。

### 解決する2つの問題

#### メモリ断片化 (Memory Fragmentation)

空きヒープ空間が小さな断片に散在する問題。「14バイトの空きがあるが、7バイト2つに断片化」していると、12バイトのオブジェクトを確保できない。コンソールでは「ソークテスト」（数日間の連続実行）で「忍び寄る断片化やメモリリーク」によるクラッシュが発生する。

#### 割り当て性能 (Allocation Performance)

パーティクルなど多数のオブジェクトの生成・破棄は、メモリアロケータへの頻繁なリクエストとなり、パフォーマンススパイクを引き起こす。

### 動作原理

1. 起動時に大きな連続メモリブロックを確保し、オブジェクトを「未使用」に初期化
2. 要求時に利用可能スロットを「使用中」にマークし初期化
3. 解放時に「未使用」状態に戻す
4. ゲームプレイ中はヒープ割り当てなし

### フリーリスト最適化 (Free List)

未使用オブジェクトのメモリ内に、次の利用可能オブジェクトへのポインタを union で格納する。追加メモリオーバーヘッドなしで **O(1)** の取得を実現。使用中のオブジェクトはそのメモリ領域を本来のデータに使い、未使用のオブジェクトは同じ領域を次ポインタに使う。

### 設計判断

| 判断ポイント | 選択肢A | 選択肢B |
|------------|---------|---------|
| **プールとの結合** | オブジェクトがプールを認識（シンプル、privateコンストラクタ可） | 汎用プール（柔軟だが別途追跡配列が必要） |
| **初期化責任** | プールが管理（完全カプセル化） | 外部で初期化（シンプルだが割り当て失敗チェック必要） |

### 制約と対策

| 制約 | 説明 | 対策 |
|------|------|------|
| **固定容量** | 同時存在オブジェクト数に上限がある | 防止（上限を十分に取る）・優雅な劣化（古いものを消す）・低優先度の強制置換 |
| **固定サイズ** | 異種オブジェクトはメモリを浪費する | サイズ別の個別プールを用意 |
| **再利用時の初期化** | 前回のメモリが残存する | 取得時に完全な再初期化を必須とする |

> 📎 出典: reference_object-pool-pattern.md, reference_10-game-design-patterns.md

## サービスロケーター (Service Locator) と依存性注入 (DI)

### 概要

ゲーム開発では、オーディオ・ロギング・セーブシステムなど「どこからでもアクセスしたいサービス」の提供方法が設計上の重要な判断となる。サービスロケーター (Service Locator) と依存性注入 (Dependency Injection, DI) はどちらもこの問題を解決するパターンだが、アプローチとトレードオフが異なる。

### シングルトンの問題

シングルトンは最も単純なグローバルアクセス手段だが、以下の問題がある。

- テスタビリティの低下（モック差し替えが困難）
- 初期化順序の暗黙的な依存
- サービスクラスの内部設計に影響（クラス自身がシングルトンであることを意識する必要がある）

### サービスロケーター

具体的な実装を知らずにサービスにアクセスできる一元化ポイント。「サービスを必要とするコードを、それが**誰**であるか（具体的実装型）と**どこ**にあるかの両方から分離する」。

#### シングルトンとの根本的な違い

両方がグローバルアクセスを提供するが、サービスロケーターでは具体的サービスクラスがロケーター経由でアクセスされていることを **知らない**。シングルトンはサービスクラスの内部設計に影響するが、サービスロケーターはしない。

#### 3つの実装手法

| 手法 | 説明 |
|------|------|
| **Simple Locator** | 登録済みサービスを返すだけ。最速、コンパイル時インライン化可能 |
| **Null Service** | 何もしない実装でnullポインタクラッシュを防止。未登録時の優雅な劣化 |
| **Decorated Service** | ラッピング（LoggedAudio等）でコア実装を変更せずにランタイム行動を修正 |

#### サービス配置方法

| 方法 | 利点 | 欠点 |
|------|------|------|
| **外部登録** | 高速、柔軟 | 初期化順序の規律が必要 |
| **コンパイル時バインド** | 可用性保証 | 柔軟性なし |
| **ランタイム設定** | 最大の柔軟性 | パフォーマンスオーバーヘッド |

#### 使用すべきケース（慎重に）

- モジュールのパブリックAPIの一部にすべきでないシステム（ロギング、メモリ管理）
- 根本的に唯一のファシリティ（オーディオデバイス、ディスプレイ）
- オブジェクトの多層受け渡しが不要な複雑さを生む場合
- **デフォルトはオブジェクトの直接受け渡し**（サービスロケーターは最後の手段）

### 依存性注入 (DI)

クラスが自身の依存関係を作成/探索する代わりに、外部から提供されるパターン。`new SaveService()` や `GameManager.Instance` の代わりに、コンストラクタパラメータで必要なものを受け取る。

#### なぜゲームに重要か

内部レイヤーを外部レイヤーから独立させるアーキテクチャモデルを実現する。例えば、Core層が `IPersistence` インターフェースを定義し、DIがローカルファイル/クラウド/テストダブルに接続する。3つの利益: **レイヤー独立性**、**テスタビリティ**、**コード変更なしの柔軟性**。

#### 危険な罠: シングルトンの偽装

「DIがシングルトンを修正する、ゆえにDIが問題を解決する」は危険な誤解。`GameManager.Instance` に依存する40クラスを `IGameManager` のインジェクションに変換しても、**依存グラフは同一**。アクセス方法を変えただけでは本質的な改善にならない。

**「注射器 (Syringe) は薬ではない。アーキテクチャモデルが薬。DIは単なるデリバリーメカニズム」**

#### DIを有効に機能させるための原則

1. **神インターフェースを小さな焦点契約に分割**: `IGameManager` → `IPlayerStats`, `IQuestLog`, `IAudioService`
2. **レイヤー境界を尊重**: Core層はCoreインターフェースのみ解決
3. **依存表面を最小化**: 5つ以上の依存関係を持つクラスは責任が多すぎる可能性

### サービスロケーター vs DI: 比較

| 観点 | サービスロケーター | 依存性注入 |
|------|-----------------|-----------|
| **依存関係の明示性** | 暗黙的（ロケーター経由で何でも取得可能） | 明示的（コンストラクタで宣言） |
| **テスタビリティ** | ロケーターのモック化が必要 | インターフェース差し替えが容易 |
| **セットアップコスト** | 低い | DIコンテナの設定が必要 |
| **実行時パフォーマンス** | 直接アクセスに近い | リフレクション等のオーバーヘッドがありうる |
| **適用規模** | 小〜中規模プロジェクト | 中〜大規模プロジェクト |

### よくあるアンチパターン

| アンチパターン | 問題 |
|-------------|------|
| **サービスロケーターの乱用** | コンテナ内で任意のクラスが何でも解決 → 設計臭 |
| **ライフサイクル管理ミス** | 依存関係が意図されたスコープを超えて存続 |
| **過剰インジェクション** | 設計問題を多すぎる依存関係で隠蔽 |
| **グローバル状態としてのコンテナ** | コンテナ自体がシングルトン化し、元の問題に回帰 |

> 📎 出典: reference_service-locator-pattern.md, reference_dependency-injection-games.md

## セーブシステム (Save System)

### 概要

セーブシステムはゲームの永続化基盤であり、プレイヤーの進捗データを安全に保存・復元する。プロジェクト**開始時から**設計に組み込むべきシステムで、中盤以降の後付けは初期化フローとゲームロジック全体のリファクタリングを引き起こす。

### アーキテクチャ

セーブシステムは層構造で設計する。

#### 4層アーキテクチャ

| 層 | 責務 |
|----|------|
| **プレゼンテーション層** | セーブUI・プレイヤーインタラクション |
| **アプリケーション層** | セーブ操作のオーケストレーション・ビジネスロジック |
| **ドメイン層** | セーブデータ構造・バリデーションルール |
| **インフラ層** | ファイル操作・シリアライゼーション・プラットフォーム固有ストレージ |

#### 3層アーキテクチャ（実装視点）

| 層 | 責務 |
|----|------|
| **永続システム (Persistent Systems)** | セーブを必要とするゲームモジュール |
| **セーブゲームマネージャー** | 中央調整。各システムを登録し個別のバケットを付与 |
| **ストレージ層** | 実際の永続化処理（ファイル/クラウド） |

### ISaveable インターフェースパターン

`CaptureState()` / `RestoreState()` / `GetSaveId()` を実装するコンポーネントが、SaveManagerのコード変更なしに自動的にセーブシステムに統合される。ゲーム固有ロジックからセーブインフラを分離する鍵。

### 設計目標と優先順位

1. **堅牢性 > 新しさ**: 「5分前の正しいセーブが30秒前の破損セーブより良い」
2. **使いやすいAPI**: 各システム開発者がセーブ対応を容易に追加できること
3. **パフォーマンス目標**: CPU・帯域・メモリの制約を明確に設定
4. **後方互換性**: 古いセーブデータでも正しくロードできること

### シリアライゼーション方式の比較

| 形式 | ファイルサイズ | パフォーマンス | 最適な用途 |
|------|-------------|-------------|----------|
| **JSON** | 中 | 良好 | 開発ビルド、デバッグ、Web統合 |
| **Binary** | 小（30-40%減） | 優秀 | プロダクションビルド、モバイル |
| **XML** | 大 | 普通 | 外部ツール互換性 |

### バージョニング戦略

セーブデータのフォーマットは開発中に変化するため、バージョニングが不可欠。

- **すべてをバージョン管理**: マジックナンバーではなく名前付きマクロ（例: `FIRST_SAVE_GAME_VERSION_NUM`）で変更を自動文書化
- ロード時に全システムの存在とバージョン番号の最低要件を検証
- 各バージョン遷移にマイグレーション関数を実装
- 暗黙的データ依存を避ける: ビットパック整数より個別のboolフラグ
- 実際のプレイヤーデータでマイグレーションパスをテスト
- 開発者セーブはデータスキーマ変更時に自動ワイプ。完全な後方互換性はリリース前まで遅延可

### セーブトリガー戦略

#### オートセーブ

複数トリガーを同時実装する:
- **時間ベース**: 30-60秒間隔の定期セーブ
- **イベントベース**: レベル完了・アイテム取得でトリガー
- **ライフサイクルベース**: 一時停止/終了時

#### ダーティフラグ (Dirty Flag) パターン

システムが自身を「ダーティ」とマークする2段階:
- **即時**: 重要変更で即座にセーブ
- **次バッチ**: 通常変更でスロットル化されたセーブ

マネージャーがストレージを叩きすぎずに定期的ロジックを実行。ゲーム終了時には**フラッシュ関数**で強制セーブ。

### よくある落とし穴

| 落とし穴 | 問題 | 対策 |
|---------|------|------|
| **ロード完了前のセーブ** | 未初期化システムが空バケットをシリアライズ → データ全消去 | 永続システムまたはマネージャー側のアサーションで保護 |
| **遅延統合** | 既存システムへのセーブの後付けはリファクタリングを誘発 | プロジェクト開始時から実装する |
| **破損セーブの上書き** | 書き込み中のクラッシュでセーブデータが失われる | 一時ファイルに書き込み後リネーム、複数スロットのローテーション |

### モバイル向け考慮事項

- アプリバックグラウンド移行時のセーブが必須
- 積極的な圧縮でファイルサイズ70-80%削減
- JSONよりBinaryでCPU使用量を低減
- デルタ同期で変更データのみクラウドアップロード

### 高度な概念

- **空間的永続性 (Spatial Persistence)**: プレイヤーが離れたエリアの状態を保存・復元する仕組み
- **セッション永続性 (Session Persistence)**: ゲーム再起動を跨いだデータの維持
- マルチプレイヤーシステムは読み書きアクセス要件を宣言し、意図しないデータ操作を防止

> 📎 出典: reference_save-system-architecture.md, reference_save-game-implementation-practical.md

## コンポジション (Composition over Inheritance)

### 概要

「継承より合成 (Composition over Inheritance)」は、オブジェクトの振る舞いを継承ツリーではなく、小さな再利用可能なパーツの組み合わせで表現する設計原則。ECSのコンポーネントモデル、能力システム、パワーアップシステムなど、ゲーム開発の多くの場面で中核的な役割を果たす。

### 継承の問題: ダイヤモンド問題と組み合わせ爆発

`Enemy → FlyingEnemy → FireFlyingEnemy` のような継承ツリーは、新しい組み合わせ（氷を吐く飛行敵、地上の火を吐く敵...）のたびにクラスが指数的に増加する。

### コンポジションによる解決

振る舞いを小さなコンポーネントに分解し、必要なものを組み合わせる。

- エンティティが一部の振る舞いのみ共有する場合に特に有効
- ECSのComponent設計と直接対応する
- データ駆動で新しい組み合わせを追加可能

### 実践事例: 能力システム設計

カードゲームの事例では、200種類以上の能力効果をコンポジションで実現している。

#### Targeting パイプライン（Head + Pipe 構造）

- **Head** (8種類): 対象選択の起点（自分、敵全体、隣接マスなど）
- **Pipe** (18種類): フィルタチェーン（例: 「敵の前列HP3以下」）

#### Effect (12種類)

ダメージ、回復、移動、能力付与など、個別の効果を表す。

#### Calculator

加算・乗算・固定値等による数値補正パーツ。

#### Trigger

発動条件を宣言的に指定するパーツ。

すべてJSON化可能で、**アプリアップデート不要のコンテンツ追加** を実現。コード変更なしに新しいカード・能力を追加できる宣言的データ駆動設計の好例。

### 関連パターンとの組み合わせ

- **Component + Factory** = データ駆動エンティティ生成
- **Composition + Strategy** = 柔軟な条件表現（Head + Pipeパイプライン）
- **Composition + Observer** = パーツ単位のイベント購読

## コンポーネントパターン (Component Pattern)

### 概要

コンポーネントパターンは、複数ドメイン（物理・グラフィックス・AI・サウンド・入力処理など）にまたがるゲームエンティティを、ドメイン境界に沿ってスライスすることでモノリシッククラスを分解するパターン。Robert Nystrom の『Game Programming Patterns』で体系化された。ECS のコンポーネント概念の直接的な起源であり、「コンポジション > 継承」原則の具体的な実装パターンに該当する。

### モノリシッククラスの問題

ゲームエンティティの振る舞いを1つの巨大クラスに詰め込むと、以下の問題が発生する:

- クラスが数千行に膨張しナビゲーションが困難になる
- 1ドメインの変更が他ドメインに予期せず波及する
- 複数チームメンバーが同一ファイルを編集しコンフリクトが頻発する
- スレッド境界をクリーンに引けない

### 基本構造

コンテナオブジェクト（GameObject）がコンポーネントへの参照を保持し、各フレームで `update()` を呼び出す。典型的な更新順序は `InputComponent → PhysicsComponent → GraphicsComponent`。

コンポーネントを抽象基底クラス（インターフェース）の背後に隠すことで、異なる実装の差し替えが可能となる。例えばデモモードでは `PlayerInputComponent` を `DemoInputComponent` にスワップでき、物理・レンダリングシステムは違いを認知しない。

### コンポーネント間通信の3戦略

| 戦略 | 仕組み | 利点 | 欠点 |
|------|--------|------|------|
| **共有ステート (Shared State)** | コンテナの公開プロパティを変更 | シンプル | 暗黙的順序依存、未使用属性のメモリ浪費 |
| **直接参照 (Direct Reference)** | コンポーネント同士がポインタを保持 | 高速・シンプル | 特定ペア間の結合を再導入 |
| **メッセージパッシング (Message Passing)** | コンテナ経由で全コンポーネントにブロードキャスト | デカップリング維持 | 複雑さ増加、フィードバックループの危険 |

### コンポーネントのインスタンス化

| アプローチ | メリット | デメリット |
|-----------|---------|----------|
| **自己生成** | 必須コンポーネントの保証 | 再構成の柔軟性が低下 |
| **外部提供** | ランタイム行動変更可能。ファクトリーパターンと相性が良い | コンテナの具象型への結合を削減 |

### 適用すべき場面

- クラスが分離したい複数ドメインに触れている
- クラスが巨大になり作業困難
- 継承の複雑化なしに多様なオブジェクトが異なる機能サブセットを共有する必要がある

### 利点と欠点

| 利点 | 欠点 |
|------|------|
| ドメインをデカップルし独立開発を可能にする | 間接性とワイヤリングによる複雑さの増加 |
| コンポジションによるコード再利用 | エンティティごとに複数オブジェクトの初期化が必要 |
| ランタイムでのコンポーネントスワップによる行動変更 | ポインタ追跡のパフォーマンスコスト |
| マルチスレッドアーキテクチャとの自然な親和性 | シンプルなオブジェクトへの過剰エンジニアリングの危険 |

### 実世界の採用例

- **Unity**: GameObject がコンポーネントを中心に全面設計
- **XNA**: Game クラスが GameComponent インスタンスを所有
- **Delta3D**: GameActor と ActorComponent 基底クラス

ECS（Entity Component System）はこのパターンをさらに拡張し、コンテナオブジェクトの代わりに数値 ID を使用する形態である。

### ECS・コンポジションとの関係

コンポーネントパターンは、本ナレッジベースで扱う以下の概念と直接関連する:

- **ECS**: コンポーネントパターンのデータ指向進化形。コンテナオブジェクトを数値 ID に置き換え、システムがデータを横断処理する
- **コンポジション > 継承**: コンポーネントパターンはこの原則の最も代表的な実装形態。多重継承の複雑さをコンポーネントの組み合わせで解消する
- **オブザーバー / メッセージパッシング**: コンポーネント間通信の第3戦略としてメッセージパッシングが使われ、オブザーバーパターンと共通する設計思想を持つ

> 📎 出典: reference_component-pattern.md

## その他の重要パターン

### フライウェイトパターン (Flyweight)

類似オブジェクト数千個のメモリ膨張を解決する。大量のオブジェクトが共有できるデータ（テクスチャ、メッシュ、ステータステーブルなど）を一箇所に集約し、個々のインスタンスは固有データのみを保持する。森林の木々、草のブレード、地形タイルなどで有効。

### ファクトリメソッド (Factory Method)

散在するハードコードされた生成ロジックを一元化する。生成ロジックが複雑、またはランタイムデータに依存する場合に使用する。

- **Component + Factory** = データ駆動エンティティ生成（JSONやスクリプタブルオブジェクトから動的にオブジェクトを構成）
- 敵のウェーブ生成、アイテムドロップ、レベル構築などで活用

### ストラテジーパターン (Strategy)

ランタイムで交換が必要なアルゴリズムをカプセル化する。振る舞いをコアロジックの変更なしにスワップできる。

- AIの行動選択アルゴリズムの動的切り替え
- ダメージ計算方式の差し替え
- 対象選択ロジック（Head + PipeパイプラインはStrategyの応用）

### デコレーターパターン (Decorator)

修飾子やパワーアップによる指数的サブクラス化を防ぐ。動的なモディファイアのスタッキングを実現する。

- ダメージバフ/デバフの重ね掛け
- 武器エンチャント効果の累積
- サービスロケーターの Decorated Service（LoggedAudio等）も同じ原理

> 📎 出典: reference_10-game-design-patterns.md, reference_service-locator-pattern.md, reference_card-game-command-architecture.md

## パターンの組み合わせと実践ガイドライン

### パターン組み合わせマトリクス

複数のパターンを組み合わせることで、単独では実現しにくい高度な設計を構築できる。

| 組み合わせ | 用途 | 効果 |
|-----------|------|------|
| **Command + Queue** | 入力バッファリング | 格闘ゲームの先行入力、ネットワーク同期 |
| **Observer + Event Queue** | 非同期イベント処理 | 大規模プロジェクトの疎結合化 |
| **Component + Factory** | データ駆動エンティティ生成 | JSON/設定ファイルからの動的構成 |
| **State + Command** | フェーズ管理 + 操作記録 | ターン制ゲームのリプレイ・Undo |
| **Strategy + Factory** | モード切替 | 新モード追加が容易 |
| **Composition + Strategy** | 能力・スキルシステム | 再利用可能パーツで多様な効果 |
| **Command + Observer** | 操作通知 | コマンド実行時のUI・サウンド自動更新 |

### 実践事例: パターン統合アーキテクチャ

カードゲーム対戦機能の設計で、複数パターンがどのように統合されるかの実例。

| 課題 | 採用パターン | 効果 |
|------|------------|------|
| フェーズ管理 | **State** (33種類) | 独立クラスで可読性向上 |
| 同期・リプレイ | **Command** (29種類) | 履歴再実行で完全復現、署名検証でチート対策 |
| 能力発動通知 | **Observer** (40種Trigger) | UI疎結合 |
| 対象選択 | **Strategy** (Head+Pipe) | 柔軟な条件表現 |
| モード切替 | **Strategy + Factory** (7種Facilitator) | 新モード追加が容易 |
| 能力多様性 | **Composition** (200種類以上) | 再利用可能パーツで効果を合成 |

### ゴールデンルール

1. **パターンは問題を解決してから適用する**: 事前にパターンを適用せず、問題が発生してからリファクタリングで導入する
2. **出荷された乱雑なコードは、完璧だが未完成のアーキテクチャに勝る**: パターンはコードを増やすためではなく、より少ないコードでより多くを実現するためにある
3. **過剰な抽象化を避ける**: YAGNI (You Aren't Gonna Need It) の原則。今必要でないパターンは導入しない
4. **プロファイリングで検証する**: パフォーマンス関連のパターン（Object Pool, ECSのアーキタイプ設計等）は、実測データに基づいて判断する
5. **宣言的データ駆動を志向する**: 能力定義やエンティティ構成をデータ（JSON等）で表現することで、コード変更なしのコンテンツ追加を実現する

> 📎 出典: reference_10-game-design-patterns.md, reference_card-game-command-architecture.md

## 実践メモ

### パターン選択の判断基準

| 状況 | 推奨パターン | 理由 |
|------|------------|------|
| 「毎秒大量のオブジェクト生成・破棄」 | Object Pool | メモリ断片化・GCスパイク防止 |
| 「明確で排他的な振る舞いモード」 | State Machine | if-elseスパゲッティの排除 |
| 「1トリガーに複数システム反応」 | Observer / Event Queue | 疎結合化 |
| 「Undo/Redo・リプレイ・入力バッファ」 | Command | 操作のオブジェクト化 |
| 「継承ツリーの爆発」 | Composition (ECS/Component) | 柔軟な振る舞い合成 |
| 「グローバルサービスへのアクセス」 | Service Locator / DI | 結合度低減・テスタビリティ |
| 「データの永続化」 | Save System (ISaveable) | 層分離・バージョニング |
| 「物理の安定性・決定論」 | 固定タイムステップ + 補間 | フレームレート非依存 |
| 「大量の共有データ」 | Flyweight | メモリ効率 |
| 「ランタイムの振る舞い切替」 | Strategy | アルゴリズムの差し替え |
| 「動的なモディファイア重ね掛け」 | Decorator | サブクラス爆発の防止 |
| 「生成ロジックの一元化」 | Factory Method | ハードコード排除 |

### プロジェクト初期に検討すべきこと

- **セーブシステムは最初から組み込む**: 後付けは全体のリファクタリングを引き起こす
- **ゲームループのタイムステップ方式を早期に決定**: 固定更新+可変描画が推奨
- **メッセージング基盤の検討**: 大規模プロジェクトではEvent Queue/Message Busの早期導入が効果的
- **DIかService Locatorかの方針統一**: プロジェクト内で混在させない

### 注意すべきアンチパターン

- **シングルトンの乱用**: グローバルアクセスの便利さに依存し、テスト不可能なコードが増殖する
- **過剰なパターン適用**: YAGNI原則に反し、不要な抽象化レイヤーを積む
- **DIによるシングルトンの偽装**: アクセス方法を変えただけで依存グラフが改善されていない
- **状態爆発**: ステートマシンの状態数が増えすぎたら、階層化FSMやビヘイビアツリーを検討する
- **固定容量プールの枯渇**: 上限値の見積もりとフォールバック戦略を事前に定める
