## 問題提起

多くの開発者が入力処理を後回しにし、キーバインドをゲームプレイロジックに直接ハードコード → 3つの重大問題:

1. **アクセシビリティ障壁**: 非標準キーボードレイアウト（Colemak、分割キーボード）で操作不能
2. **フレーム依存バグ**: 入力ステート（up/pressed/down/released）の混同で意図しないインタラクション
3. **劣悪なプレイヤー体験**: カスタマイズ不能がフラストレーションとゲーム評価低下を引き起こす

### 実例: Cyberpunk 2077

戦闘中にストレイフ → 照準が死体上を通過 → ロックされたキーバインドがリバインドと競合 → 意図しないピックアップアニメーション発生。

## 推奨アーキテクチャ

### 第1層: InputState 抽象化

離散ステート列挙でフレームタイミングバグを排除:

| ステート | 意味 |
|---------|------|
| Up | 押されていない（持続） |
| Pressed | 今フレームで押された（一過性） |
| Down | 押し続け中（持続） |
| Released | 今フレームで離された（一過性） |

### 第2層: アクション-バインディングマッピング

ハードウェア非依存のゲーム固有アクションを定義:

```
GameBinding: Jump, Sprint, Left, Right, ...
```

2つの並列データ構造を維持:
- `input_map`: 物理キー → アクション enum
- `input_state`: 各アクションの現在ステート

→ ゲームプレイコード修正なしにリバインド可能。

### 第3層: 設定ファイルパターン

#### デフォルト設定の自動生成

```ini
[bindings]
Jump = SPACE
Sprint = LEFT_SHIFT
Left = A
Right = D
```

#### 無効状態時の自動回復

config.ini が存在しない/破損 → デフォルトをプログラム的に生成 → ディスク書込 → 即座にパース適用。

## フレームごとの更新パターン

```
各キーについて:
  IsKeyPressed → Pressed
  IsKeyDown → Down
  IsKeyReleased → Released
  else → Up
```

フレームワーク固有の入力API（Raylib/SDL3/GLFW）を正規化。

## ゲームプレイ実装

```
if input_state[.Jump] == .Pressed && player.is_grounded {
    // ジャンプロジック
}
```

→ ハードコードされたキー定数がゲームロジックから**完全に排除**。

| 指標 | 値 |
|------|-----|
| 実装時間 | 15分未満 |

| フレームワーク依存 | なし |
| GUI不要 | プレイヤーが.iniファイル直接編集可能 |

## 主要な利点

1. **アクセシビリティ**: 非標準レイアウト即座対応
2. **コード明確性**: 意味のあるアクション名を参照
3. **バグ防止**: ステートマシンがフレームタイミング混乱を排除
4. **拡張性**: 新アクション追加が最小変更で可能
5. **オフライン設定可能**: GUIメニューなしで設定変更可能
