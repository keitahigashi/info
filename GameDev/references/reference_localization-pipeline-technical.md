## ロケールエンコーディング

単純な言語列挙を超え、2桁コードの組合せでロケール識別:

- **言語コード**: ISO 639-2 準拠
- **地域コード**: ISO 3166 準拠
- 例: `en-US`（英語・米国）、`fr-CA`（フランス語・カナダ）

→ 正確な地域バリエーション対応が可能。

## パイプラインアーキテクチャ

### 1. Excel ベース文字列管理

- A列: 識別子文字列（ID）
- 以降の列: 各ロケールの翻訳
- シート上部にエンコーディングID
- 翻訳者/デザイナーがスプレッドシート上で直接作業

### 2. エクスポートツール処理

| 機能 | 説明 |
|------|------|
| バイナリ圧縮変換 | Excelデータ → ランタイム用バイナリ（`en-US.lang`等） |
| 欠落文字列検出 | 未翻訳エントリの自動検出 |
| 最適フォント構築 | 必要な文字のみを使用したフォント生成 |
| フォーマットエラー検出 | 不正文字・書式エラーの特定 |

### 3. ランタイムロード

プレイヤーのロケール選択に基づき、必要な言語ファイルのみをメモリにロード。

## 文字列実装パターン

コード内にリテラルテキストを埋め込まず、文字列識別子を渡す:

```
DrawString("Hello_Message");  // リテラルではなくID
```

→ 表示とコンテンツの分離 → 翻訳ワークフローと保守が容易に。

## アイコン統合

括弧記法でアイコンを文字列内に埋め込み: `Press [START] to continue.`

### プラットフォーム非依存アプローチ

- システム固有ボタン名を禁止
- ゲーム記述的識別子を使用: `[TARGETING]`, `[FIRE]`
- プラットフォームごとに別個のアイコンマッピングファイル
- → 冗長な文字列重複を排除

## コンテキストパラメータ

翻訳者向けに記述的コンテキストを提供:

```
"Use {s-Name} to continue"  // s-Name = キャラクター名
```

→ パラメータの役割を明確化し、翻訳ミスを防止。

## テキストエンコーディング

| 方式 | 適用場面 |
|------|---------|
| UTF-8 | メモリ制約のある環境 |
| UTF-16 | メモリ制限が厳しくないプロジェクト |

## QA考慮事項

| チェック項目 | 内容 |
|-------------|------|
| 翻訳長変動 | ソースから通常20%の変動を監視 |
| パラメータ順序保持 | 言語間でパラメータ順が変わる場合の検証 |
| アイコンレンダリング | プラットフォーム固有アイコン表示の検証 |
| 日付/数値フォーマット | 地域ごとのフォーマット検証 |
| 文字エンコーディング | 全サポートロケールでの検証 |

## ベストプラクティス

- 翻訳ベンダーとの明確なコミュニケーションチャネル
- 編集中のファイルロック（バージョン管理衝突防止）
- コンテンツとコード表示層の分離
- 初期ロケールサポート選定時に拡張を見越した設計
