## Observerパターンの定義

あるオブジェクト（Subject）が、具体的な受信者を意識せずに「何か興味深いことが起きた」と通知する仕組み。一片のコードが別のコードを気にかけることなく、イベント発生を伝達する。

## ゲームでの具体例: アチーブメントシステム

### 問題設定
- 達成条件がゲームの異なるシステム（物理、戦闘、UI等）で発生する
- 各システムにアチーブメント機能を直接組み込むと過度な結合が生じる

### 解決
- 物理エンジンは「何かが落ちた」と通知するだけ
- アチーブメントシステムがそれを受け取って条件判定する
- 物理エンジンはアチーブメントの存在を知らない

## 実装構造

### Observer インターフェース
- 仮想メソッド `onNotify(Entity, Event)` を定義
- パラメータは用途に応じてカスタマイズ可能

### Subject クラス
- オブザーバーのリスト（配列やリンクリスト）を保持
- `addObserver()` / `removeObserver()` で登録管理
- `notify()` で全オブザーバーに通知（protectedメソッド）

### 通知の流れ
```
Physics(Subject) --notify()--> [Observer1, Observer2, ...]
                                  |            |
                          Achievements    AudioSystem
```

## メモリ・パフォーマンス最適化

### 侵襲的リンクリスト (Intrusive Linked List)
- 各Observer自身が「次のノードへのポインタ」を保持
- Subject側での配列動的確保が不要
- 登録/削除時のメモリアロケーションを完全排除
- **制約**: 1つのObserverは1つのSubjectのみ観察可能
- **制約緩和策**: オブジェクトプールにリストノードを事前割当

### パフォーマンス特性
- 通知送信 = メソッド呼び出しのリスト走査（高速）
- 動的割当は登録時のみ、送信時はゼロオーバーヘッド

## 重大な問題点

### 1. Lapsed Listener (メモリリーク)
- Observer削除時にSubjectのリストから外さないと、オブザーバーが生存し続ける
- 非表示UIでも通知を受け続け、CPUを無駄に消費
- **対策**: デストラクタで自動的に全Subjectから登録解除

### 2. 破棄時の管理
- Subject削除時 → Observerが無効な参照を保持
- Observer削除時 → Subjectが無効ポインタを参照
- **対策**: 最終通知の送出、または逆方向の参照リスト保持

### 3. 可視性の低下
- 静的にコードを読むだけでは通知フローを追跡困難
- 実行時の動的な接続関係がデバッグを複雑化
- **指針**: 送信側と受信側を同時に理解する必要があれば、Observerの使用を避ける

## 設計判断

### 適した場面
- ほぼ無関係なモジュール間の通信（物理 <-> アチーブメント）
- 複数の無関心なシステムが同一イベントに応答する場合

### 避けるべき場面
- 通信フローの追跡が開発に必須な場合
- 同一機能内での密接な相互作用

## 言語別実装の違い

| アプローチ | 言語 | 特徴 |
|-----------|------|------|
| インターフェースベース | C++, Java | Observer IF実装を強制 |
| 関数ベース（デリゲート） | C#, JavaScript | 関数参照で柔軟に登録 |
| クロージャベース | JS, Python | 無名関数で状態キャプチャ |

## 発展パターン

- **データバインディング**: UI要素と状態変化の自動同期
- **関数型リアクティブプログラミング (FRP)**: 宣言的なデータフロー定義
- ただし実装複雑性が高く、ゲームでは非クリティカルな領域に限定されやすい

## 関連パターン

- Event Queue: 非同期通知のキューイング
- Message-Driven Architecture: より疎結合な通信
- Mediator: 双方向通信の仲介
