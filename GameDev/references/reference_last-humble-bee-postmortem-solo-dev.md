## ソロ開発で正気を保つ方法

著者: Jacob Weersing (2024年12月5日)
ゲーム: The Last Humble Bee（2024年11月21日 Steam発売、ハック&スラッシュ型サバイバル）

### 背景

- フルタイム職を持ちながら2年間開発（前作 Bass Monkey の経験を活かした2作目）
- 当初予定: 9ヶ月 → 実際: 27ヶ月（3倍）
- リリース時 ウィッシュリスト約3,000件（前作500件から6倍成長）

### 3つの精神的教訓

#### 1. 誰もあなたのゲームを気にしない（それでも大丈夫）

最初のゲームが大成功する期待は非現実的である。

事例による裏付け:
- Scott Cawthon (FNAF作者): 成功作まで**70作品**制作
- Jake Birkett (Grey Alien Games): 「**11年間無一文**だった」

→ **設計上の示唆**: 成功の基準を「完成させること」と「前作より改善すること」に設定し、市場の反応に一喜一憂しない習慣をつける。

#### 2. 喜びは成果にあるのではなく、仕事そのものにある

Csikszentmihalyi のフロー理論を引用: 目標達成時の幸福感は短命であり、実際の幸福は**作業の過程**で生じる。

→ **実践的含意**: スコープ拡大・完璧主義・比較による不安よりも、「今日のタスクに集中する」サイクルを優先する。

#### 3. 始めたことを完成させる価値がある

27ヶ月という3倍の工数をかけても完成させた価値は:
- 技術的スキルの蓄積（次作への基盤）
- ポートフォリオとしての実績
- ウィッシュリスト・コミュニティの獲得

**参考文献として推奨**:
- Rick Rubin 著『The Creative Act』
- Austin Kleon 著『Keep Going』

### 技術・制作上の教訓

| 領域 | 教訓 |
|------|------|
| エンジン | GameMaker で開発したが次作では Unity への移行を検討（スケール対応） |
| アート | itch.io の資産パック購入で開発時間を大幅短縮。ソロ開発では「時間効率」を最優先 |
| スコープ | 当初9ヶ月予定が27ヶ月に。機能追加を継続的に断る意思決定が必要 |

### マーケティング戦略の転換

**旧戦略**: 広範なアウトリーチ（無効だと判明）

**新戦略**: 「be so good they can't ignore you」原則に基づく品質集中

実施した施策（絞り込みリスト）:
1. プレスキット作成
2. ゲームトレーラー制作
3. Steamページ最適化
4. ゲームフェスティバル参加
5. メールリスト構築

→ **教訓**: ソロ開発者のマーケティングリソースは有限。「全方位」より「精度の高い少数施策」を選択する。

### スケジュール管理の示唆

- 個人プロジェクトの工数見積もりは**3倍バッファ**を前提に設定する
- マイルストーンをリリース日固定でなく「機能セット完成」ベースで設定することでスコープ調整を可能にする
