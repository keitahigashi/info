## 5つのクラフトシステム類型

### Type 1: 通貨の別名 (Money by Another Name)
収集リソースを固定・透明な交換で所望アイテムに変換。

| 項目 | 内容 |
|------|------|
| **メカニクス** | ワークベンチ/NPCでの固定レート交換 |
| **利点** | 即座に理解可能、既存ショップUIで実装容易、テーマ的風味付け、バランスツール |
| **欠点** | 没入感の制限、リソース通貨が多すぎると退屈、使い過ぎると余計に感じる |
| **代表作** | Dragon Age: Origins, Skyrim (鍛冶), Terraria, BioShock |
| **適用** | 既存マーケットへの補足的風味付け |

### Type 2: レシピ発見 (Find the Recipe)
レシピ/設計図/ブループリントを取得後にクラフトが可能に。

| 項目 | 内容 |
|------|------|
| **メカニクス** | 探索・報酬・購入でレシピ発見 → クラフト解禁 |
| **利点** | ナラティブ風味、報酬多様性の拡大、コンプリート欲、コンテンツゲーティング |
| **欠点** | レア度とアイテム価値のバランス調整が必要、過剰ブループリントは退屈 |
| **代表作** | WoW, Diablo III, Fallout 3/NV, Witcher 2 |
| **適用** | プログレッションベースのコンテンツゲーティング |

### Type 3: 試行錯誤 (Guess and See)
事前定義なしのプレイヤー主導レシピ発見。

| 項目 | 内容 |
|------|------|
| **メカニクス** | 組み合わせ実験で発見。複雑さは素材選択から配置まで多様 |
| **利点** | 真の発見体験、メカニクスへの直接的エンゲージメント、実験方法論の近似 |
| **欠点** | 失敗の懲罰が探索を阻害、任意の組み合わせにフラストレーション、外部ガイドに頼りがち |
| **代表作** | Minecraft, Skyrim (錬金術), Diablo II (Horadric Cube) |
| **適用** | 錬金術・実験テーマのメカニクス |

### Type 4: カスタマイズオーダー (Made-to-Order)
既知オプションからのモジュラー選択で戦略的アイテム構築。

| 項目 | 内容 |
|------|------|
| **メカニクス** | 素材・デザイン・美的オプションからの動的カスタマイズ |
| **利点** | 深いカスタマイズ、プレイヤーエージェンシー、戦略的ゲームプレイレイヤー |
| **欠点** | 高い実装複雑性、プレイヤーの高い注意力要求、最適化による選択肢狭小化 |
| **代表作** | AC: Revelations, Dead Space 3, Monster Hunter Tri |
| **適用** | カスタマイズが戦略を駆動するゲーム |

### Type 5: 無限の可能性 (Anything is Possible)
膨大な入力変数が広大なアウトカム配列を生む真のクラフト。

| 項目 | 内容 |
|------|------|
| **メカニクス** | 数千の順列を持つ動的プロセス |
| **利点** | 調査に極めてやりがい、意味あるエージェンシー、高いリプレイ性 |
| **欠点** | 開発・バランスが極めて困難、ゲームブレイキング組み合わせのリスク |
| **代表作** | Vagrant Story, Legend of Mana, Atelier シリーズ |
| **適用** | クラフトが主要フィーチャーであるゲーム |

## 設計原則

クラフトはコアゲームデザイン目標を支えるべき。明確な目的なしの実装は避ける。不必要な複雑さは深みではなく退屈を生む。
