Game Programming Patterns (Robert Nystrom): https://gameprogrammingpatterns.com/object-pool.html

## パターン定義

「再利用可能なオブジェクトのコレクション」を維持し、各オブジェクトが「使用中 (in use)」クエリを支援する。起動時に大きな連続メモリブロックを確保し、ゲームプレイ中のヒープ割り当てをゼロにする。

## 動機: 2つの問題

### メモリ断片化 (Memory Fragmentation)

空きヒープ空間が小さな断片に散在。「14バイト空きがあるが、7バイト2つに断片化」→ 12バイトのオブジェクトを確保できない。コンソールでは「ソークテスト」（数日間の連続実行）で「忍び寄る断片化やメモリリーク」によるクラッシュが発生。

### 割り当て性能 (Allocation Performance)

パーティクルなど多数のオブジェクトの生成・破棄にオーバーヘッドが発生。

## 動作原理

1. オブジェクトを「未使用」に初期化
2. 要求時に利用可能スロットを「使用中」にマークし初期化
3. 解放時に「未使用」状態に戻す
4. ゲームプレイ中はヒープ割り当てなし

## フリーリスト最適化

未使用パーティクルのメモリ内に次の利用可能パーティクルへのポインタを union で格納。追加メモリオーバーヘッドなしで **O(1)** の生成を実現。

## 設計判断

| 判断ポイント | 選択肢A | 選択肢B |
|------------|---------|---------|
| **プールとの結合** | オブジェクトがプールを認識（シンプル、private コンストラクタ可） | 汎用プール（柔軟だが別途追跡配列必要） |
| **初期化責任** | プール管理（完全カプセル化） | 外部初期化（シンプルだが割り当て失敗チェック必要） |

## 制約

- **固定容量**: 同時存在オブジェクト数に上限。対策: 防止・優雅な劣化・低優先度の強制置換
- **固定サイズ**: 異種オブジェクトはメモリ浪費。対策: サイズ別の個別プール
- **再利用時の初期化**: 前回のメモリが残存。完全な再初期化が必須
