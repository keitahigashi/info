## データ駆動型 UI/UX 設計

### 中心的主張

「デザインは美的要素だけでなく、ユーザーデータという科学に基づいている」

設計者が常に最善を知っているわけではない。ユーザーデータを組み込まない設計判断は、推測に頼った危険な賭けになる。

### 事例：VIP Baloot のアラビア語化失敗と回復

- フランスとブルガリアで成功したゲームが、アラビア語版リリース時に初期失敗
- データドリブンUI/UXの適用後に回復
- 教訓：文化・地域差をユーザーデータなしに推測すると失敗する

### 5つのフレームワーク手法

#### 1. ユーザー調査 (User Research)
- アプリ分析・インタビュー・アンケートを通じた間接的なユーザー参加
- 設計チームの「推測」ではなく、実際のユーザーの声を意思決定に組み込む
- 特に文化・地域差のある市場展開では必須

#### 2. A/B および多変量テスト (A/B Testing)
- 複数デザイン案を同時に類似ユーザーグループで測定
- コンバージョン率など単一メトリクスで勝者を決定
- 多変量テストでは複数変数を同時評価し、相互作用効果も把握

#### 3. アウトサイドイン設計 (Outside-In Design)
- 組織側ではなくユーザー側の視点で意思決定を行う
- ビジネス要件よりユーザーニーズを優先した設計判断
- 「私たちが作りたいもの」から「ユーザーが必要とするもの」へ

#### 4. ユーザーストーリーの追跡 (User Story Tracking)
- ユーザー行動フロー全体を追跡し、UXの摩擦点を特定
- 優先度付きでUI再構築の課題リストを作成
- 一点的な改善ではなくフロー全体の最適化

#### 5. 定量・定性データによる検証 (Quantitative + Qualitative Validation)
- 統計情報（数値）とユーザーフィードバック（言語）の両方を活用
- 数値だけでは「何が起きているか」はわかっても「なぜか」はわからない
- 変更後は必ずユーザー調査を実施し、継続的な改善サイクルを構築

### 実践上の注意点

| フェーズ | 推奨アクション |
|---------|--------------|
| デザイン前 | ユーザー調査でターゲット像を確立 |
| デザイン中 | 複数案をA/Bテストで比較 |
| リリース後 | ユーザーストーリー追跡で摩擦点を特定 |
| 改修後 | 再度定量・定性データで検証 |

### 既存のUI/UX記事との補完関係

従来のUI/UX記事（視覚設計・HUDレイアウト・アクセシビリティ）が「何を作るか」を扱うのに対し、本記事は「どう検証・改善するか」のプロセスを提供する。
