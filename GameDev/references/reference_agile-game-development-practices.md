## 概要

ゲーム業界はワークフロー自動化、分散チーム管理、タイトなデッドラインなどの課題に直面。アジャイル手法は柔軟性、自己組織化、迅速なデリバリーを通じてこれらに対応する。

## 3つのコアフレームワーク

### 1. スクラム (Scrum)

ゲーム開発で最も広く採用されているアジャイルフレームワーク。

**チーム構造（3つの役割）:**

| 役割 | 責任 |
|------|------|
| **プロダクトオーナー (Product Owner)** | 何の作業を完了するか決定 |
| **スクラムマスター (Scrum Master)** | サーバントリーダーとしてチームと組織のスクラム効果を最大化 |
| **開発チーム (Development Team)** | プロダクトの実際の構築 |

**スプリントメカニクス:**
- 作業を短期間（スプリント）に分割。通常2〜4週間
- バックログ（プロダクトオーナーが優先順位付けしたタスクリスト）に基づく開発

**主要セレモニー:**
- **デイリースクラム**: 状況アップデート
- **スプリントレビュー**: 完了作業のデモ
- **レトロスペクティブ**: チームプロセスの検査と適応
- **スクラムボード**: タスクの整理と追跡

### 2. エクストリームプログラミング (XP)

迅速なリリースと変更への適応を優先。

**コアバリュー:** コミュニケーション、勇気、フィードバック、リスペクト、シンプリシティ

**5つの主要プラクティス:**

| プラクティス | 説明 |
|-------------|------|
| **テスト駆動開発 (TDD)** | テストを先に書き、それを満たす機能を実装 |
| **ペアプログラミング** | 2人の開発者が同じマシンで計画・議論・コーディング |
| **継続的設計** | 生産サイクル中にフィードバックを測定・学習・改善 |
| **持続可能なペース (Energized Work)** | チームが最適にパフォーマンスを発揮できる時間帯で開発 |
| **実顧客参加** | エンドユーザーがフルタイムチームメンバーとして参加 |

### 3. カンバン (Kanban)

タイムボックスなしの継続的デリバリーに焦点。

**コアコンセプト:** フローは作業が継続的に流れる様子を直接記述。スクラムの「タイムボックス」とは異なる。

**必須プラクティス:**
- **可視化**: カンバンボードで作業段階・コミットメントポイント・WIP制限を表示
- **フロー管理**: 生産段階を通じた継続的な移動
- **明示的ポリシー**: ワークフロー実行の明確なガイドライン
- **フィードバックループ**: 定期的なレビューと調整
- **WIP制限**: 同時作業アイテム数の制御

## フレームワーク選択の指針

| 特性 | Scrum | XP | Kanban |
|------|-------|-----|--------|
| イテレーション | タイムボックス（2-4週） | 短サイクル | 継続的フロー |
| 計画粒度 | スプリント単位 | リリース単位 | 作業項目単位 |
| 変更対応 | スプリント内は固定 | 柔軟 | 随時 |
| 最適な状況 | 複雑なプロジェクト | 技術的不確実性が高い | 運用・保守フェーズ |
