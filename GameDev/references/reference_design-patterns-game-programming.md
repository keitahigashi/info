## 4つのデザインパターン概要

### 1. シングルトン (Singleton)

単一インスタンスしか生成できないオブジェクト。ゲームマネージャ・オーディオシステム等のグローバルサービスに使われるが、著者は**「悪い設計を隠す傾向がある」**として警告している。

```
アンチパターンの兆候：
- テスト困難（グローバル状態の隠蔽）
- 依存関係が不透明になる
- 代替：サービスロケーターまたはDI（依存性注入）を検討
```

### 2. ファクトリ (Factory)

他のオブジェクト生成を専門とするクラス。複雑な初期化ロジックを一元管理する。

**ゲームでの典型的な使用例：**

| 使用場面 | 概要 |
|---------|------|
| 敵スポーン | 敵タイプに応じた初期化ロジックをFactoryが担当 |
| 弾丸・エフェクト | オブジェクトプールと組み合わせた生成管理 |
| セーブデータ復元 | シリアライズデータからオブジェクトグラフを再構築 |

ファクトリは**コンストラクタの知識をクライアントコードから隠蔽**し、生成ロジックの変更を局所化する。

### 3. オブザーバー (Observer)

オブジェクトの状態変化を購読者（リスナー）に通知する仕組み。ゲームのイベント駆動設計に最適。

```
Subject（被観察者）→ notify() → Observer（購読者）
例：プレイヤーHP変化 → UIヘルスバー更新 + ゲームオーバー判定
```

疎結合を実現するが、通知チェーンが深くなりすぎると**デバッグが困難**になる点に注意。

### 4. ステート (State)

オブジェクトの状態を専用クラスで表現し、条件分岐を排除する。

```
悪い例：if (state == IDLE) ... else if (state == RUNNING) ...
良い例：currentState.Update() → IdleState / RunState / AttackState がそれぞれ振る舞いを定義
```

状態遷移ロジックの局所化により、新状態追加が他状態のコードに影響しない。

## 選定指針

- **シングルトン** → 使用を避ける。代わりにサービスロケーターかDIを使う
- **ファクトリ** → 生成ロジックが複雑・変化しやすい場合に採用
- **オブザーバー** → UIとゲームロジックの分離、イベント通知に適用
- **ステート** → キャラクターAI・入力処理・ゲームフローに有効
