## 出典

Game Programming Patterns (Robert Nystrom): https://gameprogrammingpatterns.com/type-object.html

## 定義

型定義を言語の固定的なクラス階層から、**実行時に変更可能なオブジェクト**へと移行させる設計手法。

## 従来のOOPの問題点

- 新種族（ドラゴン、トロル等）追加ごとに新サブクラス作成が必要
- 数百種類の敵追加時に頻繁な再コンパイル
- デザイナーの数値調整に毎回プログラマ介入が必要

## コア設計

```
Breed（型オブジェクト）
├─ 開始時の体力
├─ 攻撃テキスト
└─ newMonster() ファクトリメソッド

Monster（型付きオブジェクト）
├─ 現在の体力（インスタンス固有）
└─ Breedへの参照
```

## データ駆動設計への移行

- 新種族を**データ（JSON等）として定義**（コンパイル不要）
- 非技術者のデザイナーも種族を作成・調整可能
- 実行時パフォーマンスとのトレードオフで開発生産性を向上

## 継承メカニズム

### 3オプション比較

| 継承形式 | 利点 | 欠点 |
|---------|------|------|
| **なし** | 最もシンプル | データ重複 |
| **シングル** | バランス | 属性検索時に遅延 |
| **マルチプル** | 重複なし | 複雑・保守困難 |

### Copy-down委譲
コンストラクション時に親から値をコピー。ルックアップ不要で高速だが、親変更時に子の自動更新なし。

## 設計判断ポイント

### 型オブジェクトのカプセル化
- **非公開**: 種族情報は実装詳細。条件付きで動的変更可能
- **公開**: ファクトリメソッド経由で外部から生成

### 型の変更可能性
- **固定型**: デバッグ容易、概念的に直感的
- **変更可能型**: オブジェクト再生成削減（例: 死体→ゾンビ変身）

## 技術的背景

> C++の仮想関数テーブル（vtable）は、タイプオブジェクトパターンの自動化版

コンパイラが通常処理する型システムを、プログラマが明示的に実装し直すもの。ボトルネックが「処理能力」から「コンテンツ制作量」へシフトした現代において重要性が増大。
