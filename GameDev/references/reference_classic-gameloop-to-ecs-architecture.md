## アーキテクチャ進化の3段階

### 第1段階：ゲームループとゲームオブジェクト

基本的な「状態更新・描画」サイクルからゲームエンジンが誕生した経緯。反復可能な構造として定式化されたゲームループが起点。

### 第2段階：コンポーネントシステムへの移行

継承ベース設計が招く問題：

- **機能組み合わせパターンの爆発** — クラス階層が深くなるほど、機能の組み合わせごとにサブクラスが必要になる
- 解決策：「継承より包含 (Composition over Inheritance)」— 機能をコンポーネントとして切り出し、エンティティに付け替える

### 第3段階：ECS（Entity Component System）アーキテクチャ

大量オブジェクト処理時の課題である「メモリ上のデータが散在する」問題への対応：

| 要素 | 役割 |
|------|------|
| Entity | 単なる整数ID |
| Component | 値型の配列（データのみ、ロジックなし） |
| System | ロジック関数（ComponentのArrayを処理） |

## ECSの設計原則

### データ構造：Struct of Arrays (SoA)

```
// Array of Structs（非推奨）
[{pos, vel, hp}, {pos, vel, hp}, ...]

// Struct of Arrays（推奨）
{positions: [...], velocities: [...], hps: [...]}
```

SoA構造により：
- **キャッシュ効率の向上** — 同種データが連続メモリに配置
- **並列処理の効率化** — SystemがComponent配列を独立して処理可能
- **シリアライズが容易** — 値型の配列として直接保存可能

## トレードオフと適用指針

### ECSが有効なケース
- 大量（数千〜数万）のエンティティをリアルタイム処理する場合
- Unity DOTS、Unreal Engine Mass Entityなど現代エンジンの実装で採用

### 注意点
- **親子関係処理** — ヒエラルキー構造を持つ場合は既存コンポーネントシステムとの併用を検討
- **学習コスト** — データとロジックの分離という思考転換が必要

## 汎用化ポイント

エンジン固有の実装（Unity DOTS等）から抽象化できる汎用パターン：

1. **データとロジックの分離** — パフォーマンスクリティカルな処理でComponent（データ）とSystem（処理）を分ける
2. **キャッシュ指向設計** — 処理するデータを連続メモリに配置することでCPUキャッシュ効率を最大化
3. **段階的移行** — 全体を即座にECSにする必要はなく、ホットパスのみ適用することも有効
