Game Programming Patterns (Robert Nystrom): https://gameprogrammingpatterns.com/game-loop.html

## 定義

ゲームループは「ユーザー入力をブロックせずに処理し、ゲーム状態を更新し、ゲームを描画する」連続サイクル。異なるハードウェア間で一貫したゲームプレイ速度を維持するために時間を追跡。

## 基本動作の3フェーズ

1. **入力処理 (Process Input)** — 入力を待たずにユーザーコマンドを処理
2. **更新 (Update)** — AI・物理を含むゲームシミュレーションを進行
3. **描画 (Render)** — 現在のゲーム状態をプレイヤーに表示

## 4つのタイムステップアプローチ

| アプローチ | 説明 | 問題 |
|-----------|------|------|
| **固定（同期なし）** | 最もシンプルだがハードウェアで速度が変化 | 現代マシンで速すぎ、旧マシンで遅い |
| **固定（同期あり）** | sleep/delay でFPSをキャップ | 電力効率的だが処理超過時にゲームプレイ減速 |
| **可変タイムステップ** | フレームごとの経過時間でスケール | 非決定論的物理、丸め誤差蓄積、不安定性 |
| **固定更新+可変描画（推奨）** | 固定ステップでゲームロジック、任意タイミングで描画 | 安定性とビジュアルの滑らかさを両立 |

## 更新/描画分離 (Update-Render Decoupling)

固定タイムステップ更新と可変レート描画の分離が、不安定な物理問題を解決しつつスムーズなビジュアルを維持。更新と描画間の「残留ラグ」は**補間 (interpolation)** で補償 — シミュレーションフレーム間の部分的進捗(0-1.0)をレンダラーに渡す。

## 電力管理

| プラットフォーム | 戦略 |
|----------------|------|
| デスクトップ | 最大速度で実行 |
| モバイル | 30-60 FPSにクランプし残りをスリープ |

## ループの所有者

- **プラットフォームのイベントループ**: シンプルだが制御が限定
- **ゲームエンジンのループ**: 実装負担なしだが柔軟性低下
- **カスタム実装**: 最大制御だがプラットフォーム調整が必要
