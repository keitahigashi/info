## 問題の規模

調査対象開発者の70%以上が「スコープが大きすぎる」をデッドラインの逸脱やプロジェクト放棄の主要因として特定。才能の不足ではなく、過剰なフィーチャーが原因で野心的なプロジェクトが失敗する。

## 主要な原因

### リソーススケーリングの誤解

フィーチャー追加は比例的な時間増加で済むと仮定しがち。実際は複雑さが**指数的に増加**。例: クラフティングメカニクスはレシピバランス、UIデザイン、ゲームプレイループとの統合を必要とする。

## 5つの予防戦略

| 戦略 | 説明 |
|------|------|
| **コアループ定義 (Define Core Loop)** | 最も楽しい1つのメカニクスを完成させてからセカンダリフィーチャーを追加 |
| **バーティカルスライス (Vertical Slice)** | 多数のフィーチャーに薄く広がるのではなく、1つの磨かれた完全なセクションを作成 |
| **MoSCoW優先度 (MoSCoW Prioritization)** | Must/Should/Could/Won't に分類。除外について容赦なく判断 |
| **タイムボックス (Timeboxing)** | フィーチャーごとに厳格な開発期限。超過時はカットまたは簡素化 |
| **早期プレイテスト (Early Playtesting)** | 最終段階ではなく開発全体を通じてフィードバック収集 |

## 成功モデル: Stardew Valley

ソロ開発者 Eric Barone がコアの「Harvest Moon」メカニクスに集中し、能力とプレイヤーフィードバックに基づいて段階的に拡張。フィーチャーコンプリートではなく**ポリッシュされた状態**でローンチ。

## 核心的哲学

フィーチャーをカットすることは自己認識とクオリティへのコミットメントを示す**強み**であり、弱みではない。
