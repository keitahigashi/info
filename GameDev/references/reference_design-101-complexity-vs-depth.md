## 複雑さ vs. 深さ フレームワーク

著者: Dan Felder (2015)

### 前提：複雑さの誤解

「複雑なゲームは奥が深い」という誤解が広く存在するが、複雑さは深さではない。むしろ**複雑さはコスト (budget)** であり、使うほどプレイヤーへの負担が増加する。

### 3種類の複雑さ

**1. 理解複雑さ (Comprehension Complexity)**
- 定義: デザイナーの意図をプレイヤーが理解するのがどれほど難しいか
- 例: 不明瞭なルール、分かりにくい UI、目標の説明不足
- 影響: **常に有害** — プレイヤーを混乱させ、ゲームを諦めさせる

**2. 追跡複雑さ (Tracking Complexity)**
- 定義: 複数の要素を同時に認知的に把握する難しさ
- 例: 10人の NPC の名前と関係性を記憶する必要があるゲーム
- 影響: **基本的に有害** — 認知負荷を高め、重要な意思決定から注意をそらす

**3. 深さ (Depth)**
- 定義: 最適な行動を決定することがどれほど難しいか（戦略的奥深さ）
- 例: チェスでの最善手の探索、TCGでのデッキ構築の判断
- 影響: **プレイヤーの楽しさの源泉** — 習熟するほど新しい可能性が見えてくる

### 「複雑さはバジェット」メタファー

```
設計目標の達成に必要なコスト = 理解複雑さ + 追跡複雑さ + 深さ

理想: 最小の理解・追跡複雑さで最大の深さを生み出す
```

### 意思決定の優先順位

1. 同等の深さを生む選択肢があれば**理解・追跡複雑さが低い方**を選ぶ
2. 同等であれば**開発コストが低い方**を選ぶ
3. 不確実な場合は**常にシンプルな案を採用**する（複雑さは後で追加できるが削除は困難）

### 実践的な検証質問

- このルールをプレイヤーは直感的に理解できるか？（理解複雑さ）
- プレイヤーは同時にいくつの要素を追跡する必要があるか？（追跡複雑さ）
- このメカニクスで「唯一最適の戦略」は存在するか、しないか？（深さの有無）

### 関連概念との接続

- **ゲームデザインにおける深さ**: 複雑さなしに深さを生む設計（例: チェス・囲碁）
- **学習曲線**: 理解複雑さが低いほど参入障壁が下がり、深さが長期リテンションを支える
