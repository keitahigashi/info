## 概要

ファイバー (fiber) とは、OSのプリエンプティブスケジューリングに依存せず、アプリケーションコードが自らスタックメモリ間をジャンプできる軽量な実行単位。「ファイバーとは、通常のリターン命令を使わずに別のスタックメモリへジャンプする仕組み」と定義される。

## スレッド・コルーチンとの比較

| 特性 | OS スレッド | C++20 コルーチン | ファイバー |
|------|------------|----------------|-----------|
| スケジューリング | OSが決定 | コンパイラが変換 | アプリが制御 |
| コンテキスト切り替え | カーネルモード遷移 | レジスタ保存のみ | レジスタ保存のみ |
| yield 可能深度 | N/A | コルーチン呼び出しのみ | **任意の呼び出しスタック深度** |
| RAII保証 | あり | あり | プログラマ責任 |

**ファイバーの優位点**: コルーチンと異なり、任意の呼び出しスタック深度で yield できる。これがゲームエンジンでファイバーが採用される主な理由。

## x64実装アーキテクチャ

ファイバーコンテキスト構造体は callee-saved レジスタのみ保存：

```cpp
// 最小コンテキスト構造（64バイト）
struct FiberContext {
    void* rbx, *rbp, *r12, *r13, *r14, *r15;  // callee-saved
    void* rsp;  // スタックポインタ
    void* rip;  // 命令ポインタ
};
```

caller-saved レジスタは呼び出し規約上、呼び出し側が保存するため不要。

## ゲームエンジンにおけるジョブシステム設計パターン

```
[物理コア固定スレッド群]
    ↓ 実行
[ファイバープール]
    ↓ タスク取得
[ジョブキュー]
    ←→ 依存関係解決時にyield/resume
```

- スレッドを物理コアに固定（スレッドアフィニティ）
- ファイバープールをタスク数以上用意
- ファイバーは `while(true){ DoTask(); SwitchFiber(...); }` で無限ループ
- ブロッキングI/Oは専用バックグラウンドスレッドに委譲
- タスクが依存する入力待ちの場合はブロックせずyield → 入力完了時にresume

## 実装上の落とし穴

1. ファイバーエントリ関数は通常のreturnで終わってはいけない（必ず別ファイバーにyieldする）
2. スマートポインタのRAII保証が壊れる（ファイバー破棄タイミングで未実行の場合）
3. コンパイラのTLSキャッシュ最適化がファイバースイッチをまたぐと破綻する
4. ブロッキング呼び出しはコアに固定されたスレッドを飢餓状態にする
5. デバッガから中断ファイバーが見えない（標準ツールの制限）

## クロスプラットフォーム考慮点

- Windows: `ConvertThreadToFiber` / `CreateFiber` / `SwitchToFiber` のネイティブAPIあり
- macOS: `ucontext` が非推奨のためカスタムアセンブリ実装が必要
- ARM64: x64と同原理、レジスタセットが異なる
