## 出典

Game Developer (Wolfgang Walk, 2015): https://www.gamedeveloper.com/design/from-mda-to-dde

## MDAの限界

2004年発表の MDA フレームワーク（Mechanics/Dynamics/Aesthetics）は11年以上更新されていない。現代ゲーム開発における2つの課題に対応できない：

1. **「メカニクス」定義の過度な広さ**: コード実装・UI・コンセプトデザインがすべて同じ「メカニクス」に分類されてしまう
2. **ナラティブの位置づけ不明**: ナラティブデザインを MDA のどこに実装するかが不明確

## DDE フレームワーク

| 層 | 構成要素 | 説明 |
|----|---------|------|
| **Design** | Blueprint | 世界観・ルール・美的指針の概念設計（ナラティブもここに統合） |
| | Mechanics | コード実装レベルの処理 |
| | Interface | プレイヤーが知覚する視覚・音響情報 |
| **Dynamics** | — | 設計要素の実行時の相互作用・振る舞い |
| **Experience** | 感覚的旅 | 視覚・聴覚への影響 |
| | 感情的旅 | 恐怖・喜び・悲しみなどの感情体験 |
| | 知的旅 | 判断・課題解決・学習の体験 |

## MDA vs DDE 比較

| 観点 | MDA | DDE |
|------|-----|-----|
| 用語精度 | 「メカニクス」が実装〜概念を全包括 | Blueprint / Mechanics / Interface に分離 |
| ナラティブ | 実装方法が不明確 | Blueprint に統合 |
| プレイヤー体験 | 「美学 (Aesthetics)」が曖昧 | 感覚・感情・知的の3旅として具体化 |
| 対立設計 | 言及なし | ゲームを「価値ある対抗者 (worthy adversary)」として概念化 |

## 具体例：グラフィックスタイルガイド

- MDA では「メカニクス」に分類 → データ表現レベルの定義が曖昧
- DDE では「Blueprint」内に明確に位置づけ → 設計段階の成果物として管理可能

## ナラティブ統合の実践

Blueprint にナラティブを統合することで：
- ゲームプレイメカニクスとストーリーの矛盾（ルドナラティブ・ディソナンス）を設計段階で検出できる
- 「この Dynamics がどの Experience（感情旅）を引き起こすか」を明示的に設計できる

## 活用場面

- 設計レビュー会議での共通言語として
- ナラティブデザイナーとゲームプログラマーの役割分担の明確化
- ゲームの体験目標を「3旅の組み合わせ」として定義する際のテンプレート
