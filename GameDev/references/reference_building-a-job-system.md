## 設計理念

単一スレッドのゲームループでは複数コアを活用不可。スレッド固定分割は負荷不均衡を招くため、ワーカースレッドがキューからジョブを動的に取得する方式を採用。

## スレッドプールの進化

### 初期問題
共有キューへの同時アクセス → 競合による深刻なボトルネック。

### 解決: ローカルジョブバッファ + ワークスティーリング
各ワーカーがローカルバッファを所有。別スレッドからの盗取機能で、同じバッファの異なる端から操作し競合を最小化。

## JobBuffer実装

### データ構造
- `head`（UnsafeCell）: ローカルスレッドのみアクセス
- `tail`（Mutex）: 盗取スレッドが保護
- `jobs`（Vec<Mutex<Option<Job>>>）: 個別ロック

### 最小限インターフェース
`push()` / `pop()` / `steal()` のみ公開。`peek()` は競合状態を招くため**意図的に除外**。

### リングバッファ
容量固定でメモリ再割り当てなし。満杯時はジョブを即座実行（ブロック回避、進捗保証）。

## ワーカースレッドのライフサイクル

### 初期化
- CPU親和性設定で各スレッドを特定コアに制限
- 盗取対象バッファの配列を環状配置

### 実行ループ
```
1. ローカルキューからpop
2. 空なら → 他スレッドからsteal
3. それでも空なら → yield
```

### 終了
終了フラグ設定後、全スレッドが`join()`されるまで待機。

## JobFuture（非同期結果取得）

ジョブの戻り値を`SettableJobFuture`に格納。待機側は`JobFuture`で値を取得。

**重要パターン**: `wait()` 時にスピンロック + ペンディングジョブ実行を組み合わせ。待機スレッドをブロックせずジョブシステムの進行を継続。

## Rust所有権への対策

| 課題 | 解決策 |
|------|--------|
| ミュータブル値 | ジョブに所有権移譲 → 戻り値で返却（単一所有者の不変式維持） |
| イミュータブル参照 | 独自`Ref<T>`構造体（UnsafeCell + AtomicUsize参照カウント） |
| 関数ポインタの異種保存 | `Box<dyn FnOnce()>` でカプセル化 |

## テスト戦略

100行のバッファコードに対して**1180行のテストコード**。「複数スレッドの同時push/steal」「空状態での各操作」など極限ケースを網羅。550日以上の安定運用を確認。

## アーキテクチャ全体

メインスレッドが「God Job」を実行。各フレームのシステム（入力/ロジック/出力）を並列ジョブとして投入。依存関係は`JobFuture`で明示的に待機。

## 汎用化ポイント

- ローカルバッファ + ワークスティーリングのパターンはRust以外でも適用可能
- リングバッファの固定容量 + 満杯時即実行は汎用的な設計判断
- 待機中のペンディングジョブ実行でデッドロックを回避する手法は広く応用可能
