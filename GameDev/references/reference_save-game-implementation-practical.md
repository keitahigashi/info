## 3層アーキテクチャ

1. **永続システム (Persistent Systems)**: セーブを必要とするゲームモジュール
2. **セーブゲームマネージャー**: 中央調整。各システムを登録し個別のJSONバケットを付与
3. **ストレージ層**: 実際の永続化処理

## 4つの設計目標

1. **堅牢性 > 新しさ**: 「5分前の正しいセーブが30秒前の破損セーブより良い」
2. **使いやすいAPI**
3. **パフォーマンス目標** (CPU/帯域/メモリ)
4. **後方互換性**

## バージョニング戦略

- **すべてをバージョン管理**: マジックナンバーではなく名前付きマクロ（例: `FIRST_SAVE_GAME_VERSION_NUM`）で形式変更を自動文書化
- ロード時に全システムの存在とバージョン番号の最低要件を検証
- 暗黙的データ依存を避ける: ビットパック整数より個別のboolフラグ

## ダーティフラグによるセーブトリガー

システムが自身を「ダーティ」とマークする2段階:
- **即時**: 重要変更で即座にセーブ
- **次バッチ**: 通常変更でスロットル化されたセーブ

マネージャーがストレージを叩きすぎずに定期的ロジックを実行。ゲーム終了時には**フラッシュ関数**で強制セーブ。

## よくある落とし穴

### ロード完了前のセーブ

未初期化システムが空バケットをシリアライズ → データ全消去を保証。永続システムまたはマネージャー側のアサーションで保護。

### 遅延統合

既存システムへのセーブの後付けは、初期化フローとミッションロジックパス全体のリファクタリングを引き起こす。

## 実践的アドバイス

- プロジェクト**開始時から**実装する（中盤以降ではない）
- 操作契約を強制する広範なアサーション
- 開発者セーブはデータスキーマ変更時に自動ワイプ。完全な後方互換性はリリース前まで遅延可
- マルチプレイヤーのセーブルールをシンプルかつ理解可能に

## 高度な概念

- **空間的永続性** (Spatial Persistence): プレイヤーが離れたエリアのデータ
- **セッション永続性** (Session Persistence): ゲーム再起動時のデータ
- マルチプレイヤーシステムは読み書きアクセス要件を宣言し意図しないデータ操作を防止
