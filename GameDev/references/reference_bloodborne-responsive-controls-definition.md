## レスポンシブネスの再定義

著者: Stanislav Costiuc、2019年5月8日、Game Developer

一般に「レスポンシブなコントロール」は「ボタンを押した瞬間にアクションが完了すること」と誤解されるが、正しくは**「ボタン押下からフィードバックを受け取るまでの時間が短いこと」**である。

---

### Bloodborneによる実証

**軽攻撃（R1）の例**

- R1ボタン押下 → 即座にキャラが前進するウィンドアップ（予備動作）アニメーションが開始
- 実際のヒット判定はその後に発生するが、プレイヤーは「入力が受け付けられた」という即時フィードバックを受け取る
- カメラの揺れ・効果音も同期して始まる

**チャージ攻撃（R2）の例**

- R2長押し → 即座に構えポーズと効果音が発生
- リリース時に低速の大攻撃アニメーション＋音響＋カメラ追従

**重要な結論：**
ウィンドアップ（予備動作）アニメーションは遅延ではなく、**即時フィードバックの一形態**である。

---

### 2要素モデル

```
レスポンシブネス = 即時フィードバック × カテゴリ内一貫性
```

| 要素 | 意味 |
|---|---|
| 即時フィードバック | 入力受付を確認させる視覚・聴覚信号（ウィンドアップ等） |
| アクション完了 | 実際のゲームロジック処理（遅れても可） |
| カテゴリ内一貫性 | 同種の操作は同じフィードバックパターンを持つ |

---

### 一貫性（Consistency）の重要性

レスポンシブネスには「カテゴリ内の機械的一貫性」も必要。

- Bloodborneでは全攻撃が「ウィンドアップ→実行」パターンに従う
- アイテム拾得は即座に完了するが、それは**別カテゴリのルール**として成立しているため矛盾しない
- 同一カテゴリ内でルールが混在すると「何か操作が引っかかる感覚」が生まれる

---

### 設計への示唆

```
NG: ボタン押下 → 何も起こらない → 200ms後にアクション開始
OK: ボタン押下 → 即座にウィンドアップ開始（フィードバック） → アクション完了
```

ウィンドアップをフィードバックチャンネルとして活用することで、重量感のあるアクション（ソウルライクの重攻撃等）でも「レスポンシブに感じる」設計が成立する。
