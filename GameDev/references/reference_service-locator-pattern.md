Game Programming Patterns (Robert Nystrom): https://gameprogrammingpatterns.com/service-locator.html

## 定義

具体的な実装を知らずにサービスにアクセスできる一元化ポイント。「サービスを必要とするコードを、それが**誰**であるか（具体的実装型）と**どこ**にあるかの両方から分離する」。

## シングルトンとの根本的な違い

両方がグローバルアクセスを提供するが、サービスロケーターでは具体的サービスクラスがロケーター経由でアクセスされていることを**知らない**。シングルトンはサービスクラスの内部設計に影響。

## 使用すべきケース（慎重に）

- モジュールのパブリックAPIの一部にすべきでないシステム（ロギング、メモリ管理）
- 根本的に唯一のファシリティ（オーディオデバイス、ディスプレイ）
- オブジェクトの多層受け渡しが不要な複雑さを生む場合
- **デフォルトはオブジェクトの直接受け渡し**

## 3つの実装手法

| 手法 | 説明 |
|------|------|
| **Simple Locator** | 登録済みサービスを返すだけ。最速、コンパイル時インライン化可能 |
| **Null Service** | 何もしない実装でnullポインタクラッシュを防止。未登録時の優雅な劣化 |
| **Decorated Service** | ラッピング（LoggedAudio等）でコア実装を変更せずランタイム行動を修正 |

## 設計判断

### サービス配置方法

| 方法 | 利点 | 欠点 |
|------|------|------|
| **外部登録** | 高速、柔軟 | 初期化順序の規律が必要 |
| **コンパイル時バインド** | 可用性保証 | 柔軟性なし |
| **ランタイム設定** | 最大の柔軟性 | パフォーマンスオーバーヘッド |

## 核心的トレードオフ

即時のコード結合の可視性とランタイム柔軟性のトレード。実装の容易なスワップを可能にするが、依存関係を不明瞭にし時間的結合（正しい順序での初期化が必須）を導入。
