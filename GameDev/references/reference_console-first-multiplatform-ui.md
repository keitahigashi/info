## マルチプラットフォームUI設計の3アプローチ

### 問題提起

開発リソースを考慮しながら、プレイヤーが使用するプラットフォーム・入力方式に適切に機能するUIを設計するにはどうすればよいか。

### コンソール優先が有効な理由

- テレビから離れて見やすい設計 → PCモニターでも必ず読みやすい
- コントローラーで機能する操作体系 → マウス/キーボードへの適応が容易（逆は困難）

### 3つのアプローチ比較

| アプローチ | 事例 | 概要 | トレードオフ |
|-----------|------|------|------------|
| シンプルプロンプト切り替え | Dishonored 2 | ボタンアイコンのみ変更 | 最も経済的。UIはコンソール仕様のまま |
| PCリーニング | Fortnite | マウス優先設計＋コントローラー対応 | PC体験は良好。コンソールは妥協点あり |
| 秘密のコンソールファースト | Overwatch | レイアウト共有 + 入力方式別インタラクション | PCユーザーにコンソール設計が伝わらない最適解 |

### 「秘密のコンソールファースト (Secretly Console First)」の実装原則

1. **レイアウトは共有** — 画面配置・情報密度はコンソール基準で設計
2. **インタラクション層を分離** — 入力方式に応じたホバー/フォーカス/スクロール挙動を切り替え
3. **PCユーザーに気づかせない** — マウス専用の利便性（ホバーツールチップ等）を追加し差を埋める

### 適用判断フロー

```
マルチプラット対応予算が少ない
  └→ シンプルプロンプト切り替え（最速）

PCが主要プラットフォーム
  └→ PCリーニング

コンソール・PCを同等品質で出したい
  └→ 秘密のコンソールファースト
```

### 設計上の注意点

- 「フォントサイズ10メートルテスト」：テレビ距離で読める文字サイズはPC近距離でも問題ない
- コントローラーUIにマウスオーバー状態を追加するより、マウスUIにフォーカス状態を追加する方がコスト低
- 入力方式検出はリアルタイムで行い、ゲーム中にシームレス切り替えを実現する
