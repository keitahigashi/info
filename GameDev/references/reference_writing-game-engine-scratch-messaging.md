## カスタムゲームエンジン設計の基本方針

「既存エンジン（Unity、Unreal等）で十分な場合が多いが、学習目的や特殊機能が必要な場合はカスタムエンジン開発も有効。」最大の敵は「できるだけ早くゲームプレイを見たい」という衝動であり、グラフィックスから始めると後でコードをすべて捨てることになる。

## 設計プロセス 4ステップ

1. エンジンの必要機能と不要機能を明確化する
2. それらを論理的なシステムに整理する
3. システムを結びつくアーキテクチャを設計する
4. 1〜3を繰り返す（設計に十分な時間をかける）

## レイヤー構造（ラザニア (lasagna) モデル）

「スパゲッティ依存」を避けるため、階層化構造を採用する。

| レイヤー | 内容 |
|----------|------|
| フレームワーク層 | OpenGL / DirectX などの外部ライブラリラッパー |
| システム層 | GUI・入力・レンダラーなどの高レベル機能 |
| ゲームロジック層 | フレームワークと独立した純粋なゲームロジック |

各システムは他のシステムを直接参照しない。通信はすべてメッセージバス経由で行う。

## メッセージバス (message bus) システム

```
class System {
    void handleMessage(Msg *msg);
    MessageBus *msgBus;
}
```

- すべてのシステムはメッセージバスへの参照を持つ
- メッセージ型に応じて処理を振り分ける
- システム間の物理的な接続（直接参照）が不要になる

### メッセージバスで無料で得られる機能

| 機能 | 仕組み |
|------|--------|
| デバッグコンソール | すべてのメッセージを記録・再現 |
| ゲーム内シネマティクス | 時系列でメッセージを送信 |
| リプレイ機能 | 入力メッセージを記録・再生 |
| マルチスレッド対応 | メッセージ処理をスレッド化 |
| フレームレート最適化 | キューで次フレームに遅延処理 |

## 実装推奨順序

1. メッセージバスとコンソール
2. GUI
3. フレームワークインターフェース
4. その他のシステム

この順序により実装途中でも常に機能をテスト・デバッグできる。

## 注意点

メッセージシステムはオーバーヘッドを生じさせるため、高速応答が必要な処理（FPS の衝突判定など）では即座メッセージ処理（即時呼び出し）も併用する。
