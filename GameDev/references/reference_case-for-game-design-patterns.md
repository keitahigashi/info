Game Developer (Bernd Kreimeier, 2002): https://www.gamedeveloper.com/design/the-case-for-game-design-patterns

## ゲームデザインパターンとは

本稿はソフトウェア工学のパターン言語（Christopher Alexander起源）をゲームデザイン分野に適用することを提唱した初期の重要論文。

> "パターンは単に反復する設計上の決定を記述・文書化するための慣例 (A pattern is just a convention for describing and documenting recurring design decisions)"

### パターンテンプレートの4要素

| 要素 | 内容 |
|------|------|
| **名称 (Name)** | 設計語彙を高める識別子 |
| **問題 (Problem)** | 文脈・前提条件・障害を含む課題の説明 |
| **解決策 (Solution)** | 問題を解決する抽象的な構造・原則 |
| **帰結 (Consequences)** | 解決策のトレードオフ・副作用・適用範囲の限界 |

## 具体的なパターン事例

### Proxy（代理操作）
- **問題**: プレイヤーが直接制御できない他キャラクターを操作したい
- **解決策**: メインキャラを通じて他キャラを制御可能にするオーナーシップ機構
- **事例**: Munch's Oddyssee の所有オーブ、Oddworld シリーズ

### Predictable Consequence（予測可能な結果）
- **問題**: プレイヤーが行動前に結果を予測できず、不公平感が生まれる
- **解決策**: 物理挙動・AI行動・ゲームルールに一貫性を与え、習熟による予測を可能にする
- **事例**: Super Mario 64 の物理挙動（ジャンプの初速・重力・壁すり抜け判定が常に同じ）

### Weenie Chain（目標の連鎖）
- **問題**: プレイヤーが次に何をすべきか分からなくなる
- **解決策**: ディズニーの「carrot on a stick」から着想した段階的な可視化道標。近くの目標達成で次の目標が見えてくる
- **事例**: テーマパーク設計 → レベルデザインへの応用

## パターン文書化の価値

1. **語彙の共有化**: 設計議論で「これはWeenie Chainが必要な問題だ」と素早く通じる
2. **トレードオフの明示**: 解決策の副作用が文書化されるため、盲目的な適用を防ぐ
3. **知識の蓄積**: 業界全体での設計知識の再利用と進化を促す
4. **直感的で柔軟**: 厳密な形式主義ではなく、実践者が使いやすい「慣例」として機能する

## 現代での意義

この2002年の論文が現在のゲームデザインパターン研究の基礎となっている。ゲームデザインパターンはソフトウェアの「GoFパターン」と異なり、プレイヤー体験レベルの設計判断を対象とする。
