## UXフローの必要性

「建築には設計図が必要なように、ゲームUIにもフロー図が必要」。プロトタイプ製作前の低コスト反復を実現し、チーム間の認識を統一する。

### 解決する課題
- プロトタイプ製作は時間がかかる
- 全体構造を一目で把握しにくい
- 設計上の矛盾を見落としやすい

## UXフローの4分類

### 1. ナビゲーションフロー (Navigation Flow)
- **定義**: 異なる画面間のアクセスと終了方法の表現
- 画面やオーバーレイ間の移動関係を図示
- メインメニュー → サブ画面への接続構造を可視化
- **用途**: 全体の画面遷移マップ作成

### 2. 情報アーキテクチャフロー (Information Architecture Flow)
- **定義**: 各画面を構成する情報要素と表示方法の表現
- 画面内に配置される情報の階層構造を詳細に記述
- **事例**: Immortals Fenyx Risingのインベントリ画面分析
- **用途**: 画面内の情報配置と優先度設計

### 3. インタラクションフロー (Interaction Flow)
- **定義**: プレイヤーの行動とゲームシステム状態変化の表現
- ユーザー操作がトリガーする状態遷移を追跡
- **事例**: A Plague Tale: Requiemの武器変更機能
- **用途**: 操作に対するシステム応答の設計

### 4. ワイヤーフロー (Wireflow)
- ワイヤーフレームを図式化した表現
- プロトタイプなしで構造を理解可能
- **用途**: 視覚的なレイアウトと遷移の統合表現

## 3段階のスコープレベル

| レベル | 範囲 | 目的 |
|--------|------|------|
| **ハイレベル** | ゲーム全体の構造 | 目標・方針・一貫性の確認 |
| **ミッドレベル** | 関連機能群のシステム | 部分間の統一性の検証 |
| **ローレベル** | 個別機能の詳細動作 | 具体的なインタラクション設計 |

**事例**: Mario Kart Tourのハイレベルフローで主要画面群を整理。

## フロー設計の6つのベストプラクティス

| # | プラクティス | 説明 |
|---|-------------|------|
| 1 | 交差の回避 | 同類のフロー線が交差しないよう配置 |
| 2 | 重複排除 | 同じ画面を複数表示しない |
| 3 | フロー線の統合 | 可能な限り線を結合させる |
| 4 | 最短経路化 | フロー線は直線的に |
| 5 | 間隔の一貫性 | 均等な要素配置 |
| 6 | 視覚的多様性 | 色・形・太さで区別（色数は制限） |

「道路地図やメトロ路線図のように整理する」。

## 拡張的活用法

### 制作進度追跡
各画面の開発段階を色分け表示:
- Next Sprint → Proto → Skinned → Polished → QA

### フィードバック配置マッピング
- 重要なUI信号・フィードバックが必要な箇所を地図化
- 開発優先順位付けへの活用

## 主要メリット

- **低コスト反復**: プロトタイプ前に構造を検証
- **ドキュメント化**: 再利用と拡張が可能
- **信頼できる基準点**: チーム全体の認識統一
- **優先順位付け**: 機能依存性を考慮した制作計画
- **効率的プロトタイピング**: フロー図完成後の実装が高速化
