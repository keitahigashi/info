## マルチウィンドウUIの設計課題

養蜂シミュレーションゲーム APICO では「複数のメニューを同時に開き、画面上で自由に配置できる」設計を採用した。この判断の背景と実装上の知見を紹介する。

### 既存ゲームへの問題意識

Terraria 等では単一メニューのみ開ける設計が主流。しかし複雑な資源管理・クラフトシステムでは、「A から B にアイテムを移したいときに A を開けない」という使いにくさが生じる。APICO はこれを解決するためにマルチウィンドウを採用。

## マルチウィンドウシステムの設計決定

### OS窓パターン (OS Window Pattern) の採用

- デスクトップOSのウィンドウ操作をモデルとした親しみやすいインターフェース
- タイトルバーのドラッグで移動、×ボタンで閉じるという普遍的操作体系
- プレイヤーが学習コストなしに操作できる既存メンタルモデルを活用

### ウィンドウ管理の実装

| 機能 | 設計内容 |
|------|---------|
| **ドラッグ可能なメニューバー** | プレイヤーが好きな位置に配置可能 |
| **Z オーダー管理** | クリックしたウィンドウを最前面に移動 |
| **重なりの視認性** | 半透明化・縁取りでオーバーラップ時の識別を確保 |
| **ピン留め機能** | リリース後に追加。特定ウィンドウを固定位置に留める |

### UX の重要な設計判断

#### アクセシビリティを考慮したボタン配置

左利きユーザーや親指操作に配慮し、主要インタラクションボタンを左側に配置。「デフォルトが右」という慣例を疑い、プレイヤーの多様性に対応。

#### ターゲットメニューシステム

- シフトクリックでアイテムを「転送」する場合、どのメニューに送るかを指定する仕組み
- 複数ウィンドウが開いている状態でも目的のコンテナに直接転送できる
- アクティブウィンドウを「ターゲット」として明示することで混乱を防止

## ニールセン原則との対応

著者はニールセンの10の使いやすさ原則を設計プロセスに明示的に統合：

- **システム状態の可視性**: 転送先ターゲットメニューを視覚的にハイライト
- **現実との対応**: OS ウィンドウ操作という馴染みのある比喩を採用
- **エラー防止**: 誤転送を防ぐターゲット選択と確認フロー
- **フィードバック**: ホバー、クリック、転送完了のアニメーション

## 汎用化ポイント

APICO 固有の事例だが、以下のパターンは他ジャンルに応用可能：

### マルチウィンドウUIが有効なゲームジャンル

- **ターン制RPG/ストラテジー**: パーティメンバーとインベントリを並べて参照
- **ファーミング/農業シム**: 複数ストレージを並べてアイテム整理
- **クラフト重視ゲーム**: レシピと素材リストを同時表示
- **デッキ構築**: コレクションとデッキを並べて編集

### 実装時の考慮事項

1. **Z オーダー管理**: 最前面ウィンドウの明確な視覚的表現が必須
2. **最大ウィンドウ数の制限**: 無制限だと画面が溢れる。上限と通知が必要
3. **スクロール干渉**: ウィンドウ内スクロールとカメラ移動の衝突を解決
4. **モバイル対応**: タッチ操作でのドラッグ実装はPCと異なる考慮が必要
5. **画面解像度への対応**: 小画面ではウィンドウが多すぎると機能しない
