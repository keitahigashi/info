## 設計の根本原則

「アイテムの収支を矛盾なく行うシステムであること」— インベントリシステムの最重要原則。

## 3層アーキテクチャ（関心の分離）

| 層 | 責務 |
|----|------|
| **データ層** | アイテム定義・インベントリ状態・永続化 |
| **ロジック層** | ゲームプレイルール・操作検証・ネットワーク同期 |
| **UI層** | 表示・ユーザーインタラクション |

各層が他層の詳細を知らずに自身の役割に集中する設計。

## アイテム定義 vs インスタンスの分離

### Item Definition（定義/テンプレート）

不変または準不変のテンプレート。アイテムの静的プロパティを記述:
- 名称、タイプ、レアリティ、スタッカビリティ
- 装備可能スロット、基本ステータス
- JSON / ScriptableObject / データベースで管理

### Item Instance（インスタンス）

プレイヤー所有のアイテム実体。テンプレートを参照し、可変フィールドをコピー:
- 耐久度、カスタムステータス、エンチャント
- 一意ID（インスタンスごと）

## コンポーネント型アイテム設計（Fragment/Component Pattern）

アイテム効果を小さなコンポーネントとして表現:

```
例: 剣アイテム = StatModifier(+10 STR)
              + OnEquip(GrantAbility: "Slash")
              + DurabilityTracker(max: 100)
```

**利点**: 新アイテムタイプごとにモノリシックなクラスを作成する必要がない。フラグメントの組合せで多様なアイテムを構成。

## インベントリコンテナ

ストレージの抽象化（バッグ/チェスト/装備スロット/倉庫）で容量とルールを強制。

### 核データ構造

- `List<ItemId>`: 所持アイテム識別子
- `List<int>`: 各アイテム保持数
- 整数配列ではなくリスト: 「整列順がID順に限定されない」柔軟性

### 基本操作

| 操作 | 説明 |
|------|------|
| Add | アイテム追加（スタック可能なら既存にマージ） |
| Remove | アイテム除去（部分除去対応） |
| Swap | スロット間のアイテム交換 |
| Merge | 同一アイテムのスタック結合 |
| Sort | タイプ/ID等でフィルタリング・ソート |

## スタッキングルール

- `StackSizeMax`: アイテムタイプごとの最大スタック数
- スタック不可アイテム → インスタンスごとに固有（耐久度/カスタム攻撃力等）
- スタック可能アイテム → 同一扱い＋カウント管理

## 装備システム

### EffectStack パターン

- **装備品**: 装備中は有効なバフ効果（持続的スタック）
- **消費アイテム**: 1回だけ実行されるエフェクト（一時的スタック）

統一的なエフェクトスタック解釈で装備/消費の処理を共通化。

## クラス責務分担

| クラス | 責務 |
|--------|------|
| InventoryManager | Add/Remove/Move/Merge操作 |
| EquipmentManager | 装備/解除、修正値の適用/除去 |
| SaveSystem | アイテムインスタンス＋レイアウトのシリアライズ |
| NetworkSync | インベントリ変更の差分送信（マルチプレイ） |

## カスタマイズ層（抽象クラスパターン）

スロットクリック時の挙動を抽象クラスで定義し、継承で詳細表示・ショップ・クラフト等を実装。`internal` 修飾子でパッケージ内部実装を保護しつつ、拡張ポイントを公開。
