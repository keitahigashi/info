## 核心的な問題

> 「技術者がツールを設計するが、最も技術的でない人々がそれを使用する」

ゲームパイプラインのツールは実装モデル（コード・データ構造）に基づいて設計されるため、アーティストやレベルデザイナーには複雑すぎる。この乖離がパイプライン全体のボトルネックになる。

## パイプラインのマッピング

### 目的
ワークフローを依存関係グラフ (dependency graph) として可視化し、ボトルネックを特定する。

### 手順
1. 各ステージとその依存関係を列挙（例：設計→モデリング→テクスチャリング→リギング→アニメーション→ビヘイビア）
2. 最長パス（クリティカルパス (critical path)）を特定
3. 最初のアセット完成までの時間を算出
4. 以降のアセットは「最も時間がかかるステージの期間」で量産可能になる

## ユーザビリティ測定ツール

### System Usability Scale (SUS)
- 10項目の声明文に対する5段階評価
- スコア範囲：0〜100
- 70〜80以上：平均的なユーザビリティの基準

### Single Usability Metric (SUM)
より詳細な測定方法。4指標を測定：
1. タスク完了率
2. タスク完了時間
3. エラー数
4. ユーザー満足度

## ツールの抽象化レベル

| レベル | 特徴 | 利点・課題 |
|--------|------|----------|
| テキストエディタ型 | 直接データ編集 | 最も寛容性が低い；反復が必要 |
| 入力制限型 | 有効値のみ受け入れ | タイポを防止；ロジックエラーは残存 |
| 抽象化型 | データを概念的に表現 | 最も効率的；反復時間を大幅短縮 |

**例（3Dモデリングツール）**
- 実装レベル：頂点座標・UV マッピング・RGBA 値
- 抽象化レベル：視覚的なモデル表現

→ ユーザーの理解度に合わせた抽象化層を設けることで、同じ結果をより少ないタスク数で達成できる。

## ゴール指向設計とユーザーモデル

### ユーザーモデル (User Model) の構築
具体的な架空のユーザーを作成する：
- 名前・年齢・教育背景
- 個人的なゴール（昇進志向など）
- 仕事上のモチベーション

**実例「Brad」（レベルデザイナー）**
- 27歳、英文学学位、残業を避けたい
- ゴール：難易度の高いゲームを制作したい

### プライマリユーザーモデルの重要性
「すべてのユーザーに対応する」のではなく、**中級ユーザーを対象**に設計する。個別の要求ではなくユーザーモデルの「ゴール」に照らし合わせてフィードバックを判断することで、設計の一貫性を保つ。

## 実装プロセス（5ステップ）

1. **ボトルネック特定**: パイプラインマップで最も遅いステージを洗い出す
2. **ユーザビリティ測定**: SUS または SUM で現状を数値化
3. **ユーザーモデル定義**: プライマリユーザーを具体的に設定
4. **データ抽象化**: ユーザーの理解度に合わせた表現方法を実装
5. **フィードバック検証**: ユーザーモデルのゴールに照らし合わせて改善案を評価
