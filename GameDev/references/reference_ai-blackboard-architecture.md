## ブラックボードとは

AIシステム間で情報を交換するための**共有キーバリューストア**。各キーは名前（例: `TargetLocation`）と型（int/float/vector/bool）を持ち、型安全性を強制。

## 単一エージェントから多エージェント協調へ

単一エージェントのビヘイビアツリーを超え、**分散型マルチエージェント協調**を実現。階層的コマンド構造ではなく、共有知識に基づく分散型意思決定。

「エージェントが共有データを読むことで、自然に役割を分担し衝突を回避 — 明示的なスクリプトなし」

## 2種類のブラックボード（Jeff Orkin分類）

| タイプ | 説明 | 例 |
|--------|------|-----|
| **Static（静的）** | ゲーム状態の事前定義リポジトリ。アクセサー/モディファイアで読み書き | ターゲット位置、エージェント速度、現在HP |
| **Dynamic（動的）** | ゲーム状態に関する情報の記録。エージェントが実行したアクションを記録 | 「特定アクションを実行済み」→ 他エージェントが重複回避 |

## タスク協調メカニズム

明示的ブロッキングでタスク重複を防止:

1. エージェントが利用可能タスクを評価し適切なアクションを選択
2. 選択されたタスクをブラックボード上で**利用不可にマーク**
3. 他エージェントが即座に認識し代替を選択

→ 「複数NPCが同じドアを突破しようとする」等の無駄な重複を排除。

## Utilityシステムとの統合

エージェントがスキル・ペルソナ・緊急度に基づいてタスク適合度を**Utilityスコアで評価**。ブラックボードがクレーム済みタスクを利用不可にマークして重複を防止。

## 実践例: 戦術分隊

| エージェント | 自然に選択される役割 |
|------------|------------------|
| Agent A | 突破（ブリーチング） |
| Agent B | 援護射撃（カバーファイア） |
| Agent C | 側面攻撃（フランキング） |

→ 中央集権的割り当てなしに**有機的に出現**。

## 5つの主要利点

| 利点 | 説明 |
|------|------|
| シンプルさ | キーバリューストアの基本構造 |
| 柔軟性 | 新エージェント/タスク追加が容易 |
| スケーラビリティ | エージェント数増加に線形対応 |
| 創発性 | 単純ルールから複雑な協調が出現 |
| 統合互換 | BT/Smart Objects/Utilityと統合可能 |

## 中央集権型AIとの比較

| 観点 | 中央集権型コマンダーAI | ブラックボード |
