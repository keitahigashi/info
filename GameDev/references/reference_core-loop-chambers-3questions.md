## コアループ設計：3問フレームワーク

### コアループの定義

> プレイヤーのアクション（動詞）が次々と接続し、繰り返し可能な環状構造を形成したもの。この環状構造への参加動機がエンゲージメントの源泉となる。

### Andrew Chambersの3問フレームワーク

設計時に必ず問うべき3つの質問：

1. **「プレイヤーは何をしているのか？」** — 実際の繰り返し行動の同定
2. **「何が彼らを止めているのか？」** — 障壁・リスク・コスト構造
3. **「なぜそれをしているのか？」** — 動機・報酬・意味の確認

### 実装ガイドライン

| ガイドライン | 詳細 |
|---|---|
| **要素数を絞る** | コアは **2〜5要素**に保つ。過剰設計 (bloat) を意識的に避ける |
| **広義の動詞を選ぶ** | 「射撃する」より「**戦闘に従事する**」など、拡張可能な抽象度で定義 |
| **過剰設計を避ける** | コアに含めるべきでない要素はメタループやセカンダリシステムへ移動 |

### 実例: Bullet Heaven / アクションローグライク

Survivor.io を例に、最小構成のコアループ：

```
Kill（敵の撃破）→ Collect（ドロップ収集）→ Upgrade（能力強化）→ Kill ...
```

3動詞のみでゲームプレイの本質を成立させる設計。コアの簡潔さが反復耐性と拡張余地を同時に生む。

### 関連フレームワークとの接続

- **リソースループ (Resources Loop)** — リソース循環設計への接続
- **MDAフレームワーク (MDA Framework)** — Mechanics-Dynamics-Aesthetics
- **One Pager** — コンセプト文書化ツールとしてループ設計を記述

### 設計上の注意

- 動詞の抽象度が高すぎると設計が曖昧になる。「戦闘に従事する」が何を指すかをチームで明示的に合意する必要がある
- 「何が止めているのか」の問いは、難易度設計・ゲートシステム・リソース制約の設計指針として機能する
