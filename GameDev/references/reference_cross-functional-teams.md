## クロスファンクショナルチームの定義

「完全な目標を最小限の外部依存で達成するためのツールを持つチーム」。職能（アニメーション、プログラミング、アート）ではなく、機能目標でチームを編成する。

## 組織構造の比較

### 従来の職能別組織 (Departmental Model)

**「ショットガンゾンビの死亡」機能の例**:
- 機能がサブコンポーネントに分解される
- アニメーション部門 → プログラミング部門 → エフェクト部門 → モデリング部門 → サウンド部門を順次通過
- **結果**: 5スプリントにまたがる実装
- バックログが「機能レベル」ではなく「実装詳細」で管理される

```
[企画] → [アニメ] → [プログラム] → [エフェクト] → [モデル] → [サウンド]
          Sprint1    Sprint2       Sprint3      Sprint4    Sprint5
```

### クロスファンクショナルモデル

同じ機能を1つの統合チームに割り当て:
- **結果**: 1スプリントで完了（5スプリントの代わりに）
- 外部調整が最小限
- フラットな管理構造

```
[統合チーム: アニメ+プログラム+エフェクト+モデル+サウンド]
                    Sprint1 で完了
```

## 4つの主要利点

| # | 利点 | 説明 |
|---|------|------|
| 1 | **高速イテレーション** | 完了までの時間短縮 → 品質改良の余地拡大 |
| 2 | **調整オーバーヘッド削減** | 内部での依存解決 → 管理負担の最小化 |
| 3 | **スケーラブルな成長** | チーム追加が複雑性の指数的増加を招かない |
| 4 | **明確なプロダクトビジョン** | バックログが機能フォーカスを維持（タスク断片化を回避） |

## バックログ管理の違い

| 組織形態 | バックログ内容 | 問題 |
|---------|--------------|------|
| 職能別 | 実装詳細（「アニメーション作成」「シェーダー調整」） | 機能全体の優先度が見えない |
| クロスファンクショナル | 機能目標（「ゾンビ死亡演出の実装」） | 機能レベルでの優先度判断が可能 |

## 導入の課題

- 既存の組織構造の解体が必要
- 専門性のサイロ化からの脱却
- チームメンバーの多様なスキルセットが前提
- 職能リーダーの役割再定義

## 汎用化ポイント

この原則はスタジオ規模を問わず適用可能:
- **インディー (1-5人)**: 本質的にクロスファンクショナル（全員が複数役割）
- **小規模 (5-20人)**: 2-3のクロスファンクショナルポッドに分割
- **中規模 (20-100人)**: 職能横断チーム + Communities of Practice（専門知識共有）
- **大規模 (100人+)**: フィーチャーチーム + 職能リードによるクラフトメンターシップ
