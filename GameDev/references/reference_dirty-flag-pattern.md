## 出典

Game Programming Patterns (Robert Nystrom): https://gameprogrammingpatterns.com/dirty-flag.html

## 定義

一次データの変更から派生データの再計算を遅延させ、**結果が必要になるまで処理を先送りする**最適化パターン。

## 基本構造

| 要素 | 役割 |
|------|------|
| **一次データ** | 時間とともに変化する基本情報 |
| **派生データ** | 一次データから計算される結果 |
| **ダーティフラグ** | 両者の同期状態を示す真偽値 |

### 処理フロー
1. 一次データ変更時にフラグを設定
2. 派生データ必要時にフラグをチェック
3. 設定済み → 再計算してフラグクリア
4. 未設定 → キャッシュされた派生データを使用

## シーングラフの事例

### 階層的変換構造
海賊船（親）→ カラス巣（子）→ 海賊（孫）→ オウム（曾孫）

各オブジェクトは相対変換（ローカル変換）を保持。絶対位置（ワールド変換）は全親チェーンの変換を合成して計算。

### 最適化効果
- 非効率: 4オブジェクト移動で**10回計算**（不要6回）
- 最適化後: **4回に削減**

## パフォーマンストレードオフ

| 側面 | トレード関係 |
|------|------------|
| 時間 vs 記憶 | 計算速度向上のため派生データをメモリに保持 |
| 遅延コスト | 計算先送りするが、必要時に処理待機が発生可能 |
| キャッシュ無効化 | 一次データ変更時に**必ず**フラグを設定 |

## フラグクリアタイミング3種

| 方式 | 特徴 | 適用 |
|------|------|------|
| **結果必要時** | 未使用なら計算スキップ | 軽量計算向け |
| **チェックポイント** | ローディング画面等の自然な遅延ポイント | 中量計算向け |
| **バックグラウンド** | タイマーベース定期実行 | 重量計算向け |

## トラッキング粒度

- **細粒度**: 変更データのみ処理（オーバーヘッド大）
- **粗粒度**: 不変データも含む（固定コスト削減）

## 適用条件
- 一次データ変更頻度 > 派生データ利用頻度
- 段階的更新が困難
- 計算コストが高い
