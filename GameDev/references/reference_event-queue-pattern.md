## 出典

Game Programming Patterns (Robert Nystrom): https://gameprogrammingpatterns.com/event-queue.html

## 意図 (Intent)

「メッセージやイベントの送信と処理を時間的にデカップルする」。

## コア問題

同期APIが直接リクエストを処理する際の3つの問題:
1. **ブロッキング実行**: API呼び出しが完了まで呼び出し元を停止
2. **個別リクエスト処理**: 複数リクエストの最適化・集約ができない
3. **スレッド問題**: 同期呼び出しが呼び出し元のスレッドで実行、複数システム同時呼び出しで並行性問題

## 解決アーキテクチャ

リクエスターとプロセッサー間のバッファとしてキューを導入。送信者はエンキューして即座にリターン。別プロセスが自分のペースでキューからアイテムを取得し非同期処理。

### リングバッファ実装
配列要素のシフトではなく、headとtailポインタがmodulo演算でラップアラウンド。キャッシュ効率を維持しつつ連続的エンキュー/デキューをサポート。

### メッセージの具象化 (Reification)
リクエストが処理されるまで持続する離散データ構造（例: `PlayMessage`にSoundIdとvolume）に変換。

## 設計判断

### キュー内容タイプ

| タイプ | 説明 | 特徴 |
|--------|------|------|
| **イベント（通知型）** | 過去の出来事（「敵が死亡」） | 複数リスナーにブロードキャスト。広いスコープ |
| **メッセージ（要求型）** | 望む未来のアクション（「音を再生」） | 単一受信者。より限定的ルーティング |

### 読み取りアクセスモデル

| モデル | 説明 |
|--------|------|
| **シングルキャスト** | キューは読み取り側のプライベート |
| **ブロードキャスト** | 全リスナーが全アイテムを受信。イベントフィルタリングが必要 |
| **ワークキュー** | 各アイテムを1リスナーに配布。スケジューリングロジックが必要 |

### 書き込みアクセスモデル

| モデル | 説明 |
|--------|------|
| **単一ライター** | 暗黙的な起点。サイクル防止だが柔軟性に制限 |
| **複数ライター** | 任意システムがアクションを要求。メッセージ内での送信者ID追跡が必要 |

### オブジェクト寿命

| 戦略 | 説明 |
|------|------|
| **所有権移転** | 送信者がメッセージを放棄、受信者がクリーンアップ担当 |
| **共有所有権** | 参照カウントで全参照が消えるまでメッセージを維持 |
| **キュー所有権** | 事前割当プールスロットにメッセージが永続。送信者が事前割当オブジェクトを参照 |

## 重大な考慮事項

### 状態の陳腐化 (State Staleness)
キューに入れたイベントは未知の間隔で遅延通知。その間に世界が変化（エンティティ解放、環境変更）。エンキュー時に必要なコンテキストをすべてキャプチャすべき。

### フィードバックループ
非同期システムは同期コードならクラッシュするサイクルを隠す可能性。一般的な安全策: イベント処理中のイベント送信を禁止。ログで問題パターンを特定。

### グローバル状態の危険性
中央イベントキューはグローバル変数として機能 — 隠れた依存関係、予測不能な状態相互作用、追跡困難なバグのリスク。

## いつ使うか

**使うべき場面:**
- 送信者が不規則なタイミングでリクエストをプッシュ
- 受信者が便利な瞬間にプルする必要がある
- 複数リクエストの集約が有益
- スレッドがコア間で関心を分離

**使わないべき場面:**
- 送信者が即座の応答を必要とする
- 状態がリクエスト時点の世界条件を反映すべき

## 関連パターン

- **Observer**: リスナーを送信者からデカップルする同期的代替
- **Command**: キューに入れて遅延実行可能
- **Update Method**: キュー評価の処理サイクルを提供
- **Object Pool**: 固定サイズキューの効率的バッキングストア
