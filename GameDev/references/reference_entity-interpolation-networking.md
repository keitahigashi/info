## 核心課題

サーバーが低頻度（例: 100ms間隔）で位置更新を送信する状況で、他プレイヤーをスムーズにレンダリングする方法。

## サーバータイムステップアーキテクチャ

- ゲームワールドは「クライアント入力の有無と量に関係なく、予測可能なレートで更新」
- 入力頻度とワールドシミュレーションを分離
- CPU＋帯域幅を最適化

## 時間オフセット戦略

核心的洞察: **自分のキャラクターはリアルタイム**（予測）、**他プレイヤーは遅延表示**。

```
サーバー更新 t=900 と t=1000 の間:
→ 次の区間で「t=900からt=1000の間の実際の移動データ」を表示
→ 制御されたレイテンシで「実際の移動データ」を使用
```

## 2つのアプローチ比較

### Dead Reckoning（デッドレコニング）

予測可能なエンティティ向け（レーシングゲーム等）:
- 速度と方向から将来位置を推定
- 「車の方向と加速度が一定のまま」と仮定
- サーバーデータ到着時に修正
- プレイヤー行動が仮定と一致する場合に有効

### Entity Interpolation（エンティティ補間）

予測不能な動き向け（シューター/アクション）:
- 予測を完全に放棄
- 受信済みサーバー位置**のみ**を使用
- 連続スナップショット間の線形補間でスムーズな動きを生成
- プレイヤー意図を推測しない

## 帯域幅最適化

- 10ms間隔でサンプリングされた位置データの送信
- 移動軌跡の送信（位置列ではなく）
- **デルタ圧縮**で過剰帯域幅消費を回避

## トレードオフと限界

各プレイヤーが異なる世界状態を知覚することは許容可能。ただし射撃等の競技的アクションでは精度要求が露出:

- ターゲットが100ms前の歴史的位置を反映
- エイム課題が発生
- → **ラグ補償**で対処（サーバーが過去を再構築）
