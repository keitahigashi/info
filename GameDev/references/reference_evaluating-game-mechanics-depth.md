## メカニクスの「深さ」を評価する

### 「浅い」フィードバックの3分類

プレイヤーの「つまらない」というフィードバックは原因によって対策が異なる：

| 問題のタイプ | 主なキーワード | 対応方法 |
|-----------|-------------|--------|
| ゲーム全体の多様性不足 | 「一本調子」「バリエーションがない」 | メカニクスの種類を増やす |
| 個別メカニクスへの不満 | 「つまらない」「見栄えが悪い」 | 演出・音響・報酬を強化 |
| **メカニクスの浅さ** | 「簡単すぎる」「すぐ飽きる」「深みがない」 | メカニクス自体に深さを追加 ← この記事の対象 |

### 「深さ (depth)」の定義

> プレイヤーがそのメカニクスへの習熟を**繰り返し発揮できる**ポイント

深さのための必須要素：

1. **明確な目標 (Clear Objective)** ── プレイヤーが「何を達成すべきか」を理解している
2. **意味のあるスキル (Meaningful Skills)** ── 目標を完了状態へ導く、実質的な技術

### 意味のあるスキルの条件

- 「基本的すぎるスキル」ではない（A地点からB地点へ移動するだけ等は除外）
- 目標それ自体と混同していない
- 他のスキルと機能的に重複していない

## 活動ステートメント (Activity Statement)

メカニクス設計時に、目標と意味のあるスキルを組み合わせた1文を書く：

```
「プレイヤーに [意味のあるスキル群] を使って [目標] を達成させたい」

例：
・「二段ジャンプとグライダーを組み合わせて敵を回避させたい」
・「属性相性と武器の組み合わせを選択して効率的に敵を倒させたい」
```

## ケーススタディ：Ratchet & Clank 2 のトラクタービーム

**初期設計の問題：**
- 目的が「物体をA→Bへ移動させる」だけで浅かった
- ロボット・爆弾・エネルギースリングショットなど複数の目的を追加したが、プレイヤーは混乱
- 目的を増やしただけではメカニクスの深さは生まれない

**改善アプローチ：**
```
冗長な要素を排除
  ↓
1つの意味のあるスキルに集中
  ↓
Activity Statement を改善
  ↓
プロトタイプ → プレイテスト → 繰り返し
```

## 深さを評価するチェックリスト

**目標の確認：**
- [ ] 各目標を列挙できているか
- [ ] 機能的に重複する目標はないか
- [ ] 冗長なものは削除されているか

**意味のあるスキルの確認：**
- [ ] 本当に「意味のある」スキルか（基本的すぎない）
- [ ] 他スキルとの機能的重複がない
- [ ] 目標と混同していない

## 深さを追加するプロセス

```
1. 新しい意味のあるスキルを追加
2. すべてのチャレンジの Activity Statement を改善
3. プロトタイプ作成
4. プレイテスト
5. 問題解決まで繰り返し
```

## 深さが不要な場面

深さの追加が常に適切とは限らない：

| 状況 | 理由 |
|------|------|
| 幼年向けゲーム | 浅いメカニクスの多様性が適切（例：Lego Star Wars） |
| カジュアルゲーム | 深さより活動のバリエーションが重要 |
| キャラクター表現重視 | 深さが個性・感情表現を損なう可能性 |
| ナラティブメカニクス | 難度より感情的共鳴が優先 |

> 「幾何学は真実ではなく、便利である」（アンリ・ポアンカレ）── このフレームワークも唯一の正解ではなく、実用的なツールとして使うべきである。
