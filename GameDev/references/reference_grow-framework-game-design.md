## GROWフレームワークとは

1990年代にコーチング業界で開発された目標設定・問題解決手法をゲーム機能設計に応用したもの。ゲーム開発で「なんとなく機能を作り始め、後で迷走する」混乱を防ぐための構造化アプローチ。

## 4段階フレームワーク

### G — Goals（目標設定）

すべてのゲーム機能には明確な存在目的が必要。目的が曖昧なまま設計を進めると後に膨大な時間が無駄になる。

SMART 基準で目標を定義する：

| 項目 | 内容 |
|------|------|
| Specific（具体的） | 機能が達成すべき特定の目的 |
| Measurable（測定可能） | KPI などの定量指標 |
| Attainable（達成可能） | 利用可能なリソース内で実現できるか |
| Relevant（関連性） | 製品全体の目標と整合しているか |
| Timeboxed（期限付き） | 開発スケジュール内に収まるか |

### R — Reality（現状把握）

出発点を正確に把握する段階。

- 既存機能との依存関係をマッピングする
- 未解決の設計課題を列挙・優先順位付けする
- 技術リソースとチームスキルを確認する
- 利用可能なツール・アセットを整理する

### O — Options（選択肢の洗い出し）

目標達成への複数の経路を検討する。選択肢を生成する5つのアプローチ：

1. **Five Options Approach**: 複数のアイデアから最良のものを選別
2. **Brainstorming Approach**: 高レベルな目標からアイデアを創出
3. **Obstacle Approach**: 障害を排除した状態を想像して問題解決
4. **Ideal Future Approach**: 理想的な結果から逆算して計画立案
5. **Transformative Approach**: 既存機能を改良・応用する

### W — Will（実行計画）

最適な選択肢を決定し、アクションプランに落とし込む。

- 影響度と実装難度のマトリクスで優先順位を評価する
- 具体的なアクションと担当者・期限を設定する

## 活用のポイント

「このフレームワークは魔法の解決策ではなく、各チームの現実に適応させるべきツール。」規模に関わらず、機能設計の議論を始める前に G（目標）を共有することだけでも、開発の混乱を大幅に減らせる。
