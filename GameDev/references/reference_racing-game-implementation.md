## 2つの実装アプローチ

### 物理ベース実装

車輪を独立物理オブジェクトとしてシミュレート（質量/摩擦/サスペンション）。「車はホイールにトルクを適用して加速」し、物理エンジンが力を自然に解決。

#### 主要課題

| 課題 | 詳細 |
|------|------|
| **安定性** | 低速で車両が転倒。アンチロールバー効果のシミュレーションが必要 |
| **速度依存ステアリング** | 全レーシングゲームが速度で最大操舵角を制限（低速10°→高速2°） |
| **トラブルシュート** | 設定変数と実際の挙動間の間接層が多い。操舵感が悪い原因は質量/トルク/速度/サス/タイヤスリップの任意の組合せ |

#### 創発的ゲームプレイ

MotorStormのように凹凸地形でのトラクション喪失を**抑制せず活かす**トラックデザイン → 物理がゲームプレイを生む。

### アーケード実装

最小限のベースラインから目標複雑度まで構築。「何もないところから始め、個別メカニクスを追加」。

#### 柔軟なメカニクス

全レーシングゲームが同一システムを必要としない:
- Mario Kart: タイヤスリップなし
- Rocket League: 単一BoxCollider
- TrackMania: ブーストフォーカス

#### サークルベース旋回

「現在の接線速度に基づくサークル上の車の位置」を計算:
- 半径 = ステアリング入力量に依存
- ドリフト時はさらにタイトな半径
- 速度依存半径で非現実的な高速旋回を防止

## ドリフトメカニクス

「ブレーキ＋ターン = ドリフト意図」と仮定。シミュレーションが通常旋回の代わりにドリフトを開始。

## ビジュアル補償

アニメーション/エフェクトが機械的簡略化をマスク:
- サスペンションの「偽装」: 実サスペンションシミュレーションなしに、現在の運動量/加速度に基づいてビジュアルを傾斜/バウンス

## 設計トレードオフ

| 観点 | 物理ベース | アーケード |
|------|-----------|----------|
| リアリズム | 高い | 意図的に低い |
| アクセシビリティ | 上級者向け | 幅広い |
| 創発的ゲームプレイ | 自然に発生 | 意図的に設計 |
| イテレーション | 多くの変数チューニング | 直接的な制御 |

## 核心的洞察

- 位置計算が「100%トラクション」感に優れ、力適用はスライド感を生みがち
- 両アプローチの境界は曖昧 → ハイブリッドが成功
- 「磨き上げられた満足のいく操作感は、選択手法に関係なく大量の作業とイテレーション」が必要
