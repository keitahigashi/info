## 百万パーティクルシステムの構築

### 問題提起：CPUパーティクルの限界

従来のCPUパーティクル実装はGPUへのデータ転送帯域幅の制約から、**1フレームあたり約10,000パーティクル**が実質的な上限だった。GPUパーティクルシステムはこの制約を排除し、リアルタイムで最大100万パーティクルのレンダリングを可能にする。

### GPU vs CPU パーティクルの比較

| 側面 | CPUパーティクル | GPUパーティクル |
|------|----------------|----------------|
| 最大数 | ~10,000 | ~1,000,000 |
| シミュレーション場所 | CPU | GPU（コンピュートシェーダ） |
| CPU負荷 | 高い | 最小限 |
| デバッグ難易度 | 低い | 高い |
| モバイル対応 | 容易 | 制限あり |
| パーティクル間相互作用 | 可能 | 可能（GPU上で） |

### 6段階の処理パイプライン

```
1. 生成・消滅処理   → 新規パーティクル生成、寿命切れの削除
2. 速度テクスチャ更新 → 各種力（重力・吸引・フロー場）を適用
3. 位置テクスチャ更新 → 速度から位置を積分
4. ソート処理      → アルファブレンディングのための深度順ソート
5. テクスチャ→頂点変換 → GPUメモリ内でパーティクルデータを頂点化
6. レンダリング    → ポイントスプライトで描画
```

### テクスチャベースシミュレーション

**ストリーム処理モデル:**
```
入力テクスチャ（位置・速度） → ピクセルシェーダ → 出力テクスチャ（新位置・新速度）
```

**ダブルバッファリング (Double Buffering):**
- 位置テクスチャペアと速度テクスチャペアを交互使用
- フレームNの出力がフレームN+1の入力になる
- CPU-GPU間のデータ転送を排除し、全シミュレーションをGPU上で完結

### 物理演算の並列化

**速度更新で実装される力の種類:**

| 力の種類 | 説明 |
|---------|------|
| 全域的加速度 | 重力など全パーティクルに一様に作用 |
| 局所的吸引・反発力 | 特定座標を中心とした引力・斥力 |
| フロー場 (Flow Field) | テクスチャからサンプリングした速度場 |
| 衝突反応 | 平面・球体・高さマップとの衝突処理 |

**数値積分法:**

```
オイラー積分: velocity += force * dt; position += velocity * dt;
ベルレ積分:   position_new = 2*position - position_old + force * dt²;
             (速度を陰的に計算、メモリ使用量削減)
```

### 並列ソートの実装

アルファブレンディングには深度順のソートが必要。GPUでは **奇偶マージソート (Odd-Even Merge Sort)** を採用:

- log²(n)回のレンダリングパスで並列ソートを実現
- フレーム間の一貫性を活用して複数フレームに処理を分散
- ソートを何フレームかに分割することでフレームレートへの影響を最小化

### レンダリング最適化

**ポイントスプライト (Point Sprite) の活用:**
- 三角形や四角形の代わりにポイントスプライトを使用
- 頂点数を1/3〜1/4に削減
- スクリーンスペースでの四角形ビルボードに自動展開

### ハードウェア要件

- 浮動小数点対応ピクセルパイプライン（32ビット精度）
- テクスチャ→頂点データ転送機構（頂点テクスチャフェッチ）
- 現代のゲームエンジンでは Unity VFX Graph（GPU）・Unreal Niagara（CPU/GPU選択可能）が相当機能を提供

### 適用ユースケース

| 向いている用途 | 向かない用途 |
|--------------|------------|
| 大規模VFX（爆発・砂嵐・群衆） | シンプルな小規模エフェクト |
| PC・コンソール向け高品質エフェクト | モバイル（GPU制約） |
| パーティクル間相互作用が必要な場合 | CPU側でのロジック連携が頻繁な場合 |
