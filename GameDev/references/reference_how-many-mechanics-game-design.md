## 出典

Game Developer (Bobby Lockhart, 2016): https://www.gamedeveloper.com/design/how-many-mechanics-should-a-game-have-

## 中心的主張

「ゲームに必要なメカニクスの数は、現在想定しているものの **半分以下** にすべきだ」

開発者が過剰なメカニクスを想定する原因:
- 現実世界の複雑性をゲームに再現したいという衝動
- AAAゲームの多機能さへの誤解（彼らは長プレイ時間と慣例に依存できる）

## 事例比較

### スーパーマリオブラザーズ（シンプルの深さ）

- コアメカニクス: **「ジャンプ」** のみ
- そのジャンプ1つから生まれる結果: 隙間越え・敵回避・敵踏み潰し・ブロック破壊
- 追加メカニクス（きのこ・ファイアフラワー）も最小限に絞り、相互作用で深さを生成

### Jetpack Joyride（多層的設計）

- コアメカニクス: 「上昇加速制御」（ジェットパック）
- サポートメカニクス①: 乗り物パワーアップ（複数機能を1つのシステムに集約）
- サポートメカニクス②: 実績システム（行動に目標を追加し、既存メカニクスの活用を促進）
- 結果: 少ないメカニクス数で豊富な **ダイナミクス（dynamics）** を生成

## メカニクス追加の評価フレームワーク

サポートメカニクスを追加する前に、以下の4基準すべてを満たすか問う:

| 基準 | 問い |
|------|------|
| 感情的一貫性 | 目指す感情トーン・体験と合致しているか |
| 多目的性 | 複数の目的を同時に果たすか |
| 相互作用 | 既存メカニクスと豊かに絡み合うか |
| 連鎖性 | 同時実行または連続実行が可能か |

## 「ゲームがまだ楽しくない」問題

> 「ゲームがまだ楽しくないなら、メカニクス追加は解決策ではない」

コアメカニクスが確立されていない段階でサポートメカニクスを積み重ねても、楽しさの核が存在しないため問題は解決しない。まずコアを固め、そこから検証で積み上げる。

## 設計プロセス

1. コアメカニクス（行動）と感情（目指すトーン）を定義する
2. コアの結果（ダイナミクス）を十分に探索・検証する
3. サポートメカニクスを4基準で評価してから追加する
4. マーケティング要求（「機能リスト」への圧力）に抗い、ダイナミクスで価値を伝える

## ダイナミクス（dynamics）の概念

メカニクス同士が相互作用することで生まれる複合的な遊戯体験。ゲームの「深さ」はメカニクス数ではなくダイナミクスの豊かさで決まる。MDAフレームワークにおけるDynamicsと同義。
