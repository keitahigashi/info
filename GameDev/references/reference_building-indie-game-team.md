## 必須職種と責務

| 職種 | 主要責務 | 注意点 |
|------|---------|--------|
| プログラミング | ゲーム本体・ツール・Webサイト・インストーラー | ゲーム外開発も含む |
| デザイン | GDD管理・最終決定権・チームの相談役 | デザイナーが最終決定権を持つべき |
| アート | ゲームアート・マーケティング資材・Webデザイン | プログラマーとの密接連携が必須 |
| サウンド | BGM・SE・雰囲気構築 | 軽視されがちだが重要。開発初期に整備 |
| QA | バグ修正・継続テスト | 非プログラマーによるテストが効果的 |
| ビジネス | マーケティング・販売計画・キャッシュフロー管理 | 小規模でも早期から担当者を置く |

## 運営の4原則

### 1. 共同作業環境
可能であればチーム全体を同じ場所に配置する。迅速なテストと動機付けの相乗効果。

### 2. 責任の明確化
各メンバーの作業時間と関与度を**事前に合意する**。不均衡な貢献は対立の温床になる。

### 3. 人格適合性 (personality fit)
> 「人格は能力を上回るほど重要」

強い個性を持つ開発者は共通目標を個人的なビジョンより優先させる必要がある。採用時に技術力より人格を重視すべき場面がある。

### 4. 紛争解決メカニズム
- 対話：まず率直に話し合う
- 妥協：双方が譲歩できる中間点を探す
- 投票：合意形成が難しい場合の最終手段

## 外注で補完できるもの

| カテゴリ | 外注候補 |
|---------|---------|
| エンジン | Unity・UDK等の既製エンジン活用 |
| サウンド | サウンドライブラリ・効果音素材 |
| アート | 一部のアセット・UI素材 |

外注により開発期間短縮とコスト最適化が期待できるが、品質管理とコミュニケーションコストに注意。

## 汎用化ポイント

- 「デザイナーが最終決定権を持つ」原則は、エンジニア主導になりがちな小規模チームで特に重要
- QA を全員で担当する文化は、バグを「誰かが見つけるもの」ではなく「全員の責任」として内面化させる
- 人格適合性の重視は、スキルセットが不完全でもチームの持続力につながる
