## アーキテクチャ基盤

**クライアントサイド同期型**を採用。各プレイヤーのクライアントで独立して盤面管理し、「すべてのプレイヤーのクライアントで全く同じ盤面進行が行われる」ことを前提とする。決定論的処理 (Deterministic Processing) とコマンド履歴の共有により、ネットワーク遅延を最小化しながら整合性を保証。

## 4つのコアデータ構造

| 構造 | 役割 | 規模 |
|------|------|------|
| **BattleData** | HP・攻撃力・位置情報・状態異常・フィールド座標・プレイヤーパラメータの統合管理 | — |
| **Operation** | 「実行前の可能性」を表現。選択可能マス情報と事前バリデーション結果でUIを支援 | 12種類 |
| **Command** | 盤面更新の不変オブジェクト (Immutable)。全進行を記録可能、JSON直列化対応 | 29種類 |
| **Trigger** | イベント。能力発動条件判定とUI通知の二重用途 | 40種類 |

## ゲームフロー制御: State Pattern

**33種類のState**で複雑なターン構造を管理:
- バトル開始・終了（5種類）
- ターン制御（4種類）
- 移動・攻撃・召喚フェーズ（各複数）
- 能力割り込み機構

自動遷移と入力待機状態を組み合わせ、能力による中断・連鎖発動に対応。

## 能力システム: Composition over Inheritance

継承ではなくコンポジションで200種類以上の効果を再利用可能パーツで表現:

### Targeting パイプライン
**Head + Pipe**構造:
- **Head** (8種類): 対象選択の起点（自分、敵全体、隣接マスなど）
- **Pipe** (18種類): フィルタチェーン（例:「敵の前列HP3以下」）

### Effect (12種類)
ダメージ、回復、移動、能力付与など

### Calculator
加算・乗算・固定値等による数値補正

### Trigger
発動条件を宣言的に指定

すべてJSON化可能で、**アプリアップデート不要のカード追加**を実現。

## Facilitator/Facilitatee パターン

プレイモード抽象化層:

| Facilitator (7種) | 用途 |
|-------------------|------|
| オンライン | 通常対戦 |
| デバッグ | 開発テスト |
| チュートリアル | ガイド付きプレイ |
| AI | CPU対戦 |
| サンドバッグ | テスト用ダミー |
| リプレイ | 対戦再生 |
| 機械学習 | AI学習用 |

ネットワーク/ローカル、プレイヤー/AIの差分をこの層で吸収し、BattleModel本体は純粋ロジックに専念。

## 再接続・復元機能

切断時に過去のコマンド履歴をサーバーから取得 → 高速リプレイで盤面復元 → 追加コマンドの即時実行。途中参加にも対応。

## 設計パターンの集約

| 課題 | パターン | 利点 |
|------|---------|------|
| フェーズ管理 | **State** | 33種類の独立クラスで可読性向上 |
| 同期・リプレイ | **Command** | 履歴再実行で完全復現、署名検証でチート対策 |
| 能力発動 | **Observer** | Trigger購読でUI疎結合 |
| 対象選択 | **Strategy** | Head+Pipeで柔軟な条件表現 |
| モード切替 | **Strategy+Factory** | 新モード追加が容易 |
| 能力多様性 | **Composition** | 再利用可能パーツで200種類以上の効果 |

## 汎用化ポイント

1. **Commandパターンによる決定論的リプレイ**: カードゲームに限らず、ターン制ゲーム全般でコマンド履歴による完全再現が可能
2. **Facilitatorパターン**: 対戦ロジックとプレイモード（AI/オンライン/リプレイ）の分離は、あらゆるマルチモードゲームに適用可能
3. **Composition over Inheritance**: 能力・スキルシステムの設計で継承ツリーの爆発を防ぐ汎用的アプローチ
4. **宣言的データ駆動設計**: 能力定義のJSON化により、コード変更なしのコンテンツ追加を実現する設計思想
