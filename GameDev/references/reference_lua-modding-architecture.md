## 階層アーキテクチャ

```
Engine Layer (C/C++) → Embedded Lua VM → Mod Scripts
```

核心原則: 「Luaは実行フローを制御しない。エンジン駆動のイベントに反応し、明示的に公開されたネイティブ関数を呼び出す」。この制御の反転が安定性の鍵。

## API公開戦略

APIの境界がパワーと安全性の両方を定義する:
- エンティティ操作と状態クエリ
- イベント登録システム
- UIプリミティブとユーティリティ
- 制限された標準ライブラリ

モッドがエンジン状態を破壊できないことを保証しつつ、意味のある拡張性を提供。

## サンドボックス化とセキュリティ

必須の制限事項:

| 制限 | 理由 |
|------|------|
| OSシステムコール禁止 | ホスト環境の保護 |
| ファイルI/Oアクセス制限 | データ漏洩防止 |
| デバッグ/リフレクションライブラリ無効化 | 内部状態への不正アクセス防止 |
| グローバル名前空間の制御 | 名前衝突・汚染の防止 |
| エンジン側バリデーション層 | 不正な値・型の排除 |

「サンドボックス化は、特にマルチプレイヤーやUGC環境において、Luaモディングシステムの絶対的な要件」。

## モッド構成パターン

```
mod_root/
├── init.lua        (エントリーポイント)
├── config.lua      (設定)
├── events.lua      (イベントハンドラ)
└── entities/       (モジュラー行動)
```

関心の分離と保守性を促進する構造。

## イベント駆動実行モデル

- プレイヤースポーン/切断イベント
- エンティティライフサイクルフック
- フレーム/ティック更新
- 入力およびネットワークメッセージ
- 暴走スクリプトを防止し、決定論性を向上

## ライフサイクル管理

1. エンジンがモッドディレクトリをロード
2. Lua VMが `init.lua` を実行
3. エンジンがライフサイクルコールバックを呼び出し
4. モッドがイベント/フックを登録

エンジンが実行順序と障害処理を制御。

## パフォーマンス考慮事項

### 高リスクパターン
- フレームごとの重い処理
- 過度なテーブル割り当て
- 深い再帰
- ホットパスでの文字列操作

### 推奨原則
「パフォーマンスクリティカルなシステムは可能な限りネイティブに残し、Luaは制御とオーケストレーション層として機能させる」。

## Luaが選ばれる理由（他言語との比較）

| 比較項目 | Lua | JavaScript | Python | C# |
|---------|-----|-----------|--------|-----|
| フットプリント | 極小（<300KB） | 中 | 大 | 大 |
| 決定論的実行 | ○ | × | × | △ |
| 公開機能の制御粒度 | 高 | 中 | 低 | 中 |
| 組み込みの容易さ | 高 | 低 | 中 | 低 |

## 汎用化ポイント

- イベント駆動 + サンドボックス化のパターンは任意のスクリプト言語に応用可能
- モッド構成パターン（init/config/events分離）はプロジェクト構造の汎用テンプレート
- API公開のレイヤー設計はプラグインシステム全般に適用
