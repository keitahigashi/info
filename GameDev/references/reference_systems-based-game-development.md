## システムベースのゲーム開発ガイド

### 概要

システムベースの設計とは、個別の「用途固定のメカニクス」ではなく「ゲーム世界全体が相互作用するシミュレーション」を作ることである。プレイヤーの行動に対してゲーム環境が深い一貫性をもって反応する仕組みを構築することが目標。

> 「ゲームに良いシステムがあるとは、本質的に深いメカニクスがあるということだ。」

### 中心的設計原則

#### 1. システムの独立動作
各システムは最初に「それ単独の空間で完全に機能する」よう開発する。その後、外部からの入力を受け付けるインターフェースを追加する。

```
間違ったアプローチ: 全ての組み合わせを事前にプログラムする
正しいアプローチ: 各システムが独立動作 → 入力受付インターフェースを後付け
```

#### 2. モジュール性と疎結合
「新機能追加時に既存システム全体を更新しなくてよい」設計を目指す。武器とプロジェクタイルを例にすると:
- 両者を完全に独立させる
- 中間構造（インターフェース）で連携させる
- これにより新しい武器・弾薬タイプを追加しても他システムへの影響が最小になる

#### 3. 一貫性の最優先
システムの結果が再現可能であることが最優先条件。一貫性がないと「実験的プレイ」を促せない。プレイヤーは「この組み合わせを試したら何が起きるか」を学習・予測できなくなる。

### 相互作用設計のアプローチ

全ての組み合わせをブルートフォース的に予測・プログラムするのではなく、各システムが独立して動作し、共通インターフェースを通じて反応する構造にする。これにより設計者が意図しない組み合わせによる「創発的プレイ (Emergent Play)」が自然に生まれる。

### 実装の3段階プロセス

1. **最小実装**: 各システムの基本機能を単独で完成させる
2. **統合判断**: どのシステム間を接続し、何を意図的に制限するか決定する
3. **アーキテクチャ再設計**: 後の拡張性を損なわない構造に調整する

### 具体的ゲーム事例

#### Dishonored 2
- エミリーのドミノ能力で敵を鎖状につなぐ
- ブラッドフライ群で大規模な連鎖反応を起こす
- 限定されたツール（5〜6種の能力）でも、組み合わせによって無限の可能性が生まれる

#### Steel Hunters（事例）
- 跳弾がバレルを破壊 → 弾薬庫着火 → 環境全体が反応
- プレイヤーが予測不可能な状況に直面し、その場で戦術を変更する協力型プレイ

### システム設計の報酬

このアプローチの最大の成果は**「プレイヤーが他者に語りたくなるような予期しない体験の生成」**である。

> 「その機能が可能だとは知らなかった」—というプレイヤーの反応を引き出すことが目標。

### 関連概念との接続

| 関連概念 | 接続ポイント |
|---------|------------|
| 創発的プレイ (Emergent Gameplay) | システム間の相互作用から予期しない体験が生まれる |
| コンポーネントパターン | 独立コンポーネントの組み合わせがシステムベース設計の基盤 |
| ECSアーキテクチャ | データ指向でシステムを分離・組み合わせる実装手法 |
| デザイン理論（MDA） | ダイナミクスがシステム間相互作用から生まれるという点で対応 |
