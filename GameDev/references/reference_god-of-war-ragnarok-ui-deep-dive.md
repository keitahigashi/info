## God of War Ragnarök のUI設計詳解

### UIチーム構成 (11名)

| 役割 | 人数 | 主な担当 |
|------|------|---------|
| Technical Designer | 2 | UIアーキテクチャ・実装 |
| 3D Artist | 2 | Maya でUI アートを実装 |
| 2D Artist | 2 | Photoshop / Illustrator でアート制作 |
| Gameplay Engineer | 1 | C++ API 提供 |
| UX/Accessibility Designer | 1 | アクセシビリティ機能設計 |

### 3つの開発目標

1. **アクセシビリティの強化**: 70以上のアクセシビリティ機能 (ボタンリマップ・ハイコントラスト・フルキャプション) を一から設計
2. **技術的負債の解消**: UIアーキテクチャの約80%を再構築し、58MBのメモリ制約に対応
3. **UI規則の統一**: ポーズメニュー・ベンダー画面など全体のユーザーフローを一貫化

### HUDアーキテクチャ設計

#### 2層構造

```
HUD
├── Combat HUD（左下）
│   ├── ルーニック能力スロット
│   ├── HP バー
│   └── レイジゲージ
└── UI Messages（その他の領域）
    ├── エリアアナウンス
    ├── クエスト更新
    ├── ジャーナル更新
    └── アイテムティッカー
```

#### 単一キューイングシステム

全 UI メッセージを 1 つのキューで管理することで HUD の過負荷を防ぐ:
- 同時に 1 つのメッセージのみ表示
- ポーズメニュー・カットシーン・ベンダー画面でのキュー処理も統一管理
- 課題: メッセージが溜まると表示に時間がかかる → 将来は画面領域ごとの独立キューに改良予定

#### ボスHP バー管理

- 専用マネージャーファイルがレイアウトとアニメーションを制御
- ボスの種類・同時登場数 (最大3体) に応じて動的に変化
- 「最初は1体しか想定しなかった」 → 設計のスケーラビリティの重要性を示す教訓

### フォントスケーリングシステム

アクセシビリティ機能として Font Scaling (デフォルト〜XX-Large) を全 UI に対応:
- Dynamic Layouts: 他の要素の位置を基準に自動でレイアウトを決定
- 大きなフォント (MASSIVE_TEXT) は段階的なスケールアップ、小さなフォント (MICRO_TEXT) はより急なスケールアップ
- 目的: すべてのサイズで UI テキストのバランスを維持

### 段階的 UI 開示 (Progressive Disclosure)

プレイヤーの進捗に合わせて UI 要素を隠す・解放する設計:
- ゲーム開始時: ルーニック・リリックスロットを非表示 → 入手時に出現
- カテゴリ全体: アイテムが 1 つも入るまでベンダーに非表示
- Transmog・Skill Mods: ゲーム中盤〜後半に遅延開放 → 「新鮮な驚き」として再エンゲージメントを促す

### チュートリアル設計の知見

- ゲーム内チュートリアル: 80以上のチュートリアル、うち13個がポーズメニュー内
- インゲームチュートリアルは 1-2 ステップが理想
- メニュー内チュートリアルは最大 7 ステップまで (それ以上はスキップされる)

### UIとシステムデザインの関係

> 「多くの意味でシステムデザインはUIデザイン。複雑なシステムを構築しても、最終的にプレイヤーに説明・提示する必要がある」

アミュレットメニュー (Amulet of Yggdrasil) は 4 回の反復設計を経て完成 → UIは UX と情報提供のバランスを反復で見つける。

### 汎用化ポイント

これらの設計原則は特定エンジン/タイトルに依存しない:
- **単一キューイングシステム**: どのゲームでも複数の UI メッセージ競合を解決できる
- **Dynamic Layouts**: フォントスケーリングや多言語対応に必要な基本パターン
- **段階的 UI 開示**: チュートリアル設計・コンテンツ解放設計に普遍的に適用可能
- **専用マネージャーパターン**: 複雑な UI 状態変化を一元管理する設計アプローチ
