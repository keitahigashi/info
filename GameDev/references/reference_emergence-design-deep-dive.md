## 創発の定義

「複雑で魅力的で予測不可能なゲームプレイ体験が、比較的単純なゲームメカニクスの相互作用から生じる現象」。複雑系の特性として、システム全体の振る舞いが個々の部品の振る舞いからは予測できない。

## 3種の複雑性

| 種類 | 説明 |
|------|------|
| **ルールベースの複雑性** | 単純なルールでも相互作用により複雑な行動を生成 |
| **システム的複雑性** | 物理・経済・社会動態の相互作用 |
| **プレイヤー駆動の複雑性** | プレイヤーの選択と行動による複雑さの生成 |

## 創発の設計手法

### 複雑システムの構築
- ゲームメカニクスの**層化** (layering): 単純メカニクスを組み合わせて複雑な相互作用を生成
- **フィードバックループ**の導入: あるシステムの出力が別のシステムの入力に
- プレイヤーの選択肢と**自律性**の提供

### 環境の動的化
- 手続き生成: 毎回異なるワールド構成
- 動的シミュレーション（天候、昼夜サイクル）
- プレイヤーによる環境修正の許可

### プレイヤーエージェンシー
- **プレイヤー駆動のストーリーテリング**: 選択により独自のナラティブを創造
- **レスポンシブメカニクス**: プレイヤー行動への即座の応答
- 分岐ナラティブ、動的対話システム、選択と結果の提供

## 複雑性の管理

創発は望ましいが、圧倒を避けるため調整が必要:

| 管理手法 | 説明 |
|---------|------|
| インターフェース簡素化 | 複雑なシステムを直感的に操作 |
| チュートリアルとガイダンス | 段階的なシステム理解の促進 |
| ジェントルなガイダンス | ヒント・提案で方向付け |
| フィードバックメカニズム | 報酬・ペナルティで行動を導く |
| プレイテストと反復 | 創発の過不足エリアを特定 |

## 具体的なゲーム事例

| ゲーム | 創発の表れ方 |
|--------|------------|
| **Minecraft** | 基本ブロック操作 → 複雑な自動化機械・建築 |
| **The Sims** | 複雑シミュレーション → プレイヤー駆動の独自ストーリー |
| **Dwarf Fortress** | 複雑シミュレーション → 創発ナラティブ（予想外の出来事） |
| **RimWorld** | 複雑シミュレーション → 個性的なコロニーストーリー |

## コア設計原則

複雑さと指導のバランスが成功の鍵。ゲームは十分に複雑で創発的でありながら、プレイヤーがシステムを理解・操作できる**明確性**を提供する必要がある。

テストにおいて「創発が不足または過剰なエリアを特定し調整する」ことが不可欠。
