## 体験中心設計のフレームワーク

### 手法の概要

「何を作るか（機能・メカニクス）」でなく「プレイヤーにどんな体験をさせたいか」を起点にする設計アプローチ。

**手順：**
1. チームでターゲット体験を言語化・文書化する（1〜2文の具体的な体験記述）
2. その体験を実現するために何が必要か逆算する
3. 設計判断に迷ったときは常に「この選択は目標体験に近づくか」で判定する

### 4つの利点

| 利点 | 説明 |
|------|------|
| 優先順位の明確化 | 「どの機能から着手するか」の決断が速くなる |
| 執着からの解放 | 単一の要素・アイデアへの固執を防ぐ |
| チーム共通目標 | 全員が同じゴールを向く |
| 時間制約下での機能 | ゲームジャムのような短期開発でも有効 |

### 事例1：Laundromallard（GGJ 2017）

**設定：** 最大8人でアヒルを洗濯機で洗浄・乾燥させるパーティゲーム

**ターゲット体験（明文化した体験）：**
「複数プレイヤーが互いに叫び、一緒に遊び、ゲーム内の混乱で笑う」

**体験から逆算した設計要素：**
- シンプルで習得しやすい操作（スティック移動・ボタンで跳びつかみ）
- 序盤は低難度 → 段階的に難度上昇してカオスを演出
- 共有・対戦の両方が成立するルール設計で多様なグループに対応

### 事例2：Who Will...（Indie Gameleon 2015）

**設定：** 3〜7人参加、故人の遺産を誰が相続するか投票するパーティカードゲーム

**ターゲット体験（明文化した体験）：**
「パーティ中の会話の合間に気軽にプレイできる補助的なゲーム体験」

**体験から逆算した設計要素：**
- ランダム属性の組み合わせ（スキル差を平準化しつつ笑いを生む「不条理な組み合わせ」）
- シンプルなフロー（デバイスを手渡すだけで次のプレイヤーへ進行）
- 1ゲームが短時間で完結（パーティのメインになることを避ける設計）
- 不適切なユーモアを排除し「不条理から生まれる笑い」に特化

### 実践的なポイント

**体験記述の書き方：**
- 「〇〇を感じる」「〇〇な状況になる」という主語はプレイヤー
- 抽象的すぎず具体的すぎず（感情・状況の言語化）
- チーム全員が読んで同じ絵が浮かぶレベルが目標

**設計停滞時の使い方：**
- 行き詰まったときに体験記述を読み直す
- 「この問題を解決しなくても目標体験は成立するか？」と問い直す
- 体験記述と合わないアイデアは保留ではなく積極的に除外する
