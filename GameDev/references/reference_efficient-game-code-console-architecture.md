## ゲームコード効率化の実践ガイド

### 基本原則

> "The most important optimization you can make is to reduce the size of your data."

最重要最適化はデータサイズの削減。CPU はメモリよりはるかに速く、パフォーマンスボトルネックの大部分はキャッシュミスに起因する。

---

## 4つの最適化の柱

### 1. メモリアクセス最適化 (Memory Access Optimization)

**キャッシュ階層の理解**:

| キャッシュレベル | アクセスコスト | 備考 |
|--------------|-------------|------|
| L1 キャッシュ | 1〜4 サイクル | 最速、数十 KB |
| L2 キャッシュ | 10〜40 サイクル | 中速、数百 KB |
| メインメモリ | 200〜400 サイクル | **キャッシュミス = 壊滅的なコスト** |

**実践的指針**:
- ゲーム状態を L2 キャッシュサイズ以下に収める（512 KB 超過で急激な性能低下）
- **構造体配列 (SoA)** を採用する — `AoS: {x, y, z, x, y, z}` より `SoA: {xxx, yyy, zzz}` がキャッシュ効率的
- ホットデータ（頻繁にアクセスするデータ）とコールドデータを分離する

---

### 2. 分岐命令の回避 (Branch Avoidance)

分岐命令（if/switch・仮想関数呼び出し・関数ポインタ）はパイプラインストールを引き起こす。

**分岐ミスの影響**:
- パイプラインストール → 命令のルックアヘッド失敗
- 仮想関数呼び出し = 間接参照 = 予測不可能な分岐

**代替手法**:
```
三項演算子の活用: (condition) ? valueA : valueB
テンプレートの活用: コンパイル時に分岐を解決
データ駆動ディスパッチ: 関数テーブルを連続メモリに配置
```

**設計原則**: クリティカルパス（毎フレーム実行されるループ内）の分岐を最小化する。

---

### 3. SIMD とスーパーベクトル (SIMD & Supervectors)

**SIMD（Single Instruction, Multiple Data）**:
- 1つの命令で複数データを同時処理する並列演算
- ベクトル演算（位置・速度・色・行列計算）に特に有効

**スーパーベクトル (Supervectors)**:
- 16〜32 要素の大きな演算単位
- 読み後の依存遅延 (read-after-write latency) を吸収する効果
- SoA レイアウトと組み合わせることで最大効果を発揮

**インライン化の活用**:
- 関数呼び出しオーバーヘッドの削減
- 関数本体の拡大によるスケジューリング効率化
- ヘッダファイルでの定義を活用

---

### 4. スレッド管理 (Thread Management)

**基本原則**: ハードウェアリソース数以上のスレッドを生成しない。

> "Spawn only as many threads as there are hardware resources"

**効率的なスレッド設計**:
- **相補的タスクの組み合わせ**: CPU バウンドタスクと I/O バウンドタスクをペアリング
- **実行計画 (Execution Plan)**: タスク間の依存関係を定義し、フレームレートを保証する
- タスクグラフを用いてスレッド間のデータ依存を管理する

---

## アーキテクチャ的考慮事項

### メモリ管理

- `malloc` / `new` の使用を禁止（断片化の原因）
- 専用アロケータの実装: プールアロケータ・スタックアロケータ等を用途別に使い分け
- ゲームオブジェクトのライフタイムに基づいたメモリ管理

### シリアライゼーション

- ゲームデータを事前にコンパイルしてランタイムのロード時間を短縮
- バイナリフォーマットの活用でパース処理を排除

---

## 汎用化ポイント

本稿は2005年の記事だが、以下の原則は現代のゲーム開発にも適用可能:

| 原則 | 現代への適用 |
|------|------------|
| キャッシュ効率のためのデータサイズ削減 | ECS / DoD の SoA レイアウト |
| 分岐命令の最小化 | GPU シェーダー最適化 |
| SIMD / ベクトル演算の活用 | SIMD intrinsics・GPU コンピュートシェーダー |
| スレッド数 = ハードウェアリソース数 | ジョブシステム・タスクグラフ設計 |
| malloc 禁止 + 専用アロケータ | ゲームエンジンのカスタムアロケータ |

**核心的教訓**: 「大きいほど効率的」— パイプライン活用を最大化する長い関数本体、キャッシュ局所性の確保、並列処理の最大化が共通テーマ。**計測なき最適化は推測**であり、プロファイラーによる実測が最適化の出発点。
