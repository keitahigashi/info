## 出典

Philipp Hagenlocher: https://philipphagenlocher.de/post/data-aware-dialogues-for-video-games/

## コアアーキテクチャ

ダイアログを**純粋関数**として扱う: 開始ラベル＋入力変数を受け取り、終了ラベル＋出力変数を返す。

### グラフ構造: ポリツリー（非循環有向グラフ）

2種類のエッジを持つ:
- **タイムラインエッジ**（青）: 順次的ダイアログフロー（左→右）
- **依存エッジ**（色付き）: データ関係（右→左）

### 設計哲学

「体験は可能な限り非技術的であるべき。ダイアログを書くことはプログラミングではなく、書くことのように感じるべき」

## 変数管理

フラットなキーバリューアーキテクチャ。3型をサポート:
- **Boolean** / **Integer** / **String**

グラフ内の構造的位置を活用し、明示的な命名を最小化。

## 8種のノードタイプ

| ノード | 機能 |
|--------|------|
| **Dialogue** | キャラクター発話を順次表示（選択肢なし） |
| **Choice** | プレイヤー選択肢を提示。Boolean依存接続で条件付き表示/非表示 |
| **Switch** | 順序付きBoolean条件で実行パスを分岐。デフォルトフォールバックでデッドロック防止 |
| **Flag** | タイムラインイベントをマーク → 下流条件ロジック用Boolean出力 |
| **Operation** | 条件付き算術（`+=`, `-=`, `set to`）と否定を実行 |
| **Constant** | 依存グラフにリーフノードとして静的値を注入 |
| **Check** | 整数→Boolean変換（比較演算子: >, <, ==）。定数またはランダム範囲使用 |
| **If-Then-Else / Aggregation** | 型変換・複数入力結合（AND/OR/XOR、SUM/PRODUCT） |

## タイムライン vs データ依存の分離

- タイムラインエッジ: 出力接続に1つのみ接続可能（決定論的順序付け）
- 依存エッジ: 入力接続に複数接続可能（データ収束）
- 依存評価は**遅延評価 (call-by-need)**: 下流ノードが消費するときのみ値を解決

## 静的分析能力

非循環制約により、ループ付きシステムでは**不可能な形式検証**が可能:

| 分析タイプ | 内容 |
|-----------|------|
| 到達可能性分析 | Boolean代数で特定入力条件下の到達可能ノード/エンディングを計算 |
| 整数境界分析 | シンボリック方程式書換えで操作を通じた値範囲を追跡 |
| 確率計算 | 一様分布ランダムチェックで正確な分岐確率を計算 |

### ループ排除の理由

「ループとcheck/operationノードの組合せはWHILEプログラムと等価な関数を構成でき、決定不能性問題を引き起こすため、ループの排除が必要」

## 出力形式

- **JSON中間表現**: エンジン非依存＋バージョン管理互換
- エディタはGodot Engineで構築（スタンドアロン/プラグイン両対応）

## 実用的限界

| 課題 | 説明 |
|------|------|
| 国際化 | 重複テキストのユニークID解決が必要 |
| ゲームインタラクション | 効果ノードの形式仕様策定が未完 |
| ダイアログ横断データ永続化 | メタダイアログインフラが必要 |
| 意味論的仕様 | サードパーティインタプリタ開発に正式仕様が未策定 |

## 概念的位置づけ

このシステムは「ループなしのグラフベースプログラミング言語」として機能。純粋なダイアログ提示に簡素化するか、完全なゲームロジック置換に拡張するかのアーキテクチャ的問いを提起。
