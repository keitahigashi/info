## 出典

Game Programming Patterns (Robert Nystrom): https://gameprogrammingpatterns.com/command.html

## コア概念

「コマンドは具現化されたメソッド呼び出し (reified method call)」— メソッド呼び出しをファーストクラスオブジェクトに変換し、保存・受け渡し・遅延実行を可能に。

## 従来アプローチの問題

ボタン入力をゲーム関数に直接ハードワイヤ:
```
if (isPressed(BUTTON_X)) jump();
```
→ **設定可能なボタンマッピング**が不可能。プレイヤーはカスタマイズ可能な操作を期待。

## Command ベースの解決策

### 構造

1. **Base Command インターフェース**: `execute()` メソッドを持つ抽象クラス
2. **Concrete Commands**: 具体的アクション実装（JumpCommand, FireCommand等）
3. **Input Handler**: 各ボタンが関数直接呼び出しではなく Command ポインタを参照

### リファクタリング後

```
if (isPressed(BUTTON_X)) buttonX_->execute();
```

## 入力システムへの主要利点

### デカップリング

入力ハンドラが特定のアクション実装に依存しなくなる:

| 機能 | 説明 |
|------|------|
| **ランタイムリバインド** | コアコード修正なしにボタン-アクションマッピングを変更 |
| **柔軟なアクター制御** | 同じコマンドで異なるキャラクターを制御（GameActorパラメータ） |
| **Null Object パターン** | 未バインドボタンのNULLチェック不要 |

### 遅延実行

コマンドを即座に実行せず返却することで、入力検出とアクション実行を分離:
- コマンドキューイング
- ネットワーク越しのプレイヤー入力送信
- AI駆動のコマンドストリーム

## Undo/Redo 実装

`execute()` と `undo()` の両メソッドを実装:
1. 変更前の状態を保存
2. アクションを実行
3. Undoで状態変更を巻き戻し

コマンド履歴スタックで複数レベルのUndoが可能。

## リプレイシステム

完全なゲーム状態ではなく**コマンドシーケンスを記録** → メモリ消費を劇的に削減しつつ完全なゲームリプレイを実現。

## 実装考慮事項

| 観点 | 詳細 |
|------|------|
| 言語パラダイム | C++: クラスベース / 関数型言語: クロージャで同等に実現可 |
| 状態管理 | 特定アクター向けコマンド（MoveUnitCommand）は必要な状態を内部保存 |
| 再利用性 | 汎用入力コマンドは任意のアクターで動作 |
