## コードとアート資産の二元戦略

### コード管理
- **主要ツール**: Git
- **保存内容**: スクリプト、プリファブ、最適化済みアセット
- **拡張**: Git LFS（大容量バイナリ対応）

### アート資産管理
- **課題**: 高解像度テクスチャ、3Dモデル、PSDファイルなどの大容量化
- **戦略**: 原本ファイルと最終出力物を分離
- **保管**: 専用リポジトリで原本管理、ゲーム側では最適化版のみ保存

## ツール比較

| ツール | 特徴 | 適用規模 |
|--------|------|---------|
| **Git LFS** | シンプル、Git統合 | 小〜中規模 |
| **Perforce** | 業界標準、大規模資産対応 | 中〜大規模 |
| **Plastic SCM** | アーティスト向けUI、Unity統合 | 中規模以上 |
| **クラウドストレージ** | 手動管理 | ソロ開発者 |

## ベストプラクティス

### リポジトリ構成
- ゲームプロジェクト用: Unity/スクリプト/最適化アセット
- アート専用: 元プロジェクトファイル（PSD、Blend、Kra）

### バージョン同期
- タグ付け: `v1.0.1_art` と `v1.0.1_code` で統一
- Gitコミット記録で追跡可能性確保

### アセット最適化
- 効率的形式への変換（PNG、FBX、OGG）
- ファイルサイズ制限の遵守

### 定期的な同期
- アーティスト側での最適化版エクスポート
- デベロッパーによる統合と実装
