## 核心テクニック：グレースケールテスト

スクリーンショットを脱色（グレースケール化）して確認することで、色彩に頼らずゲーム要素が識別できるかをテストする。

**前提**：色が認識される前に「明度 (value)」が認識される。明暗のコントラストが形状・深度・注意誘導の根本を担う。

## 明度 (Value) の役割

明度コントラストを適切に配置することで実現できる3つの効果：

### 1. 深度感の表現
暗い背景・明るい前景という配置で空間的奥行きを生み出す。カメラに近い要素を明るく、遠い要素を暗くすることで距離感が自然に伝わる。

### 2. 形状と形態の表現
同じ明度のオブジェクトが隣接すると境界が消える。異なる明度を配置することでシルエットを明確化し、キャラクター・敵・インタラクティブ要素を区別する。

### 3. プレイヤーの注意誘導
最も明るい（または最も暗い）エリアに自然と視線が集まる。重要な要素（敵・アイテム・目標）は背景と大きなコントラストを持たせる。

## 実例：DOTA 2 vs 初期DOMARD

**DOTA 2**：背景を暗くし、英雄・敵を明るく配置。グレースケールで見ても各要素が明確に識別できる。

**初期DOMARD（Before）**：背景・敵・歩行可能な地表が同じグレー値で混在。グレースケールでは全て同じ明度になり、プレイヤーが何を操作すべきかわからない状態。

**改善後DOMARD（After）**：コントラストを整理し、各要素に異なる明度を割り当てることで可読性が大幅向上。

## 段階的な視覚的優先順位の設計

高コントラストを多用すると全てが同等の重要度に見えてしまい、かえって混乱を招く。意図的な優先順位の組織化が必要：

```
最高コントラスト（最重要）: プレイヤーキャラクター・主要敵・目標
中コントラスト（重要）:     インタラクティブオブジェクト・UI要素
低コントラスト（補助）:     背景・装飾・環境テクスチャ
```

## 実践的なワークフロー

1. **スクリーンショット撮影** → グレースケール変換
2. **識別テスト**：敵・プレイヤー・重要アイテムが区別できるか確認
3. **問題箇所の特定**：同じグレー値に潰れた要素を発見
4. **修正**：明度差を意図的に広げ、優先順位を視覚的に表現

## アクセシビリティとの関連

グレースケールテストは色覚異常 (color blindness) のプレイヤーへの対応にもなる。色のみに依存した設計はアクセシビリティを損なうが、明度ベースの設計は色盲のプレイヤーにも機能する。
