## 概要

アーティストがプログラマーに依存せず GPU パフォーマンスの基礎を理解するための実践ガイド。「ちょっとした知識が大きな影響をもたらす」という観点から、アセット制作の判断が直接フレームレートに影響するメカニズムを解説する。

## GPUレンダリングパイプライン基礎

```
頂点バッファ/インデックスバッファ
     ↓
頂点シェーダー (Vertex Shader)     ← 3D 座標をクリップ空間に変換
     ↓
ラスタライゼーション (Rasterization) ← 三角形をピクセルに変換
     ↓
ピクセルシェーダー (Pixel Shader)   ← 各ピクセルの色を計算
     ↓
レンダターゲット出力
```

シェーダープログラムはマテリアルの見た目を決定し、GPU 上で並列実行される。

## ドローコール (Draw Call) のコスト

CPU は GPU に「何を・どのように描画するか」の指示をドローコールで送信する。

| 指標 | 目安 |
|------|------|
| コンソールゲームの許容ドローコール数 | フレームあたり 2,000〜3,000 |
| ドローコール削減手法 | テクスチャアトラス・インスタンシング |

**GPU の状態変更（テクスチャ・シェーダーの切り替え）は必ずドローコールを消費する**ため、マテリアル統合がアーティストにとっての直接的な最適化手段となる。

## 主要なGPUボトルネック

### オーバードロー (Overdraw)

同一ピクセルを複数回シェーディングすること。半透明オブジェクト・パーティクルエフェクトの大量配置は深刻なオーバードローを引き起こす。

### オーバーシェーディング (Overshading)

非常に小さい三角形は GPU のラスタライゼーション効率を低下させる。**「三角形のサイズが小さいほど、面積あたりの処理コストが増加する」**。

LOD (Level of Detail) システムが有効な理由:
- 遠距離オブジェクトの頂点数を削減 → 頂点シェーダー負荷軽減
- 画面上の三角形サイズを適正化 → オーバーシェーディング回避

### テクスチャ帯域幅

| 最適化手法 | 効果 |
|-----------|------|
| ミップマップ (Mipmap) | 遠距離オブジェクトでの帯域幅削減・エイリアシング低減 |
| BC 圧縮 (Block Compression) | テクスチャサイズを 1/4〜1/6 に削減 |
| 不要テクスチャフェッチの削除 | GPU キャッシュミスを削減 |

**キャッシュミス**: GPU はキャッシュに載らないテクスチャデータを VRAM から読み込む。頻繁なキャッシュミスは帯域幅を大量消費する。

## メッシュマージの判断基準

| 判断 | 条件 |
|------|------|
| マージ推奨 | 小さく密集したオブジェクトのグループ |
| マージ非推奨 | 広い範囲に散在 → バウンディングボリュームが大きくなりカリングが無効化される |

## アーティストへの実践ガイドライン

1. **マテリアル数を最小化**: 異なるマテリアルを使うほどドローコールが増加する
2. **三角形ポリゴン密度の管理**: 近距離用・遠距離用 (LOD) の両方を用意する
3. **テクスチャ解像度の根拠を持つ**: 画面上での最大表示サイズに基づき適切な解像度を選択
4. **プロファイリングツールの活用**: NVIDIA Nsight・RenderDoc 等で実際のボトルネックを特定
5. **プログラマーとの協働**: 複雑な最適化判断はプログラマーの専門知識と協力して実施

## 開発チームへの示唆

- **アーティストも性能責任を持つ**: メッシュ複雑さ・テクスチャ選択・マテリアル設計は直接ゲームの安定性に影響する
- **パフォーマンスバジェット**: 60 fps = 1 フレームあたり 16.66 ms の予算内に全レンダリングを収める必要がある
- **早期計測**: パフォーマンス問題はリリース直前ではなく、アセット制作段階から計測・管理する
