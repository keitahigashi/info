## Game Engine Architecture — Real-Time Object Updates（リアルタイムオブジェクト更新）

著者: Jason Gregory（Naughty Dog プログラマー、Uncharted シリーズ担当） | 公開: 2009年11月

書籍: 『Game Engine Architecture』（第14章抜粋）

### 概要

ゲームエンジンが毎フレームすべてのゲームオブジェクトの状態をどのように更新するかを解説する。単純な仮想 Update() 関数の限界から始まり、商用エンジンが採用する高度な更新パターンを体系化する。

---

### コア問題: 状態更新とは何か

> 「ゲームオブジェクトの状態はある特定の瞬間の設定を記述する。ゲームオブジェクト時間は連続的ではなく離散的である」

毎フレーム全オブジェクトの状態を更新する必要があるが、単純なアプローチでは性能と正確性の問題が生じる。

---

### 単純アプローチの限界

**仮想 Update() の問題:**
- ゲームオブジェクト内部からサブシステムを直接呼び出すとパフォーマンスボトルネックを生む
- キャッシュ効率が低下し、不必要なリソース再割り当てが発生する

---

### 解決パターン集

#### 1. バッチ更新 (Batched Updates)

**原理:** 同種タスクをまとめて処理してキャッシュ効率を最大化する

**効果:**
- キャッシュコヒーレンシ (cache coherency) の向上
- 計算の重複最小化
- 効率的なパイプライン処理

#### 2. フェーズ更新 (Phased Updates)

**原理:** 複数の更新ステージに分割してサブシステムの計算順序を制御する

```
フェーズ例:
1. アニメーション更新
2. 物理演算
3. コリジョン検出
4. ゲームロジック更新
```

アニメーションが物理に先行するなど、依存関係の制約を守った安全な更新順序を保証する。

#### 3. バケット更新 (Bucketed Updates)

**原理:** 依存関係レベルでオブジェクトをグループ化し、競合する更新要件を解決する

**ユースケース:** 互いに依存するオブジェクト群を、依存関係グラフに基づいたバケットに分類して処理する。

#### 4. 状態キャッシュ (State Caching)

**原理:** 前フレームの状態を保存し、更新ループ中の安全なオブジェクト間クエリを可能にする

**問題:** 更新ループ中はオブジェクト状態が不整合になり「1フレームずれ (one-frame-off lag)」バグが発生する

**解決策:** 前フレームの状態をキャッシュして参照用に保持しつつ、現フレームを更新する

#### 5. タイムスタンプ (Time-Stamping)

**原理:** 時間的整合性を検証し、同期エラーを防止する

---

### 並列化の考慮

| アプローチ | 説明 |
|-----------|------|
| 現状 | 多くのエンジンはシングルスレッドのオブジェクトモデルを維持 |
| 戦略 | 並列化は低レベルサブシステム（物理・レンダリング等）に集中させる |
| 非同期思考 | ブロッキングリクエストを非ブロッキングに変換してCPU待機時間を最大活用 |

---

### 設計への示唆

- Update() を単純化するほど個々のオブジェクトの責務は小さくなるが、エンジン全体の調整コストは上昇する
- バッチ更新・フェーズ更新・状態キャッシュを組み合わせることで、性能と正確性を同時に達成できる
- 更新順序の依存関係は明示的に文書化し、暗黙の実行順序への依存を避ける
