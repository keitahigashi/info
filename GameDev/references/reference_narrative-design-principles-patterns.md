## 核心命題

> 「ナラティブデザインは動詞と物語が交わる場所に生きる」

メカニクスが生み出す感情と情報ニーズを先にマッピングし、その後シーンを書く。ナラティブを後付けするアプローチとは逆順。

## 動詞中心アプローチ (Verb-Centered Design)

**何をプレイヤーが「する」かを最初に定義する。**

| メカニクス | ナラティブ要件 |
|-----------|--------------|
| ステルス | 短いフィードバックループ・読みやすい結果 |
| クラフト | 発見感・段階的なヒント |
| 戦闘 | 明確なスケート・テンポよい情報 |

→ システムの動詞から逆算してナラティブを設計する

## エージェンシーの2階層

| 階層 | 内容 | 実装コスト |
|------|------|-----------|
| **表現的エージェンシー (Expressive Agency)** | 結果の装飾的スタイリング | 低 |
| **分岐エージェンシー (Branching Agency)** | 状態を変える決断 | 高 |

**実践指針**: 2〜3の高インパクトポイントのみハードな結果を実装し、その他はコスメティックながら味わい深い表現にする。

## データハイジーン (Data Hygiene First)

コンテンツが増殖する前に安定した基盤を確立:
- 命名規則 (naming conventions)
- メタデータテンプレート
- エクスポートフォーマット

→ 後のローカライゼーションや統合フェーズでのコスト高騰を防ぐ

## 分岐システム設計パターン

### 実装の4原則

1. 「プレイヤーが**合理的な期間内に永続的な変化を知覚できる**」場合のみ分岐を使う
2. **段階的な結果**: コスメティックライン → ローカルシーン変更 → グローバルフラグ
3. 実装前に**状態図 (state diagram)** でナラティブロジックをモデル化 → 矛盾を早期発見
4. すべての分岐は**2シーン以内**に可観測なシグナルを持つ（変化した対話・変更された目標・修正された価格）

### 段階的結果テーブル

```
コスメティックライン → ローカルシーン変更 → グローバルフラグ
```

## クエストアーキテクチャ (Quest Architecture)

### 推奨構造

```
Hook（フック）→ Decision（決断）→ Complication（複雑化）→ Payoff（報酬）→ [Epilogue（後日談）]
```

### 依存関係の設計

- **トークンベースのアンロック**: 複数のクエストが要件を満たせる設計
- **リカバリーパス**: 失敗をリセットではなく「別目標へのリダイレクト」に変換
- クロスクエスト依存は最小化する

## 環境ストーリーテリング (Environmental Storytelling) の構造

空間はレイアウトリズムと空間シーケンスで語る:

| 空間形状 | ナラティブ効果 |
|---------|--------------|
| スパイラル形状 | 期待感の構築と視点変化 |
| 十字路 | 選択と危険の招待 |

**プロップの活用**: 意図的なシーケンスによる「ナラティブ文」

```
倒れた椅子 + こぼれた紅茶 + 未完成のメモ
  → モノローグよりも効果的なストーリーテリング
```

## モジュールビートによるペーシング (Modular Beat Pacing)

固定的な曲線ではなく**ゲームが再シーケンス可能なモジュールタイル**を推奨:

| ビートタイプ | 役割 |
|------------|------|
| Setup（セットアップ） | 状況説明・期待構築 |
| Escalation（エスカレーション） | 緊張増大 |
| Breath（ブレス） | 緊張緩和・消化時間 |
| Twist（ツイスト） | 期待の裏切り |
| Resolution（解決） | 締めくくり |

**ガイドライン**: 2〜3回のエスカレーションごとに必ず1回のリカバリービートを置く。

## 制作パイプライン整合

### 語彙の統一 (Vocabulary Alignment)
ライター・クエストデザイナー・システムチームが共通の用語で話す → 手戻りの削減。

### プロダクション優先事項
1. 1つの結果を証明する垂直スライス (vertical slice) から開始
2. シグナルがテスト可能になってから拡張
3. すべての選択が「2シーン以内に可観測」であることを QA で検証可能にする

### スコープコントロール
- 1チャプターあたりのグローバル変更をキャップ
- 収束ポイント (convergence points) を事前計画
- クロスクエスト依存を制限して分岐爆発を防ぐ

## テレメトリの活用

**データドリブン (data-driven) ではなくデータインフォームド (data-informed)**:
- テレメトリは仮説検証の情報源
- 創造的意図の指示者ではない
- 摩擦ポイントを発見するために使用し、デザインを機械的に置き換えない

## ローカライゼーション準備

早期からパイプライン要件を組み込む:
- 一意 ID
- 性別・複数形対応のメモ
- プレースホルダートークンフォーマット

→ 後からの遡及的修正を防ぎコストを削減
