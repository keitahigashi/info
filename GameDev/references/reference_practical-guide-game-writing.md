## ゲーム執筆の実践ガイド

著者: Darby McDevitt（Assassin's Creed: Bloodlines 等担当ゲームライター, 2010）

### 中心的な主張

ゲームライターは「準デザイナー (quasi-designer)」として扱われるべきであり、ナラティブはゲームプレイと同等に重要。ライターはデザインチーム全体の**継続的パートナー**として機能することが成功の鍵。

### 3段階ワークフロー

#### 1. プリプロダクション段階

| 作業 | 内容 |
|------|------|
| 高層ナラティブサマリー | 1〜4ページの概要。クライアント・ディレクターの承認を早期取得 |
| 主要ロケーション/レベル設計への参加 | 空間設計段階からナラティブを統合 |
| 世界観・キャラクター概要の策定 | 後続工程の変更コストを最小化 |

#### 2. プロダクション段階

| 作業 | 内容 |
|------|------|
| 詳細なストーリーアウトライン | シーン・クエスト・分岐の全体マップ |
| ストーリー提示計画 (Story Delivery Plan) | どのメディア（ダイアログ・環境・ドキュメント）で何を伝えるかの設計 |
| キャラクター一覧と関係図 | NPCの背景・動機・役割の体系化 |
| テキストデータベース整理 | 多言語対応と実装連携のための文字列管理 |

#### 3. 実装段階

| 作業 | 内容 |
|------|------|
| スクリプト第一稿完成 | ゲームプレイと整合した完全脚本 |
| ボイスオーバー録音管理 | ディレクション・リテイク判断・口語最適化 |
| 校正・非ダイアログテキスト修正 | UI・チュートリアル・ドキュメントの文言統一 |

### 失敗事例と教訓

**設定変更による不整合の例**: デザインチームが「黒い森」のロケーションを「Walmart」に変更した結果、ライターの書いたセリフ・環境ストーリーの大部分が無効化。

→ **教訓**: ライターをデザイン決定プロセスの外に置くと、後工程での修正コストが爆発する。ライターは設定・メカニクス変更の意思決定に参加すべき。

### 協働パートナーシップの原則

1. **早期関与**: プロトタイプ段階からライターをチームに参加させる
2. **双方向のコミュニケーション**: デザイナーは設計変更をライターに即時共有し、ライターはナラティブ制約をデザイナーに伝える
3. **ナラティブ優先度の可視化**: ストーリーが犠牲にできる要素と譲れない要素をリスト化し、チーム全員が把握できる状態にする
4. **「準デザイナー」としての権限**: ライターが実装上の懸念を伝えられる発言権を確保する

### ゲームと他メディアの違い

映画・小説のライターとは異なり、ゲームライターは「インタラクティブな制約の中で機能するナラティブ」を設計する必要がある。プレイヤーのアクション（移動・戦闘・スキップ）がいつでも起こり得ることを前提に、ナラティブは「どのタイミングで体験されても意味をなす」ように設計する。
