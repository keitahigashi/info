## ヒューリスティック評価（ヒューリスティック分析）の概要

**ヒューリスティック評価 (heuristic evaluation)** は、専門家が事前定義の使いやすさ原則（ヒューリスティクス）に照らしてUIを審査する手法。大規模ユーザーテストを実施する前の段階で、低コストかつ短期間にUI問題を特定できる。

### なぜゲーム開発でヒューリスティック評価が有効か

- 形式的なユーザーテストに比べて圧倒的に安価・迅速
- 開発反復の初期段階（プロトタイプ・アルファ段階）から適用可能
- 外部テスター招集なしに問題を発見できる
- 発見した問題を形式的テスト前に解消することで、テストの精度が上がる

## ニールセンの10の使いやすさ原則のゲームUI適用

| 原則 | ゲームUIへの適用例 |
|------|-----------------|
| **システム状態の可視性** | HP・弾薬・クールダウン等を常にプレイヤーに提示 |
| **現実との対応** | ゲーム世界のルールと整合した用語・メタファー使用 |
| **ユーザーコントロールと自由** | メニューのキャンセル、誤操作からの復帰手段 |
| **一貫性と標準** | UI要素の配置・色・動作の統一 |
| **エラー防止** | 不可逆なアクション（アイテム削除等）の確認ダイアログ |
| **記憶への依存を減らす** | ツールチップ、文脈ヘルプで情報を必要時に提示 |
| **柔軟性と効率** | 初心者向けガイドと上級者向けショートカットの両立 |
| **美的で最小限のデザイン** | 不要な情報を排除し、重要要素を際立たせる |
| **エラーの認識・診断・回復支援** | 明確なエラーメッセージと解決策の提示 |
| **ヘルプとドキュメント** | コンテキスト適切なヒント、チュートリアルとの連携 |

## ケーススタディ分析

### The Sharpest Blade — 問題点の指摘

- **フォント選択**: ゲームテーマと不整合なフォント使用（一貫性と標準の原則違反）
- **ナビゲーション構造**: メニュー階層が直感的でない（記憶への依存を増大）

### Hearthstone — 優れた実装例

| 観点 | 具体的評価 |
|------|----------|
| **ダイアログクローズ規約** | 一貫したESCキー/右クリックでのキャンセル（標準準拠） |
| **システムフィードバック** | ターン終了・カード効果の明確なアニメーション通知 |
| **検索機能** | カードコレクション内の効率的フィルタリング |
| **非対称ボード配置** | 相手フィールドを上部、自分フィールドを下部に固定し混乱防止 |
| **音声キュー** | 視覚のみに依存せず聴覚でも状態変化を伝達 |

## ヒューリスティック評価の実施手順

1. **評価者を選定**: 複数人（3〜5人）で独立して評価することで発見率が上がる
2. **評価項目の定義**: 適用するヒューリスティクスを事前に共有
3. **独立評価の実施**: 評価者がゲームをプレイしながら問題をリスト化
4. **問題の集約・重大度評価**: 発見問題を集約し、重大度（0〜4）を評価
5. **優先度付きの修正計画**: 重大度・修正コストに基づいて優先順位を決定

## 他の評価手法との使い分け

| 手法 | 適したタイミング | コスト | 発見できる問題 |
|------|--------------|--------|-------------|
| **ヒューリスティック評価** | 開発初期〜中期 | 低 | UI設計の体系的問題 |
| **認知的ウォークスルー** | 設計段階 | 低 | 学習フロー・タスク完了の問題 |
| **ユーザーテスト** | 中期〜後期 | 高 | 実際のプレイヤー行動・感情 |
| **A/Bテスト** | 後期・リリース後 | 中 | 最適なUI選択肢の特定 |

## 適用における注意点

- ヒューリスティック評価はユーザーテストの**代替ではなく補完**
- 評価者の専門性によって発見される問題の質が大きく変わる
- ゲームジャンル固有のヒューリスティクスを追加することで精度向上
- 発見された問題をすべて修正する必要はない（プレイ感・意図的な制約との兼ね合い）
