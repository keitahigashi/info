## SCRUM の基本哲学

> 「ゲーム開発に魔法の公式はない。試行→結果→学習のサイクルが重要」

SCRUM の最大の利点：
1. **チーム責任性 (accountability) の強化**: 各メンバーがプロジェクト全体に対して責任を持つ
2. **学習重視**: ゲーム自体よりも継続的な学習機会を優先

Thousand Gears が採用したのは **Scrum-ban**（Scrum + Kanban の組み合わせ）。

## ペルソナ (Personas) の活用

プレイヤー像を詳細に定義し、機能の優先度付けの基準にする：

| ペルソナ | 特徴 |
|---------|------|
| ニュービー | 初日プレイヤー；チュートリアルが重要 |
| カジュアル | 通常の継続率・低エンゲージメント |
| アドバンス | 高継続率・高エンゲージメント；深い機能を求める |

## プレイヤーストーリー (Player Story) フォーマット

```
[対象者] として、[要望] がしたい。なぜなら [目的] だから。
KPI: [測定指標]
```

例：
- ニュービーとして、タクシーが自動移動してほしい。なぜならゲーム操作に集中したいから。KPI: エンゲージメント
- カジュアルとして、獲得額を表示してほしい。なぜなら達成感を実感したいから。KPI: エンゲージメント

## カンバン (Kanban) の4段階

```
To Do → Doing → Test → Done
```

- **Done の明確な定義が必須**: 「完成」の基準を事前に合意しないとタスクが終わらない問題が発生

## スプリント管理

- **期間**: 1週間の短期反復
- **推定精度**: チームは計画の約 60% を正確に推定可能（過信しない）
- **スプリントレビュー**: 完了したストーリーをステークホルダーに見せる機会

## マイルストーン構成

1. **プリプロトタイプ**: 基本ループの実装
2. **プロトタイプ**: 技術・ツール選定
3. **デモ**: レベル・アート制作
4. **アルファ版**: コンテンツ改善・バグ修正
5. **ベータ版**: 最終バグ修正

## ドキュメント戦略：GDD (Game Design Diary) 方式

- 大量文書作成を避ける
- スプリントごとに軽量な章を更新
- 変更履歴を透明に保つ
- アジャイル開発では「生きたドキュメント」が必要

## SCRUM 導入時の注意点

> 「SCRUM はメソッドではなく哲学。実行に先立ちベストプラクティスを研究し、チームの文化に適応させることが肝要」

- 標準化ではなくチームへの最適化を目指す
- 小チームでは全員が SCRUM の価値観（透明性・検査・適応）を理解する必要がある
- 経験豊富なスクラムマスターがいない場合は段階的に導入する
