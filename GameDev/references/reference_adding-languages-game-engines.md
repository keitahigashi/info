## 概要

ゲームエンジンにカスタムスクリプト言語を組み込む実践的手法を、COG（Jedi Knight: Dark Forces IIで使用されたスクリプト言語）の設計事例を通じて解説する。「デザイナーが自律的に実装できる環境を作る」ことが主目的。

## なぜカスタムスクリプト言語か

| 課題 | スクリプト言語による解決 |
|------|----------------------|
| プログラマがデザイン変更ごとにビルドが必要 | デザイナーが直接スクリプトを書ける |
| C++の誤りがエンジン全体をクラッシュさせる | VM隔離でスクリプトのミスを局所化 |
| 変更のたびに再コンパイル・再起動が必要 | ホットリロードで即時確認 |
| ネットワーク同期の検証が困難 | アクセス制御で同期可能な操作に限定 |

## 設計方針（COGの場合）

### 構文の選択
- **C風の構文**を採用: 「複雑だが非プログラマも学習可能」なバランス点
- シンプルすぎると表現力不足、複雑すぎるとデザイナーが使えない
- 既存プログラミング知識の転用可能性を重視

### イベント駆動アーキテクチャ
- 各スクリプトが**独立した状態**（変数・スタック・実行ポインタ）を持つ
- ゲームイベント（「スイッチが押された」「プレイヤーが接触した」等）をトリガーにコードが実行される
- 複数スクリプトが並列して自分の状態を管理

## VM（仮想マシン）実装

### コンパイル戦略
インタプリタとコンパイル済みコードの中間解として**スタックマシン (stack machine) オペコード**へのコンパイルを採用:

- **インタプリタより高速**: ソースコードを毎回パースしない
- **ネイティブより柔軟**: VM命令セットはエンジンと独立して設計できる
- **デバッグ容易**: オペコードレベルで実行トレースが可能

### パーサ生成ツールの活用
- **lex**（字句解析器）と **yacc**（構文解析器）を使ってパーサを自動生成
- 言語仕様の変更をlex/yaccルールの変更だけで対応できる
- 手書きパーサより開発速度が大幅に向上

## アクセス制御の設計

**スクリプトから直接メモリ操作を禁止**し、すべてのエンジン操作をライブラリ関数呼び出しに限定する:

```
スクリプト → [ライブラリ関数] → エンジン内部
                    ↑
              ここで検証・ログ・同期処理
```

**メリット:**
- スクリプトのバグがエンジンを破壊しない
- 全操作をログに記録できる（デバッグ・リプレイ）
- ネットワーク同期が必要な操作を一元管理
- チート検出・セキュリティ検証の実装ポイントを集約

## 組織的・プロセス的な効果

スクリプティングシステムの本質的価値は技術的優雅さより**チーム構造の変革**にある:

- デザイナーがプログラマを経由せずにゲームロジックを実装できる
- 「ゲームプレイ上の問題」と「エンジン上の問題」が分離され、デバッグしやすくなる
- 複雑なプロジェクトでプログラマのボトルネックを軽減し、並列作業を可能にする

## 汎用化ポイント

エンジン・言語を問わず適用できる原則:
- VM分離 + アクセス制御パターンはスクリプティング層の設計全般に適用可能
- 「デザイナーが自律的に作業できる」ことがゲーム開発チームの生産性の鍵
- lex/yaccの活用はDSL（ドメイン固有言語）設計一般の標準的アプローチ
