## GOAPの起源と背景

- Jeff Orkin（Monolith Productions AIリード）が、1971年のSTRIPS計画をゲームAIに適応
- F.E.A.R.（2005）で商用ゲーム初のGOAP実装
- 従来のFSM（Half-Lifeは80以上の状態を列挙）の複雑性問題を解決

## 自動計画の基盤

| 要素 | 説明 |
|------|------|
| Facts/Predicates | ゲーム世界の情報（ドアの開閉、キャラ位置） |
| State | ある時点で真である述語の集合 |
| Actions | 関連オブジェクト + 前提条件 + 効果の3要素で定義 |

計画の本質: 「何をすべきか、どの順序で行うか」を自動決定

## 3状態FSMアーキテクチャ

GOAPは**わずか3状態**のFSMで全行動を表現:

1. **GoTo**: 特定位置への移動アニメーション（歩行/走行/ジャンプ/登攀）
2. **Animate**: キャラクター表現のアニメーション再生（射撃/アイドル/合図）
3. **Use Smart Object**: ワールドオブジェクトとのインタラクション（ドア開閉/スイッチ/着席）

### 核心的洞察

知的なNPC行動は「適切なアニメーションを、適切なタイミングで、正しい順序で再生する」ことから**創発**する。計画システムは「どの」アニメーションを「いつ」実行するかを決定し、ゲームエンジンが実行詳細を処理する。

## Smart Objects

- ワールド内のオブジェクトがキャラクターのアニメーションシステムに「何をすべきか」を伝達
- 環境デザインとキャラクター行動の**密結合**を実現
- 明示的な状態列挙なしに洗練されたインタラクションが可能

## 設計上の利点

| 特性 | 説明 |
|------|------|
| データ駆動 | 述語とアクション効果でハードコード不要 |
| スケーラブル | 新アクション追加時にコアFSM修正不要 |
| 柔軟 | 計画システムがゴール到達のアクション列を自動発見 |
| 保守容易 | 従来のBT/巨大FSMより単純な状態構造 |

## 業界への影響

F.E.A.R.のGOAP実装はNPC AI開発のアプローチを根本的に変え、その後数十年の業界慣行に影響を与えた。ロボティクス、風力タービン制御、発電所管理、軍事ロジスティクスでも計画技術は1970年代から活用されており、ゲーム以外での堅牢性が実証されている。
