## 概要

Stephen Trinh が提唱する、ゲームメカニクスを技術的に分析・文書化するための4要素フレームワーク。MDA フレームワーク（Mechanics-Dynamics-Aesthetics）の「M」をより実装レベルで分解したアプローチ。設計ドキュメント作成ツールとして位置づけられる。

## 4 要素の定義

### 1. 数量 (Quantity)

ゲーム内の計測・追跡可能な値。

| 例 | 説明 |
|----|------|
| HP | キャラクターの生命力 |
| スコア | プレイヤーの点数 |
| 弾薬数 | 残弾 |
| 通貨 | ゲーム内の経済単位 |

- 増減のルールと上下限が設計ポイント
- 「どの数量が存在するか」が次の 3 要素の動作条件になる

### 2. 空間 (Spatial)

オブジェクトの位置・衝突判定・移動範囲に関わる要素。

| 例 | 説明 |
|----|------|
| 座標系 | 2D/3D、グリッド/連続 |
| 衝突判定 | キャラクターと壁・敵の重なり検出 |
| 移動域 | キャラクターが移動可能な空間 |
| 距離 | 射程・索敵範囲の計算基盤 |

- ゲームの物理的ルールをすべて含む
- アクションゲームでは最も直接的な体験層を担う

### 3. 状態 (State)

キャラクターやシステムの条件変化を管理する要素。

| 例 | 説明 |
|----|------|
| ステータス効果 | 毒・無敵・スタン |
| モード切替 | 地上/水中、昼/夜 |
| ゲームフェーズ | 探索/戦闘/イベント |
| フラグ | ドアを開けた/クエストを受注済み |

- ゲームの「現在の条件」を定義する
- 状態遷移の設計が複雑なシステムの核心

### 4. アクション (Action)

プレイヤーや機械的プロセスが起こす変化。

| 例 | 説明 |
|----|------|
| ジャンプ | 空間要素を操作するアクション |
| 攻撃 | 数量（HP）と状態を変化させるアクション |
| アイテム使用 | 数量を消費して状態・数量を変化させる |
| 移動 | 空間要素の継続的変化 |

- プレイヤーが「できること」はすべてここに分類される
- 4 要素の中で唯一プレイヤーが直接操作できる要素

## マリオの体力システムによる事例分析

スーパーマリオの「大きい（Super）/小さい（Small）マリオ」システム：

| 要素 | 適用 |
|------|------|
| **数量** | — （このシステムには数値 HP なし） |
| **空間** | 大マリオは小マリオより大きい当たり判定 |
| **状態** | Super/Small の 2 状態 |
| **アクション** | マッシュルーム取得でSmall→Super、被弾でSuper→Small |

シンプルな 2 状態システムでも 4 要素が明確に機能していることが分かる。

## 設計文書への応用

4 要素フレームワークはゲームデザインドキュメント（GDD）に活用できる：

```
メカニクス名：[名前]
- 数量要素：[関与するパラメータ]
- 空間要素：[位置・当たり判定仕様]
- 状態要素：[状態一覧と遷移条件]
- アクション要素：[プレイヤーアクション・トリガー]
```

既存メカニクスの分析ツールとしても、新メカニクスの設計仕様書としても機能する。
