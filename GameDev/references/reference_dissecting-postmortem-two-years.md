## 出典

Game Developer (Ara Shirinian, 2011): https://www.gamedeveloper.com/audio/dissecting-the-postmortem-lessons-learned-from-two-years-of-game-development-self-reportage

## 研究概要

2008年2月〜2010年1月にGame Developer誌に掲載されたポストモーテム24件を横断分析。240個のデータポイント（成功事例120・失敗事例120）を収集し、ゲーム開発で繰り返される失敗パターンを定量化した。

### 調査対象プロジェクトのプロフィール

- チームサイズ: 5人〜265人（4クラスターに分布）
- 平均開発期間: 2.4年
- マルチプラットフォーム割合: 全体の約1/3

## 主要な発見

### 成功・失敗要因の分布比較

| カテゴリ | 成功時の言及率 | 失敗時の言及率 |
|---------|------------|------------|
| プロダクション / プロセス | 38% | 57% |
| デザイン | 19% | - |
| テスティング | - | 増加 |
| 外部要因 | - | 増加 |

→ 失敗時にプロダクション問題が突出して増加。**「開発プロセスの管理が最も重要な要素」** であることが数値で裏付けられた。

### 一般的傾向（24プロジェクト横断）

| 指標 | 該当率 |
|-----|-------|
| スコープ問題の報告 | **71%** |
| プロトタイピングを有効と評価 | **75%** |
| 後期機能追加（スコープクリープ） | 50% |
| チーム管理問題 | 50% |
| 納期延長を経験 | 38% |
| クランチ（過密労働）を報告 | 38% |
| 通信障害（Communication breakdown） | 38% |
| アジャイル / Scrum 導入 | 21% |

### プロダクション問題の内訳

- スコープ・フィーチャークリープ（Feature Creep）: 問題の23%
- チーム間コミュニケーション障害: 10%以上

### アセットパイプライン問題

9件のポストモーテムが明示的にパイプライン問題を報告:
- パイプラインが作業を過度に複雑・時間のかかるものにした
- プロジェクト中期になってもパイプラインが未稼働だった
- チームの実際の作業フローを適切にサポートしていなかった

## 失敗の根本原因

**「スコープと日程の過度な楽観主義」**: 開発チームは一貫して必要な時間とリソースを過小評価する傾向。71%のプロジェクトがスコープ問題を報告した事実がこれを裏付ける。

### アウトソーシング（Outsourcing）のリスク

- 7件中4件（57%）がアウトソーシング関連の問題を報告
- 主な問題: パイプライン統合の失敗・受注先の準備不足
- 計画段階でのパイプライン設計とコミュニケーション体制の整備が必要

## 成功要因

1. **柔軟な計画**: 75%がイテレーション・ラピッドプロトタイピング（Rapid Prototyping）を有効と評価
2. **能力に合わせたスコープ設定**: 意図的にチーム能力に合わせた企画を実施したプロジェクトはほぼ2年以内に完了
3. **良好なパブリッシャー関係**: 良好な関係があるケースで40%が肯定的な外部要因として記載
4. **適応的な設計**: 硬直した設計方針（変更不可能な仕様）はプロジェクト長期化のリスクを高める

## 結論

> 「良好な管理は他の不十分さを補うことができるが、悪い管理は優れた創造的作品さえも破滅させる可能性がある」

プロダクション管理の優先度はデザインやアートよりも上位に置くべき。定量データが示す最大の教訓は、**スコープ管理と反復プロセスの導入**が成功率を最も高める要因であること。
