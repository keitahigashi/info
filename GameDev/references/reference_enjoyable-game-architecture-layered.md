## 3層アーキテクチャ

- **Visualレイヤー**: ゲームエンジンコンポーネント（MonoBehaviour/Godotノード）。表示のみを担当する「薄い層」。入力を状態機械に転送
- **GameLogicレイヤー（2分割）**:
  - Visual Game Logic: 状態機械・ビヘイビアツリーでコンポーネント制御
  - Pure Game Logic: ドメインルールを担当するRepositoryクラス（ゲームオーバー判定等）
- **Dataレイヤー**: ネットワーク・ファイルI/O・永続化ストレージ

**制約**: 各オブジェクトは直下の層とのみやり取りし、「層スキップ」を禁止。

## データフロー（コイン収集の例）

1. 衝突検知 → 状態機械に通知
2. 状態機械 → アニメーション再生 + Repository通知
3. Repository → `CoinCollected`イベント発火
4. UI・ゲームオーバーシステムがイベントを購読して反応

## 階層型状態機械（Hierarchical State Machine）

- コードからUML図を自動生成
- 入力/出力をビジュアル層と疎結合に保ち単独テスト可能
- ネストされた状態のenter/exitコールバック

## ツリーベースの依存性注入（Dependency Injection）

- エンジンのノードツリー構造を活用してDIを解決
- 親ノードが依存性を提供し、子孫ノードが取得
- グローバル状態・サービスロケーターを回避

## 2フェーズ初期化

Setup（状態機械・Repositoryの生成）→ OnResolved（バインドとシステム起動）に分離。テスト時のモック注入を可能にする。

## 機能単位のファイル構成（Feature-Based Structure）

`src/coin/`配下にコンポーネント・シーン・状態機械・オーディオをまとめるフィーチャーベース構成。型別分類でなく機能別分類でチームの認知負荷を下げる。

## テスト戦略

単体テストの主目的は「動作検証」より「アーキテクチャ整合性の強制」。境界違反はテスト失敗として現れる。100%コードカバレッジを目標とする。

## 汎用化ポイント

3層アーキテクチャ（Visual/GameLogic/Data）・階層型状態機械・2フェーズ初期化・フィーチャーベースファイル構成はGodot以外のUnity・Bevy・自作エンジンにも汎用適用可能。DIコンテナの実装方式はエンジン固有だが、「親が依存を提供し子が取得する」設計思想はノードツリー構造を持つあらゆるエンジンで応用できる。
