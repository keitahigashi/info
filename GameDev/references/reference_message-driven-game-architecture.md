## 従来方式の問題点

直接メソッド呼び出しアプローチ（例: `GameInformation.AddScore(10)`）では「依存度の高さ」が課題。呼び出し元がクラス定義やメソッド内容に依存し、結合度 (coupling) が高まる。

## メッセージベース駆動の基本

従来の直接メソッド呼び出しではなく、メッセージをキューイングしてシステムが統一的に処理する手法。

## 主要な利点

### 1. 低依存度 (Low Coupling)

「処理をコールする側は、コール先のクラス構成やその内容に依存していない」。メッセージ形式のみを定めれば、実装詳細は隠蔽される。

### 2. 受信意思表明パターン (Subscription Pattern)

モジュール側が「こういうメッセージが来たら配信してくれ」と宣言。送信側は具体的な受取先を指定しない。オブザーバーパターン (Observer Pattern) の応用。

### 3. ブロードキャスト配信 (Broadcast)

単一メッセージで複数モジュールに配信可能。例: 「敵全滅」メッセージを送ると、スコア計算・UI更新・サウンド再生がそれぞれ独立に反応。

### 4. スレッド・ネットワーク透過性

同一プロセス内、スレッド間、ネットワーク越しの配信を透過的に処理可能。マルチプレイ対応やロックレスマルチスレッディングの基盤。

## 実装効果

- 大規模チーム開発でのイテレーション速度維持
- モジュール間の独立性確保（チーム間の依存関係低減）
- マルチプレイ対応の基盤
- テスタビリティの向上

## 汎用化ポイント

本記事はCygamesの実プロダクション事例に基づくが、メッセージベースアーキテクチャの原則（Pub/Sub、ブロードキャスト、疎結合）はエンジン非依存の汎用設計パターンとして適用可能。
