# Article Collector Skill (GameDev)

ゲーム開発ナレッジベースへの記事自動収集ワークフロー。

## トリガー

以下のいずれかでこのワークフローを開始する:
- 「記事収集して」「ゲーム開発の記事探して」「集積して」「続けて」
- `/loop` でのスケジュール実行
- URL直接指定

## 制約

- **1回の実行で最大5記事**
- **収集済みURLへの重複追加禁止**
- ナレッジベース（00-09）への追記は所定フォーマットに従う
- ログは必ず `log/{YYYY-MM-DD}.md` に作成する
- **エンジン固有の記事は汎用パターンとして抽象化可能な場合のみ採用**
- **英語の専門用語は原語を併記する**（例: "ジュース (juice)"）

## ワークフロー

### Phase 1: 重複チェック準備

1. `REFERENCES.md` を読み込み、収集済みファイル名の一覧を取得する
2. これを重複判定のソースとして使用する（ファイル名の一致で判定）

### Phase 2: 記事検索

1. Web検索で3キーワードを**並列実行**する
   - **うち少なくとも1つは英語キーワード**を含める
   - 前回の収集ログ（`log/` 内の最新ファイル）を確認し、同じキーワード・カテゴリの繰り返しを避ける
   - 8カテゴリからローテーション選出する（下記「検索キーワード戦略」参照）
2. 検索結果からPhase 1のURL一覧で重複を除外する
3. 残った候補から最大5件を選定する

### Phase 3: 記事取得・詳細保存

各記事について:
1. `WebFetch` で記事本文を取得する
2. `references/reference_{slug}.md` に以下のフォーマットで保存する:

```markdown
---
name: {記事タイトル}
description: {1行要約}
type: reference
---

## 出典

{ソース名}: {URL}

## {記事内容の構造化要約}

（見出し・箇条書き・テーブル・コード例を含む）
（英語記事の場合は日本語で要約を記述する）
```

3. エンジン固有の記事を採用した場合は「汎用化ポイント」セクションを追記する

### Phase 4: メタデータ更新

1. `REFERENCES.md` の「収集済み記事」テーブルに行を追加する:
   ```
   | reference_{slug}.md | {YYYY-MM-DD} |
   ```

2. `log/{YYYY-MM-DD}.md` にデイリーログを作成/追記する:
   ```markdown
   # {YYYY-MM-DD} 収集ログ

   収集件数: {N}件

   ## 検索キーワード
   - 「キーワード1」
   - 「キーワード2」
   - 「キーワード3」

   ## 収集記事

   ### 1. {タイトル}
   - **URL**: {url}
   - **ソース**: {source}
   - **反映先**: {target}
   - **サマリー**: {2-3行の要約}
   ```

### Phase 5: 完了

- ファイル更新まで実行し、commit/push はユーザー確認を待つ
- **ログにPR情報（URL・番号等）を記載しない**（ブランチマージ後のログ更新で再マージが必要になるため）

## 検索キーワード戦略

### 基本キーワード（8カテゴリ・ローテーション）

| カテゴリ | キーワード例（日本語） | キーワード例（英語） |
|---------|---------------------|---------------------|
| デザイン理論 | `ゲームデザイン 理論 2026`, `MDA フレームワーク ゲーム` | `game design theory GDC 2026` |
| メカニクス | `ゲームメカニクス バランス調整`, `コアループ 設計` | `game mechanics design balance` |
| レベルデザイン | `レベルデザイン 手法 実践`, `空間設計 ゲーム` | `level design methodology 2026` |
| ナラティブ | `ゲーム ナラティブデザイン 分岐`, `インタラクティブ ストーリー` | `narrative design branching dialogue` |
| UI/UX | `ゲーム UI デザイン`, `ゲーム UX アクセシビリティ` | `game UI UX design accessibility` |
| プログラミング | `ゲーム プログラミングパターン ECS`, `ゲーム アーキテクチャ` | `game programming patterns 2026` |
| 技術システム | `ゲーム AI パスファインディング`, `手続き生成 プロシージャル` | `procedural generation game dev` |
| プロダクション | `ゲーム プロトタイピング プレイテスト`, `ゲーム開発 イテレーション` | `game prototyping playtest iteration` |

### ソース優先度

1. GDC Vault / Game Developers Conference 講演
2. Gamasutra (Game Developer)
3. 80 Level
4. Qiita / Zenn のゲーム開発タグ
5. 開発者ブログ（ポストモーテム等）

### キーワード選定ルール

- 直近3回のログで使ったキーワード・カテゴリは避ける
- 収集が少ないカテゴリを優先する
- 時事性の高いトピック（GDC講演、大型タイトルのポストモーテム等）があれば優先する

## エンジン固有記事の取り扱い

Unity / Unreal / Godot 等の記事でも、設計パターンやアプローチがエンジン非依存で応用可能なら採用する。

**採用OK例:**
- Unity の ECS 実装解説 → 汎用的な ECS パターンとして抽出可能
- Unreal の Behavior Tree 設計 → ゲームAI設計の汎用知見として活用可能

**採用NG例:**
- Unity エディタの特定ウィンドウの使い方 → エンジン固有すぎる
- Unreal Blueprint の具体的な操作手順 → 他エンジンに応用不可

## ファイル構成

```
GameDev/
├── references/          ← 記事詳細（Git管理）
│   └── reference_*.md
├── REFERENCES.md        ← 収集済み記事テーブル（重複チェックの正）
├── log/                 ← デイリー収集ログ
│   └── YYYY-MM-DD.md
└── 00-09               ← ナレッジベース本体
```
