# Article Collector Skill (GameDev)

ゲーム開発ナレッジベースへの記事自動収集ワークフロー。

## トリガー

以下のいずれかでこのワークフローを開始する:
- 「記事収集して」「ゲーム開発の記事探して」「集積して」「続けて」
- `/loop` でのスケジュール実行
- URL直接指定

## 制約

- **1回の実行で収集する記事数は厳密に5件**（多くも少なくもしない）
- **収集済みURLへの重複追加禁止**
- ナレッジベース（00-09）への追記は所定フォーマットに従う
- ログは必ず `log/{YYYY-MM-DD}.md` に作成する
- **エンジン固有の記事は汎用パターンとして抽象化可能な場合のみ採用**
- **英語の専門用語は原語を併記する**（例: "ジュース (juice)"）
- **コンテキスト制限を理由に途中で中断しない**。本当に制限に近い場合は、現時点までの作業をコミットした上でユーザーに状況を伝える。勝手な早期中断は禁止

## ワークフロー

### Phase 1: 重複チェック準備

1. `REFERENCES.md` を読み込み、収集済みファイル名の一覧を取得する
2. これを重複判定のソースとして使用する（ファイル名の一致で判定）

### Phase 2: 記事検索・取得（サブエージェント委譲）

Web検索と記事本文の取得・要約は **Explore サブエージェントに委譲する**（メインのコンテキストに検索結果・記事全文を載せず、メインは指揮官役に専念するため）。

サブエージェントへの指示に含めること:
- Phase 1 で取得した収集済みファイル名一覧（重複回避用）
- 検索キーワード3つ（下記「検索キーワード戦略」参照。**うち少なくとも1つは英語キーワード**。8カテゴリからローテーション選出し、前回ログと重複させない）
- 抽出して返すべき情報（記事タイトル / URL / ソース名 / 1行要約 / 構造化要約 / 反映先候補(00-09) / 汎用化ポイント）

サブエージェントに委譲する作業:
1. 3キーワードを**並列**でWeb検索する
2. 渡された収集済みファイル名一覧で重複候補を除外する
3. 残った候補から最大5件を選定し、各記事を `WebFetch` で取得する
4. 各記事の構造化要約を抽出して返す（英語記事は日本語で要約する）
5. エンジン固有の記事は「汎用パターンとして抽象化可能か」を判定し、可能なら汎用化ポイントを併記する

メインはサブエージェントが返した候補リストを受け取り、採否を最終判断する（エンジン固有すぎるものは除外）。

### Phase 3: 記事詳細の保存

サブエージェントが返した採用記事ごとに、メインが `references/reference_{slug}.md` を以下のフォーマットで作成する:

```markdown
---
name: {記事タイトル}
description: {1行要約}
type: reference
---

## 出典

{ソース名}: {URL}

## {記事内容の構造化要約}

（見出し・箇条書き・テーブル・コード例を含む）
（英語記事の場合は日本語で要約を記述する）
```

3. エンジン固有の記事を採用した場合は「汎用化ポイント」セクションを追記する

### Phase 4: メタデータ更新

1. `REFERENCES.md` の「収集済み記事」テーブルに行を追加する:
   ```
   | reference_{slug}.md | {YYYY-MM-DD} |
   ```

2. `log/{YYYY-MM-DD}.md` にデイリーログを作成/追記する:
   ```markdown
   # {YYYY-MM-DD} 収集ログ

   収集件数: {N}件

   ## 検索キーワード
   - 「キーワード1」
   - 「キーワード2」
   - 「キーワード3」

   ## 収集記事

   ### 1. {タイトル}
   - **URL**: {url}
   - **ソース**: {source}
   - **反映先**: {target}
   - **サマリー**: {2-3行の要約}
   ```

### Phase 5: 完了

- ファイル更新まで実行し、commit/push はユーザー確認を待つ
- **ログにPR情報（URL・番号等）を記載しない**
- **`## Pull Request` ブロックをログに追記しない**（PR URL・番号・ラベル等を含む）
- **コミット/PR作成後にログファイルへの追記は一切行わない**
- 理由: PRマージ後のブランチへの追記はmainに反映されず、不要な差分を生む

### 失敗時の挙動

- 5件揃わない場合や WebFetch が失敗した場合は、その旨をログに記録し、取得できた分だけコミットする（中途半端な状態で放置しない）
- ユーザーから「次のラウンド」を依頼された場合は、Phase 1 から再実行する

## 出力フォーマット

実行完了時、以下を表示する:
- 今回の検索キーワード
- 追加した記事タイトル（5件）
- 累計記事数
- 次回避けるべきキーワード・カテゴリ

## 検索キーワード戦略

### ソース優先度

1. GDC Vault / Game Developers Conference 講演
2. Gamasutra (Game Developer)
3. 80 Level
4. Qiita / Zenn のゲーム開発タグ
5. 開発者ブログ（ポストモーテム等）

### キーワード選定ルール

- 直近3回のログで使ったキーワード・カテゴリは避ける
- 収集が少ないカテゴリを優先する
- 時事性の高いトピック（GDC講演、大型タイトルのポストモーテム等）があれば優先する

## エンジン固有記事の取り扱い

Unity / Unreal / Godot 等の記事でも、設計パターンやアプローチがエンジン非依存で応用可能なら採用する。

**採用OK例:**
- Unity の ECS 実装解説 → 汎用的な ECS パターンとして抽出可能
- Unreal の Behavior Tree 設計 → ゲームAI設計の汎用知見として活用可能

**採用NG例:**
- Unity エディタの特定ウィンドウの使い方 → エンジン固有すぎる
- Unreal Blueprint の具体的な操作手順 → 他エンジンに応用不可

## ファイル構成

```
GameDev/
├── references/          ← 記事詳細（Git管理）
│   └── reference_*.md
├── REFERENCES.md        ← 収集済み記事テーブル（重複チェックの正）
├── log/                 ← デイリー収集ログ
│   └── YYYY-MM-DD.md
└── 00-09               ← ナレッジベース本体
```
