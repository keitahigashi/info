## 空間設計

空間設計はレベルデザインの基盤であり、プレイヤーの行動・感情・意思決定を物理的な環境構成によって制御する技術である。ゲームプレイメカニクスや難易度調整だけでなく、**建築的選択そのものがプレイヤー体験を形成する**という認識が重要となる。

### 空間構成 (Spatial Composition) の基本

空間構成には2つのアプローチがあるが、レベルデザインではとくに前者が重視される。

| アプローチ | 定義 | 重要度 |
|-----------|------|--------|
| **空間構成 (Spatial Composition)** | 複数の視点を考慮した3Dマスの全体配置 | **高** — レベル全体の体験を決定 |
| **ショット構成 (Shot Composition)** | 特定カメラ位置からのレベルの見え方 | 低 — スクリーンショット映えと混同しやすい |

ショット構成に依存した「リーディングライン (Leading Lines)」は、レベルデザインとしては**無効**とする見解がある。理由は、プレイヤーはスクリーンショットの無意識的視覚手がかりではなく、**パターン認識とメンタルマッピング**でナビゲートするためである。「スクリーンショットはレベルではない。レベルはプレイヤーが使う空間」という指摘は核心を突いている。

### コントラストによる空間階層

異なるエリア間に重要度の感覚を生み出すには、**コントラスト**が鍵となる。

| コントラスト手法 | 説明 | 効果 |
|----------------|------|------|
| **高さの変化** | 密集空間内の高い構造物、開放空間内の小さなオブジェクト | 垂直方向の視線誘導 |
| **密度の差異** | 開放空間と密集した狭い構造の対比 | エリアの性格づけ |
| **方向の断絶** | 周囲のグリッドパターンを破る角度のある要素 | 注目ポイントの生成 |
| **形状の区別** | 矩形マスの中に円形を導入 | 記憶に残る焦点の創出 |

焦点 (Focal Point) は「空間構成内に新しい中心を生む」。空間の幾何学的中心から外れていても、コントラストがあれば重要に感じられる。

#### コントラスト設計の実践例

- **ダンジョンの宝箱部屋**: 狭い通路（高密度）の連続後に広い空間（低密度）を配置 → 密度コントラストで「ここが重要」と感じさせる
- **城の玉座の間**: 周囲の矩形回廊に対して円形ドームを導入 → 形状コントラストで権威を伝達
- **塔のランドマーク**: 平坦な地形に突出した垂直要素 → 高さコントラストでナビゲーションの目印を生成

### ランドマーク (Landmark)

「レベル内のユニークで記憶に残る形状、マス、または場所」。単なる装飾的なスカイボックス要素ではなく、プレイヤーの**方向感覚とナビゲーションに機能的に関連**する必要がある。ランドマークが機能するためには、周囲の環境とのコントラストが不可欠である。

### サイトラインとビスタ (Sightline / Vista)

- **サイトライン (Sightline)**: 別の空間の視界を提供する開放空間の軌跡。状況的ツールとして、プレイヤーがゲームプレイ目標に関連する時に使用される — ただし保証された行動指示ではない
- **ビスタ (Vista)**: 戦略的概要を提供する深いシーン構成。高台から次の目的地を俯瞰させる「ご褒美の景色」は、探索のモチベーションと方向の確認を同時に果たす
- **アプローチ (Approach)**: ビスタを含む移動可能なパス。プレイヤーが目的地に向かう過程で徐々に情報を得る「段階的開示」の空間的実装

サイトラインの設計で重要なのは、**見えるけれどすぐには到達できない**という距離感の制御である。目的地が見えることで方向感覚を維持しつつ、到達までの道程が探索意欲を維持する。

### Prospect-Refuge 理論 — 空間構成によるペーシング制御

環境心理学の **Prospect-Refuge 理論** をレベルデザインに応用したフレームワークがある。空間の開放度/閉鎖度がプレイヤーのペーシング（テンポ）を直接制御するという研究成果に基づく。

| 空間タイプ | 特徴 | ペーシング効果 | 適用場面 |
|-----------|------|--------------|---------|
| **Prospect 空間** | 開放的なエリア | **加速** — 動きの感覚と緊迫感を高める | 戦闘エリア、チェイスシーケンス、タイムプレッシャー場面 |
| **Refuge 空間** | 囲まれた/保護されたエリア | **減速** — 慎重な動きと探索を促す | セーフルーム、探索エリア、ナラティブポイント |

この理論の実践的意義は、**戦闘難易度や敵配置に頼らず、建築的選択だけでプレイヤーのテンポを操作できる**点にある。メカニカルな難易度調整と空間構成によるペーシング操作を組み合わせることで、より多層的な体験設計が可能になる。

#### 交互配置の設計パターン

Prospect空間とRefuge空間は単独で機能するのではなく、**交互に配置されることでリズムを生む**。典型的なパターン:

1. **Refuge（安全地帯で情報収集）** → **Prospect（開放的な戦闘エリア）** → **Refuge（休息と報酬）**
2. **Prospect（広大なフィールドを横断）** → **Refuge（洞窟内のパズル）** → **Prospect（ボスアリーナ）**

この交互配置はペーシングセクションの「ダイナミクスを交互に」という原則と直接対応しており、空間設計とペーシングが不可分であることを示している。

### 垂直性 (Verticality)

垂直性はゲーム空間内での上下移動の流れと感覚であり、ナビゲーションと心理的目的の両方を果たす。水平方向の抽象的な方向よりも、**上下は直感的に進行を理解させる**。

#### フロアプレーンとチャンキング

高さの変化を **フロアプレーン（階層面）** にグループ化し、細かい標高変化を単一の面に統合する。

- **最大3層ルール**: 底 (Bottom)・中 (Middle)・上 (Top) の構造が最適
- 「4層目は新しいダイナミクスを追加しない。単にもう1つの中間層を増やすだけ」

#### 方向別フローメカニクス

| 方向 | 特性 | 設計上の活用 |
|------|------|------------|
| **下降フロー** | 重力が自然に促進。一方通行の落下が戦略ツール | バックトラック防止、前進の強制。パズル（認知負荷の制限）やシューター（戦闘トリガー）に不可欠 |
| **上昇フロー** | 階段・スロープ・梯子・ジャンプメカニクスによる意図的な設計が必要 | Quake 3のジャンプパッドが空中戦闘を生成、狙撃ポジションがカウンタープレイを提供 |

#### ジャンル別垂直性設計

| ジャンル | 垂直性のアプローチ |
|---------|-----------------|
| **競技シューター** (CS系) | 垂直性を最小化し照準精度を維持。スタッキングで連携を報酬 |
| **コンソールシューター** (Halo/CoD系) | 高い高低差を回避（ゲームパッドの垂直エイムが困難）。浅い傾斜と平面を優先 |
| **アリーナゲーム** | 限定的で慎重な垂直性の適用 |

設計原則はコンテキスト依存であり、過度な垂直変化は特定のプレイヤー層をフラストレーションさせるが、他の層では魅力的なゲームプレイを可能にする。

#### 垂直性とナラティブの関係

垂直方向の移動には暗黙のナラティブが伴う。**上昇は達成・進歩・啓示**を、**下降は危険・秘密・堕落**を連想させやすい。この心理的効果を意図的に活用することで、レベル構成そのものが物語を語る手段となる。

- 城の最上階に囚われた姫 → 上昇 = 目標への接近
- 地下深くの魔王の間 → 下降 = 危険への没入
- 崩壊する建物からの脱出 → 下降 = 緊迫感の加速

### 空間設計の定量的基準

空間設計を感覚だけでなく数値で裏づけるための基準。

| 要素 | 基準 |
|------|------|
| 壁の高さ | スケールフィギュア（プレイヤーキャラクター）の**150-200%** |
| 通路・ドア | 幅の**最低2倍** |
| 道幅 | 最低8m（MOBAの場合） |
| 高低差 | 2m以上で認識可能 |
| 主要 vs 二次建築要素 | 主要は二次の**約2倍のサイズ**（1m差ではビジュアルインパクト不足） |

## ペーシング (Pacing)

ペーシングとは「レベル内のアクティビティとイベントの一般的な順序とリズム」である。プレイヤーの緊張と弛緩、学習と実践、挑戦と報酬のバランスを時間軸に沿って設計する技術であり、レベルデザインの中核をなす。

### 計画の5つの軸

ペーシングを設計する際に考慮すべき5つの視点。

| 軸 | 説明 | 設計上の問い |
|----|------|------------|
| **スコープ (Scope)** | レベルごとのプレイヤー能力範囲 | そのレベルでプレイヤーは何ができるか？ |
| **階層 (Hierarchy)** | 最も重要なレベルセクション | どのビートが「山場」か？ |
| **因果関係 (Causality)** | 論理的な順序付けの根拠 | なぜこの順番で体験するのか？ |
| **情報 (Information)** | 戦略的なプレイヤーへの情報提供タイミング | いつ何を知らせるか？ |
| **強度 (Intensity)** | 注意のピークと回復期間 | テンションの波はどう描くか？ |

### ビート (Beat) 構造

ビートは「レベルの自己完結したチャンク」であり、個々のエリア、イベント、アクティビティを指す。音楽の小節のように、独立した意味を持ちながら全体の一貫性を形成する。

#### 音楽理論からのバリエーション技法

| 技法 | 説明 | ゲームでの例 |
|------|------|------------|
| **パルス (Pulse)** | 繰り返しパターンの確立 | 特徴的な出口ドアの反復 |
| **アクセント (Accent)** | 特定の瞬間を強調 | 発見困難な出口の配置 |
| **レスト (Rest)** | 低強度期間でコントラスト | 戦闘後の静寂・安全地帯 |
| **モチーフ (Motif)** | 短い反復シーケンス | 出口前のボス遭遇パターン |
| **バリエーション (Variation)** | 異なる実行での反復 | 同じ構造に新要素を追加 |
| **シンコペーション (Syncopation)** | 予想外のオフビート | ボスの割り込み、偽の出口 |

### セットピース (Set Piece)

「ユニークなコンセプトや記憶に残るアクティビティを持つ精巧なビート」。映画のプロダクションセットに由来する高投資のスペクタクル要素。

- **特徴**: 高コスト（ユニークアセット）、スキップ不可、重度のスクリプティング
- **実例**: ボス戦、複雑なパズル、振り付けされたシーケンス
- **助言**: プロジェクトごとに最低1つのセットピースを設計する

### Teach-Test-Twist パターン

メカニカルマスタリーのための3ビートフレームワーク。

| ステップ | 目的 | 具体例 |
|---------|------|--------|
| **Teach（教える）** | 明確なガイダンスでアクティビティを紹介 | Portalの「フリング」メカニクスの安全な導入 |
| **Test（試す）** | プロンプト付きで実行を要求 | 学んだメカニクスの応用課題 |
| **Twist（捻る）** | サポートなしで想起を要求し、複雑さをエスカレート | 複合的な応用、予想外の文脈 |

このパターンはパズルデザインにおける難易度プログレッションとも一致する。パズルでは「コアメカニクスの孤立した導入 → バリエーションによる拡張 → 以前の知識と新概念の組み合わせ」という3段階が基本構造となる。

#### Teach-Test-Twist の拡張応用

このパターンは個別メカニクスだけでなく、**レベル全体の構成にもスケールアップ**できる。

| スケール | Teach | Test | Twist |
|---------|-------|------|-------|
| **メカニクス単位** | 新しいジャンプの導入 | ジャンプが必要な障害 | タイミングシビアなジャンプ |
| **エリア単位** | 安全なチュートリアルエリア | メカニクスを活用するダンジョン | 既習の全メカニクスを組み合わせるボスエリア |
| **ゲーム全体** | 序盤のワールド（基礎の学習） | 中盤のワールド（応用と発展） | 終盤のワールド（マスタリーの証明） |

### 強度曲線 (Intensity Curve) の設計原則

#### ゆっくり始める

低強度コンテンツから開始し、雰囲気を確立して説明を可能にする。

- Half-Life: 6分間のトラム移動
- Portal: コントロールテスト空間

#### ダイナミクスを交互に

持続的な高強度シーケンス（10分以上）は消耗感を与える。ダウンタイムを「パレットクレンザー (Palate Cleanser)」と報酬として使用する。Prospect空間（開放的）とRefuge空間（閉鎖的）の交互配置は、メカニクスに頼らず空間構成だけでこのダイナミクスを実現する手法である。

ダウンタイムの機能は複合的である:

- **回復**: プレイヤーの認知的・感情的リソースを補充
- **対比**: 次の高強度セクションのインパクトを増幅
- **報酬**: 高強度セクションを乗り越えたことへのご褒美
- **情報提供**: ナラティブ・探索・環境ストーリーテリングの配置に最適な時間帯
- **予兆**: 次の挑戦への心理的準備（武器/リソースの提供、敵の遠景確認など）

#### 最大強度フィナーレを避ける

古典的なナラティブ理論（英雄の旅、五幕構造）はクライマックス後の下降を規定する。

| タイトル | フィナーレの設計 | 効果 |
|---------|---------------|------|
| **Dark Souls** | 最終ボスは高HPだがシンプルなメカニクス | 達成感の余韻を残す |
| **Portal** | 最終パズルは明示的で簡明な解法 | 爽快な締めくくり |
| **Half-Life 2** | 低ストレスの過剰パワー遭遇 | カタルシスの提供 |

### ゲームタイプ別ペーシング

| タイプ | ペーシングの特徴 | デザイナーの制御度 |
|--------|----------------|-----------------|
| **シングルプレイヤー** | デザイナーが直接制御。ビートシートと強度グラフで計画 | **高** — ビートの順序と強度を完全に設計 |
| **マルチプレイヤー** | ゲームプレイループとモードルールから**創発的**に生まれる。マップレイアウト・移動時間・フローバランスに依存 | **低** — 空間設計とルールで間接的に制御 |
| **オープンワールド** | 同心円オーバーレイパターン（Assassin's Creedモデル）。目的地に近づくにつれ各セグメントが小さく密になり、レイヤード防御突破でテンションを段階的に上昇 | **中** — 全体構造は設計するがプレイヤーの進行順は不定 |
| **MOBA/対戦** | マッチの制限時間（例: 3分間）と空間の対称構造がペーシングを規定。ステージ自体は静的だがプレイヤー同士の相互作用が動的ペーシングを生む | **低** — 空間とルールのみ |

### ペーシングのドキュメンテーション

| 手法 | 用途 | 形式 |
|------|------|------|
| **ビートシート (Beat Sheet)** | テキストベースまたはビジュアルアウトライン | リスト/表 |
| **フローチャート (Flowchart)** | 因果論理を可視化。パズルゲーム・非線形進行に必須 | 図 |
| **強度グラフ (Intensity Graph)** | 時間 vs 強度（0-10）でビートをプロット | グラフ。Valveの分類: Explore, Combat, Choreography, Puzzle |

## 難易度曲線 (Difficulty Curve)

難易度曲線はペーシングと密接に関連するが、とくにプレイヤーのスキル習得と挑戦度のバランスに焦点を当てる。適切な難易度曲線は「フロー状態 (Flow State)」を維持し、退屈とフラストレーションの間の最適帯にプレイヤーを保つ。

### 難易度プログレッションの3段階

パズルデザインの研究から導かれた、メカニクス導入の普遍的構造。

| 段階 | 内容 | 設計のポイント |
|------|------|--------------|
| **1. 孤立した導入** | コアメカニクスを単独で紹介 | 他の要素を排除し、新しい概念に集中させる |
| **2. バリエーションによる拡張** | メカニクスの応用パターンを提示 | 同じコアの異なる使い方を経験させる |
| **3. 知識の組み合わせ** | 以前の知識と新概念を統合 | 既習メカニクスとの複合課題 |

Lorelei and the Laser Eyesの例が示すように、最も高い満足感は「より難しいパズルを解くこと」ではなく「一見無関係な手がかりを統合すること」から生まれる場合がある。

この3段階はTeach-Test-Twistパターンと本質的に同じ構造であり、レベルデザイン全般に適用可能な普遍的原則である。重要なのは、各段階の**境界を明確にする**ことである。プレイヤーが「今は学習フェーズだ」「今は実力を試されている」と無意識に認識できる空間的・演出的手がかりを配置する。

### エンカウンターにおける段階的難易度

God of War (2018) のヴァルキリーボスシリーズは、難易度曲線の優れた実例である。

| ボス | 段階 | 設計意図 |
|------|------|---------|
| **Gunnr** | チュートリアル | 黄色ビジュアルフィードバックと長い予備動作（2-3秒）でパリィを促進 |
| **Kara** | 拡張 | 強制的な回避/ブロック判断を導入。タイミングミスが脆弱性を生む |
| **Geirdriful** | 統合 | 前2体のムーブを統合しつつ固有攻撃を追加。タイミング再学習を強制 |

9体のオプションボスがアニメーションとムーブセットを異なるシーケンスで再利用し、同じリソースから段階的に複雑さを増す構造を実現している。

### 引き算の美学 — 複雑さの制御

難易度を上げる方向（足し算）だけでなく、**不要な複雑さを削る判断（引き算）** がレベルデザインの品質を決定する。

#### よくある足し算の問題

| 問題パターン | 具体例 | なぜ問題か |
|------------|--------|----------|
| **ランダム要素の追加** | ルーレット、ランダムギミック | プレイヤーの技術に無関係 → 敗因が不明瞭 |
| **ゲージ管理の増加** | 酸素ゲージの追加 | 管理項目が増える = 認知負荷の直接的上昇 |
| **速度低下ギミック** | 水中での全行動速度低下 | 直感的にストレスが高く、工数も増加 |
| **全員のアイデアを足す** | チーム制作での折衷案 | 引き算判断の欠如が体験の質を低下させる |

「ステージ制作はゲームデザインの体験を最大限に引き出す空間設計」であり、作り手の創意よりも**プレイヤーが直感的に理解でき、技術で競える環境構築**が優先される。

### 敵パレット (Enemy Palette) による複雑度管理

エンカウンターあたりの敵タイプ数が認知負荷と直結する。

| 敵タイプ数 | 機能 | プレイヤー体験 |
|-----------|------|--------------|
| **1** | チュートリアル/休息 | 孤立した集中。新メカニクスの学習に最適 |
| **2** | 通常エンカウンター | 敵同士が意味ある相互作用を生む |
| **3** | 複雑なエンカウンター | 複数の関係管理が認知的に困難に |
| **4** | 乱闘 | 2陣営の2v2構成が生まれやすい |
| **5+** | 殲滅 | カオス。結局プレイヤーは小サブセットを分離して対処 |

> 📎 出典: reference_puzzle-design-guide.md, reference_encounter-design-combat-story.md, reference_level-design-subtraction-compass.md

## 環境ナラティブ (Environmental Storytelling)

環境ストーリーテリングは「空間内の個々の事物やその配置を描くことでストーリーをプレイヤーに伝える手法」である。直接的な出来事表現ではなく、**「そこでなにかが起きたことを表すもの」を描く**アプローチ。

### 映画・マンガとの比較 — mise-en-scene との関係

映画やマンガの **mise-en-scene**（舞台設定による物語表現）に類似するが、決定的な違いがある。

| メディア | 事物の提示順序 | デザイナーの制御 |
|---------|-------------|----------------|
| **映画・マンガ** | 固定的（作者が制御） | 完全 |
| **ビデオゲーム** | プレイヤーの行為に依存し**可変** | 間接的 |

この違いが、ゲームにおける環境ストーリーテリングの最大の強みであり同時に最大の課題でもある。プレイヤーが「自身でストーリーの手がかりを集めて再構成する感覚」を生み出せる一方、情報の取得順序を制御できない。

### プレイヤー体験への効果

- **能動的発見**: プレイヤーが自らの探索行為で物語を再構成する感覚がナラティブ体験の質を高める
- **インタラクティブ性と物語の両立**: ストーリー自体は固定的な時系列を保つため、インタラクティブ性と物語の直線性との葛藤を回避できる
- **没入感の維持**: 明示的なテキスト・カットシーンに頼らないため、ゲームプレイの流れを中断しない

### 代表的タイトル

| タイトル | 環境ストーリーテリングの特徴 |
|---------|--------------------------|
| **Myst** | 環境内のオブジェクトと空間構成だけで世界の謎を伝える |
| **Gone Home** | 家の中の遺留品から家族の物語を再構成 |
| **ドラゴンクエストシリーズ** | 「ただのしかばね」のような場面も該当。最小限の表現で物語を暗示 |
| **Dishonored** | 環境から敵の行動パターンやルートを推測させ、ステルスプレイと統合 |

### 環境ストーリーテリングの実装技法

| 技法 | 説明 | 例 |
|------|------|-----|
| **状態の痕跡** | オブジェクトの配置・破損状態で過去の出来事を暗示 | 倒れた椅子、割れた窓、血痕 |
| **時系列の断片** | 異なる場所に散在する手がかりで時間軸を再構成させる | 日記の断片、録音テープ |
| **対比による変化** | 同じ場所の「以前」と「以後」を対比 | 平和な写真と荒廃した現実 |
| **不在の語り** | 本来あるべきものの不在が物語を語る | 空の額縁、使われていないベッド |
| **文化的文脈** | 生活用品や装飾品でNPCの人格・文化を伝達 | 本棚の内容、壁の装飾、食卓の状態 |

### デザイン課題

実装の最大の困難は、**強制性を感じさせずにプレイヤーに必要な手がかりを集めさせる**ことにある。具体的な課題と対策:

| 課題 | 対策 |
|------|------|
| 手がかりを見逃す | 視線誘導テクニック（ライティング、カラーリング）と連携 |
| 情報の取得順序が不定 | 順序依存しない手がかり設計。どの順序でも物語が成立するように |
| 探索の動機づけ | 環境の「違和感」を演出し、好奇心を喚起 |
| 過度な説明 | 暗示にとどめ、プレイヤーの解釈余地を残す |

これらの課題は視線誘導・空間設計・ライティングとの緊密な連携によって解決される（後述「プレイヤー誘導」セクション参照）。

> 📎 出典: reference_environmental-storytelling.md, reference_encounter-design-combat-story.md

## プレイヤー誘導 (Player Guidance)

プレイヤー誘導は、レベルデザインにおいて「自由な探索感」と「意図した体験への導き」を両立させる技術である。最も洗練された誘導は、プレイヤーが「自分で選んでいる」と感じながら、実は制作者の綿密な配置設計に従っている状態を実現する。

### 無意識的誘導 (Invisible Guidance) の原則

環境・空間設計を通じた無意識的な誘導が、明示的な指示（矢印、マーカー、ミニマップ）よりも自然で没入感を維持する。核心は**プレイヤーの自発性 (Sense of Agency) を損なわずに意図した体験を提供する**ことにある。

- 「プレイヤーはパズルを解きたいのであって、パズルがどこに隠されているか探したいわけではない」
- 利便性のための通路追加は、プレイヤーの自然な動きを支援し、結果として意図した経路への誘導となる
- サイバーパンク2077の開発では、制作後半に「この通路がないと不便」として追加された経路が、動線設計 (Flow Line Design) の重要性を示した

### 視覚的誘導テクニック

レベル内でプレイヤーの視線を間接的にコントロールする6つの主要テクニック。

| テクニック | 目的・効果 | 実装のポイント |
|-----------|----------|--------------|
| **三分割法 (Rule of Thirds)** | 画面を3x3に分割し、交点に重要アイテムを配置 | カメラの基本視野を考慮して配置 |
| **ライティング (Lighting)** | スポットライトやハイライトで手がかりを浮き上がらせる | 自然光源を装った演出が効果的 |
| **カラーリング (Coloring)** | コントラストの高い色で注意を喚起 | 各キャラクターに固有色 (Signature Color) を設定し、関連アイテムで色の関連性を暗示 |
| **対数螺旋構図 (Logarithmic Spiral)** | 視線を画面全体に行き渡らせて目標物に導く | 自然界のパターンを活用した視線の流れ |
| **被写界深度 (Depth of Field)** | 浅い被写界深度でキーアイテムを強調 | カメラ制御が可能なゲームで特に有効 |
| **ミスディレクション (Misdirection)** | 偽情報で本当の手がかりを隠蔽 | 適度な使用が重要。過度な使用はフラストレーションに |

### 動的要素による視線制御

- **アニメーション・パーティクル**: 動く要素は静的要素より強く視線を引きつける。これを利用してプレイヤーの注視点を制御する
- **ネガティブスペース (Negative Space)**: 暗い領域に手がかりを配置しつつ完全には隠さない。「見つけた」感覚を維持しながら発見可能性を保証する

### 空間構成による誘導

視覚的テクニック以外に、空間そのものの構成がプレイヤーを誘導する。

| 手法 | メカニズム |
|------|----------|
| **通路の幅と方向** | 広い通路は「進むべき道」と認識される |
| **ライティングの勾配** | 明るい方向へ自然に進む傾向 |
| **高低差** | 下降は自然に促進、上昇は意図的な目標設定が必要 |
| **右カーブの優先** | カメラフレーム最適化のため右カーブを意識した空間配置（横スクロールやサードパーソン視点で特に有効） |
| **開放から閉鎖への遷移** | Prospect-Refuge理論に基づく空間変化がペーシングと誘導を同時に実現 |

### 誘導とエンカウンターの統合

エンカウンターデザインにおけるプレイヤー足場確保 (Player Footholds) は、誘導と戦闘体験を統合する手法である。

| 手法 | 説明 | 誘導効果 |
|------|------|---------|
| **プレイヤーが敵を待ち伏せ** | 入口から無警戒の敵が視認可能 | 計画時間を付与し、増援が段階的に出現 |
| **敵がプレイヤーを待ち伏せ** | 空のアリーナに誘引後、閾値通過でトリガー | アリーナ空間が「進むべき場所」として機能 |
| **一方通行入口のビスタ** | 高所から攻撃不能な敵を俯瞰後、降下 | コミットメント感を創出。戻れない一方通行が前進を強制 |

> 📎 出典: reference_visual-guidance-techniques.md, reference_invisible-player-guidance.md, reference_spatial-composition-level-design.md, reference_encounter-design-combat-story.md

## メトリクス駆動レベルデザイン

メトリクス (Metrics) は戦闘レベルデザインの背骨であり、「レベルの空間、カバーの高さ、廊下の幅などすべてを決定する」定量的寸法である。感覚的な調整ではなく、測定可能な基準に基づいてレベルを構築するアプローチ。

### 基礎計測領域

メトリクス確定前に「チームメイトとテストするための空間を作成」し、ジム的環境で推測ではなく**協働実験**を通じて適切な感覚を決定する。

| 計測対象 | 目的 |
|---------|------|
| プレイヤー移動速度 | 空間スケールの基準 |
| ジャンプ高さ（シングル/ダブル） | 垂直方向の到達可能範囲 |
| 廊下幅 | 通行とカバーの可能性 |
| ドア・窓の寸法 | エントリーポイントと射線 |
| 複数フロア間の垂直間隔 | 垂直移動の設計 |

### 戦闘空間メトリクスの5要素

| 要素 | 説明 | 設計上の意味 |
|------|------|------------|
| **カバーの高さと幅** | 使用可能なカバーの明確な定義 | プレイヤーの安全感と射線管理 |
| **カバー間隔 (バッファゾーン)** | カバーポイント間の距離制御 | **最も重要なメトリクスの一つ**。プレイヤーに実行可能なカバールートを理解させる |
| **建築物上のカバールール** | ドア・窓・構造物とカバーの統合 | 環境の一貫性と戦術的機能の両立 |
| **武器射程** | 多様なプレイスタイルを促進する空間設計 | 射程に応じた空間の広さ・構造 |
| **敵アーキタイプ** | 異なるAI行動を考慮した配置 | 多様な戦術の許容 |

### カバー設計の詳細

#### 間隔設計

- 適切な間隔がプレイヤーに実行可能なカバールートを理解させる
- AIが効果的にナビゲートし脅威を与えられる距離を確保する

#### 建築要素との統合

| 建築要素 | カバーとの関係 | 戦術的機能 |
|---------|-------------|----------|
| **ドア** | 付近に防御的カバーが必要 | プレイヤーが無防備に空間に入ることを防止 |
| **窓** | スナイパーポジションとして機能 | 長距離交戦の起点 |
| **柱・壁** | 中距離カバーポイント | 移動中の一時的な安全地帯 |

### 武器射程と空間設計

武器の有効距離が異なるため、レベル空間は「特定のプレイスタイルを促進」するように設計する。

| 空間タイプ | プレイスタイル | 設計手法 |
|-----------|-------------|---------|
| **スナイパーネスト** | 長距離交戦 | 見通しの良い高所、窓付き |
| **フランクルート** | 近接ショットガンプレイ | 敵の側面・背面に回り込む狭い経路 |
| **開放エリア** | 中距離アサルト | 散在するカバーで移動しながらの交戦 |

### ビジュアル区別の原則

主要建築要素は二次的要素の**約2倍のサイズ**にすべきである。1メートル差ではビジュアルインパクトが不足し、2倍にすることでプレイヤーの明確な認識を確保できる。

### ドキュメンテーション基準

- メトリクスドキュメントは「非常に読みやすく」、明確な視覚スケール付き
- 参照テーブルを含め、「何ページもスクロールせずに」情報を見つけられるように
- 設計者はアーキテクチャがメトリクスに準拠しているか常に検証

### 統合設計の原則

「建築の信憑性とテーマを、ゲームプレイの優れた感覚と組み合わせる」。戦闘空間とナラティブ環境を別物として扱わず統合する。本フレームワークは**射撃ベースの遠距離戦闘**に特化しており、近接戦闘、ターン制戦闘、異なるカメラパースペクティブには別のアプローチが必要となる。

> 📎 出典: reference_combat-level-metrics.md, reference_encounter-design-combat-story.md

## エンカウンターデザイン (Encounter Design)

エンカウンターとは「多様なプレイヤー戦術 (Tactics) をサポートする、システム的 (Systemic) チャレンジのシーケンス (Sequence)」である。主にアクション/シューター/RPGのシングルプレイヤー/PvE戦闘を指すが、パズルや対話など非暴力的形態も含む。

### 戦闘ナラティブ (Combat Story) の3段階

エンカウンターは戦闘を通じてプロットされるナラティブとして機能する。ジャッキー・チェンのスタントのように、動きと位置取りが因果関係を観客（プレイヤー）に伝達する。

#### 1. 開始 (Beginning / Pre-Fight)

| 設計要素 | 問い |
|---------|------|
| 敵の状態 | すでに交戦中か、無警戒か、接近中か？ |
| プレイヤーの主体性 | 戦闘開始タイミングを選べるか？ |
| 事前情報 | レイアウトと敵構成をどこまで把握できるか？ |

#### 2. 中盤 (Middle / During Fight)

| 設計要素 | 問い |
|---------|------|
| 持続時間 | 何ウェーブ構造か？ |
| スクリプトイベント | ゲームプレイに影響するイベントはあるか？ |
| ルート示唆 | 設計を通じて戦略やルートを暗示できるか？ |

#### 3. 終了 (Ending / Post-Fight)

| 設計要素 | 問い |
|---------|------|
| 終了シグナル | 戦闘終了がプレイヤーに明確に伝わるか？ |
| 結果バリエーション | 勝利/敗北以外の結末はあるか？ |

### プレイヤーペルソナ (Player Personas)

各プロジェクトはペルソナを定義し、各エンカウンターがどの戦略をサポートするか決定する必要がある。

| ジャンル | ペルソナ例 |
|---------|----------|
| **プラットフォーマー** | ランナー、ジャンパー、グライダー、スイマー |
| **シューター** | スナイパー、ショットガナー、スプレー&プレイ、キャンパー |
| **オープンワールド** | 探索者、戦闘者、ドライバー、コレクター |
| **ストラテジー** | ラッシャー、タートラー、拡張主義者 |

### エンカウンターのウェーブ設計

ウェーブ (Wave) は中盤のペーシングを構成する基本単位であり、エンカウンター内の小さなビートとして機能する。

| ウェーブ構造 | 効果 | 適した場面 |
|------------|------|----------|
| **単一ウェーブ** | 短い交戦、明確な終了 | 雑魚遭遇、チュートリアル |
| **段階的増援** | テンションの段階的上昇 | メインエンカウンター |
| **サプライズウェーブ** | シンコペーション効果 | セットピース、ボスフェーズ変化 |
| **撤退ウェーブ** | 敵が退く → プレイヤーに追撃判断を委ねる | 戦略的選択の付与 |

### ケーススタディ: Dishonored のステルスボス

- スニークと戦闘の両方を許容するボス設計
- **フローチャートアプローチ**で異なるプレイヤー能力インタラクションとエッジケースをマッピング
- 環境ストーリーテリングとエンカウンターが統合され、環境から敵の行動パターンを推測させる
- 複数のプレイヤーペルソナ（ステルス派・戦闘派・パシフィスト）それぞれに成立するルート設計

### エンカウンター設計の横断的原則

エンカウンターデザインは空間設計・ペーシング・難易度曲線・プレイヤー誘導のすべてが交差する領域であり、以下の原則がそれらを統合する:

- **空間がエンカウンターを定義する**: カバー配置、射線、高低差がプレイヤーの戦術的選択肢を決定
- **ペーシングがエンカウンターを配置する**: ビート構造の中でエンカウンターは高強度ビートとして機能
- **難易度がエンカウンターを調整する**: 敵パレット、ウェーブ構造、プレイヤー足場が認知負荷を制御
- **誘導がエンカウンターに導く**: ビスタ、サイトライン、ライティングが戦闘空間への遷移を準備

> 📎 出典: reference_encounter-design-combat-story.md, reference_combat-level-metrics.md

## パズルデザイン (Puzzle Design)

パズルは「身体的実行よりも精神的明快さを優先する」ゲームプレイ形態であり、論理・パターン認識・実験を重視する。単一のメカニクスではなく「制約の中で考えることへの認知的フレームワーク」として理解すべきである。

### 3つの基礎要素

| 要素 | 説明 | 欠如した場合 |
|------|------|------------|
| **明確性 (Clarity)** | 目的が即座に識別可能 | プレイヤーが何をすべきか分からず放置 |
| **目的 (Purpose)** | 各パズルがワールドビルディングまたは進行を支援 | パズルが「ただの障害」に感じられる |
| **報酬 (Reward)** | 解決が感情的・知的満足を生む | 解いても達成感がない |

### 3つの推論モード (Reasoning Modes)

パズル設計は、プレイヤーに要求する**推論の種類**によって根本的に異なる性質を持つ。

#### 1. 演繹的推論 (Deductive Reasoning) — 論理パズル

「前提が真であれば結論が偽であることは不可能」。数学的・論理的思考を要求し、不可能性の排除で結論に到達する。

- **代表作**: Professor Layton, The Witness, Minesweeper
- **特徴**: 単一の正解。線形的な設計に適する

#### 2. 帰納的推論 (Inductive Reasoning) — 試行錯誤/探索

「メカニズムやオブジェクトを実験してから解決する」。環境探索、仮説テスト、パターンの一般化に依存する。

- **代表作**: Limbo, Fez, Gorogoa, Machinarium
- **特徴**: 観察と実験に基づく。失敗からの学習が設計に組み込まれる

#### 3. アブダクション (Abductive Reasoning) — 調査

「特定の時間と場所で何が起こったかについて最も妥当な結論を推論する」。手がかりの解釈と最も蓋然性の高い説明の選択。

- **代表作**: Return of the Obra Dinn, Disco Elysium, Ace Attorney
- **特徴**: 複数の解釈が可能な手がかりから最良の推論を選ぶ

#### 推論モードの統合

最も洗練されたパズルゲームは3つの推論タイプを**ブレンド**し、メカニクス・ナラティブ・問題がすべて推論を通じて発見される「完全なパズルゲーム」を生む。異なるプレイヤーのスキルセットが異なる推論タイプに適合するため、**演繹ロジックに長けた人が帰納的探索やアブダクティブ調査に秀でるとは限らない**点を考慮して設計する。

### 線形 vs 非線形設計

| 構造 | 特徴 | 適した推論タイプ |
|------|------|----------------|
| **線形** | 単一の正解 | 演繹的推論 |
| **非線形（発見型）** | 複数の解法を発見する | 帰納的推論、アブダクション |
| **非線形（発明型）** | 解法を自ら創出する | Human Resource Machine等のコーディングゲーム |

### 言葉なき教授法 (Show, Don't Tell)

Portalに代表される「環境デザインによる明示的指示なしの教育」。空間レイアウトとビジュアルフィードバックでメカニクスを段階的に導入し、「問題解決」自体が学習メカニズムとなる。チュートリアルテキストより環境学習を優先することが、パズルデザインの7つの実践原則の筆頭に挙げられる。

### 7つの実践原則

| # | 原則 | 詳細 |
|---|------|------|
| 1 | チュートリアルより**環境学習**を優先 | 空間レイアウトとビジュアルフィードバックでメカニクスを段階的に導入 |
| 2 | 進行を**ナラティブ構造**のように扱う | 起承転結のある体験を設計。パズルにもペーシングが必要 |
| 3 | オーディエンスのスキルレベルにチャレンジを**適応** | ヒントシステム、複数の解法、難易度段階の用意 |
| 4 | **ソリューションから逆算**してデザイン | 正解の行動を先に決め、そこに至る空間と制約を構築 |
| 5 | 複雑さより**エレガンス**を重視 | 少ないルールで深い思考を生む設計。引き算の美学と同じ発想 |
| 6 | 不公平・誤解を招く要素を**排除** | 解法が理不尽に感じられる場合、パズルではなくただの障害 |
| 7 | バックトラックの**最小化**で摩擦を低減 | 失敗時のリトライコストを抑え、実験を奨励する |

### パズルと空間設計の関係

パズルデザインは空間設計と密接に結びついている。パズルの要素（入力装置、ギミック、ゴール）の**配置と視認性**がパズルの難易度を大きく左右する。

| 空間的要素 | パズルへの影響 |
|-----------|-------------|
| **部屋の広さ** | 広すぎると要素が散在して全体像を把握しにくい。狭すぎると閉塞感 |
| **視線の到達範囲** | すべてのパズル要素が一望できるか、一部が隠れているかで難易度が変化 |
| **垂直方向の配置** | 上下に分散した要素は認知負荷が高い。下降フローは認知負荷を制限する効果がある |
| **ライティング** | パズル要素を照らすか暗くするかで「発見」と「解法」の難易度を分離できる |

## ブロックアウト / グレイボックス (Blockout / Greybox)

ブロックアウト（blockmesh / graybox）は、詳細なアートアセットなしにシンプルな幾何形状で構築される3Dレベルのラフドラフトである。「安い（修正容易な）」段階で設計を証明し、「高い（最終アート）」段階に移行するための不可欠なプロセス。

### 5段階ワークフロー

| ステップ | 内容 | 注意点 |
|---------|------|--------|
| **1. レイアウトをスケッチ** | シンプルな描画で計画の基盤を確立 | 紙でもデジタルでも可 |
| **2. グラウンドプレーン・スケールフィギュア・壁を追加** | 空間コンテキストとプレイヤー比率を確立 | プレイヤーサイズとの比率が最重要 |
| **3. プレイテスト** | フル重力・コリジョンで空間を歩く | **エディタで飛ばない**。実際の移動感覚を確認 |
| **4. 分岐してイテレーション** | フィードバックに基づきセクションを再構築 | 部屋の削除・拡大も許容 |
| **5. ステップ4を繰り返す** | 継続的改善 | 「99%のブロックアウトはプレイテストを生き残らない」 |

### 4つの重点領域

| 領域 | 内容 | 検証する問い |
|------|------|------------|
| **マッシング (Massing)** | 形状によるボリュームと重みの伝達 | 空間の性格は伝わるか？ |
| **メトリクス (Metrics)** | スケール・寸法・プロポーションの確立 | プレイヤーサイズとの比率は適切か？ |
| **ウェイファインディング (Wayfinding)** | プレイヤーナビゲーションとマップ構造の学習 | 迷わずに進めるか？ |
| **プレイテスト (Playtesting)** | エンカウンター機能・バランス・目的達成の検証 | 楽しいか？ |

### イテレーションの哲学

「99%の場合、ブロックアウトはプレイテストを生き残らない」。部屋の削除・空間の拡大・セクション全体の再構築を受け入れることが重要である。

**Neon White の例**: 出荷100+レベルに対し約2倍がボツ。1レベルに3年間で50+の変更が加えられた。

### 構築手法5種

| 手法 | 特徴 | 適したケース |
|------|------|------------|
| **プリミティブ** | 基本キューブ・ボックスの配置 | シンプルだが制限あり |
| **ブラシ/モデリング** | エディタ内ローポリ形状 | 最大のコントロール |
| **モジュラーキット** | プリメイドピースのスナップ | **初心者推奨** |
| **スカルプティング** | 有機的形状のペイント | 地形・ランドスケープ |
| **スプライン** | パスに沿ったジオメトリ生成 | 道路・河川 |

### 開発プロセスにおける位置づけ

1. レベルデザイナーが**グレイボックス段階**でアイデアを盛り込む
2. 背景アーティストが仕上げ
3. 繰り返し**フォーカステスト**を実施する協働体制

この段階で視線誘導テクニック（ライティング、カラーリング等）の基本方針も検討しておくことで、最終アート段階での手戻りを最小化できる。

### ブロックアウトでの検証ポイント

グレイボックス段階で検証すべき項目と、検証に失敗した場合の対処。

| 検証項目 | 成功基準 | 失敗時の対処 |
|---------|---------|------------|
| **移動時間** | エリア間の移動が退屈でなく、かつ急かされない | 通路の長さ・幅を調整、ショートカットの追加 |
| **視認性** | 重要な要素がプレイヤーの自然な視線内にある | オブジェクトの位置変更、高さ調整 |
| **空間認知** | プレイヤーが現在地と目的地を把握できる | ランドマーク追加、サイトライン確保 |
| **戦闘フロー** | カバー間の移動が自然で、行き止まりがない | カバー間隔・配置の再設計 |
| **スケール感** | 空間がゲームプレイに対して適切なサイズ | メトリクス基準に照らして拡大/縮小 |

### スケール調整のガイドライン

| 問題 | 対処法 |
|------|--------|
| マップが小さすぎる | 外壁を拡大してから中央を埋める |
| マップが大きすぎる | 不要な部屋を削除・圧縮 |
| 始められない | まず**1部屋だけ**作る |

> 📎 出典: reference_blockout-greyboxing-guide.md, reference_visual-guidance-techniques.md

## オープンワールドと非線形設計

レベルデザインには、プレイヤーの自由度と制作者の制御度のバランスに基づく3つの基本構造パターンが存在する。リニアからオープンワールドまで、構造の選択がプレイヤー体験を根本的に規定する。

### 3つのレベル構造パターン

#### 1. リニア設計 (Linear Design)

「プレイヤーが事前定義されたパスを使ってAからBへ移動」する構造。デザイナーの制御度が最も高い。

**エスカレーション曲線**: 1本の連続的な難易度ピークではなく、複数のビートを配置する。ピーク（激しいゲームプレイ）とバレー（静かな瞬間）の交互配置がプレイヤーの疲弊を防ぎエンゲージメントを維持する。赤エリア（ゲームプレイ空間）と緑エリア（静かな内省空間）の明確な区分がリズムを生む。

本ドキュメントのペーシングセクションで述べた「ダイナミクスを交互に」の原則と直接対応する。

#### 2. マルチパス設計 (Multi-Path Design)

「環境内の複数ルート」で、目的地や重要エリアをつなぐパスを提供する構造。

| 課題 | 解決戦略 |
|------|---------|
| プレイヤーの混乱と方向感覚の喪失 | 特定の場所と関連ゲームプレイを個別のプログレッションビートに割り当て |
| ルートに関係なく一貫したプログレッション維持 | 複数パスでも意味ある瞬間で収束させるノード設計 |
| エージェンシー提供とデザイナー意図の両立 | 各パスに個別キャラクター（個性）を付与しつつ構造を維持 |

マルチパスの利点は、全ルートで一貫したプレイヤープログレッションを保ちつつ、各パスに独自の体験を与え、プレイヤーエージェンシーを提供できる点にある。

#### 3. オープンワールド / サンドボックス設計

プレイヤーの自由度が最も高く、デザイナーの直接制御が最も低い構造。

### POI磁力 (POI Magnetism)

POI (Point of Interest) はオープンワールドにおいて **磁力的引力 (magnetic pull)** として機能する。プレイヤーは以下の要因を無意識に比較検討してPOIを選択する:

- **距離**: 現在位置からの物理的距離
- **相対的関心度**: POIの視覚的・情報的魅力
- **事前知識**: NPCの会話やマップから得た情報
- **過去の訪問**: 既に探索済みかどうか

POI設計の鍵は、プレイヤーの視界に常に次の「行きたい場所」が存在するよう配置すること。これは空間設計セクションで述べたランドマークとサイトラインの原則をオープンワールドスケールに拡張したものである。

### オニオン構造 (Onion Structure)

ワールドを地域に分割し、プレイヤープログレッションをレイヤー化する構造。外側ゾーンから内側へ（または中心から外側へ）、プログレッション曲線に基づいたコンテンツ体験を保証する。

各POIは以下の **レイヤーシステム** で構成される:

| 要素 | 説明 |
|------|------|
| **無限のアプローチ角度** | オープンワールドからの多方向接近を許容 |
| **複数エントリポイント** | プログレッションレイヤー間に複数の入口を設置 |
| **2-4の内部目標レイヤー** | 段階的なアクセスポイントと難易度で内部を構造化 |

コンポーネントが多すぎると「追跡が非常に困難」になるため、好ましいルートを特定し、過度な複雑性をバランスするのではなく**徹底的に磨く**べきとされる。

### ハブとセトルメント設計 (Hubs & Settlements)

ハブは安全地帯として機能し、オープンワールドの **休息と準備の拠点** となる。Dying Light シリーズの開発知見に基づく設計パターン:

**全ハブ共通の一貫した要素**:
- 休息エリア（セーブ・回復）
- インベントリキャッシュ（アイテム管理）
- 商店（売買・装備）
- クラフトマスター工房（強化・改造）

**動的バリエーション**: 各ロケーションにプレイヤーの決定に基づく複数バージョンを用意する。陣営アライメントがセトルメントの外観やNPCの気分に影響し、選択の重みを可視化する。

### プレイヤー選択の統合 (Player Choice Integration)

オープンワールドにおいてワールドがプレイヤーの決定に動的に応答する設計:

| 設計要素 | 効果 |
|---------|------|
| **陣営アライメント反映** | セトルメントの外観・NPCの態度が変化 |
| **NPC再登場** | 助けたNPCが後のクエストやイベントで再び現れる |
| **世界観の可視的変化** | 選択の結果が環境として目に見える形で現れる |

### NPC生活感と日夜サイクル (NPC Living World)

ワールドの説得力を高めるため、NPCがプレイヤーインタラクションに関係なく日常タスクを実行するシステム:

- **昼間**: 各NPCが独自の日課を実行（巡回、作業、会話）
- **夜間**: NPCがシェルターを探し、焚き火の周りに集まり、UVライトを使用
- **帰還時バリエーション**: プレイヤーが拠点に戻るタイミングに応じて異なるシナリオを提供

動的NPC予定管理ツールがスポーンと行動を管理し、プログラミング・ナラティブ・オーディオチームの協働で実現される。

### 設計開発プロセス

オープンワールドの開発は3段階で進行する:

| ステップ | 内容 |
|---------|------|
| **1. クリエイティブブリーフ** | アート・ゲームプレイ・クエストデザインチームからインプットを収集 |
| **2. ペーパーデザイン** | フロアプランと参照資料による計画 |
| **3. エンジンブロックアウト** | 処理能力を考慮した主要エリアのジオメトリ構築 |

### 横断的設計原則

3つの構造パターンに共通する原則:

- **コミュニケーション価値**: 明確な構造図がレベルデザインを概念的に理解しないステークホルダーへの伝達を助ける
- **変数 vs 定数**: 各パターンに制御可能な変数（場所サイズ、レイヤー数、難易度）と固定定数（プログレッションビート、目標配置）が存在する
- **シンプリシティの優先**: いずれの構造でも、過度な複雑性を避け、好ましいルートを磨き上げるアプローチが効果的

### 他のセクションとの関係

- **ペーシング**: リニア設計のエスカレーション曲線は、ペーシングセクションの強度曲線設計原則と直接対応する
- **空間設計**: POI磁力はランドマークとサイトラインの原則をオープンワールドスケールに拡張したもの
- **環境ナラティブ**: プレイヤー選択の世界観反映は、環境ストーリーテリングの「状態の痕跡」技法をインタラクティブに展開したもの
- **エンカウンターデザイン**: オニオン構造の内部レイヤーは、エンカウンターのウェーブ設計と同様の段階的テンション上昇を空間的に実現する

## 実践メモ

### レベルデザインの統合チェックリスト

レベルを設計・レビューする際の横断的な確認項目。

#### 空間設計

- [ ] Prospect空間とRefuge空間のバランスは適切か？
- [ ] ランドマークはナビゲーションに機能的に関連しているか？
- [ ] 垂直性は最大3層ルールを守っているか？
- [ ] コントラスト（高さ・密度・方向・形状）で空間階層が明確か？

#### ペーシング

- [ ] ビートシートまたは強度グラフを作成したか？
- [ ] 低強度から開始しているか？
- [ ] 10分以上の持続的高強度区間はないか？
- [ ] クライマックス後の下降があるか？
- [ ] Teach-Test-Twist パターンでメカニクスを導入しているか？

#### プレイヤー誘導

- [ ] 明示的なUI指示なしで目的地に到達できるか？
- [ ] ライティング・カラーリングで重要物が識別可能か？
- [ ] 動線設計は自然な移動を支援しているか？

#### 戦闘空間（該当する場合）

- [ ] カバー間隔は適切か？
- [ ] 武器射程に対応した空間バリエーションがあるか？
- [ ] エンカウンターあたりの敵タイプ数は2-3に制限されているか？
- [ ] 戦闘開始前にプレイヤーに足場（情報と計画時間）を与えているか？

#### パズル（該当する場合）

- [ ] 明確性・目的・報酬の3要素が揃っているか？
- [ ] ソリューションから逆算してデザインしたか？
- [ ] バックトラックは最小化されているか？
- [ ] 要求する推論タイプ（演繹/帰納/アブダクション）を意識しているか？

#### 引き算の確認

- [ ] 不要なギミック・ゲージ・ランダム要素を追加していないか？
- [ ] プレイヤーが直感的に理解できる仕組みか？
- [ ] チーム全員のアイデアを足し合わせただけになっていないか？

### 制作フローの要約

```
コンセプト → ビートシート/フローチャート → ブロックアウト → プレイテスト
    ↑                                                          ↓
    ←←←←←←←←←←←  イテレーション  ←←←←←←←←←←←←←←
                                                               ↓
                                              プレイテスト合格 → 本制作
```

1. **プリプロダクション**: ペーシング計画（ビートシート、強度グラフ）を先に策定
2. **ブロックアウト**: プリミティブ形状でレイアウト → メトリクス確認 → プレイテスト
3. **イテレーション**: 「99%のブロックアウトはプレイテストを生き残らない」を受け入れる
4. **本制作**: 背景アーティストと協働。フォーカステストを継続

### セクション間の相互参照マップ

本ドキュメントの各セクションは相互に深く関連している。以下はセクション間の主な依存関係。

| 設計要素 | 関連するセクション | 関係性 |
|---------|------------------|--------|
| Prospect-Refuge理論 | 空間設計 + ペーシング | 空間の開放度がペーシングを制御 |
| Teach-Test-Twist | ペーシング + 難易度曲線 + パズル | メカニクス導入の普遍的パターン |
| 視線誘導 | プレイヤー誘導 + 環境ナラティブ | 手がかりの発見と経路の暗示を同時に実現 |
| 敵パレット | エンカウンター + 難易度曲線 | 認知負荷の定量的制御 |
| カバー設計 | メトリクス + エンカウンター | 戦術的選択肢の空間的基盤 |
| ブロックアウト | 全セクション | すべての設計仮説を検証する最初の場 |
| 引き算の美学 | 難易度曲線 + パズル + エンカウンター | 複雑さの上限を制御する横断的原則 |

### よくある失敗パターン

| パターン | 原因 | 対処 |
|---------|------|------|
| プレイヤーが迷子になる | ランドマーク不足、サイトライン未設計 | コントラストのあるランドマーク配置、ビスタの追加 |
| 「退屈」と感じる | ペーシングの単調さ、ダイナミクスの不在 | ビートのバリエーション技法（レスト、シンコペーション等）を活用 |
| 「難しすぎる」と感じる | 難易度の急上昇、足し算の過多 | Teach-Test-Twist の徹底、引き算の実行 |
| 「何をすればいいかわからない」 | 誘導不足、パズルの明確性欠如 | ライティング・カラーリングの見直し、環境学習の強化 |
| 戦闘が単調 | 敵パレットの貧弱さ、空間の均質性 | 敵タイプ2-3の組み合わせ、空間バリエーションの追加 |
| ブロックアウトから進めない | 完璧主義、プレイテスト不足 | まず1部屋だけ作る。プレイテストを早期に実施 |
| 環境ストーリーテリングが伝わらない | 手がかりの視認性不足、文脈の欠如 | ライティングで手がかりを浮き上がらせ、複数の裏付け手がかりを配置 |
| カバーが機能しない | メトリクス未検証、間隔の不適切さ | テスト空間での協働実験、バッファゾーンの再計測 |
