## ゲームUIの基本概念

### UIとUXの違い

- **UI (User Interface)**: ビジュアル要素・インタラクティブ要素そのもの。ボタン、メニュー、HUD、テキストなど
- **UX (User Experience)**: 製品やサービスに対するユーザーの**全体的な体験**。UIを含む、より広い概念

ゲームにおいては、プレイヤーの**没入感 (immersion)** と**満足度 (satisfaction)** がUX設計の最重要指標となる。UIは単なる情報表示装置ではなく、世界観を構築し、感情を喚起する手段として機能する。

### ゲームUIとWebUIの設計思想の違い

ゲームUIとWebUIは根本的に異なる設計原則で成り立っている。両者の違いを理解することで、それぞれの長所を活かした設計が可能になる。

| 観点 | WebUI | GameUI |
|------|-------|--------|
| **優先事項** | 再現性、標準化、アクセシビリティ | 没入感、ブランド表現、世界観構築 |
| **デザイン方針** | 業界標準フレームワーク準拠 | タイトル固有のデザイン |
| **アニメーション** | 状態変化を示す最小限のもの | 世界に繋ぎ止める演出を常に含む |
| **インタラクション** | 明確な選択肢で誤操作防止 | テンポや快適性を優先した柔軟な実装 |
| **UIの更新** | ユーザー離脱リスクとして認識 | 新しい体験として肯定的に受容 |

#### 両分野の知見の相互活用

- **Web側へ**: SaaS等で「ブランド価値を表現する体験」が重視されつつある。記憶に残るサービス設計にゲーム的アプローチが有効
- **ゲーム側へ**: クロスプラットフォーム対応の拡大に伴い、**普遍的ユーザビリティ (universal usability)** がGameUIにも求められている

> 📎 出典: reference_game-ui-vs-web-ui.md, reference_game-ux-design-guide.md, reference_game-ui-lessons-for-ux.md

## UIの4分類

ゲームUIは、ゲーム世界との関係性によって4つのタイプに分類される。これらを理解し使い分けることが、没入感と情報伝達のバランスを取る鍵となる。

### ダイエジェティック (Diegetic)

ゲーム世界の**一体的な部分**として存在し、プレイヤーとキャラクターの**両方**に見えるUI。

| 項目 | 内容 |
|------|------|
| **強み** | 没入感を破るオーバーレイを排除。キャラクターとのリアルなインタラクション |
| **弱み** | 情報の明確性が低下する可能性。アクションシーンで読みにくい |
| **代表例** | Dead Space（RIGスーツにHP/弾薬表示）、Far Cry 2（物理的に手持ちする地図・コンパス） |
| **適したジャンル** | 没入感とナラティブ一貫性を優先するゲーム（ホラー、サバイバル） |

#### ダイエジェティックインターフェースの設計原則（4つの基盤）

| 原則 | 説明 |
|------|------|
| **一貫性 (Consistency)** | ゲームの確立されたロアとビジュアルスタイルに合致させる |
| **機能性 (Functionality)** | 情報を明確かつ簡潔に提示する |
| **統合 (Integration)** | インターフェースを環境にシームレスに埋め込む |
| **フィードバック (Feedback)** | プレイヤーのアクションに対する即座の直感的な応答を返す |

ダイエジェティックUIは**没入感の乗数 (immersion multiplier)** として機能する。情報が有機的に提示されることで、プレイヤーはキャラクターを離れずに情報にアクセスでき、**認知負荷 (cognitive load)** を軽減する。外部観察者ではなく、アバターの視点でワールドを体験させる効果がある。

#### 実装上の課題と解決策

| 課題 | 解決策 |
|------|--------|
| 不明瞭性 (Obscurity) — 情報が伝わらない | 厳密なユーザーテストで検証 |
| 侵入性 (Intrusiveness) — 操作を邪魔する | 反復的な洗練 |
| 不整合な適用 — 場面に合わない | 文脈に応じた情報の優先順位付け |
| 過度な依存 — 全てをダイエジェティックにしすぎる | カスタマイズオプションとアクセシビリティ配慮 |

**核心的助言**: 微妙さ (subtlety) を受け入れること。複雑さでプレイヤーを圧倒するのではなく、最小限の戦略的要素を使用する。

#### 代表的なダイエジェティックUI事例

| タイトル | ダイエジェティックUI | 注目ポイント |
|---------|-------------------|-------------|
| **Dead Space** | RIGスーツにHP/弾薬表示 | 従来のHUDを完全に排除 |
| **Metro 2033** | 腕時計で時間・放射線レベル表示 | サバイバル感の象徴 |
| **Mirror's Edge** | 環境色彩でルートを誘導 | ランナーの視覚体験と一体化 |
| **Far Cry 2** | 地図・コンパスを物理的に手持ち | 探索感の強化 |
| **Fallout (Pip-Boy)** | 腕装着デバイスで全情報管理 | ワールドビルディングの一部 |

### 非ダイエジェティック (Non-Diegetic)

**プレイヤーのみに表示される**グラフィックオーバーレイ。キャラクターは認識できない、最も一般的なUIタイプ。

| 項目 | 内容 |
|------|------|
| **強み** | 明確でアクセシブルな情報提供。迅速な戦略判断を支援 |
| **弱み** | 人工的・侵入的に感じうる。画面の乱雑化リスク |
| **代表例** | World of Warcraft（HP/マナ/アビリティのHUD） |
| **適したジャンル** | ストラテジー、シミュレーション、競技ゲーム |

### スペーシャル (Spatial)

ゲーム世界内に表示されるが、キャラクターには見えない。ダイエジェティックと非ダイエジェティックの**ハイブリッド**。

| 項目 | 内容 |
|------|------|
| **強み** | 没入感と機能性のバランス。自然なナビゲーション支援 |
| **弱み** | 実装が複雑。やや人工的に感じうる |
| **代表例** | Assassin's Creed（敵の頭上の名前）、Splinter Cell: Conviction（ミッションガイドの光路） |
| **適したジャンル** | ナビゲーション支援を没入感を損なわず提供したいゲーム |

### メタ (Meta)

キャラクターの状態を反映するが、ゲーム世界の「物理的オブジェクト」としては表示されない。画面エフェクトで状態を伝える手法。

| 項目 | 内容 |
|------|------|
| **強み** | 侵入的なHUDなしにドラマティックな緊張感。直感的な危険伝達 |
| **弱み** | 繊細すぎて重要情報を見逃すリスク。特定ステータスに限定 |
| **代表例** | Call of Duty（被弾時の画面血しぶき） |
| **適したジャンル** | アクション・シューターでの強度と緊迫感の演出 |

### UIタイプの選択指針

実際のゲームでは、複数のUIタイプを組み合わせるのが一般的。選択の判断基準は以下のとおり。

| 判断基準 | ダイエジェティック寄り | 非ダイエジェティック寄り |
|---------|---------------------|----------------------|
| 没入感の重視度 | 高 | 低 |
| 情報量 | 少ない | 多い |
| ジャンル | ホラー、サバイバル、探索 | RTS、MOBA、RPG |
| ターゲット層 | コアゲーマー | 幅広い層 |
| 開発コスト | 高（テスト工数大） | 低（標準パターン活用可） |

> 📎 出典: reference_game-ui-4types.md, reference_diegetic-interfaces-design.md, reference_hud-design-guide.md

## HUD設計

HUD (Heads-Up Display) は、ゲーム画面に重ねて表示されるインターフェースレイヤー。HP、スタミナ、インベントリ、目標、スコア、ミニマップ等の重要なゲームプレイ情報を伝達する。

### 情報階層と認知負荷 (Information Hierarchy & Cognitive Load)

HUD設計の根幹は「プレイヤーがその瞬間に必要なものだけを表示する」こと。

- **プライマリ情報**: 生命線（HP、弾薬、目標）はプレイヤーの主要視線上に配置
- **セカンダリ情報**: メニューやコンテキスト表示でオンデマンドにアクセス可能にする
- **認知過負荷の防止**: 一度に表示する情報量を意識的に制限する

認知心理学の知見によれば、**作業記憶 (working memory)** の容量は限られ、相当な注意を必要とする。記憶保持は時間とともに指数関数的に低下するため、プレイヤーが覚えるべき情報を最小化し、必要に応じてリマインダーを提供すべきである。

### ミニマリスト設計原則

- 不要なインジケーターで画面を汚さない
- インターフェースを独立したウィジェットではなく**統一されたビジュアルシステム**として扱う
- 最良のHUDは「背景に溶け込む (disappearing into the background)」

### 動的・コンテキスト型HUD (Dynamic & Contextual HUD)

モダンエンジンが可能にするアダプティブインターフェースの例:

| パターン | 動作 |
|---------|------|
| ステルスインジケーター | 隠れている時のみアクティブ化 |
| クエスト目標表示 | 完了時にフェードアウト |
| 戦闘UI | 戦闘開始時に表示、平時は非表示 |
| コンテキストアクション | 操作可能オブジェクト接近時に表示 |

画面の清潔さを維持しつつアクセシビリティを確保する手法として、現在のHUD設計の主流になりつつある。

### ジャンル別HUD配置戦略

| ジャンル | HUD戦略 | 重視すべきポイント |
|---------|---------|-----------------|
| **FPS/TPS** | 最小限、コーナー固定で即座の読み取り | 視界の妨げにならないこと |
| **RPG** | 重い情報量をレイヤードメニューで収容 | 情報の階層整理 |
| **モバイル** | 親指到達ゾーンを考慮した大きなタップターゲット | サムゾーン設計 |
| **バトルロイヤル** | ゾーンタイマー・チームHP・インベントリを同時表示 | 情報の優先順位付け |
| **ストラテジー** | 大量のステータスをパネルとオーバーレイで管理 | 情報密度と視認性の両立 |

### HUDレイアウトの認知心理学的根拠

**ゲシュタルトの近接の法則**: 「互いに近い要素は同じグループに属すると解釈される」。この原則をHUD設計に適用すると:

- 関連するアイコン・数値を**近くにグループ化**する（HP + シールド + 回復アイテムなど）
- 無関係な情報は**十分な間隔**で分離する
- 形態が機能に合致するようデザインする（攻撃系は攻撃的な形状、防御系は安定した形状）

### 代表的HUD事例

| タイトル | HUDの特徴 |
|---------|----------|
| **Dead Space** | スーツ搭載HPでHUD完全排除。ダイエジェティックの模範 |
| **Metroid Prime** | ヘルメットバイザーのビジュアルで世界観に統合 |
| **Persona 5** | 赤黒の大胆なスタイリングと明確性の両立。UIそのものがゲームの個性 |
| **Far Cry 2** | 画面マップの代わりに物理的なワールド内ナビゲーション |

## メニュー設計

### 設計の基本方針

メニューはゲームプレイの「外側」に位置するが、プレイヤー体験の重要な一部。世界観を壊さず、かつ機能的で直感的な設計が求められる。

#### 操作シンプル化の原則

NieR:Automata のメニュー設計哲学が示す、シンプル化の優れた実践:

- **最小操作で全機能アクセス**: スティック移動と決定/キャンセルボタンのみで全機能を操作可能にする
- **ボタンヘルプは最小限**: 画面上のボタンガイドを減らし、直感的に操作できるようにする
- **幅広い層への対応**: ゲーム初心者や高齢層でも操作可能であることを想定
- **パワーユーザーへの配慮**: 隠しショートカット（L1/R1/△/□等）でゲーム慣れした層にも対応
- **脱出の容易さ**: どの階層からでもOPTIONSボタン1つでゲーム復帰可能にする

### 世界観統合メニュー

メニューの機能は同等でも、表現を世界観に合わせることで没入感が大きく変わる。

| タイトル | メニュー表現 | 効果 |
|---------|------------|------|
| **Skyrim** | スキルツリーを星座として表示 | ファンタジー世界への没頭度向上 |
| **NieR:Automata** | 楽譜モチーフの装飾、グリッドテクスチャ | SF感とファンタジー感のバランス |
| **Fallout** | Pip-Boy端末上のメニュー | レトロフューチャー世界の一部 |

### ビジュアル表現のポイント

NieR:Automataの事例から学ぶ、メニューのビジュアル設計:

- **隠れたモチーフ**: 楽譜（五線譜・ヘ音記号等）を採用し、世界観に深みを与える
- **カラー設計**: 温かく柔らかいベージュを基調色に。パラメーター増減はフォント太さ・濃度で表現。強調色は淡い赤橙のみ
- **質感の付加**: フラット基調にカメラレンズの歪み・周辺減光を加え、「モニター越しに見ている」実在感を演出
- **独創的表現**: 自爆時に見える破壊されたUI、ノスタルジックフィルタなど、UI自体をナラティブの手段にする

### ケーススタディ: ベヨネッタ オリジンズの絵本モチーフUI

PlatinumGames の「ベヨネッタ オリジンズ」は、**世界観ファーストUI (World-First UI)** の優れた実践例である。UIアーティスト近藤知之が「ゲーム内すべてが絵本内で起きている」というコンセプトに基づき、各画面を「絵本の1ページに感じられる」デザインとして設計した。

#### 設計哲学

- イラスト・文字を主役に配置
- 装飾は最小限
- 余白を区切り要素として活用（罫線やボーダーの代替）
- 西洋風おしゃれな絵本の雰囲気を目標とする

#### コンテンツ別カラーコーディング (Color Coding)

内容に応じた基調色の切り替えが、情報構造を暗黙的に伝達する:

| 基調色 | 用途 |
|--------|------|
| **白紙** | 共通要素（汎用メニュー） |
| **緑** | 森シーン関連 |
| **セピア調** | シナリオ系（ストーリー画面） |
| **水色** | ティルナノーグ関連 |

この手法は NieR:Automata のカラー設計と同様に、プレイヤーが現在どのコンテキストにいるかを色で直感的に認知できるようにしている。

#### イラスト配置戦略

| 画面タイプ | 手法 |
|-----------|------|
| **メニュー画面** | セレッサ・チェシャの描き下ろしイラスト |
| **イラストなし画面** | 物語シーン関連のシルエット（ヒトダマ、探索風景） |
| **深度表現** | 「システム」→「設定」でシルエットの前後感演出 |
| **登場人物** | ランダム形ボタン、森イラスト囲み |
| **ワールドマップ** | ランドマーク描写。未開放時は生き物・ヒトダマ配置 |
| **ストーリー連動** | 進行で画面イラスト変化 |

#### HUDのキャラクター分離

2キャラクター同時操作のゲーム構造に対応し、セレッサとチェシャで**色・配置を明確に区別**:

- 体力強化: アート班制作アイテム（キャラクター世界観寄り）
- 魔力強化: UI班デザインのゲージ（対比表現）
- 属性関連: チェシャデザインに統一
- 視認性: コントラスト維持しつつアイコン化

#### 2Dアニメーション哲学: CG感の排除

「CG感」を排除する **2D的表現** を一貫して採用し、絵本の触感を維持:

| 手法 | 説明 |
|------|------|
| **ページめくり** | 画面切り替えトランジション |
| **じわっと描かれる出現** | 段階的描画エフェクト |
| **パラパラ漫画風** | アイコンアニメーション |
| **2D的強調** | CG発光を回避した強調表現 |

#### 汎用的設計パターンとしての抽出

ベヨネッタ オリジンズの事例から、他のタイトルにも適用可能な4つの汎用パターンが導出できる:

| パターン | 説明 |
|---------|------|
| **世界観ファーストUI** | ゲーム世界のメタファー（絵本の1ページ）がUI全体の設計基盤となる |
| **コンテンツ別カラーコーディング** | 画面内容に応じた基調色の切り替えが情報構造を暗黙的に伝達 |
| **アニメーション言語の統一** | UI遷移・エフェクトのスタイルを統一し全画面で一貫した触感を実現 |
| **HUDのキャラクター分離** | 複数操作キャラクターのHUD要素を色・位置で明確に分離 |

> 📎 出典: reference_bayonetta-origins-ui-design.md

### 段階的情報提示 (Progressive Disclosure)

情報を一度に全て表示するのではなく、プレイヤーの行動に応じて段階的に開示する手法。

- **ゼルダの伝説 BotW の例**: カエルに焦点を当てると種別情報が表示。「見つけやすすぎず、見つけられないほど難しくもない」バランス
- **原則**: 複数の注目点 (POI: Points of Interest) で段階的にユーザーを誘導する
- **効果**: 認知負荷を下げつつ、探索の楽しさを維持する

### カスタマイズによる複雑性の解決

複雑なゲームでは、万能の解決策よりも**ユーザーに自分仕様にカスタマイズさせる**アプローチが有効。

- **World of Warcraft の例**: UIの配置・サイズ・表示項目をユーザーがカスタマイズ可能
- **原則**: 主要機能のカスタマイズの自由度を提供する。特に情報量の多いジャンル（MMO、ストラテジー）で有効

> 📎 出典: reference_nier-automata-ui-design.md, reference_game-ui-lessons-for-ux.md, reference_game-ux-design-guide.md

## フィードバック設計

（視覚・聴覚・触覚フィードバック）

### フィードバックの役割

フィードバックはプレイヤーの行動に対するゲームの**反応**であり、シグナル (Signals) とセットで機能する。

- **シグナル**: 視覚・聴覚的な合図でゲーム内の状況を伝える（例: 敵の出現音、アイテムの光）
- **フィードバック**: プレイヤーの行動に対するゲームの応答（例: 攻撃ヒット時のエフェクト、メニュー選択時の効果音）

両者は「プレイヤーがゲームの世界に対して感じる**感情的なつながり**を増大させる」役割を果たす。

### 直感的フィードバックの設計原則

情報を数値やテキストで示すのではなく、直感的・感情的に伝える設計がUXを高める。

| フィードバック手法 | 例 | 効果 |
|------------------|---|------|
| **メタUI (画面エフェクト)** | CoD: 被弾時の画面血しぶき | HPゲージを見ずに危険度を認知 |
| **キャラクター反応** | CoD: 操作キャラが痛みにあえぐ | 感情移入と状況認知の同時達成 |
| **環境変化** | Mirror's Edge: ルート誘導の色彩 | 世界観を壊さないナビゲーション |
| **アニメーション** | メニュー遷移のモーション | テンポとフィーリングの向上 |
| **触覚 (ハプティクス)** | コントローラー振動パターン | 視覚に頼らない状況伝達 |

### ゲームフィール (Game Feel)

インタラクションの触覚的感覚は感情的体験に大きく影響する。「ゲームフィール」は以下の要素の組み合わせで構成される:

- **入力応答性**: ボタン押下からアクション実行までの遅延
- **視覚的重み**: アニメーションの慣性・反動
- **聴覚的質感**: 効果音のタイミングと音色
- **触覚的確認**: ハプティクスによる物理的フィードバック

### 多層フィードバックの活用

ゲームUIが他のUIと一線を画す点は、**複数の感覚チャネルを同時に活用**できること。

- 攻撃ヒット時: ヒットエフェクト（視覚） + ヒット音（聴覚） + 振動（触覚） + ダメージ数値（情報）
- アイテム取得時: 光るエフェクト（視覚） + 取得音（聴覚） + UI表示（情報）
- 危険警告: 画面の赤み（視覚） + 警告音（聴覚） + 強振動（触覚）

1つのチャネルに依存すると、そのチャネルにアクセスできないプレイヤー（聴覚障害者など）が重要情報を逃す。**複数モダリティでの冗長な伝達**がアクセシビリティにも寄与する。

## 認知心理学とゲームUX

### 知覚とゲシュタルト原則 (Perception & Gestalt)

知覚は客観的ではなく、**主観的でコンテキスト依存**。同じ視覚刺激でも状況によって異なる解釈が生まれる。ゲシュタルトの法則をUI設計に活かすことで、プレイヤーの自然な知覚パターンに沿ったデザインが可能になる。

| ゲシュタルト法則 | UI設計への適用 |
|---------------|--------------|
| **近接の法則** | 関連するHUD要素を近くにグループ化 |
| **類似の法則** | 同種の情報に同じ視覚スタイルを適用 |
| **閉合の法則** | 不完全な形状でもプレイヤーが全体像を補完できるようにする |
| **連続の法則** | ナビゲーション要素を視線の自然な流れに沿って配置 |

### 記憶システム (Memory Systems)

| 記憶タイプ | 持続時間 | 容量 | UI設計への示唆 |
|-----------|---------|------|--------------|
| **感覚記憶 (Sensory)** | 数分の1秒 | 大 | 瞬間的なフィードバック（ヒットエフェクト等）はここに作用 |
| **作業記憶 (Working)** | 短期間 | 約4-7チャンク | HUDの同時表示情報はこの限界内に収める |
| **長期記憶 (Long-term)** | 潜在的に無制限 | 無制限 | 操作スキーマの習得、アイコンの意味の記憶 |

**設計上の要点**: 記憶保持は時間とともに指数関数的に低下する。プレイヤーが覚えるべき情報を最小化し、頻繁にリマインダーを提供すべき。

### 注意と非注意盲目 (Inattentional Blindness)

脳の注意リソースはスポットライトのように**限られ選択的**。有名な「ゴリラ実験」が示すように、集中した注意は注意されていない情報への知覚を妨げる。

**UI設計への示唆**:
- 重要な情報は**プレイヤーの注意が向いている場所の近く**に配置する
- 戦闘中に画面端の小さなアイコン変化は見逃される可能性が高い
- 重要な変化は**複数のチャネル**（視覚 + 聴覚 + 触覚）で伝達する
- **オンボーディング中の認知負荷管理**が特に重要。新しい概念を一度に多く提示しない

### 感情と動機付け (Emotion & Motivation)

プレイヤーは以下の**心理的欲求**が満たされると深くエンゲージする（自己決定理論に基づく）:

| 欲求 | 内容 | UI/UXでの実現方法 |
|------|------|-----------------|
| **有能感 (Competence)** | マスタリー・進歩の実感 | プログレッションバー、スキルツリー、達成表示 |
| **自律性 (Autonomy)** | 意味ある選択ができる | カスタマイズ可能なUI、複数の攻略ルート |
| **関係性 (Relatedness)** | 他者とのつながり | マルチプレイヤーUI、共有機能 |

### ユーザビリティ + ゲームフローのフレームワーク

| 要素 | 目的 | 指針 |
|------|------|------|
| **ユーザビリティ** | フラストレーションの排除 | 記憶・知覚・注意の人間的限界を尊重する |
| **ゲームフロー** | 楽しいエンゲージメントの創出 | チャレンジと意味あるインタラクションで没頭状態を維持する |

この2軸のバランスが取れているとき、開発者が意図したデザインとプレイヤー体験が整合する。

> 📎 出典: reference_cognitive-psychology-game-ux.md, reference_game-ux-design-guide.md

## アクセシビリティ

障害のあるプレイヤーは米国ゲーマーの**31%**を占める。アクセシビリティは倫理的な要請であると同時に、市場拡大にも寄与するビジネス上の合理的判断でもある。

### アクセシビリティの4カテゴリ

#### 視覚障害 (Visual Impairments)

| 機能 | 説明 |
|------|------|
| ハイコントラストモード | 背景とUI要素のコントラスト比を強化 |
| 色覚特性対応設定 (Colorblind Settings) | 色に依存しない情報伝達。形状・パターンの併用 |
| テキスト読み上げ (Text-to-Speech) | メニュー・ダイアログの音声読み上げ |
| 調整可能なテキストサイズ | フォントサイズの拡大・縮小 |
| スクリーンリーダー互換性 | 外部支援技術との連携 |

#### 聴覚障害 (Auditory Disabilities)

| 機能 | 説明 |
|------|------|
| カスタマイズ可能な字幕・キャプション | サイズ・色・背景の調整が可能 |
| 視覚的手がかり | オーディオ信号の代替（アイコン、方向矢印） |
| テキストベースコミュニケーション | マルチプレイヤーでの音声チャット代替 |

#### 運動/身体障害 (Motor/Physical Disabilities)

| 機能 | 説明 |
|------|------|
| リマッピング可能なコントロール (Remappable Controls) | ボタン配置の自由な変更 |
| 特殊周辺機器対応 | Xbox Adaptive Controller、フットペダル等 |
| 片手操作レイアウト (One-Handed Layouts) | 片手のみで全操作が可能 |

#### 認知障害 (Cognitive Disabilities)

| 機能 | 説明 |
|------|------|
| 簡潔な言語と明確な指示 | 専門用語の回避、段階的な説明 |
| 一時停止可能なチュートリアル | 自分のペースで学習可能 |
| 難易度カスタマイゼーション | ゲームスピード・敵の強さ等の調整 |
| スキップ機構 | 困難なセクションの回避オプション |

### アクセシビリティ設計のベストプラクティス

1. **プレイテストに障害のあるプレイヤーを参加させる** — 想像ではなく実際のフィードバックに基づく
2. **アクセシビリティファースト (Accessibility-First)** — 後付けではなく設計段階から組み込む
3. **複数の入出力モダリティを提供** — 視覚・聴覚・触覚の冗長な伝達
4. **認知的休息のための一時停止機能** — カットシーン中を含め、いつでも一時停止可能に
5. **一貫した読みやすいフォントとレイアウト** — 装飾的なフォントよりも可読性を優先

### 主要な設計パターン

| パターン | 説明 |
|---------|------|
| **カスタマイズ可能なゲームプレイ** | 難易度設定、クールダウン期間の調整 |
| **ダイナミックスケーリング (Dynamic Scaling)** | リアルタイムでチャレンジを調整 |
| **UIシンプリフィケーション** | メニュー複雑性を低減するモード |
| **アクセシビリティプリセット** | 障害タイプ別の推奨設定セット |

### 業界事例

| タイトル | アクセシビリティ機能 |
|---------|-------------------|
| **The Last of Us Part II** | 60以上のプリセット（リマッピング、読み上げ、ハイコントラスト等）。業界のベンチマーク |
| **Hearthstone Access** | 視覚障害者の競技プレイを可能にするオーディオMod |
| **Sea of Thieves** | 完全カスタマイズ可能なコントロールと色覚対応モード |

> 📎 出典: reference_game-accessibility-best-practices.md, reference_game-ux-design-guide.md

## オンボーディング

### オンボーディングの目的

新規プレイヤーがゲームのルール、操作、世界観を**フラストレーションなく学べる**ようにすること。認知心理学の知見から、オンボーディング中は認知負荷が最も高くなるため、特に慎重な設計が必要。

### 認知負荷管理の原則

| 原則 | 説明 | 実装例 |
|------|------|--------|
| **段階的開示** | 一度に教える概念を最小限に | ゲーム進行に応じて新メカニクスを1つずつ導入 |
| **文脈に即した学習** | 抽象的な説明より実践 | 操作を実際に行わせるチュートリアル |
| **反復とリマインダー** | 忘却曲線に対応 | 重要操作の定期的なリマインド表示 |
| **安全な失敗環境** | 失敗のペナルティを低く | チュートリアル中はゲームオーバーにならない |

### 非注意盲目への対策

オンボーディング中、プレイヤーはゲーム世界の美しさや新しい環境に注意を奪われ、UIの指示を見逃しやすい。対策として:

- 重要な指示は**画面中央**に表示する
- テキストだけでなく**矢印やハイライト**で注目点を明示する
- **操作をブロックして**指示を読ませる場面と、**自由に操作させながら**学ばせる場面を使い分ける
- **音声やサウンドエフェクト**で注意を喚起する

### 5つのコアテクニック

効果的なオンボーディングはプレイヤーリテンションと満足度に不可欠である。ローンチ直前ではなく、**プリプロダクションまたは早期プロダクション段階**でオンボーディング計画を開発すべきである。

| # | テクニック | 説明 | 事例 |
|---|----------|------|------|
| 1 | **学ぶべきことを特定する** | 基本操作から複雑なシステムまで、必須知識の包括的リストを作成 | Zelda: BotW は Great Plateau で移動からグライディングまで段階的に進行 |
| 2 | **難易度と重要度で分類する** | ゲーム要素を複雑さと進行への必要性で分類し、必須メカニクスを優先 | Celeste は基本スキル（ジャンプ、ダッシュ）を早期導入し、高難度メカニクスを段階的に追加 |
| 3 | **導入タイミングを計画する** | 認知的過負荷を防ぐため、新情報を戦略的に間隔を空けて配置 | Portal は各テストチェンバーが特定の概念を教えるよう設計 |
| 4 | **「何を」ではなく「なぜ」に焦点** | メカニクスの背後にある目的を理解させ、感情的な没入を促進 | The Witness は島の探索を誘い、プレイヤーが自力でルールを発見 |
| 5 | **プレイヤー体験に基づき監視・調整** | プレイテスト、フォーラム、アナリティクスで継続的にフィードバック収集 | Don't Starve はベータ版で新規プレイヤーの苦戦を特定しガイダンスを強化 |

> 📎 出典: reference_game-onboarding-5techniques.md

### 8つのオンボーディング原則

従来のチュートリアルを超えた、UIガイダンス・暗黙的学習・スキルプログレッション戦略を含む包括的な教育手法。

| # | 原則 | 説明 |
|---|------|------|
| 1 | **差別化 (Differentiation)** | プレイヤーのスキルレベル・経験・失敗耐性は異なる。柔軟な難易度やカスタマイズ可能なガイダンスで対応 |
| 2 | **複数の手法 (Multiple Methods)** | 複数の学習アプローチを提供。異なるテクニックの組み合わせが類似した冗長なアプローチより効果的 |
| 3 | **画面の簡素化 (Simplify the Screen)** | 教育場面での視覚的煩雑さを最小化。**プレイヤーは熟読ではなくスキムすると想定する** |
| 4 | **いつでもオンボーディング (Onboarding at Any Time)** | 教育はゲーム開始時だけでなく全体を通じて発生（ローディング画面のヒント、ゲームオーバー画面のガイダンス等） |
| 5 | **プレイテスト重視 (Playtesting Focus)** | 難易度リバランスではなく、メカニクスをより早い段階で教えることが解決策になる場合がある |
| 6 | **スキャフォールディング (Scaffolding)** | ゲームプレイを段階的な学習ステージに分割。能力の発達に伴い段階的にガイダンスを撤回する |
| 7 | **設計による教育 (Teach by Design)** | レベルデザインと敵/パズルの配置を通じて直感的理解を構築する |
| 8 | **楽しさへの近道 (Fast-Track to Fun)** | インタラクティブプレイ前の受動的オンボーディングを最小化。プレイヤーは即座のエンゲージメントを望む |

#### 任天堂の3段階導入法 (設計による教育の実践)

| ステージ | 目的 | 説明 |
|---------|------|------|
| **分離 (Isolation)** | 新メカニクスの単独導入 | 他要素なしで新概念に集中させる |
| **バリエーション付き強化** | 理解の定着 | 変奏を加えて適用範囲を広げる |
| **チャレンジ増加** | 習得の確認 | 高難度環境でのメカニクス応用を求める |

> 📎 出典: reference_onboarding-principles-design.md

### UX問題解決のイテレーション

オンボーディングの質はプレイテストなしには検証できない。以下のサイクルを回す:

1. **問題特定**: プレイテストで躓きポイントを発見
2. **解決策検討**: 複数アプローチを比較
3. **実装**: 変更を適用
4. **テスト**: 新しいプレイヤーで再テスト
5. **モニタリング・イテレーション**: 継続的な改善

> 📎 出典: reference_cognitive-psychology-game-ux.md, reference_game-ux-design-guide.md

## 操作体系設計

### プラットフォーム別の操作設計

ゲームUIの操作設計は、対象プラットフォームによって根本的に異なる。特にモバイルでは物理的制約が大きい。

### モバイル操作: サムゾーン設計 (Thumb Zone Design)

研究によると、**49%の人が片手**でスマートフォンを保持し、**75%のインタラクションが親指駆動**。自然な親指の動きパターンに基づくデザインが不可欠。

#### 3つのアクセシビリティゾーン

| ゾーン | 説明 | 配置すべき要素 |
|--------|------|-------------|
| **容易到達 (Easy)** | 自然な親指エリア（画面下部中央付近） | 頻繁に使うアクション（攻撃、移動） |
| **中間 (In-between)** | 中程度の快適さ（画面中央付近） | 準頻繁な操作（アイテム使用） |
| **困難 (Hard-to-reach)** | 上部角・端（ストレッチが必要） | まれに使う機能（設定、メニュー） |

左右利き**両方**のユーザーを考慮した複合ゾーンを設計すること。

#### ナビゲーションパターン

| パターン | 特徴 | 適したケース |
|---------|------|------------|
| **スティッキーフッターメニュー** | サムゾーン内に重要ナビゲーションを配置 | リンク数が少ないアプリ |
| **フルスクリーンオーバーレイ** | 整理されたスペースで親指フレンドリーな配置 | 広範なナビゲーション |
| **混合アプローチ** | 使用頻度でリンクを優先順位付け | 中程度の機能数 |

#### タッチターゲットの仕様

| 項目 | 推奨値 |
|------|--------|
| 最小タッチターゲットサイズ | **48px** |
| ターゲット間の最小間隔 | **8px** |
| WCAG 2.1準拠サイズ | **44 x 44 CSSピクセル** |
| 最小スワイプエリア | **45px x 45px** |

**ジェスチャー設計の注意点**: ユーザーは端から中央に向かって斜め下方向にスワイプする傾向がある。この自然な動作パターンに沿ったジェスチャー割り当てが望ましい。

### コンソール・PC操作の設計指針

- **リマッピング可能なコントロール**: 全てのボタンをユーザーが再割り当て可能にする（アクセシビリティの観点からも必須）
- **片手操作レイアウト**: 身体障害を持つプレイヤーへの対応
- **キーボード＋マウス / ゲームパッド両対応**: PC版では両方の入力方式に対応
- **入力切替の即時反映**: デバイスを切り替えた瞬間にUIのプロンプト表示も切り替わる

> 📎 出典: reference_thumb-zone-mobile-design.md, reference_game-accessibility-best-practices.md, reference_hud-design-guide.md

## ローカライゼーションとi18n

UIの国際化 (internationalization: i18n) とローカライゼーション (localization: l10n) は、グローバル展開に不可欠な設計配慮。後付けではなく、**UI設計の初期段階から組み込む**ことで手戻りを防ぐ。

### 7つのベストプラクティス

#### 1. マーケットリサーチ

地域法規、文化的感受性、ローカル慣習（日付形式、計量単位）の理解が基盤。翻訳だけでなく**文化適応 (cultural adaptation)** が必要。

#### 2. スケーラブルなテキストフィールド

英語比で**最大30%長いテキスト**に備える。

| 対策 | 説明 |
|------|------|
| 動的スケーリング | テキスト量に応じてフィールドサイズを自動調整 |
| 2行改行 | フォントサイズ縮小ではなく改行で対応 |
| レスポンシブスペーシング | 余白の自動調整 |

#### 3. フォントサポート

- 複数文字セット対応の**Unicode互換フォント**を選択
- 異なるアルファベット混在時のフォント不整合に注意
- バイトベースではなく**文字数ベース**の文字列長チェック

#### 4. RTL言語対応 (Right-to-Left)

アラビア語・ヘブライ語ではインターフェース要素のレイアウトを**ミラーリング**する。

- メニュー配置の反転
- スクロール方向の反転
- LTRレイアウトの強制は厳禁

#### 5. データフォーマット

| 項目 | 注意点 |
|------|--------|
| 日付 | MM/DD/YYYY vs DD/MM/YYYY vs YYYY/MM/DD |
| 時間 | 12時間表記 vs 24時間表記 |
| 通貨 | 記号の位置、小数点/桁区切り |
| 数値 | 桁区切り（カンマ vs ピリオド） |

#### 6. アイコン戦略

「多くのアイコンに確立された標準的使用法はない」ため:

- **普遍的に明確な意味を持つアイコンのみ**単独で使用
- 意味が曖昧なアイコンには**ラベルを併記**
- 文化固有の画像（ハンドジェスチャー、宗教的シンボル等）にはローカライゼーションを検討

#### 7. 包括的テスト

- デザインレビューツール（Figma統合等）でのビジュアル検証
- **ネイティブスピーカーによるゲームプレイテスト**で不整合と文化的ミスステップを検出
- 擬似ローカライゼーション (pseudo-localization) でレイアウト崩れを事前検出

### よくある落とし穴

| 落とし穴 | 影響 |
|---------|------|
| スケール不能なボタン | テキストオーバーフロー、ボタンからはみ出す |
| RTL言語にLTRレイアウト強制 | 不自然な読みにくさ、操作感の違和感 |
| 日付フォーマット設定の見落とし | 誤解を招く日付表示（1/2は1月2日? 2月1日?） |
| ハードコードされた文字列 | 翻訳対象から漏れる |
| 連結文字列で文章を構築 | 語順が異なる言語で破綻する |

## UX改善プロセス

### ゲームUX開発の5段階

| 段階 | 内容 | UIへの関わり |
|------|------|------------|
| **1. リサーチ & プランニング** | ターゲット層の調査、競合分析 | UI方針の決定 |
| **2. コンセプト & アイデア開発** | 世界観・テーマの確立 | UIのトーン&マナー策定 |
| **3. デザイン & 制作** | ワイヤーフレーム、プロトタイプ | UI実装 |
| **4. テスト & イテレーション** | プレイテスト、フィードバック収集 | UI改善サイクル |
| **5. 発売 & 継続的改善** | ライブデータ分析、パッチ | UI更新・調整 |

### 実践的改善チェックリスト

- [ ] ターゲット層を詳細に理解しているか
- [ ] アクセシビリティ機能は十分か
- [ ] 不必要な複雑性はないか。明確性 (clarity) を重視しているか
- [ ] テスト・最適化サイクルを継続的に回しているか
- [ ] プレイヤーエンゲージメント (engagement) を計測しているか
- [ ] 難易度バランス (difficulty curve) は適切か
- [ ] トレンドと最新技術を把握しているか

### プレイテストの重要性

実ユーザーからのフィードバック収集が全てのUX改善の基盤。**早期プロトタイピング (early prototyping)** が開発プロセスを効率化し、後半での大規模な手戻りを防ぐ。

プレイテストで特に確認すべき観点:
- プレイヤーは意図通りにUIを使えているか
- 情報の優先順位は適切か
- オンボーディングで躓いているポイントはどこか
- アクセシビリティの問題はないか

## 実践メモ

### UIデザインから学べる4つの普遍的原則

ゲームUIの知見は、ゲーム以外のプロダクトにも応用可能な普遍的原則を含んでいる。

| 原則 | ゲームでの適用 | 汎用プロダクトへの応用 |
|------|--------------|---------------------|
| **直感的フィードバック** | 被弾時の画面血しぶきでHP伝達 | 操作結果を数値でなく体感で伝える |
| **世界観統合メニュー** | スキルツリーを星座で表現 | ブランドアイデンティティに合致したUI |
| **段階的情報提示** | 注視で詳細情報を開示 | 複数のPOIで段階的にユーザーを誘導 |
| **カスタマイズ自由度** | WoWのUI配置カスタマイズ | ユーザーに自分仕様にする自由度を提供 |

### UIタイプ選択の早見表

| 優先したいこと | 推奨UIタイプ | 代表例 |
|--------------|------------|--------|
| 最大の没入感 | ダイエジェティック | Dead Space, Metro 2033 |
| 情報の明確性 | 非ダイエジェティック | WoW, LoL |
| 没入感と機能性の両立 | スペーシャル | Assassin's Creed |
| 直感的な状態伝達 | メタ | Call of Duty |
| 現実的なアプローチ | 複数タイプの組み合わせ | ほとんどの商用タイトル |

### 設計時の自問チェックリスト

1. このUIは**プレイヤーがその瞬間に必要な情報だけ**を表示しているか?
2. 情報伝達は**複数の感覚チャネル**（視覚・聴覚・触覚）で冗長に行えているか?
3. **色に依存しない**情報伝達になっているか?
4. テキストは**30%長くなっても**レイアウトが崩れないか?
5. **片手操作**でも重要な機能にアクセスできるか?
6. オンボーディングで**一度に教える概念は1-2個**に絞れているか?
7. UIは世界観・ブランドアイデンティティと**一貫**しているか?
